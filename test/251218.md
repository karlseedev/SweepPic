# Gate 2 성능 테스트 결과 (2024-12-18)

## 개요

Gate 2 (Image Loading) 스크롤 성능 테스트 진행.
HitchMonitor(앱 내부 측정)와 Instruments(시스템 레벨 측정) 비교를 통해 측정 방법론 검증.

---

## 1. Release 빌드 테스트 (HitchMonitor)

### 목적
Debug 빌드의 디버거 노이즈 제거 후 실제 성능 측정

### 결과

| 항목 | Debug | Release |
|------|-------|---------|
| hitch | ~17 ms/s | **11.0 ms/s** |
| req/s | 64.5 | 47 |
| maxInFlight | 20 | 22 |
| latency avg | 41.3ms | 32.9ms |
| 등급 | Critical | Critical |

### 분석
- 디버거 오버헤드로 인해 ~35% 성능 저하 발생
- Release에서도 여전히 Critical (목표 < 5 ms/s)

---

## 2. Minimal 모드 테스트

### 목적
cellForItemAt 내부 로직(metrics, dedupe, cancel)이 hitch 원인인지 확인

### 구현
- `Gate2ViewController`에 `ExperimentMode` 추가
- `GateMenuViewController`에 "PhotoKit (Minimal)" 버튼 추가
- Minimal 모드: 이미지 적용만, 다른 로직 제거

### 결과 (Debug 빌드, 노이즈)
```
avgFrame: 14.0ms (비정상)
hitch: 14.8 ms/s [Critical]
```

### 분석
- Debug 빌드 + 디버거 연결로 인해 노이즈 발생
- 유의미한 결론 도출 불가

---

## 3. Instruments 비교 테스트 (핵심)

### 목적
HitchMonitor 대신 시스템 레벨 측정으로 네이티브 앱과 정확한 비교

### 방법
- Instruments > Animation Hitches 템플릿
- All Processes 타겟으로 측정
- 동일 기기(iPhone 13 Pro, iOS 18.6.2)에서 테스트

### 결과

#### 네이티브 Photos 앱
| 항목 | 값 |
|------|-----|
| 측정 시간 | 12.5초 |
| Hitch 개수 | 22개 |
| Hitch/초 | 1.76 |
| 최대 Duration | 58 ms |
| High severity | 2개 |

주요 Hitch:
- 58.34 ms (High) - Pre-Commit latency
- 25.00 ms (High) - Pre-Commit latency

#### Spike1Test 앱
| 항목 | 값 |
|------|-----|
| 측정 시간 | 16.7초 |
| Hitch 개수 | 38개 |
| Hitch/초 | 2.28 |
| 최대 Duration | 83 ms |
| High severity | 8개+ |

주요 Hitch:
- 83.34 ms (High) - Pre-Commit latency
- 75.01 ms (High) - Commit to Render latency
- 62.51 ms (High) - Commit to Render latency
- 37.50 ms (High) - Commit to Render latency - Delay Frame Swap

### 비교

| 항목 | Photos | Spike1Test | 차이 |
|------|--------|------------|------|
| Hitch/초 | 1.76 | 2.28 | +30% |
| 최대 Duration | 58ms | 83ms | +43% |
| High severity | 2개 | 8개+ | +300% |

## 4. 핵심 발견

### HitchMonitor의 한계

| 항목 | HitchMonitor | Instruments |
|------|--------------|-------------|
| 측정 방식 | CADisplayLink 콜백 | 시스템 렌더링 파이프라인 |
| 정확도 | 낮음 | 높음 |
| Hitch 유형 구분 | 불가 | 가능 |
| 네이티브 앱 비교 | 불가 | 가능 |

**결론: 앞으로 성능 측정은 Instruments를 기준으로 사용**

### Spike1Test의 문제점

1. **Pre-Commit latency** - 메인 스레드에서 프레임 준비 지연
2. **Commit to Render latency** - 렌더링 비용 과다
3. **High severity hitch가 네이티브 대비 4배 많음**

---

## 5. 기존 테스트의 유효성

| 테스트 | 유효성 | 이유 |
|--------|--------|------|
| Preheat ON/OFF A/B | ✅ 유효 | 상대 비교는 의미 있음 |
| L1/L2 자동 테스트 | ⚠️ 제한적 | 절대 수치는 부정확 |
| HitchMonitor 절대값 | ❌ 무효 | Instruments와 불일치 |

---

## 6. 다음 단계

### 최적화 방향
1. **Pre-Commit latency 감소**
   - 메인 스레드 작업 최소화
   - 이미지 디코딩 백그라운드 처리

2. **Commit to Render latency 감소**
   - 레이어 최적화
   - 불필요한 렌더링 제거

### 측정 방법
- Instruments Animation Hitches 템플릿 사용
- 네이티브 Photos 앱과 동일 조건 비교
- 목표: High severity hitch 네이티브 수준으로 감소
