# Gate 2 파이프라인 테스트 계획

## 목표

PhotoKit 썸네일 로딩을 **오표시 0으로 유지**하면서, 스크롤 체감(끊김/복원)을 **iOS Photos에 최대한 가깝게** 만드는 기본 파이프라인을 확정한다.

---

## 고정 전제 (모든 후보 공통)

| 항목 | 값 | 비고 |
|------|-----|------|
| UI | UICollectionView 가상화 유지 | 변경 불가 |
| 오표시 방지 | representedIdentifier 토큰 검증 | 필수 |
| Dedupe | 동일 assetID+targetSize 중복 요청 방지 | 필수 |
| 측정 원칙 | Manual + Instruments (Photos 대비) | Auto는 참고용 |

---

## 후보 세트

### Baseline (Control, 현행)

| 정책 | 값 |
|------|-----|
| Preheat | prefetch 연동 + 100ms throttle |
| DeliveryMode | opportunistic |
| Quality | 스크롤 중 50% |
| Cancel | prepareForReuse |
| MaxInFlight | 무제한 |

---

### 후보 A: 최소 변경

| 정책 | 변경 |
|------|------|
| DeliveryMode | opportunistic → **fastFormat** |
| Cancel | + **didEndDisplaying** 추가 |

**확정하려는 것**: 다중 콜백 UI churn + 화면 밖 로딩이 hitch 원인인가?

---

### 후보 D: Adaptive 2-Stage (추천 기본안)

| 상태 | 정책 |
|------|------|
| 스크롤 중 | fastFormat + targetSize 50% (필요시 30%) |
| 정지 후 | 가시 셀만 100% 업그레이드 (1회) |
| Preheat | ±1~2 + throttle 150~250ms, 스크롤 중 최소화 |
| Cancel | didEndDisplaying + 토큰 검증 |

**확정하려는 것**: Photos와 같은 2단계 로딩 정책을 기본안으로 채택할지?

---

### 후보 B: 레버별 단계 테스트 (원인 분리용)

| 단계 | 변경 | Baseline 대비 |
|------|------|---------------|
| B1 | preheat 스크롤 중 OFF | preheat만 |
| B2 | B1 + quality 30% | +quality |
| B3 | B2 + fastFormat | +deliveryMode |
| B4 | B3 + didEndDisplaying | +cancel |

**용도**: A, D로 해결 안 될 때 원인 추적

---

### 후보 C: MaxInFlight 제한 (최후 안전장치)

| 정책 | 변경 |
|------|------|
| MaxInFlight | 무제한 → **8~12개 제한** |

**조건**: 붕괴가 반복 재현될 때만 적용

---

## 테스트 (3개)

### T1: L2 실사용 패턴 (10초)

- **방법**: 플릭+감속 반복 (Manual)
- **측정**: Instruments hitch/초, max duration, high severity 개수
- **보조**: req/s, cancel/s, complete/s, maxInFlight
- **판정**: Photos 대비 체감

### T2: 훑고 멈춤 (10초)

- **방법**: 빠르게 스크롤 후 정지 2초 포함
- **측정**: 정지 후 가시 셀 선명해지는 체감 속도 (빠름/보통/느림)
- **보조**: 요청 폭주 여부

### T3: 스트레스 (5초)

- **방법**: 일부러 과격 조작
- **측정**: 파이프라인 붕괴 여부만 체크
- **붕괴 조건**: cancel/s ≥ req/s × 50% + complete/s < req/s × 80% + maxInFlight ≥ 15 (2개 이상)

---

## 합격 기준

| 항목 | 기준 |
|------|------|
| **필수** | 오표시 0, 크래시 0 |
| **제품** | T1 체감 "문제 없음" + Photos 대비 high severity 1.5배 이내 |
| **스트레스** | T3 Critical 허용, 붕괴/오표시/크래시만 없으면 OK |

## 결론 트리

```
1. Baseline으로 T1/T2/T3 수행 → 기준선 기록

2. 후보 A 적용 → T1 수행
   ├─ 개선 있음 → A 임시 채택, 후보 D 시험
   └─ 변화 없음 → 후보 D 또는 B1로 이동

3. 후보 D 적용 → T1/T2 수행
   ├─ T1 OK + T2 OK → 최종 기본안 = D 확정
   ├─ T1 OK + T2 느림 → preheat 조정 후 재시험
   └─ T1 끊김 여전함 → 후보 B로 원인 분리

4. 후보 B (단계형)
   ├─ B1에서 개선 → preheat이 원인 → D + "스크롤 중 preheat OFF" 확정
   ├─ B2에서 개선 → 디코드 비용이 원인 → D + targetSize 30% 확정
   ├─ B3에서 개선 → 콜백 churn이 원인 → D + fastFormat 고정 확정
   └─ B4에서 개선 → 화면 밖 로딩이 원인 → didEndDisplaying cancel 확정

5. T3에서 붕괴 재현 시
   └─ 후보 C (inFlight 제한) 도입 검토
```

## 후속 작업 (Gate 2 범위 외)

### 필수: 정지 후 자동 복구 로직 구현

**목표**: 극한 스크롤 후 정지 시, 재스크롤 없이 0.5~1초 내 자동 복구

**구현 방향**:

| 우선순위 | 방안 | 설명 |
|----------|------|------|
| 1순위 | 취소/미완료 추적 + willDisplay 재요청 | 근본 해결. "가시화되는 순간 자동 복구" |
| 2순위 | 스크롤 종료 시 가시 셀 재요청 | 안전망. reloadData 아님, 이미지 요청만 |

**주의**: 이 로직은 중요한 기능이므로 별도 설계 및 테스트 필요. 대충 수정 금지.

**제품 기준**:
- ✅ 허용: 극한(스트레스)에서 순간 placeholder 노출
- ✅ 필수: 멈추면 자동 복구 (0.5~1초 내)
- ❌ 불허: 재스크롤 해야만 복구되는 상태

## R1 + R2 자동 복구 로직

### R1: 재진입 복구 (willDisplay)

- **트리거**: 셀이 화면에 새로 들어올 때
- **조건**: 이미지 nil + 요청 중 아님
- **역할**: 화면 밖 → 화면 안으로 들어오는 셀 자동 복구

### R2: 정지 복구 (scrollViewDidEnd*)

- **트리거**: scrollViewDidEndDecelerating / scrollViewDidEndDragging
- **디바운스**: 200ms 후 1회 실행
- **대상**: visible 셀 중 이미지 nil인 셀만 (최대 15개)
- **역할**: 정지 시점에 이미 화면에 있는 회색 셀 자동 복구
