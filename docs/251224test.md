# AutoScroll 테스트 결과 보고서 (2025-12-24)

## 1. 테스트 목적

수동 스크롤 테스트의 높은 변동성(±54%~99%)을 해결하기 위해 **자동화된 스크롤 테스터**를 구현하고, 수동 극한 스크롤과 동등한 부하를 재현하여 일관된 성능 측정 환경 구축.

### 목표 지표 (수동 극한 스크롤 기준)
- req/s: 500~1000
- hitch: Critical 수준 재현
- gray/req 비율: ~5.27

---

## 2. 접근 방식 비교

### 2.1 1차 시도: XCUITest (기각)

**개념**: XCUITest의 `swipeUp()`/`swipeDown()`으로 연속 스크롤 자동화

**문제점**:
- 각 스와이프 후 **waitForQuiescence** (앱 idle 대기) 발생
- 스와이프 → 대기 → 스와이프 → 대기... 패턴으로 연속 스크롤 불가
- `app.setShouldWaitForQuiescence(false)` 시도 → **private API**로 접근 불가

```swift
// 시도했던 코드 (실패)
let app = XCUIApplication()
app.perform(NSSelectorFromString("setShouldWaitForQuiescence:"), with: false)
// → "unrecognized selector" 에러
```

#### XCUITest 실제 테스트 결과 (12/24 18:18)

| 항목 | 값 | 비고 |
|------|-----|------|
| Build | Debug | Release 테스트 불가 |
| L1 hitch | **0.0 ms/s [Good]** | 부하 부족으로 무의미 |
| L1 req/s | **23.5** | 수동 극한의 1/25 수준 |
| L1 grayShown | 16 | |
| L2 | 측정 안 됨 | |
| swipe 간격 | ~0.4초 | waitForQuiescence 대기 |

**비교**:
| 방법 | req/s | hitch | 연속성 |
|------|-------|-------|--------|
| 수동 극한 | 580 | 12.8 [Critical] | 연속 |
| CADisplayLink | 403 | 0.0 [Good] | 연속 |
| **XCUITest** | **23.5** | 0.0 [Good] | **불연속** |

**결론**: XCUITest는 waitForQuiescence로 인해 극한 스크롤 부하 재현 불가. **req/s 23.5**로 성능 측정에 부적합.

### 2.2 2차 시도: CADisplayLink (채택)

**개념**: 앱 내부에서 CADisplayLink로 contentOffset 직접 조작

**장점**:
- waitForQuiescence 문제 없음
- 프레임 단위 정밀 제어 가능
- 속도/프로파일 파라미터화 가능

**단점**:
- UIKit 터치/관성 시스템 우회 → 수동 스크롤과 다른 부하 패턴
- 앱 코드에 테스트 코드 포함 필요 (Release 빌드에도)

## 3. AutoScrollTester 구현

### 3.1 기술 스택
- **CADisplayLink**: 프레임 동기화된 contentOffset 직접 조작
- **Release 빌드**: 정확한 성능 측정을 위해 필수
- **트리거**: 3손가락 탭 제스처 또는 런치 아규먼트

### 2.2 스크롤 프로파일

| 프로파일 | 설명 | 파라미터 |
|----------|------|----------|
| **continuous** | 연속 일정 속도 스크롤 | speed, oscillation |
| **burst** | 플릭 리듬 (버스트+휴지 반복) | burstDuration, restDuration, restSpeedRatio |

#### 버스트 프리셋
```swift
// 수동 극한 시뮬레이션
.manualExtreme: burst=0.25초, rest=0.12초, restRatio=15%

// 일상 스크롤 시뮬레이션
.manualNormal: burst=0.40초, rest=0.20초, restRatio=10%
```

### 2.3 테스트 시퀀스 (L1 + L2)
1. **L1 (3초)**: 일상 스크롤, 8000 pt/s, manualNormal
2. **정지 (1초)**: 터치 홀드 시뮬레이션
3. **L2 (8초)**: 극한 스크롤, 30000 pt/s, manualExtreme

## 5. 핵심 발견

### 5.1 속도-요청 선형 관계 확인
```
10000 pt/s → 204 req/s
25000 pt/s → 508 req/s  (2.5배 → 2.49배)
```
속도와 req/s는 거의 선형 관계.

### 5.2 자동 vs 수동 근본적 차이

| 특성 | 자동 스크롤 | 수동 스크롤 |
|------|------------|------------|
| **hitch** | 0.0 [Good] | 12~28 [Critical] |
| **gray/req** | 9.0+ | 5.27~6.97 |
| **스크롤 방식** | contentOffset 직접 조작 | UIKit 터치/관성 시스템 |
| **UIKit 이벤트** | 없음 | scrollViewDidScroll 등 발생 |

### 5.3 회색 셀 차이 원인 분석

**수동 스크롤의 이점:**
- 손가락 터치/릴리즈 사이클에서 UIKit 내부 최적화 발동
- 터치 이벤트가 없으면 시스템이 "idle" 상태로 인식 → 백그라운드 작업 우선순위 상승
- 스크롤 관성(deceleration) 중 점진적 감속 → 파이프라인 catch-up 시간 확보

**자동 스크롤의 한계:**
- CADisplayLink는 contentOffset만 변경 → UIKit 스크롤 시스템 미개입
- 버스트 프로파일의 "휴지"도 contentOffset이 천천히 변하는 것일 뿐, 진정한 idle 아님
- 시스템 입장에서는 계속 스크롤 중으로 인식

### 5.4 hitch 0.0의 의미
- 자동 스크롤에서 hitch가 0인 것은 **측정 문제가 아님**
- contentOffset 직접 조작은 UIKit 렌더링 파이프라인 부하가 적음
- 수동 스크롤은 터치 이벤트 처리 + 관성 계산 + 스크롤 델리게이트 등 추가 부하 발생

## 6. 결론

### 6.1 자동 스크롤 테스터의 가치
- **재현성**: 동일 조건 반복 테스트 가능 (수동 ±54%~99% 변동 해결)
- **파라미터 튜닝**: 속도, 프로파일, 지속시간 등 세밀한 조정 가능
- **CI/CD 통합**: 런치 아규먼트로 자동화 가능

### 6.2 자동 스크롤 테스터의 한계
- **수동 스크롤과 동일한 부하 패턴 재현 불가**
  - hitch 발생 패턴 다름
  - gray/req 비율 다름 (자동이 더 많은 회색 셀)
- **UIKit 스크롤 시스템 우회**로 인한 근본적 차이

### 6.3 권장 사항

| 용도 | 권장 방법 |
|------|----------|
| **회귀 테스트** | 자동 스크롤 (일관된 비교 가능) |
| **절대 성능 측정** | 수동 스크롤 (실제 사용자 경험) |
| **최적화 효과 검증** | 자동 → 수동 순서로 둘 다 수행 |

## 7. XCUITest 통합 (2025-12-24)

### 7.1 목표
- 자동화 테스트를 **XCUITest에서 실행** 가능하게 하여 CI/CD 통합 준비
- hitch 개선 작업의 **전후 비교가 가능한 재현 가능한 테스트** 환경 구축

### 7.2 문제 1: fps 60Hz 고정

**증상**: Release 빌드에서 hitch가 항상 0.0 ms/s로 측정됨

| 빌드 | fps | hitch |
|------|-----|-------|
| Debug | 76.2 | 10.5~18.0 ms/s |
| Release | **59.8** | 0.0 ms/s |

**원인**: iOS에서 120Hz ProMotion을 사용하려면 Info.plist 설정 필요

**해결**: `Info.plist`에 추가
```xml
<key>CADisableMinimumFrameDurationOnPhone</key>
<true/>
```

**결과**: Release에서 **119.7 fps** 달성, hitch 재현 성공

### 7.3 문제 2: XCUITest swipe 부하 부족

**증상**: XCUITest의 `swipeDown()` 사용 시 req/s가 매우 낮음

| 방식 | req/s | 비고 |
|------|-------|------|
| 수동 극한 | 580 | 목표 |
| XCUITest swipe | **21~30** | 부하 부족 |

**원인**: `waitForQuiescence` 비활성화 후에도 swipe 간 간격 존재

**결론**: XCUITest swipe는 극한 스크롤 재현에 부적합 → AutoScrollTester 사용

### 7.4 문제 3: AutoScrollTester launchArguments 전달 실패

**증상**: `app.launchArguments`로 `--auto-scroll` 전달해도 스크롤 안 됨

**원인**:
- `setUpWithError()`에서 이미 `app.launch()` 호출됨
- `app.terminate()` → `app.launch()` 순서 문제

**해결**: `setUpWithError()`에서 테스트 이름 확인 후 조건부 launch
```swift
override func setUpWithError() throws {
    if name.contains("testAutoScrollTester") {
        launchAppWithAutoScroll()  // launchArguments 포함
    } else {
        launchAppIfNeeded()  // 기본 launch
    }
}
```

### 7.5 최종 결과

**테스트 환경**: iPhone 13 Pro, Debug 빌드, XCUITest

**AutoScrollTester 설정**:
- speed: 12000 pt/s
- duration: 12초
- direction: down
- boundary: reverse

**결과** (12초 연속 스크롤):

| 지표 | 값 | 평가 |
|------|-----|------|
| hitch | **3.5 ms/s** | Good |
| fps | **119.5** | 120Hz 정상 |
| req/s | **272.9** | 안정적 |
| latency avg | 4.0ms | 양호 |
| frames | 1455 | |
| dropped | 5 | |

**XCUITest swipe vs AutoScrollTester 비교**:

| 방식 | hitch | fps | req/s |
|------|-------|-----|-------|
| XCUITest swipe | 6.9~15.9 ms/s | 117~119 | 21~30 |
| **AutoScrollTester** | **3.5 ms/s** | **119.5** | **272.9** |

### 7.6 현재 상태

- XCUITest에서 launchArguments로 AutoScrollTester 실행 가능
- 두 방식 모두 동작하지만 **특성이 다름**:
  - **XCUITest swipe**: hitch 6.9~15.9 ms/s로 수동(12~28)과 유사, req/s가 낮음(21~30)
  - **AutoScrollTester**: req/s는 높지만(272.9) hitch가 낮음(3.5) → 수동과 다른 부하 패턴
- **최종 방식 미결정** - 추가 검토 필요

## 8. XCUITest 추가 검증 (2025-12-25)

### 8.1 velocity 8000 반복 테스트

**목적**: XCUITest swipe의 재현성 검증

| 테스트 | 측정 횟수 | hitch 패턴 | Good | Critical |
|--------|----------|-----------|------|----------|
| 4회 측정 | 4 | 0 → 17.1 → 7.1 → 0 | 2 | 2 |
| 8회 측정 | 8 | 0 → 14.9 → 4.8 → 4.8 → 17.4 → 14.6 → 14.1 → 0 | 3 | 4 |

**발견**: 동일 velocity(8000)에서도 hitch가 0~18 ms/s로 큰 변동

### 8.2 preheat OFF 진단 테스트

**가설**: `preheatAfterScrollStop()`이 다음 측정 상태에 영향

| 조건 | hitch 패턴 | Good | Critical |
|------|-----------|------|----------|
| preheat ON | 0 → 14.9 → ... → 0 | 3 | 4 |
| **preheat OFF** | 0 → 15.5 → 7.2 → 14.8 → 18.3 → 15.6 → 15.1 → 0 | 2 | 5 |

**결론**: preheat OFF가 오히려 나빠짐 → **preheat는 hitch 변동 원인 아님**

### 8.3 velocity 단계별 테스트 (1000~10000)

| velocity | req/s | hitch | 평가 |
|----------|-------|-------|------|
| 1000 | 10.9 | 변동 | - |
| 2000 | 14.3 | 변동 | - |
| ... | ... | ... | - |
| 10000 | 31.9 | 변동 | - |

**발견**: velocity ↑ → req/s ↑ (선형 관계), hitch와 명확한 상관관계 없음

### 8.4 현재 상태

- XCUITest swipe: **hitch 재현은 되지만 변동성 높음** (0~18 ms/s)
- **미완료**: 버스트 패턴 (0.25s 드래그 + 0.12s 휴지) XCUITest 구현
