# E-3 코치마크 구현 계획: 삭제대기함 스와이프 복구 안내

## Context

E-1+E-2 시퀀스(삭제 시스템 안내) 완료 직후, 사용자가 삭제대기함 탭에 머무르는 상태에서 **스와이프로 사진을 복구**할 수 있다는 것을 자동 시연 애니메이션으로 안내한다. 직전 단계에서 삭제된 사진을 실제로 복구하여 원상복구까지 완료한다.

**핵심 요구사항**: E-1+E-2 dismiss 직후 자동 트리거. 사용자가 직접 스와이프할 필요 없이 자동 시연 + 실제 복구.

---

## 기존 코드 재사용 안내

> **손가락 아이콘 + 스와이프 모션 애니메이션은 이미 구현되어 있다.**
>
> - **A 코치마크**: `performDeleteSwipe()`, `performRestoreSwipe()` — 손가락 터치다운/프레스/드래그/릴리즈 4단계 모션
> - **다시보기 E-1+E-2**: `showReplaySwipeVariant()`, `performSingleDeleteSwipe()` — 1회 자동 스와이프 + 카드 팝업
> - **D-1 Step 3**: `performD1DeleteSwipe()`, `performD1RestoreSwipe()` — 녹색 커튼 스와이프 루프
>
> E-3의 스와이프 모션은 위 기존 구현을 참고하여 **녹색 커튼 1회 자동 스와이프** 형태로 구현한다.
> `showReplaySwipeVariant`의 1회 스와이프 패턴 + D-1의 녹색 커튼 색상을 조합.

## 전체 플로우

```
[E-1+E-2 Step 3 [확인] 탭] → dismiss
    │
    └── 0.3초 후 E-3 트리거
          │
          ├── 삭제대기함에서 직전 삭제 사진 셀 탐색
          │
          [E-3 표시]
            1. 딤 배경 페이드인 + 셀 하이라이트 구멍
            2. 셀 스냅샷 + 녹색 딤드 뷰 배치
            3. 자동 스와이프 모션 (1회, 녹색 커튼)
            4. 실제 복구 (trashStore.restore)
            5. 카드 팝업:
               "사진을 밀어서 편리하게 복구할 수 있어요
                연속으로 밀어서 여러 장 복구도 가능해요"
                        [확인]
          │
          └── [확인] 탭 → dismiss + markAsShown
```

**첫 사용 / 다시보기 모두 동일 흐름**:
- 첫 사용: 사용자가 삭제 → E-1+E-2 → E-3 (자동 복구)
- 다시보기: 기계가 임시 삭제 → E-1+E-2 → E-3 (자동 복구)

## assetID 전달

E-3는 삭제대기함의 **아무 visible 셀**을 사용하므로, E-1+E-2에서 assetID를 전달받을 필요 없다.
E-3가 직접 `findFirstTrashCell()`로 셀 + assetID를 획득한다.

> ⚠️ 뷰어 삭제는 E-1+E-2를 트리거하지 않음 (`GridViewController.swift:1050` 참조).
> 따라서 E-3도 뷰어 삭제 경로에서는 발생하지 않음.

## 스와이프 모션 상세

> **기존 구현 참고**: `showReplaySwipeVariant()` + `performSingleDeleteSwipe()` 패턴을 기반으로,
> D-1 Step 3의 녹색 커튼 색상을 적용한 **1회 자동 복구 스와이프**.

### 모션 시퀀스

```
Phase 1: 포커싱 축소 (D-1 animateDFocus + C-3 animateC3FocusCircle 패턴)
  t=0.00: overlay.alpha = 0.01 (hitTest로 터치 즉시 차단)
  t=0.00: dimLayer.path = 화면 전체 크기 rect 구멍 (= 딤 없음 상태)
          시작 path: fullRect + 화면 3배 크기 rect (구멍이 화면보다 커서 딤 안 보임)
  t=0.00: window.addSubview(overlay)
  t=0.00: UIView.animate alpha → 1.0 (0.3s) 동시 진행
  t=0.00: CABasicAnimation("path") — 큰 rect → 셀 크기 rect 축소
          duration: 0.7s (C-3과 동일, D-1의 0.9s보다 짧음 — E-1+E-2 직후이므로 빠르게)
          timingFunction: easeInEaseOut
          fillMode: .forwards, isRemovedOnCompletion: false
          completion에서 dimLayer.path = endPath 확정 + removeAnimation
  t=0.70: 포커싱 완료 → 0.2s 대기

  > **구멍 shape**: rect (margin 0, cornerRadius 0) — A/C-1 셀 하이라이트와 동일
  > **참고 코드**:
  > - D-1: `animateDFocus()` (CoachMarkOverlayView+CoachMarkD.swift:319) — pill 축소
  > - D-1: `animateD1Shrink()` (CoachMarkOverlayView+CoachMarkD1.swift:576) — rect 축소
  > - C-3: `animateC3FocusCircle()` (CoachMarkOverlayView+CoachMarkC3.swift:466) — 원 축소

Phase 2: 스냅샷 + 스와이프 준비
  t=0.70: 셀 위에 스냅샷 배치 + 녹색 딤드(width 0) + 손가락 아이콘(alpha 0)
  t=0.90: 스냅샷 페이드인 (0.2s)

Phase 3: 스와이프 모션 (showReplaySwipeVariant 패턴, 녹색)
  t=1.10: performSingleRestoreSwipe() 시작
          ├── Touch Down (0.3s, curveEaseOut): finger alpha 0→1, scale 1.1→1.0
          ├── Press (0.2s, spring damping 0.7): finger scale 1.0→0.95
          ├── Drag → (0.3s, UICubicTimingParameters): finger 이동 + 녹색 딤드 width 증가
          └── Release (0.2s, curveEaseIn): finger alpha→0, scale 1.05

Phase 4: 복구 + 카드
  t=2.20: 스와이프 완료 → 실제 복구 (TrashStore.shared.restore)
  t=2.20: highlightFrame = .zero + updateDimPath() (구멍 닫기)
  t=2.20: 스냅샷+녹색딤드 페이드아웃 (0.2s)
  t=2.40: 카드 팝업 페이드인 (0.25s) — 카드 내용 + [확인] 버튼
```

## 삭제대기함 셀 탐색

### TrashAlbumViewController 참조 획득

E-1+E-2의 `findTabBarController()` 패턴과 동일:

```swift
private func findTrashViewController(from window: UIWindow) -> TrashAlbumViewController? {
    guard let tabBar = window.rootViewController as? TabBarController else { return nil }
    // 삭제대기함 = 마지막 탭 (index 2)
    guard let trashNav = tabBar.viewControllers?.last as? UINavigationController,
          let trashVC = trashNav.viewControllers.first as? TrashAlbumViewController else {
        return nil
    }
    return trashVC
}
```

> E-1+E-2 `getEmptyButtonFrame()` (CoachMarkOverlayView+E1E2.swift:755)에서 동일 경로로
> TrashAlbumViewController를 획득하고 있으므로, 검증된 패턴이다.

### 셀 탐색

삭제한 특정 사진을 찾을 필요 없음. E-3의 목적은 "스와이프 복구가 가능하다"를 시연하는 것이므로 **화면 중앙에서 가장 가까운 셀**을 사용. A 코치마크의 `findCenterCell()` 패턴과 동일.

```swift
/// E-3용: 삭제대기함 화면 중앙에서 가장 가까운 PhotoCell + assetID 반환
private func findCenterTrashCell(in window: UIWindow)
    -> (cell: PhotoCell, frame: CGRect, snapshot: UIView, assetID: String)?

    guard let trashVC = findTrashViewController(from: window) else { return nil }

    let cv = trashVC.collectionView!
    let centerPoint = CGPoint(x: cv.bounds.midX, y: cv.bounds.midY)
    var bestCell: PhotoCell?
    var bestIndexPath: IndexPath?
    var bestDistance: CGFloat = .greatestFiniteMagnitude

    // 화면 중앙에서 가장 가까운 실제 사진 셀 탐색
    for indexPath in cv.indexPathsForVisibleItems {
        guard indexPath.item >= trashVC.paddingCellCount else { continue }
        guard let cell = cv.cellForItem(at: indexPath) as? PhotoCell else { continue }
        let distance = hypot(cell.center.x - centerPoint.x, cell.center.y - centerPoint.y)
        if distance < bestDistance {
            bestDistance = distance
            bestCell = cell
            bestIndexPath = indexPath
        }
    }

    guard let cell = bestCell, let indexPath = bestIndexPath else { return nil }

    let actualIndex = indexPath.item - trashVC.paddingCellCount
    guard let assetID = trashVC.gridDataSource.assetID(at: actualIndex) else { return nil }
    let frame = cell.superview!.convert(cell.frame, to: window)
    guard let snapshot = cell.snapshotView(afterScreenUpdates: false) else { return nil }
    return (cell, frame, snapshot, assetID)

    // 셀 없으면 nil → 0.3초 후 1회 재시도, 실패 시 E-3 스킵 (markAsShown 안 함)
```

> **셀 존재 보장**: E-1+E-2 시퀀스(약 10초+) 동안 탭 전환 → `viewWillAppear` → `onStateChange` 핸들러 등록
> + `loadTrashedAssets()` 호출로 fetchResult 갱신이 완료됨. 삭제대기함에 셀이 확실히 존재한다.
> 만약 타이밍 이슈로 셀을 못 찾으면 **0.3초 후 1회 재시도** 후 스킵.

## 복구 후 UI 갱신 타이밍

```
스와이프 모션 완료
  → TrashStore.shared.restore(assetIDs: [resolvedAssetID])  // 실제 복구
  → TrashStore.notifyChange()                                // 자동 발송
  → TrashAlbumViewController가 collectionView 갱신           // 아래 레이어에서 셀 사라짐
  → highlightFrame = .zero + updateDimPath()                 // 하이라이트 구멍 닫기
  → 스냅샷+녹색딤드 페이드아웃 (0.2s)                         // 오버레이 위의 스냅샷이 덮고 있으므로 시각적 끊김 없음
  → 카드 팝업 페이드인                                        // 구멍 없는 딤 배경 위에 카드만 표시
```

> 스냅샷이 오버레이 위에 있으므로, 아래 collectionView가 갱신되어도 사용자에게 보이지 않는다.
> **복구 직후 `highlightFrame = .zero` + `updateDimPath()`로 하이라이트 구멍을 닫는다.**
> 이렇게 해야 스냅샷 페이드아웃 후 구멍 사이로 엉뚱한 셀이 보이는 문제를 방지할 수 있다.

## 검증 방법

### 첫 사용 흐름
1. UserDefaults에서 `coachMark_firstDeleteGuide`, `coachMark_trashRestore` 삭제
2. 그리드에서 사진 스와이프 삭제
3. E-1+E-2 시퀀스 진행 → Step 3 [확인]
4. E-3 자동 표시: 딤 + 셀 하이라이트 + 녹색 스와이프 시연 + 사진 복구
5. 카드 팝업 + [확인] → dismiss
6. 삭제대기함에서 해당 사진이 사라졌는지 확인 (복구됨)
7. 보관함으로 돌아가면 해당 사진 존재 확인

### 다시보기 흐름
1. 전체메뉴 → 설명 다시 보기 → E-1+E-2
2. 자동 삭제 → E-1+E-2 시퀀스 → Step 3 [확인]
3. E-3 자동 표시 + 복구 시연
4. [확인] → dismiss → 사진 원상복구 완료

### 엣지 케이스
5. E-3 표시 중 모든 터치 차단 ([확인]만 가능)
6. Reduce Motion ON: 정적 녹색 딤드 + 바로 카드
7. VoiceOver ON: E-3 미표시
8. 삭제대기함에 셀 0개: E-3 스킵 (**markAsShown 안 함** → 다음 기회에 재시도)
9. assetID로 셀 못 찾을 때: 첫 번째 visible 셀로 폴백
10. 셀 탐색 재시도 1회 후에도 실패: E-3 스킵 (**markAsShown 안 함**)
11. 앱 재실행 후: `coachMark_trashRestore` 플래그로 재표시 방지
