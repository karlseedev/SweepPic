# 코치마크 D — 사전 스캔 벤치마크 결과

> 작성일: 2026-02-17

## 목적

코치마크 D에서 사용자에게 보여줄 저품질 사진 3장을 사전에 스캔하는 최적 파이프라인을 결정하기 위한 벤치마크.

---

## 테스트 설계

### 3종 파이프라인 비교

| 테스트 | 구성 | 설명 |
|--------|------|------|
| **T1 (Full)** | MetadataFilter → Exposure → SKIP필터 → Blur → SafeGuard | QualityAnalyzer 전체 파이프라인 |
| **T2 (SKIP필터만)** | MetadataFilter → Exposure → SKIP필터 → Blur | SafeGuard 제외 |
| **T3 (최경량)** | MetadataFilter → Exposure → Blur | SKIP필터도 SafeGuard도 없음 |

### 비교 의미

- **T1 - T2** = SafeGuard 비용 (Vision 얼굴 감지)
- **T2 - T3** = SKIP필터 비용 (유틸리티/화이트배경/텍스트 감지)

### SKIP필터 내용 (QualityAnalyzer 내부)

| 필터 | 조건 | 판정 |
|------|------|------|
| `isUtilityImage()` | 극단 휘도 + 낮은 RGB표준편차 | SKIP (단색 배경 메모/문서) |
| `detectTextScreenshot()` | 극단 노출 + 텍스트 블록 다수 | SKIP (블로그 캡처 등) |
| `isWhiteBackgroundImage()` | 극밝음 + 모서리 순백 + 모서리>중앙 | SKIP (일러스트/상품 사진) |

### SafeGuard 내용 (SafeGuardChecker)

| 체크 | 조건 | 효과 |
|------|------|------|
| depthEffect | PHAsset에 포트레이트 모드 표시 | 블러 신호 무효화 |
| clearFace | Vision 얼굴 품질 >= 0.4 | 블러 신호 무효화 |

- 블러 신호가 있을 때만 실행됨
- 메타데이터 체크(depthEffect) → 빠름, 얼굴 품질 체크(Vision) → 느림

### 테스트 조건

- 각 1000장, 이미지만 (비디오 미포함)
- 순차 처리 (병렬 없음)
- Precision 모드 (Strong 신호만 저품질 판정)
- 이미지 로딩: CleanupImageLoader (짧은변 360px)

SKIP필터가 구제한 7장: 모두 `extremeBright` → 실제로는 흰 배경/유틸리티 이미지

## 분석

### SafeGuard

| | 앨범 A | 앨범 B |
|---|---|---|
| 시간 비용 | +0.18초 | +3.62초 |
| 구제한 사진 | 0장 | 0장 |

- 두 앨범 모두 SafeGuard가 구제한 사진 **0장**
- 앨범 B에서 3.62초 추가 비용 (블러 판정 48장에 대해 Vision 얼굴 감지 실행)
- **결론: 사전 스캔에서는 SafeGuard 제외**

### SKIP필터

| | 앨범 A | 앨범 B |
|---|---|---|
| 시간 비용 | -0.05초 (오차) | +1.43초 |
| 구제한 사진 | 0장 | 7장 |

- 깨끗한 앨범에서는 영향 없음
- 다양한 앨범에서는 1.43초로 7장 오판 방지
- 코치마크 D에서 흰 배경 이미지를 "저품질"로 보여주면 신뢰도 하락
- **결론: SKIP필터 포함 필수**

### 처리 시간 병목

- 평균 14~23ms/장 중 분석 자체는 ~1ms 수준
- **대부분의 시간은 이미지 로딩** (PHAsset → CGImage 변환)
- 분석 방식 변경보다 이미지 로딩 최적화가 필요

- SKIP필터로 오판 방지

## 병렬 처리 벤치마크

### 테스트 설계

T2 파이프라인을 동시 처리 수 1/2/4/8로 실행하여 1000장 처리 시간을 비교.

| 테스트 | 동시 수 | 역할 |
|--------|---------|------|
| P0 | 1 (순차) | 기준선 |
| P1 | 2 | |
| P2 | 4 | |
| P3 | 8 | |

- TaskGroup + 동시 수 제한 패턴
- 테스트 간 1초 딜레이 (GPU/캐시 안정화)

### 결과

#### 앨범 A — 깨끗한 사진 위주

| | P0 (순차) | P1 (동시 2) | P2 (동시 4) | P3 (동시 8) |
|---|---|---|---|---|
| 총 시간 | 14.29초 | 13.82초 | 13.53초 | 13.57초 |
| P0 대비 | 기준 | 1.03x | 1.06x | 1.05x |

#### 앨범 B — 다양한 사진

| | P0 (순차) | P1 (동시 2) | P2 (동시 4) | P3 (동시 8) |
|---|---|---|---|---|
| 총 시간 | 18.18초 | 16.52초 | 16.02초 | 15.73초 |
| P0 대비 | 기준 | 1.10x | 1.13x | 1.16x |

### 분석

- 두 앨범 모두 **병렬 처리 효과 미미** (최대 1.16x)
- 원인: 장당 ~14ms 중 분석 ~1ms, **이미지 로딩이 ~93%** 차지
- PHImageManager가 내부적으로 동시 요청을 직렬화/스로틀링하기 때문에 병렬 분석이 무의미
- **결론: 병렬 처리로는 속도 개선 불가능. 이미지 로딩 자체를 줄이는 접근 필요**

## 이미지 로딩 방식 벤치마크

### 테스트 설계

PHImageRequestOptions의 deliveryMode를 변경하여 로딩 속도 및 분석 결과 차이를 비교.

| 테스트 | deliveryMode | 설명 |
|--------|-------------|------|
| Q1 | `.highQualityFormat` | 현재 방식 (풀 디코딩) |
| Q2 | `.fastFormat` | 빠른 로딩 (내장 썸네일 등) |

### 결과 — 앨범 B (다양한 사진)

| | Q1 (highQuality) | Q2 (fastFormat) |
|---|---|---|
| 총 시간 | 18.14초 | 11.80초 |
| 평균/장 | 18.1ms | 11.8ms |
| 저품질 | 48장 | **168장** |
| 로딩 실패 | 0장 | 0장 |

```
속도향상: 1.54x (6.3초 절약)
판정 차이:
  공통: 48장 (Q1의 전체)
  Q1에만: 0장
  Q2에만: 120장 (모두 severeBlur)
```

### 분석

- **속도**: 1.54x 빠름 (18.14 → 11.80초)
- **판정**: fastFormat이 120장을 추가로 저품질 판정 (모두 severeBlur)
- **원인**: fastFormat이 낮은 품질의 이미지를 반환 → BlurAnalyzer가 실제로 선명한 사진도 블러로 오판
- Q1의 48장은 Q2에 전부 포함 → fastFormat이 놓치는 건 없지만 **거짓양성(false positive)이 120장**
- 정밀도: 48/168 = 28.6%로 매우 낮음
- **결론: fastFormat은 블러 오판이 심각하여 사용 불가**

## 종합 결론

### 확정 사항

- **파이프라인**: T2 (SKIP필터 포함, SafeGuard 제외)
- **로딩 방식**: `.highQualityFormat` 유지 (변경 불가)
- **처리 방식**: 순차 처리 (병렬 무의미)

### 현재 성능

- 1000장 순차: 14~18초
- 장당 평균: 14~18ms (이미지 로딩 ~93%, 분석 ~7%)

### 속도 개선 시도 및 결과

| 방안 | 결과 | 비고 |
|------|------|------|
| ~~병렬 처리 (2/4/8)~~ | ❌ 효과 없음 (최대 1.16x) | PHImageManager가 병목 |
| ~~fastFormat 로딩~~ | ❌ 블러 오판 심각 (120장 거짓양성) | 속도 1.54x 향상이나 정밀도 28% |
| ~~이미지 해상도 축소~~ | ❌ 검토 제외 | BlurAnalyzer 256x256 미만 시 업스케일링, 정확도 하락 |
| ~~Exposure만 먼저~~ | ❌ 검토 제외 | 저품질의 핵심 신호인 블러를 못 잡음 |

### 대응 전략

- **앱 시작 즉시 스캔 시작** → D 트리거까지 ~13초 → ~930장 처리 가능
- 대부분의 앨범(0.5% 이상 비율)에서 3장 확보 가능
- 극단적으로 깨끗한 앨범(0.2% 미만)에서만 부족 → 텍스트 폴백으로 대응
