## 0. 기능 개요

### 기능 흐름

```
[정리 버튼 탭]
     │
     ▼
[휴지통 비었는지 확인]
     │
     ├─ 안 비었음 → 팝업: "휴지통을 먼저 비워주세요" + [휴지통으로 이동]
     │
     └─ 비었음 → [정리 방식 선택 팝업]
                    │
                    ├─ 1. 최신사진부터 정리
                    ├─ 2. 이어서 정리 (이력 없으면 비활성화)
                    └─ 3. 연도별 정리 > (연도 선택)
                              │
                              ▼
                         [탐색 시작] → 바로 휴지통 이동 → [안내 팝업]
```

### 핵심 목표

1. **저품질 사진의 정량적 정의** - 오탐지 최소화
2. **판별 로직 및 기능** - 고성능, 사용자 친화적
3. **간소화된 UX** - 검토 화면 제거, 휴지통에서 확인/복구

### 정리 버튼 위치

- **그리드 화면 상단, 셀렉트 버튼 왼쪽**에 배치

## 1. 저품질 사진의 정량적 정의

사용자의 소중한 추억을 보호하기 위해 **'확실한 기술적 실패'**만을 저품질로 규정한다.

### A. 판정 원칙

| 원칙 | 설명 | 적용 모드 |
|-----|------|:--------:|
| **Strong** | 단일 항목만으로 즉시 저품질 확정 | 모든 모드 |
| **Conditional** | 기술적 실패지만 오탐 위험 있음 | Recall만 |
| **Weak** | 2개 이상 조합 시 저품질 확정 | Recall만 |

### B. 정량적 기준

| 구분 | 항목 | 임계값 | 근거 |
|:---:|------|-------|------|
| **Strong** | 극단 노출 (어두움) | 휘도 < 0.10 | 설계값 (Gist 0.133 기준 엄격) |
| | 극단 노출 (밝음) | 휘도 > 0.90 | GitHub Gist |
| | 심각 블러 | Laplacian < 50 | 설계값 (PyImageSearch 100의 50%) |
| **Conditional** | 주머니 샷 | 휘도<0.15 AND RGB Std<15 AND Lap<50 AND 비네팅<0.05 | 복합 조건 |
| | 극단 단색 | RGB Std<10 AND (휘도<0.15 OR >0.85) | 복합 조건 |
| | 렌즈 가림 | 모서리 휘도 < 중앙 × 0.4 | 3×3 그리드 |
| **Weak** | 일반 블러 | Laplacian < 100 | PyImageSearch |
| | 일반 노출 | 휘도 < 0.15 또는 > 0.85 | 완화 기준 |
| | 낮은 색상 다양성 | RGB Std < 15 | 설계값 |
| | 저해상도 | < 1MP | VGA 이하 |

**Weak 가중치 (iOS 16-17, Recall 모드):**
- 일반 블러: **2점**, 나머지: 1점
- 합산 >= 3점 → LOW_QUALITY

> **iOS 18+**: AestheticsScore로 대체 (Precision: <-0.3, Recall: <0)
> **isUtility**: AestheticsScore의 `isUtility == true`인 경우 스크린샷과 동일 취급 (별도 카테고리)

### C. 안전장치 (Safe Guard)

| 시점 | 조건 | 효과 |
|-----|------|------|
| **조기 필터** | 즐겨찾기, 편집됨, 숨김, 공유앨범 | 분석 건너뜀 |
| **후처리** | 심도 효과, 선명한 얼굴 (Quality >= 0.4) | 블러 판정 무효화 |

> **Face Quality 0.4**: Apple은 상대 비교용으로 권장하나, 본 기능은 최소 품질 판정 목적으로 절대 임계값 사용. 테스트 검증 필요.

### D. 스크린샷 처리

- 판별: `mediaSubtypes.contains(.photoScreenshot)`
- 처리: 저품질 로직에서 **제외** (1차 범위 외, 별도 기능으로 추후 개발)

## 2. 판별 모드

### Precision 모드 (1차 출시)
- **목적**: 오탐지 최소화
- **사용**: Strong 신호만
- **임계값**: 엄격 (휘도<0.10, Lap<50)

### Recall 모드 (추후 추가 예정)
- **목적**: 커버리지 최대화
- **사용**: Strong + Conditional + Weak 조합
- **임계값**: 완화 (휘도<0.15, Lap<100)
- **상태**: 1차 범위 외, 추후 업데이트에서 추가

## 3. 판별 파이프라인 (4단계)

```
Stage 1: MetadataFilter [비용: 최저]
  → 즐겨찾기/편집됨/숨김/공유앨범 → SKIP
  → 스크린샷 → SEPARATE
  → 저해상도 < 1MP → Weak 저장

Stage 2: ExposureAnalyzer [비용: 중간, 64×64]
  → 휘도, RGB Std, 비네팅 계산
  → 극단 노출 → LOW_QUALITY (Strong) → 조기 종료
  → 극단 단색 → LOW_QUALITY (Conditional, Recall만) → 조기 종료
  → 렌즈 가림 → LOW_QUALITY (Conditional, Recall만) → 조기 종료
  → 일반 노출, 색상 다양성 → Weak 저장

Stage 3: BlurAnalyzer [비용: 최고, 256×256]
  → Laplacian Variance (Metal GPU)
  → Lap < 50 → Strong 마킹 (Stage 4에서 Safe Guard 체크 필요)
  → Lap < 100 → Weak 저장

Stage 4: CompositeJudge + SafeGuard
  → Safe Guard 체크: 심도 효과, 얼굴 품질 >= 0.4 → 블러 판정 무효화
  → 주머니 샷 복합 판정 (Conditional, Recall만)
  → Weak 가중치 집계 (합>=3 → LOW_QUALITY)
```

> **조기 종료**: 노출/단색/렌즈 판정은 Safe Guard 대상 아님 → 즉시 LOW_QUALITY 확정
> **블러 판정**: Safe Guard 적용 가능 → Stage 4까지 진행 후 최종 판정

> **iOS 18+**: Stage 1 → AestheticsScore → Safe Guard 체크 → 최종 판정. AestheticsScore 실패 시 Stage 2-4 Fallback

## 4. 탐색 및 실행 로직

### A. 종료 조건 (우선순위)

1. **50장 찾음** → 즉시 중단
2. **1,000장 검색** → 종료
3. **연도 끝 + 1,000장 미만** → 다음 연도로 1번만 확장 (연도별 정리 제외)
4. **가장 오래된 사진** → 종료

> **연도별 정리**: 선택한 연도만 검색, 50장 못 찾아도 확장 없이 종료

### B. 정리 방식

| 방식 | 시작점 | 범위 |
|-----|-------|------|
| 최신사진부터 | 현재 | 종료 조건까지 |
| 이어서 정리 | 마지막 탐색 위치 | 종료 조건까지 |
| 연도별 | 선택 연도 12월 | 해당 연도만 |

> **연도 기준**: PHAsset의 `creationDate`를 기기 현지 타임존으로 변환하여 판단

### C. 성능 제약

| 항목 | 값 |
|-----|---|
| 배치 크기 | 100장 |
| 동시 분석 | 4개 |
| 다운샘플 (노출) | 64×64 |
| 다운샘플 (블러) | 256×256 |
| iCloud | 로컬 캐시 썸네일 사용 (`networkAccessAllowed=false`) |

### D. 특수 미디어 처리

| 유형 | 처리 방식 | 근거 |
|-----|----------|------|
| **비디오** | 로컬: 프레임 3개 추출, 중앙값 판정. iCloud-only: SKIP. 10분 초과: SKIP | 프레임 추출은 로컬 파일 필요 |
| **Live Photo** | 정지 이미지만 분석 | 사용자는 "사진"으로 인식, 삭제 시 함께 삭제됨 |
| **Burst** | 대표 사진만 분석 | PHFetchResult 기본 동작, 그룹 관리 복잡도 회피 |
| **RAW+JPEG** | JPEG로 분석, 삭제 시 함께 삭제 | PHAsset 구조상 분리 불가 |

### E. 에러 및 예외 처리

| 상황 | 처리 | 근거 |
|-----|------|------|
| 분석 실패 (Vision/Metal) | SKIP | 판단 불가 시 삭제 금지 |
| 이미지 로드 실패 | SKIP | 손상 파일은 별도 처리 |
| iCloud 썸네일 없음 | SKIP | 원본 다운로드 안 함 |

### F. 백그라운드 전환

| 상황 | 동작 |
|-----|------|
| 백그라운드 전환 | 일시정지 (상태 유지) |
| 포그라운드 복귀 | 자동 재개 |
| 앱 종료/강제 종료 | 진행 상태 소실 (처음부터) |

> **근거**: iOS 백그라운드 실행 제약 (30초), GPU 작업 비효율, 배터리 소모

## 5. UX 흐름

### 5.0 휴지통 확인 (전제조건)

휴지통이 비어있지 않으면 → "휴지통을 먼저 비워주세요" + [휴지통으로 이동]

### 5.1 정리 방식 선택

- 최신사진부터 정리
- 이어서 정리 (이력 없으면 비활성화)
- 연도별 정리 > (연도 선택)
- **선택 즉시 탐색 시작**

### 5.2 iCloud 처리

- 로컬 캐시 썸네일 (~342×256) 사용
- 원본 다운로드 없음 → **안내 팝업 불필요**

### 5.3 스캔 피드백

```
┌─────────────────────────────────┐
│  2026년 5월부터 탐색 중...       │
│  ████████░░░░░░░░  23/50        │
│                      [취소]     │
└─────────────────────────────────┘
```

### 5.4 결과 안내

| 시나리오 | 메시지 |
|---------|-------|
| 50장 발견 | "50장의 정리할 사진을 휴지통으로 이동했습니다" |
| N장 발견 | "N장의 정리할 사진을 휴지통으로 이동했습니다" |
| 0장 발견 | "정리할 사진을 찾지 못했습니다" |
| 취소 (N장) | "취소 전까지 탐색된 N장을 휴지통으로 이동했습니다" |

### 5.5 휴지통 정책

- 보관 기간: 30일
- 복구: 휴지통에서 직접

### 5.6 판별 모드 전환 UI (추후 추가 예정)

> **1차 출시**: Precision 모드만 사용, 모드 전환 UI 없음
> **추후 업데이트**: Recall 모드 추가 시 아래 UI 구현

- 신중한 정리 (기본) → Precision
- 적극적 정리 → Recall
