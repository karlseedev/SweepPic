# 추천 리워드 프로그램 v2

> 기존 클립보드 방식(v1)에서 **Offer Code URL Redemption** 방식으로 전환.
> v1 문서: `260316Reward-bak.md`

---

## 1. 핵심 구조

### 보상 테이블

| 대상 | 보상 | 적용 방식 |
|------|------|----------|
| **피추천인** (신규) | Free Trial 7일 + **14일 프리미엄 무료** (총 3주) | Offer Code URL Redemption |
| **추천인** (기존 구독자) | **14일 구독 연장** | **Promotional Offer** (앱 내 자동 적용) |
| **추천인** (비구독자/만료) | **14일 프리미엄 무료** | Offer Code (5개 Offer 순환) |

> **보상 통일: 전원 14일(2주)**
>
> **피추천인용**: Offer Code 1개 (`referral_invited`)
> - 대상: New + Existing + Expired (전체)
> - 기간: 2주 무료
> - Introductory Offer 중복: Yes
>
> **추천인용 (기존 구독자)**: Promotional Offer 1개 (`referral_extend_2week`)
> - 앱 내에서 서버 서명 기반 자동 적용 (App Store 리딤 시트 불필요)
> - Apple 제한 없이 무제한 적용 가능 → 서버에서 월 5회 상한 제어
>
> **추천인용 (비구독자/만료)**: Offer Code 5개 (`referral_reward_01` ~ `referral_reward_05`)
> - Apple "1인 1오퍼 1회" 제한 우회: 매번 다른 Offer의 코드 지급
> - 5개 Offer × 1회 = 월 최대 5회 가능
> - 합계: 1(피추천인) + 1(Promotional) + 5(Offer Code) = **7개 / 한도 10개**

> **Free Trial + Offer Code 중복 적용 가능** (Apple 공식 지원)
> ASC에서 "Introductory Offer 적용 여부 = Yes"로 설정하면,
> Free Trial 7일 → Offer Code 14일 → 정상 가격 순으로 순차 적용됨.

### 보상 기간 결정 근거 (2주 vs 1개월)

| 근거 | 데이터 |
|------|--------|
| 전환율 차이 | 14일 44% vs 30일 46% → **+1.7%p에 불과** |
| 비용 차이 | **2배** |
| 학술 실험 (Frontiers in Psychology, 2025) | 7일이 30일보다 전환율 5.6%↑, 유지율 6.4%↑, 매출 7.9%↑ |
| novelty decay | 사진 정리 앱은 탐색적 앱 → 길수록 관심 소멸 |
| 어뷰징 리스크 | 보상 클수록 저품질 유입 증가 (Marketing Science 논문) |
| 업계 포지션 | Headspace(2주), Noom(2주)과 동일 수준 |
| 프레이밍 | "14일 프리미엄 무료"가 "2주"보다 더 관대하게 느껴짐 (단위 효과) |

> **전략**: 2주로 시작 → 데이터 보고 1개월로 상향 여부 결정
> A/B 테스트: Custom Code 2개(2주용/1개월용) 반반 배분 → ASC 리포트로 전환율 비교

## 2. 전체 플로우

### Phase 1: 추천 링크 생성 (추천인 → 앱)

```
추천인이 앱 내 "친구 초대" 버튼 탭
  ↓
앱 → Supabase: 추천인 user_id로 고유 referral_code 조회 또는 생성
  ↓
추천 링크 생성: sweeppic.link/r/{referral_code}
  ↓
iOS 공유 시트로 링크 공유 (카카오톡, 인스타그램, 메시지 등)
```

### Phase 2: 피추천인 링크 클릭 → 보상 수령

```
피추천인이 링크 클릭: sweeppic.link/r/{referral_code}
  ↓
랜딩 페이지 (Supabase Edge Function 또는 별도 호스팅)
  ├─ 서버: referral_code → 추천인 식별 → referrals 테이블에 기록 (status: clicked)
  ├─ 서버: offer_codes 테이블에서 미사용 One-time Code 1개 할당
  │       (할당 시간 기록, 48시간 미리딤 시 자동 회수)
  └─ 랜딩 페이지 UI: 앱 소개 + "14일 프리미엄 무료 시작" CTA 버튼 표시
        ↓
피추천인이 CTA 버튼 탭
  ↓
302 Redirect → https://apps.apple.com/redeem?ctx=offercodes&id={APP_ID}&code={할당된_코드}
  ↓
[앱 미설치 시]                          [앱 설치 시]
  App Store 다운로드 프롬프트              바로 리딤 시트 표시
  → 다운로드 완료                        (코드 자동 입력됨)
  → 리딤 시트 자동 표시                   → "확인" 탭
  → "확인" 탭                           → 구독 시작
  → 구독 시작
```

> **코드 입력 불필요** — URL에 코드가 포함되어 자동 pre-fill됨 (Apple Tech Talk 확인)
> *"You'll notice the code input step isn't required here, as the code is part of the URL."*

### Phase 3: 리딤 감지 → 피추천인 최소 활동 → 추천인 보상 트리거

```
피추천인 앱 실행
  ↓
Transaction.updates (이미 SubscriptionStore에 구현됨)
  → 외부 리딤 트랜잭션 자동 수신 (unfinished 트랜잭션으로 즉시 전달)
  → offer.type == .code && offer.id == "referral_invited" 로 추천 리딤 식별
  ↓
앱 → Supabase: (keychain_user_id, offerIdentifier, 리딤시각) 보고
  ↓
서버: referrals 테이블에서 시간 기반 매칭 → 추천인 식별
서버: referrals 업데이트 (status: clicked → redeemed)
  ↓
[피추천인 최소 활동 조건 대기]
피추천인이 사진 N장 이상 정리 완료
  → 앱 → Supabase: "최소 활동 충족" 보고
  → 서버: referrals 업데이트 (status: redeemed → qualified)
  → 서버: 추천인 보상 트리거:
        1. referral_links.monthly_reward_count 확인
           ├─ 5회 이상 → 보상 스킵 (이번 달 상한 도달)
           └─ 5회 미만 → 계속
        2. monthly_reward_count += 1
        3. 추천인 구독 상태 확인
           ├─ 기존 구독자 → reward_type='promotional'
           └─ 비구독자/만료 → 추천인이 미리딤한 Offer 중 가장 낮은 번호 선택
                             (reward_01 사용済 → reward_02, ...)
                             → reward_type='offer_code', offer_name 결정
        4. pending_rewards에 기록
```

> **매칭 방식**: `offerIdentifier`는 오퍼 이름(개별 코드 값 아님)이므로,
> 피추천인의 `keychain_user_id` + 리딤 시각으로 referrals 테이블과 매칭.
> 초기 서비스에서 동시 리딤 충돌 확률은 극히 낮음.
> 매칭 실패 시(해당하는 clicked 레코드 없음) → 로깅만 하고 무시 (직접 코드 입수 등).

> **추가 안전장치 (향후)**: App Store Server Notifications V2의 `OFFER_REDEEMED` 웹훅을
> Supabase에서 수신하면, 앱 실행 없이도 서버에서 독립적으로 리딤 확인 가능.

### Phase 4: 추천인 보상 수령 (하이브리드)

```
추천인 보상 트리거 시 (서버):
  ├─ 추천인이 기존 구독자? → Promotional Offer 준비
  └─ 추천인이 비구독자/만료? → Offer Code 할당 (reward_01~05 순환)
  → pending_rewards에 기록 (reward_type: 'promotional' 또는 'offer_code')

추천인이 앱 실행 (포그라운드 진입)
  ↓
앱 → Supabase: pending_rewards 조회
  ├─ 보상 없음 → 정상 진행
  └─ 보상 있음 → 인앱 축하 팝업 표시
        "친구가 SweepPic에 가입했어요! 14일 프리미엄을 받으세요"
        [보상 받기] 버튼
          ↓
        [reward_type에 따라 분기]
        ├─ 'promotional' (기존 구독자):
        │     서버에서 서명 요청 → Product.purchase(options: [.promotionalOffer(...)])
        │     → 앱 내에서 바로 구독 연장 (App Store 리딤 시트 불필요)
        │     → Supabase: pending_rewards 상태 → completed
        │
        └─ 'offer_code' (비구독자/만료):
              UIApplication.shared.open(offerCodeRedeemURL)
              → App Store 리딤 시트 (코드 자동 입력)
              → "확인" 탭 → 구독 시작
              → Transaction.updates로 리딤 감지
              → Supabase: pending_rewards 상태 → completed
```

## 3. Offer Code 재고 관리

### 코드 풀 운영

```
[자동화 파이프라인]
Supabase Scheduled Function (주 1회)
  ↓
잔여 코드 수 확인
  ├─ 충분 (>1,000개) → 패스
  └─ 부족 (<1,000개)
        ↓
      App Store Connect API 호출
        POST /v1/subscriptionOfferCodeOneTimeUseCodes
        → 코드 생성 (최대 25,000개/배치)
        ↓
      GET /v1/subscriptionOfferCodeOneTimeUseCodes/{id}/values
        → CSV로 코드 값 다운로드
        ↓
      offer_codes 테이블에 INSERT
```

### 코드 할당 및 회수

```
[할당]
피추천인 랜딩 페이지 접속 시:
  offer_codes에서 status='available' AND expires_at > now() 인 코드 1개 SELECT FOR UPDATE
  → status='assigned', assigned_to={session_id}, assigned_at=now()

[회수]
Supabase Scheduled Function (매 6시간):
  1. 미리딤 코드 회수:
     status='assigned' AND assigned_at < now() - 48시간 인 코드
     → status='available', assigned_to=NULL, assigned_at=NULL
  2. 만료 코드 정리:
     expires_at < now() 인 코드
     → status='expired'
```

### App Store Connect Offer 설정

**총 7개 Offer:**

| Offer Name | 용도 | 유형 | 대상 | 비고 |
|-----------|------|------|------|------|
| `referral_invited` | 피추천인 보상 | Offer Code (One-time) | New + Existing + Expired | Intro Offer 중복: Yes |
| `referral_extend_2week` | 추천인 보상 (구독자) | **Promotional Offer** | Existing | 서버 서명, 무제한 적용 가능 |
| `referral_reward_01` | 추천인 보상 (비구독자) 1회차 | Offer Code (One-time) | New + Expired | |
| `referral_reward_02` | 추천인 보상 (비구독자) 2회차 | Offer Code (One-time) | New + Expired | |
| `referral_reward_03` | 추천인 보상 (비구독자) 3회차 | Offer Code (One-time) | New + Expired | |
| `referral_reward_04` | 추천인 보상 (비구독자) 4회차 | Offer Code (One-time) | New + Expired | |
| `referral_reward_05` | 추천인 보상 (비구독자) 5회차 | Offer Code (One-time) | New + Expired | |

> 동시 활성 Offer: 7개 / 한도 10개 → 3개 여유 (다른 프로모션용)
> 모든 Offer Code는 기간: 2주 무료로 동일 설정

## 4. 사용자 식별 체계

### 영구 User ID (Keychain 기반)

```swift
// 앱 최초 실행 시 UUID 생성 → Keychain 저장 (앱 삭제에도 유지)
// 이미 UsageLimitStore에서 동일한 Keychain 패턴 사용 중
let userId = KeychainStore.getOrCreate(key: "sweeppic_referral_id", default: UUID().uuidString)
```

- IDFV는 재설치 시 변경되므로 추천 전용 ID로 부적합
- Keychain은 앱 삭제 후에도 유지됨
- 기존 IDFV는 분석용으로 계속 사용, 추천 전용 ID만 Keychain에 추가

### 리딤 감지 코드 (iOS 버전 분기)

```swift
// offer.type은 iOS 17.2+, offerType은 iOS 15+ (deprecated)
let offerRefName: String?
let isOfferCode: Bool
if #available(iOS 17.2, *) {
    offerRefName = transaction.offer?.id
    isOfferCode = transaction.offer?.type == .code
} else {
    offerRefName = transaction.offerID
    isOfferCode = transaction.offerType == .code
}

if isOfferCode, offerRefName == "referral_invited" {
    // 피추천인의 추천 리딤 감지 → 서버에 보고
}
```

### 추천인 보상 적용 코드 (Promotional Offer — 기존 구독자)

```swift
// 서버에서 서명 요청 → Promotional Offer로 앱 내 자동 적용
func applyReferralReward(product: Product, signatureData: SignatureResponse) async throws {
    let offer = Product.PurchaseOption.promotionalOffer(
        offerID: "referral_extend_2week",
        keyID: signatureData.keyID,
        nonce: signatureData.nonce,
        signature: signatureData.signature,
        timestamp: signatureData.timestamp
    )
    let result = try await product.purchase(options: [offer])
    // 구독 연장 완료 — App Store 리딤 시트 불필요
}
```

## 5. 어뷰징 방지

| 시나리오 | 대책 |
|---------|------|
| **자기 추천** (2대 기기) | 동일 Keychain user_id 감지 (같은 Apple ID면 Keychain 공유) |
| **추천 농장** | 추천인당 월간 보상 상한 **5회** |
| **피추천인 최소 활동** | 사진 N장 이상 정리 완료 후 추천인 보상 트리거 (아하 모먼트 도달 보장 + 가짜 설치 방지) |
| **중복 리딤 (피추천인)** | Apple이 "1인 1오퍼 1회" 자동 제한 |
| **추천인 무한 리딤** | 기존 구독자: Promotional Offer → 서버에서 월 5회 제한. 비구독자: Offer Code 5개 순환 → Apple이 물리적으로 5회 제한 |
| **할당 후 미리딤** | 48시간 후 자동 회수 |
| **비구독자 무한 무료** | 보상 2주 통일로 위험 대폭 완화 (월 5회 × 2주 = 10주, 기존 5개월 대비) |

## 6. Supabase 테이블 구조

```sql
-- 추천 링크 (추천인당 1개)
CREATE TABLE referral_links (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id TEXT NOT NULL UNIQUE,        -- Keychain 기반 영구 ID
  referral_code TEXT NOT NULL UNIQUE,   -- 6자리 영숫자 (예: abc123)
  monthly_reward_count INT DEFAULT 0,   -- 이번 달 보상 횟수 (상한 5회)
  monthly_reward_reset_at TIMESTAMPTZ,  -- 월간 카운터 리셋 시점
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 추천 기록
CREATE TABLE referrals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  referrer_user_id TEXT NOT NULL,       -- 추천인
  referred_session_id TEXT,             -- 피추천인 (랜딩 페이지 세션)
  referred_user_id TEXT,                -- 피추천인 (앱 설치 후 매칭)
  offer_code TEXT,                      -- 할당된 Offer Code
  status TEXT DEFAULT 'clicked',        -- clicked → redeemed → qualified → rewarded
  clicked_at TIMESTAMPTZ DEFAULT now(),
  redeemed_at TIMESTAMPTZ,
  qualified_at TIMESTAMPTZ,             -- 피추천인 최소 활동 충족 시점
  rewarded_at TIMESTAMPTZ
);

-- 추천인 보상 대기
CREATE TABLE pending_rewards (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id TEXT NOT NULL,                -- 보상 받을 추천인
  referral_id UUID REFERENCES referrals(id),
  reward_type TEXT NOT NULL,            -- 'promotional' | 'offer_code'
  offer_code TEXT,                      -- 추천인용 Offer Code (offer_code 타입만)
  offer_name TEXT,                      -- ASC Offer Name (reward_01~05 중 어떤 것인지)
  redeem_url TEXT,                      -- apps.apple.com/redeem?... (offer_code 타입만)
  status TEXT DEFAULT 'pending',        -- pending → completed → expired
  created_at TIMESTAMPTZ DEFAULT now(),
  completed_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ                -- 보상 만료 시한 (코드 만료일 또는 생성 후 30일)
);

-- Offer Code 풀
CREATE TABLE offer_codes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code TEXT NOT NULL UNIQUE,            -- Apple이 생성한 실제 코드 값
  offer_name TEXT NOT NULL,             -- ASC Offer Name (referral_invited, referral_reward_01~05)
  status TEXT DEFAULT 'available',      -- available → assigned → used → reclaimed → expired
  assigned_to TEXT,                     -- 할당된 세션/유저 ID
  assigned_at TIMESTAMPTZ,
  used_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ               -- Apple 코드 만료일 (최대 6개월)
);
```

> offer_codes 테이블의 코드는 피추천인용(`referral_invited`)과 추천인 비구독자용(`referral_reward_01~05`)을
> `offer_name` 컬럼으로 구분. 추천인 기존 구독자는 Promotional Offer이므로 코드 풀 불필요.

## 7. 기술 인프라

| 요소 | 기술 | 비고 |
|------|------|------|
| 랜딩 페이지 | Supabase Edge Function | 커스텀 도메인 구매 (연 $5~15) + DNS → Supabase 연결 |
| 추천 데이터 | Supabase PostgreSQL | 기존 인프라 |
| 코드 생성 자동화 | Supabase Scheduled Function + ASC API | P8 키 필요 |
| 코드 리딤 | Apple Offer Code URL Redemption | apps.apple.com/redeem |

| 추천인 보상 알림 | 인앱 알림 (앱 실행 시 조회) | 푸시 불필요 |
| 분석 이벤트 | SupabaseProvider (기존) | 퍼널 추적 |

# 해결된 과제

## A. 인앱 브라우저 호환성 → 해결

**문제**: 인앱 브라우저(WKWebView)에서 `apps.apple.com/redeem` URL이 App Store를 열지 못하고 웹뷰 안에서 로드됨. 302 리다이렉트, `window.location`, `<a>` 태그 모두 동일.

**원인**: WKWebView는 `itms-appss:` 커스텀 스킴을 네이티브 앱으로 전달하지 않음. Universal Links도 인앱 브라우저에서 미지원.

### SNS별 대응 방식

| SNS | User-Agent 키워드 | 자동 전환 | 해결책 |
|-----|-------------------|----------|--------|
| **카카오톡** | `kakaotalk` | **가능** | `kakaotalk://web/openExternal?url=...` |
| **인스타그램** | `instagram` | 불가 | "Safari에서 열기" 안내 UI |
| **페이스북** | `FBAN` 또는 `FBAV` | 불가 | "Safari에서 열기" 안내 UI |
| **X (트위터)** | `twitter` | 불가 | "Safari에서 열기" 안내 UI |
| **네이버** | `naver` | 불가 | "Safari에서 열기" 안내 UI |
| **라인** | `line` | 불가 | "Safari에서 열기" 안내 UI |
| **일반 브라우저** | — | 불필요 | 바로 리다이렉트 |

> 한국 시장에서 카카오톡이 압도적 공유 채널이므로, 카카오톡 자동 전환만으로 대부분 커버됨.

### 랜딩 페이지 구현 코드

```javascript
var ua = navigator.userAgent.toLowerCase();
var redeemURL = 'https://apps.apple.com/redeem?ctx=offercodes&id={APP_ID}&code={CODE}';

if (ua.indexOf('kakaotalk') > -1) {
    // 카카오톡: 공식 스킴으로 Safari 자동 전환
    // Safari에서 다시 이 페이지 로드 → 카카오톡 아님 → else로 빠져서 리다이렉트
    location.href = 'kakaotalk://web/openExternal?url='
        + encodeURIComponent(location.href);
} else if (ua.match(/instagram|fban|fbav|twitter|line/i)) {
    // 인스타/페북/X/라인: Safari에서 열기 안내 UI 표시
    showSafariGuide();  // "우측 상단 ... → Safari에서 열기를 눌러주세요" 안내
} else {
    // 일반 브라우저 (Safari, Chrome 등): 바로 리다이렉트
    location.href = redeemURL;
}
```

> **폴백**: `kakaotalk://web/openExternal`이 동작하지 않는 경우(구버전 카카오톡),
> 1~2초 후에도 페이지가 남아있으면 "Safari에서 열기" 버튼을 수동 표시.

> **라이브러리**: [`open-external-browser`](https://github.com/hyesungoh/open-external-browser) npm 패키지가
> 카카오톡/네이버/인스타/라인 등 주요 인앱 브라우저 감지 + 외부 전환 지원.

> **출시 전 반드시 실기기 테스트 필요**: 카카오톡/인스타그램에서 전체 플로우 확인.

# 구현 시 처리할 과제

## B. 테스트 환경 제약

**문제**: Apple 공식 — *"Offer Code redemption is only testable with the app in production."* Sandbox 환경에서 Offer Code 리딤 테스트 불가.

**해결 방향**:
- 서버 로직: mock 데이터로 단위 테스트
- 클라이언트 리딤 감지 (`Transaction.updates`): 일반 구독 구매로 테스트
- 전체 E2E: TestFlight/Production에서 실제 Offer Code로 수행

## C. ASC API 키 관리

**문제**: App Store Connect API 호출에 P8 키(JWT 서명용)가 필요.

**해결 방향**:
- P8 키를 Supabase Vault(Secret Manager)에 저장
- Edge Function에서 환경 변수로 접근
- JWT 토큰 생성은 Edge Function 내부에서 처리
- 초기에는 ASC 웹 UI에서 수동 코드 생성 → CSV 다운로드 → DB INSERT로 시작

## D. 도메인 구매 (구현 전 선행)

**결정**: 커스텀 도메인 구매 (카카오톡 미리보기 브랜딩용)
- 도메인 후보: sweeppic.link, swp.link, sweeppic.app 등 → 구매 시점에 결정
- 호스팅: Supabase Edge Function (추가 비용 없음, DNS 연결만)
- OG 메타태그 설정 필요 (카카오톡/인스타 미리보기 제목/설명/이미지)

## Apple Offer Code URL Redemption

### URL 형식

```
https://apps.apple.com/redeem?ctx=offercodes&id={앱_Apple_ID}&code={코드값}
```

### 앱 미설치 시 동작 (Apple Tech Talk 원문)

> *"If the app isn't already installed, the customer will first be prompted to download the app. Once the app has downloaded, the offer details sheet will appear and the customer can then redeem and subscribe. You'll notice the code input step isn't required here, as the code is part of the URL."*

출처: [Subscription offer codes - Apple Tech Talk](https://developer.apple.com/videos/play/tech-talks/10868/)

### 리딤 감지 (Apple Tech Talk 원문)

> *"When it is completed, it would be another external transaction that your app will need to fulfill when the customer launches your app."*

> *"As noted, redemptions can occur outside your app, so it's important you initialize the TransactionObserver at launch."*

### Transaction에서 제공되는 Offer 정보

| 필드 (StoreKit 2) | 타입 | 설명 |
|-------------------|------|------|
| `transaction.offer?.id` (iOS 17.2+) | `String?` | ASC에서 설정한 Offer Reference Name |
| `transaction.offer?.type` (iOS 17.2+) | `OfferType` | `.code` = Offer Code 리딤 |
| `transaction.offerID` (deprecated) | `String?` | iOS 15~17.1 호환용 |
| `transaction.offerType` (deprecated) | `OfferType` | iOS 15~17.1 호환용 |

| 필드 (Server API) | 타입 | 설명 |
|-------------------|------|------|
| `offerIdentifier` | `string` | Offer Reference Name |
| `offerType` | `int32` | 1=Intro, 2=Promotional, **3=Offer Code**, 4=Win-Back |

> **중요**: `offerIdentifier`는 **오퍼 설정 이름**이지 개별 코드 값(XXXX-XXXX)이 아님.
> 개별 코드 식별이 필요하면 user_id + 시간 기반 매칭으로 해결.

### App Store Server Notifications V2

`OFFER_REDEEMED` 알림 타입이 존재하며, `data.signedTransactionInfo`에서 `offerType=3`, `offerIdentifier`를 확인 가능.

## 클립보드 방식 (v1) 기각 사유

### 웹 브라우저 JS Clipboard API 제약

- `navigator.clipboard.writeText()`는 **사용자 제스처(클릭/탭) 컨텍스트 내에서만** 호출 가능
- 페이지 로드 시 자동 복사 **불가** — `NotAllowedError` 발생
- Safari는 제스처 후 약 1초 이내 + 동기적 실행이어야 함

출처: [Async Clipboard API - WebKit](https://webkit.org/blog/10855/async-clipboard-api/)

### Offer Code URL이 우월한 이유

| 비교 항목 | 클립보드 방식 (v1) | Offer Code URL (v2) |
|----------|------------------|---------------------|
| 클립보드 사용 | 필수 | 불필요 |
| 시스템 팝업 | 붙여넣기 팝업 | 없음 |
| 코드 입력 | 수동 입력 필요 | 자동 pre-fill |
| 소실 위험 | 높음 (다른 복사 시) | 없음 (URL 기반) |
| 앱 미설치 처리 | 복잡 (설치 후 클립보드 읽기) | Apple이 자동 처리 |
| 추적 + 보상 | 이중 관리 (추천코드 + Offer Code) | 랜딩 페이지에서 통합 |
