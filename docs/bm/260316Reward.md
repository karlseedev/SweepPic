> 기존 클립보드 방식(v1)에서 **Offer Code URL Redemption** 방식으로 전환.
> v1 문서: `260316Reward-bak.md`

## 1. 핵심 구조

### 보상 테이블

| 대상 | 보상 | 적용 방식 |
|------|------|----------|
| **피초대자** (신규) | Free Trial 7일 + **14일 프리미엄 무료** (총 3주) | Offer Code URL Redemption |
| **초대자** (기존 구독자) | **14일 구독 연장** | **Promotional Offer** (앱 내 자동 적용) |
| **초대자** (비구독자/만료) | **14일 프리미엄 무료** | 첫 회: Offer Code → 이후: Promotional Offer |

> **보상 통일: 전원 14일(2주), 횟수 제한 없음**
>
> **피초대자용**: Offer Code 2개 (`referral_invited_monthly` / `referral_invited_yearly`)
> - monthly: 비구독/무료/monthly 구독자 대상 (기본)
> - yearly: yearly 구독자 대상 (다운그레이드 방지)
> - 기간: 2주 무료, Introductory Offer 중복: Yes
>
> **초대자용 (구독자/만료)**: Promotional Offer (`referral_extend_monthly` / `referral_extend_yearly`)
> - 앱 내에서 서버 서명 기반 자동 적용 (App Store 리딤 시트 불필요)
> - 현재/이전 구독 상품(monthly/yearly)에 맞는 오퍼 자동 선택
> - Apple 제한 없이 무제한 적용 가능
>
> **초대자용 (한 번도 구독 안 한 비구독자)**: Offer Code (`referral_reward_01`) — 첫 1회만
> - 리딤 → 구독 시작 → 해지/만료 → 이후 Promotional Offer 경로로 전환
>
> **합계: 5개 Offer / 한도 10개 → 5개 여유**

> **Free Trial + Offer Code 중복 적용 가능** (Apple 공식 지원)
> ASC에서 "Introductory Offer 적용 여부 = Yes"로 설정하면,
> Free Trial 7일 → Offer Code 14일 → 정상 가격 순으로 순차 적용됨.

### 보상 기간 결정 근거 (2주 vs 1개월)

| 근거 | 데이터 |
|------|--------|
| 전환율 차이 | 14일 44% vs 30일 46% → **+1.7%p에 불과** |
| 비용 차이 | **2배** |
| 학술 실험 (Frontiers in Psychology, 2025) | 7일이 30일보다 전환율 5.6%↑, 유지율 6.4%↑, 매출 7.9%↑ |
| novelty decay | 사진 정리 앱은 탐색적 앱 → 길수록 관심 소멸 |
| 어뷰징 리스크 | 보상 클수록 저품질 유입 증가 (Marketing Science 논문) |
| 업계 포지션 | Headspace(2주), Noom(2주)과 동일 수준 |
| 프레이밍 | "14일 프리미엄 무료"가 "2주"보다 더 관대하게 느껴짐 (단위 효과) |

> **전략**: 2주로 시작 → 데이터 보고 1개월로 상향 여부 결정
> A/B 테스트: Custom Code 2개(2주용/1개월용) 반반 배분 → ASC 리포트로 전환율 비교

## 2. 초대 프로그램 노출

### 노출 포인트

| 위치 | 대상 | 동기 강도 | UI |
|------|------|----------|-----|
| **무료삭제한도 바 팝업** 하단 | 무료 사용자 | **매우 높음** | 텍스트 + 버튼 |
| **게이트 팝업** 하단 | 무료 사용자 | **매우 높음** | 텍스트 + 버튼 |
| 정리 완료 축하 화면 | 유료 사용자 | 보통 | "친구에게도 알려주세요" + 버튼 |
| 설정 > 프리미엄 메뉴 | 공통 | 낮음 | "친구 초대" / "초대 코드 입력" / "초대 혜택 받기" 3개 항목 |

### 팝업 하단 UI

무료삭제한도 바 팝업과 게이트 팝업 2곳에 동일하게 추가:

```
┌─────────────────────────────┐
│                              │
│  (기존 팝업 내용)             │
│                              │
├──────────────────────────────┤
│  초대 한 번마다 나도 친구도    │
│  14일 프리미엄 제공!          │
│                              │
│  [초대하기]                   │
│                              │
│  이미 구독 중이어도           │
│  14일 무료 연장               │
└──────────────────────────────┘
```

> 무료 사용자가 한도에 부딪히는 순간 = 초대 동기 최고조.
> 이 시점에 "친구 초대하면 14일 프리미엄 무료"를 노출하여 바이럴 유도.

## 3. 전체 플로우

### Phase 1: 초대 링크 생성 (초대자 → 앱)

```
초대자가 "초대하기" 버튼 탭 (팝업 하단 또는 프리미엄 메뉴)
  ↓
초대 설명 화면 표시:
┌──────────────────────────────┐
│                               │
│     친구에게 초대하고          │
│     함께 프리미엄 받기!        │
│                               │
├───────────────────────────────┤
│                               │
│  👤 나                        │
│  초대 1회마다 14일 프리미엄    │
│                               │
│                               │
│  👥 친구                      │
│  14일 프리미엄 무료 체험       │
│                               │
├───────────────────────────────┤
│                               │
│  [초대하기]                   │
│                               │
│  이미 구독 중이어도           │
│  14일 무료 연장               │
│                               │
└──────────────────────────────┘
  ↓
"초대하기" 탭
  ↓
앱 → Supabase: 초대자 user_id로 고유 referral_code 조회 또는 생성
  (코드 형식: x0{6자리영숫자}9j, 예: x0k7m2x99j)
  (생성 시 6자리 내부 코드가 "x0"으로 시작하거나 "9j"로 끝나지 않도록 필터링)
  ↓
초대 링크 생성: sweeppic.link/r/{referral_code}
  ↓
iOS 공유 시트 표시:
  "SweepPic으로 사진 정리하고 14일 프리미엄 무료 받기!

   초대 코드: x0k7m2x99j

   1. 아래 링크로 앱 설치
   2. 앱 설치 후 링크를 한 번 더 누르면 무료혜택 자동 적용!

   적용이 안 되면 이 메시지를 복사해서
   앱 > 설정 > 초대 코드 입력에 붙여넣기 해주세요.

   https://sweeppic.link/r/x0k7m2x99j"
  → 카카오톡, 인스타그램, 메시지 등으로 공유
  ↓
[공유 완료 후 앱 복귀] (completionHandler completed=true)
  ↓
[Push 프리프롬프트 — 1회만 표시, 표시 여부 UserDefaults 기록]
  ├─ 이미 허용 (.authorized) → 초대 설명 화면으로 복귀
  ├─ 프리프롬프트 이미 표시한 적 있음 → 초대 설명 화면으로 복귀
  ├─ 아직 안 물어봄 (.notDetermined) → 프리프롬프트 표시:
  │     "친구가 가입하면 알려드릴까요?" [알림 받기] [닫기]
  │     → [알림 받기] 탭 → 시스템 Push 권한 요청
  │     → 허용 → device token 서버에 저장
  │     → 거부 → 닫기
  │     → [닫기] 탭 → 프리프롬프트 닫힘
  └─ 이전에 거부 (.denied) → 프리프롬프트 표시 (동일 UI):
        → [알림 받기] 탭 → 알림 꺼짐 안내 팝업:
          "알림이 꺼져 있어요. 설정에서 SweepPic 알림을 켜면
           친구 가입 소식을 받을 수 있어요"
          [설정으로 이동] → 앱 설정 페이지 열림
          [나중에] → 닫기
        → [닫기] 탭 → 프리프롬프트 닫힘

[공유 시트 취소] (completionHandler completed=false)
  → 초대 설명 화면 유지 (아무 동작 없음)
```

### Phase 2: 피초대자 링크 클릭 → 앱 설치

```
피초대자가 링크 클릭: sweeppic.link/r/{referral_code}
  ↓
랜딩 페이지 (Supabase Edge Function)
  ├─ [유효성 검증]
  │   ├─ referral_code 유효 → 분석 이벤트(referral.landing_visited) 기록 + App Store 리다이렉트
  │   └─ referral_code 무효 → App Store 앱 페이지로 리다이렉트 (초대 보상 없이)
  │
  └─ 자동 리다이렉트 → App Store 앱 페이지 (리딤 URL 아님)
        ↓
App Store에서 앱 설치 (이 시점에서는 14일 무료 적용 안 됨)
```

> **Offer Code 리딤은 Phase 2에서 하지 않음.**
> 앱 설치만 유도하고, 혜택 적용은 초대 코드 입력(Phase 3) 또는 링크 재탭(Phase 2.5)에서 처리.
> 이렇게 하면 다운로드 중 이탈 → 리딤 시트 안 뜸 문제가 원천적으로 해결됨.

### Phase 2.5: 링크 재탭 → 자동 처리 (앱 설치 후)

```
피초대자가 카톡에서 같은 링크를 한 번 더 탭
  ↓
카톡 인앱 브라우저 → Safari 전환 (kakaotalk://web/openExternal)
  ↓
Safari에서 sweeppic.link/r/{referral_code} 로드
  ↓
[앱 설치됨]
  경로 A: Universal Link 작동 → 앱이 바로 열림 → URL에서 코드 자동 추출
  경로 B: Universal Link 미작동 → 랜딩 페이지 로드
           → Custom URL Scheme(sweeppic://referral/{code})으로 앱 열기 시도
           → 앱이 열림 → 코드 자동 추출
  ↓
앱: 서버에서 상태 확인 (referrals 테이블)
  ├─ status == redeemed → 이미 처리됨 → 무시
  ├─ status == matched → 리딤 미완료 → 리딤 URL 다시 열기
  └─ 레코드 없음 → 새로 처리
        ↓
앱: 자기 초대 확인 (초대자 user_id == 내 user_id?)
  ├─ 자기 자신 → "본인의 초대 코드는 사용할 수 없습니다" → 중단
  └─ 다른 사람 → 계속
  ↓
앱: 초대 코드 자동 매칭 + 구독 상태 확인 + Offer Code 할당 + 리딤 URL 열기
  → 서버: 피초대자 구독 상태에 따라 적절한 Offer Code 할당
    (비구독/무료/monthly → referral_invited_monthly, yearly → referral_invited_yearly)
  → UIApplication.shared.open(redeemURL)
  → App Store 리딤 시트 → "확인" + Face ID → 14일 무료 시작
  → 초대자 보상 트리거
```

> **실기기 테스트 필요**: Safari에서 Universal Link(경로 A)로 앱이 바로 열리는지,
> 랜딩 페이지를 거쳐 Custom URL Scheme(경로 B)으로 열리는지 확인 필요.
> 어떤 경로든 앱은 열리고 코드는 자동 처리됨.
>
> Universal Link 설정: sweeppic.link 도메인에 apple-app-site-association 파일 배포 필요.
> Custom URL Scheme 설정: Info.plist에 `sweeppic://` 스킴 등록 필요.

### Phase 3: 초대 코드 수동 입력 → 혜택 적용 + 보상 트리거

초대 코드 입력 경로 3가지:
1. **온보딩 배너**: 앱 첫 실행 시 안내 (별도 온보딩 설계에서 결합)
2. **메뉴**: 설정 > 프리미엄 > 초대 코드 입력
3. **링크 재탭**: Phase 2.5에서 자동 처리 (수동 입력 불필요)

```
피초대자가 초대 코드 입력 메뉴 진입 (경로 1 또는 2)
  ↓
[서버에서 상태 확인 — 최우선] (referrals 테이블 기준)
  → 서버: 이 user_id가 referred_user_id로 있는 referrals 조회
  ├─ status == redeemed → "이미 초대 코드가 적용되어 있습니다." (끝)
  ├─ status == matched → "혜택이 아직 적용되지 않았어요"
  │                      [혜택 받기] → 이전 할당 코드(만료 시 새 코드)로 리딤 URL 열기 (끝)
  └─ 레코드 없음 → 붙여넣기 화면 표시
        ↓
┌──────────────────────────────┐
│                               │
│  초대 코드 입력               │
│  ┌───────────────────────┐   │
│  │ 붙여넣기              │   │
│  └───────────────────────┘   │
│                               │
│  받은 메시지 내용 전체를       │
│  복사해서 붙여넣어 주세요.     │
│  초대 코드를 자동으로          │
│  식별합니다.                  │
│                               │
└──────────────────────────────┘
  ↓
피초대자가 카톡 메시지 전체/일부/코드만 붙여넣기
  → 앱이 정규식 /x0([a-zA-Z0-9]{6})9j/ 로 코드 추출
  ↓
[자기 초대 확인]
  → 서버에 초대 코드 전송 → 초대자 user_id == 내 user_id?
  ├─ 자기 자신 → "본인의 초대 코드는 사용할 수 없습니다" → 중단
  └─ 다른 사람 → 계속
  ↓
앱: 초대 코드 매칭 + 혜택 적용
  1. 서버: referral_links에서 초대자 식별
  2. 서버: referrals에 referred_user_id = 피초대자 keychain_user_id 기록 (status: matched)
  3. 서버: 피초대자 구독 상태 확인 → 적절한 Offer Code 할당
     (비구독/무료/monthly → referral_invited_monthly, yearly 구독자 → referral_invited_yearly)
     → redeem URL 반환
  4. 앱: UIApplication.shared.open(redeemURL)
     → App Store 리딤 시트 (코드 자동 입력) → "확인" + Face ID
     → 14일 무료 구독 시작
  5. 앱 → 서버: 리딤 완료 보고 → referrals status: matched → redeemed
     → 초대자 보상 트리거
  ↓
→ pending_rewards에 기록 (reward_type/offer_code는 수령 시점에 결정)
→ 서버: 초대자에게 Push 발송
  ├─ device_token 있음 → APNs로 발송: "초대한 사람이 SweepPic에 가입했어요! 14일 무료 혜택을 받으세요"
  └─ device_token 없음 → 패스 (다음 앱 실행 시 인앱 팝업으로 폴백)
→ "초대 코드가 적용되었습니다! 초대 코드는 1회만 입력할 수 있습니다." 표시
```

> **핵심: 초대 코드 입력 = 혜택 적용 + 초대자 매칭 동시 처리.**
> 코드를 입력하지 않으면 혜택도 없고, 초대자 보상도 없음 → 일관성.
> Offer Code 할당은 입력 시점에 수행 → 코드 낭비/회수 로직 불필요.
>
> **리딤 완료 판단은 서버(referrals status)로 수행.**
> currentEntitlements는 구독 만료 시 사라지므로 부정확.
> 서버의 status=redeemed 기록은 영구적이므로 만료 후에도 정확.

### Phase 4: 초대자 보상 수령 (수령 시점 결정)

```
초대자가 앱 진입 (Push 탭 / 콜드 스타트 / 메뉴)
  ↓
[Push 탭으로 진입 시] Push payload의 action_type으로 보상 화면 직행
[콜드 스타트 시] pending_rewards 조회 → 보상 있으면 모달 팝업 자동 표시
[메뉴 진입 시] 설정 > 프리미엄 > "초대 혜택 받기" → 보상 수령 화면 (모달)
  ※ 포그라운드 복귀 시에는 팝업 미표시 (사용 중 방해 방지)
  ※ 팝업을 닫아도 메뉴에서 언제든 수령 가능
  ↓
앱 → Supabase: pending_rewards 조회
  ├─ 보상 없음 → 정상 진행
  └─ 보상 있음 → 인앱 축하 팝업 표시
        "초대한 사람이 SweepPic에 가입했어요! 14일 무료 혜택을 받으세요"
        [보상 받기] 버튼
          ↓
        앱: 현재 구독 상태 확인 (이 순간에 결정)
          ↓
        앱 → 서버: "보상 수령 요청" + 현재 구독 상태
          ↓
        서버: 구독 상태에 따라 보상 방식 결정
          ├─ 구독자 또는 만료 구독자 (이전에 구독한 적 있음):
          │     Promotional Offer 서명 생성 → 앱에 반환
          │     → 앱: Product.purchase(options: [.promotionalOffer(...)])
          │     → 앱 내에서 바로 구독 연장/시작 (App Store 리딤 시트 불필요)
          │     → pending_rewards → completed
          │
          └─ 한 번도 구독한 적 없는 비구독자:
                서버: offer_codes에서 reward_01 코드 할당 → redeem URL 반환
                → 앱: UIApplication.shared.open(redeemURL)
                → App Store 리딤 시트 (코드 자동 입력)
                → "확인" 탭 → 구독 시작
                → 이후 보상부터는 구독자/만료 → Promotional Offer 경로
                → pending_rewards → completed
```

> **수령 시점 결정**: pending_rewards에는 "보상이 있다"만 기록.
> reward_type, offer_code는 수령하는 순간에 결정.
> 비구독자가 첫 보상으로 구독자가 되면, 다음 보상부터 자동으로 Promotional Offer 경로.
> reward_01 하나만 있으면 충분 (이후는 Promotional Offer로 무제한).

## 4. Offer Code 재고 관리

### 코드 풀 운영

```
[자동화 파이프라인]
Supabase Scheduled Function (매일 새벽)
  ↓
offer_name별 잔여 코드 수 확인 (referral_invited_monthly, referral_invited_yearly, referral_reward_01 각각)
  ├─ 충분 (>5,000개) → 패스
  └─ 부족 (<5,000개)
        ↓
      App Store Connect API 호출
        POST /v1/subscriptionOfferCodeOneTimeUseCodes
        → 코드 생성 (최대 25,000개/배치)
        ↓
      GET /v1/subscriptionOfferCodeOneTimeUseCodes/{id}/values
        → CSV로 코드 값 다운로드
        ↓
      offer_codes 테이블에 INSERT
      ├─ 성공 → 완료
      └─ 실패 → 1시간 후 자동 재시도 → 실패 → 3시간 후 → 6시간 후
```

> 초기에는 매일 체크 + 5,000개 임계값으로 충분.
> 사용자 증가로 급속 소진이 발생하면 할당 시점 실시간 체크로 전환.

### 코드 할당

```
[할당 — 초대 코드 입력 또는 링크 재탭 시점에 수행]
서버 API 요청 시:
  offer_codes에서 status='available' AND expires_at > now() AND offer_name={해당 오퍼}
  → 1개 SELECT FOR UPDATE
  → status='assigned', assigned_to={user_id}, assigned_at=now()
  → redeem URL 생성하여 앱에 반환

[만료 코드 정리]
Supabase Scheduled Function (매일):
  expires_at < now() 인 코드
  → status='expired'
```

> 코드를 미리 할당하지 않음 (입력/재탭 시점에 할당) → 48시간 회수 로직 불필요.

### App Store Connect Offer 설정

**총 5개 Offer:**

| Offer Name | 용도 | 유형 | 대상 상품 | 대상 고객 | 비고 |
|-----------|------|------|----------|----------|------|
| `referral_invited_monthly` | 피초대자 (비구독/무료/monthly) | Offer Code (One-time) | plus_monthly | New + Existing + Expired | Intro Offer 중복: Yes |
| `referral_invited_yearly` | 피초대자 (yearly 구독자) | Offer Code (One-time) | plus_yearly | Existing | yearly 다운그레이드 방지 |
| `referral_extend_monthly` | 초대자 (monthly 구독/만료) | **Promotional Offer** | plus_monthly | Existing + Expired | 서버 서명, 무제한 |
| `referral_extend_yearly` | 초대자 (yearly 구독/만료) | **Promotional Offer** | plus_yearly | Existing + Expired | 서버 서명, 무제한 |
| `referral_reward_01` | 초대자 (한 번도 구독 안 한 비구독자) | Offer Code (One-time) | plus_monthly | New | 첫 1회만 |

> 동시 활성 Offer: 5개 / 한도 10개 → **5개 여유** (다른 프로모션용)
> 모든 Offer는 기간: 2주 무료
>
> **기본 상품은 plus_monthly**: 초대으로 유입된 사용자의 진입 장벽을 낮추기 위해 월간 기본.
> 체험 시작률 월간 59.8% vs 연간 54.4%, 체험→연간 강제 시 환불율 4배 (RevenueCat 데이터).
> yearly 구독자만 plus_yearly Offer를 받음 (다운그레이드 방지).
>
> **비구독자 보상이 reward_01 하나로 충분한 이유**:
> 첫 보상으로 reward_01 리딤 → 구독 시작 → 해지/만료 → "만료 구독자"
> → 이후 보상은 Promotional Offer(무제한) 경로로 전환
>
> **월간→연간 업셀 전략 (초대 프로그램 외)**:
> 월간 2~3회 갱신 시점(60~90일)에 연간 업셀 프롬프트 노출.
> 업계 평균 업셀 전환율 8~15%. 초대 유입 사용자는 LTV 16% 높고 이탈률 18% 낮아 업셀 성공률 더 높을 것으로 기대.

### 영구 User ID (Keychain 기반)

```swift
// 앱 최초 실행 시 UUID 생성 → Keychain 저장 (앱 삭제에도 유지)
// 이미 UsageLimitStore에서 동일한 Keychain 패턴 사용 중
let userId = KeychainStore.getOrCreate(key: "sweeppic_referral_id", default: UUID().uuidString)
```

- IDFV는 재설치 시 변경되므로 초대 전용 ID로 부적합
- Keychain은 앱 삭제 후에도 유지됨
- 기존 IDFV는 분석용으로 계속 사용, 초대 전용 ID만 Keychain에 추가

### 피초대자 리딤 완료 보고 (App Store에서 앱 복귀 시)

```swift
// App Store 리딤 시트에서 앱으로 복귀 시 Transaction.updates로 리딤 감지
// → 서버에 리딤 완료 보고 → referrals status: matched → redeemed
func handleReturnFromRedemption() {
    Task {
        for await result in Transaction.updates {
            guard case .verified(let transaction) = result else { continue }

            let offerRefName: String?
            let isOfferCode: Bool
            if #available(iOS 17.2, *) {
                offerRefName = transaction.offer?.id
                isOfferCode = transaction.offer?.type == .code
            } else {
                offerRefName = transaction.offerID
                isOfferCode = transaction.offerType == .code
            }

            if isOfferCode,
               (offerRefName == "referral_invited_monthly" || offerRefName == "referral_invited_yearly") {
                // 서버에 리딤 완료 보고
                await reportRedemptionComplete()
            }
        }
    }
}
```

> **리딤 완료 판단**: 서버의 referrals status로 판단 (currentEntitlements 아님).
> currentEntitlements는 구독 만료 시 사라지므로 부정확.
> Transaction.updates는 리딤 직후 감지용으로만 사용하고, 상태 관리는 서버에서 수행.

### 초대자 보상 적용 코드 (Promotional Offer — 기존 구독자)

```swift
// 서버에서 서명 요청 → Promotional Offer로 앱 내 자동 적용
func applyReferralReward(product: Product, signatureData: SignatureResponse) async throws {
    let offer = Product.PurchaseOption.promotionalOffer(
        offerID: "referral_extend_monthly", // 또는 "referral_extend_yearly" (구독 상품에 따라)
        keyID: signatureData.keyID,
        nonce: signatureData.nonce,
        signature: signatureData.signature,
        timestamp: signatureData.timestamp
    )
    let result = try await product.purchase(options: [offer])
    // 구독 연장 완료 — App Store 리딤 시트 불필요
}
```

## 6. 어뷰징 방지

| 시나리오 | 대책 |
|---------|------|
| **자기 초대** (2대 기기) | 동일 Keychain user_id 감지 (같은 Apple ID면 Keychain 공유) |
| **피초대자 검증** | Offer Code 리딤 완료 = Apple이 결제 수단 검증 완료. 별도 활동 조건 불필요 |
| **중복 리딤 (피초대자)** | Apple이 "1인 1오퍼 1회" 자동 제한 |
| **초대자 보상 제한** | 인위적 월 상한 없음. 초대 많이 하면 신규 유입이 늘어나므로 제한 불필요. 비구독자는 reward_01 1회 후 Promotional Offer 경로 전환 |
| **코드 낭비 방지** | 코드를 미리 할당하지 않음 (입력/재탭 시점에 할당) → 회수 로직 불필요 |
| **잘못된 초대 코드** | 서버에서 referral_links 조회 실패 → "유효하지 않은 코드" 안내 |
| **랜딩 페이지 코드 재고 소진** | 코드 풀 비어있으면 → App Store 앱 페이지로 폴백 (초대 보상 없이 설치 유도) |

## 7. Supabase 테이블 구조

```sql
-- 초대 링크 (초대자당 1개)
CREATE TABLE referral_links (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id TEXT NOT NULL UNIQUE,        -- Keychain 기반 영구 ID
  referral_code TEXT NOT NULL UNIQUE,   -- 형식: x0{6자리}9j (예: x0k7m2x99j), 내부 6자리에 "x0"시작/"9j"끝 제외
  device_token TEXT,                    -- APNs Push token (앱 실행 시 갱신, NULL이면 Push 미허용)
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 초대 기록
CREATE TABLE referrals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  referrer_user_id TEXT NOT NULL,       -- 초대자
  referred_user_id TEXT,                -- 피초대자 (초대 코드 입력 시 매칭)
  offer_code TEXT,                      -- 할당된 Offer Code
  status TEXT DEFAULT 'matched',         -- matched → redeemed → rewarded (clicked 상태 없음, 분석 이벤트로 대체)
  matched_at TIMESTAMPTZ DEFAULT now(), -- 피초대자가 초대 코드 입력한 시점
  redeemed_at TIMESTAMPTZ,             -- 피초대자 Offer Code 리딤 완료 시점
  rewarded_at TIMESTAMPTZ
);

-- 초대자 보상 대기
-- reward_type, offer_code 등은 수령 시점에 결정 (트리거 시점에는 미결정)
CREATE TABLE pending_rewards (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id TEXT NOT NULL,                -- 보상 받을 초대자
  referral_id UUID REFERENCES referrals(id),
  reward_type TEXT,                     -- 수령 시 결정: 'promotional' | 'offer_code'
  offer_code TEXT,                      -- 수령 시 할당: Offer Code (offer_code 타입만)
  redeem_url TEXT,                      -- 수령 시 생성: apps.apple.com/redeem?... (offer_code 타입만)
  status TEXT DEFAULT 'pending',        -- pending → completed → expired
  created_at TIMESTAMPTZ DEFAULT now(),
  completed_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ DEFAULT now() + interval '30 days'  -- 30일 후 만료
);

-- Offer Code 풀
CREATE TABLE offer_codes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code TEXT NOT NULL UNIQUE,            -- Apple이 생성한 실제 코드 값
  offer_name TEXT NOT NULL,             -- ASC Offer Name (referral_invited_monthly, referral_invited_yearly, 또는 referral_reward_01)
  status TEXT DEFAULT 'available',      -- available → assigned → used → reclaimed → expired
  assigned_to TEXT,                     -- 할당된 세션/유저 ID
  assigned_at TIMESTAMPTZ,
  used_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ               -- Apple 코드 만료일 (최대 6개월)
);
```

> offer_codes 테이블의 코드는 피초대자용(`referral_invited_monthly`, `referral_invited_yearly`)과
> 초대자 비구독자용(`referral_reward_01`)을 `offer_name` 컬럼으로 구분.
> 초대자 구독자/만료는 Promotional Offer이므로 코드 풀 불필요.

# 해결된 과제

## A. 인앱 브라우저 호환성 → 해결

**문제**: 인앱 브라우저(WKWebView)에서 `apps.apple.com/redeem` URL이 App Store를 열지 못하고 웹뷰 안에서 로드됨. 302 리다이렉트, `window.location`, `<a>` 태그 모두 동일.

**원인**: WKWebView는 `itms-appss:` 커스텀 스킴을 네이티브 앱으로 전달하지 않음. Universal Links도 인앱 브라우저에서 미지원.

### SNS별 대응 방식

| SNS | User-Agent 키워드 | 자동 전환 | 해결책 |
|-----|-------------------|----------|--------|
| **카카오톡** | `kakaotalk` | **가능** | `kakaotalk://web/openExternal?url=...` |
| **LINE** | `line` | **가능** | URL에 `?openExternalBrowser=1` 파라미터 추가 |
| **인스타그램** | `instagram` | 불확실 | App Store 앱 페이지로 리다이렉트 (열릴 수도 있고, 웹 프리뷰가 뜰 수도 있음) |
| **페이스북** | `FBAN` 또는 `FBAV` | 불확실 | 동일 (웹 프리뷰 시 "App Store에서 보기" 1탭 추가) |
| **X (트위터)** | `twitter` | **안 됨** | 2024.11 이후 WKWebView 전환, App Store 리다이렉트 차단. 웹 프리뷰 → 1탭 추가 |
| **네이버** | `naver` | 불확실 | App Store 앱 페이지로 리다이렉트 시도 |
| **일반 브라우저** | — | 불필요 | 바로 리다이렉트 |

> **핵심**: 카카오톡과 LINE만 프로그래밍적 자동 전환 가능. 나머지 SNS는 App Store 앱 페이지로 리다이렉트하면:
> - 자동으로 App Store가 열리면 → 최선
> - 웹 프리뷰가 뜨면 → "App Store에서 보기" 버튼 1탭으로 App Store 도달
> - 완전히 막히지는 않음. 실기기 테스트로 확인 필요.

### 랜딩 페이지 구현 코드

```javascript
var ua = navigator.userAgent.toLowerCase();
var appStoreURL = 'https://apps.apple.com/app/id{APP_ID}';
var appSchemeURL = 'sweeppic://referral/{referral_code}';

if (ua.indexOf('kakaotalk') > -1) {
    // 카카오톡: 공식 스킴으로 Safari 자동 전환
    location.href = 'kakaotalk://web/openExternal?url='
        + encodeURIComponent(location.href);

} else if (ua.indexOf('line') > -1) {
    // LINE: 공식 외부 브라우저 파라미터
    var separator = location.href.indexOf('?') > -1 ? '&' : '?';
    location.href = location.href + separator + 'openExternalBrowser=1';

} else if (ua.match(/instagram|fban|fbav|twitter|naver/i)) {
    // Instagram/Facebook/X/네이버: App Store 앱 페이지로 리다이렉트 시도
    // 자동으로 열리면 좋고, 안 열리면 웹 프리뷰에서 "App Store에서 보기" 1탭
    location.href = appStoreURL;

} else {
    // 일반 브라우저 (Safari, Chrome 등)
    // 앱 설치됨 → Custom URL Scheme으로 앱 열기 시도 (Phase 2.5)
    // 앱 미설치 → 타임아웃 후 App Store로 리다이렉트 (Phase 2)
    window.location.href = appSchemeURL;
    setTimeout(function() {
        window.location.href = appStoreURL;
    }, 1500);
}
```

> **폴백**: `kakaotalk://web/openExternal`이 동작하지 않는 경우(구버전 카카오톡),
> 1~2초 후에도 페이지가 남아있으면 App Store URL로 리다이렉트.
>
> **글로벌 대응**: Instagram/Facebook/X에서 App Store가 자동으로 안 열리더라도,
> apps.apple.com 웹 프리뷰에 "App Store에서 보기" 버튼이 있어 1탭으로 도달 가능.
> Phase 2.5(링크 재탭)가 안 되는 경우, 카톡 메시지의 폴백 안내로 수동 코드 입력.

> **라이브러리**: [`open-external-browser`](https://github.com/hyesungoh/open-external-browser) npm 패키지가
> 카카오톡/네이버/인스타/라인 등 주요 인앱 브라우저 감지 + 외부 전환 지원.

> **출시 전 반드시 실기기 테스트 필요**: 카카오톡/인스타그램에서 전체 플로우 확인.

# 구현 시 처리할 과제

## B. 테스트 환경 제약

**문제**: Apple 공식 — *"Offer Code redemption is only testable with the app in production."* Sandbox 환경에서 Offer Code 리딤 테스트 불가.

**해결 방향**:
- 서버 로직: mock 데이터로 단위 테스트
- 클라이언트 리딤 감지 (`Transaction.updates`): 일반 구독 구매로 테스트
- 전체 E2E: TestFlight/Production에서 실제 Offer Code로 수행

## C. ASC API 키 관리

**문제**: App Store Connect API 호출에 P8 키(JWT 서명용)가 필요.

**해결 방향**:
- P8 키를 Supabase Vault(Secret Manager)에 저장
- Edge Function에서 환경 변수로 접근
- JWT 토큰 생성은 Edge Function 내부에서 처리
- 초기에는 ASC 웹 UI에서 수동 코드 생성 → CSV 다운로드 → DB INSERT로 시작

## D. 도메인 구매 (구현 전 선행)

**결정**: 커스텀 도메인 구매 (카카오톡 미리보기 브랜딩용)
- 도메인 후보: sweeppic.link, swp.link, sweeppic.app 등 → 구매 시점에 결정
- 호스팅: Supabase Edge Function (추가 비용 없음, DNS 연결만)
- OG 메타태그 설정 필요 (카카오톡/인스타 미리보기 제목/설명/이미지)

## E. APNs Push Notification

- APNs Authentication Key (P8) 생성 필요 — ASC API 키와 별도 (JWT 생성 방식 다름)
- APNs JWT: `iss = team_id`, ASC JWT: `iss = issuer_id`
- P8 키 파일 자체는 공유 가능하지만 JWT 서명 로직은 별도 구현
- device_token은 앱 실행 시마다 갱신하여 서버에 전송 (token 변경 대응)
- Push 탭 시 앱이 보상 화면으로 직행하도록 payload에 `action_type: "referral_reward"` 포함

## F. 랜딩 페이지 OG 태그 디자인 에셋

카톡/인스타 공유 시 미리보기에 표시될 이미지와 문구 준비 필요:
- OG 이미지: "14일 프리미엄 무료" 문구가 포함된 매력적인 이미지
- OG 제목: "SweepPic - 14일 프리미엄 무료 받기"
- OG 설명: "친구가 14일 프리미엄을 선물했어요!"
- 디자인 에셋은 구현 전 사전 제작 필요

## Apple Offer Code URL Redemption

### URL 형식

```
https://apps.apple.com/redeem?ctx=offercodes&id={앱_Apple_ID}&code={코드값}
```

### 앱 미설치 시 동작 (Apple Tech Talk 원문)

> *"If the app isn't already installed, the customer will first be prompted to download the app. Once the app has downloaded, the offer details sheet will appear and the customer can then redeem and subscribe. You'll notice the code input step isn't required here, as the code is part of the URL."*

출처: [Subscription offer codes - Apple Tech Talk](https://developer.apple.com/videos/play/tech-talks/10868/)

### 리딤 감지 (Apple Tech Talk 원문)

> *"When it is completed, it would be another external transaction that your app will need to fulfill when the customer launches your app."*

> *"As noted, redemptions can occur outside your app, so it's important you initialize the TransactionObserver at launch."*

### Transaction에서 제공되는 Offer 정보

| 필드 (StoreKit 2) | 타입 | 설명 |
|-------------------|------|------|
| `transaction.offer?.id` (iOS 17.2+) | `String?` | ASC에서 설정한 Offer Reference Name |
| `transaction.offer?.type` (iOS 17.2+) | `OfferType` | `.code` = Offer Code 리딤 |
| `transaction.offerID` (deprecated) | `String?` | iOS 15~17.1 호환용 |
| `transaction.offerType` (deprecated) | `OfferType` | iOS 15~17.1 호환용 |

| 필드 (Server API) | 타입 | 설명 |
|-------------------|------|------|
| `offerIdentifier` | `string` | Offer Reference Name |
| `offerType` | `int32` | 1=Intro, 2=Promotional, **3=Offer Code**, 4=Win-Back |

> **중요**: `offerIdentifier`는 **오퍼 설정 이름**이지 개별 코드 값(XXXX-XXXX)이 아님.
> 개별 코드 식별이 필요하면 user_id + 시간 기반 매칭으로 해결.

### App Store Server Notifications V2

`OFFER_REDEEMED` 알림 타입이 존재하며, `data.signedTransactionInfo`에서 `offerType=3`, `offerIdentifier`를 확인 가능.

## 클립보드 방식 (v1) 기각 사유

### 웹 브라우저 JS Clipboard API 제약

- `navigator.clipboard.writeText()`는 **사용자 제스처(클릭/탭) 컨텍스트 내에서만** 호출 가능
- 페이지 로드 시 자동 복사 **불가** — `NotAllowedError` 발생
- Safari는 제스처 후 약 1초 이내 + 동기적 실행이어야 함

출처: [Async Clipboard API - WebKit](https://webkit.org/blog/10855/async-clipboard-api/)

### Offer Code URL이 우월한 이유

| 비교 항목 | 클립보드 방식 (v1) | Offer Code URL (v2) |
|----------|------------------|---------------------|
| 클립보드 사용 | 필수 | 불필요 |
| 시스템 팝업 | 붙여넣기 팝업 | 없음 |
| 코드 입력 | 수동 입력 필요 | 자동 pre-fill |
| 소실 위험 | 높음 (다른 복사 시) | 없음 (URL 기반) |
| 앱 미설치 처리 | 복잡 (설치 후 클립보드 읽기) | Apple이 자동 처리 |
| 추적 + 보상 | 이중 관리 (초대코드 + Offer Code) | 랜딩 페이지에서 통합 |
