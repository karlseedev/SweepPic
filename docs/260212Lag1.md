# iOS 26 정리버튼 랙 원인 분석 및 해결 계획

**날짜**: 2026-02-12
**증상**: iOS 26에서 정리버튼을 **처음 1회 클릭 시** UI 랙이 심함 (2회차부터는 발생하지 않음)

## 완료 기준

1. 최초 클릭 후 **0.5초 내** 진행 UI가 반드시 1회 이상 갱신된다
2. 최초 클릭 구간에서 **Main Thread 100ms 이상 stall이 0회**다
3. 최초 1회 총 분석시간이 기존 대비 **최소 30% 감소**한다
4. 1회차/2회차 간 체감 편차(모델 초기화 지연)가 크게 줄어든다

## 근본 원인 구조도

```
GridViewController (UIViewController → @MainActor)
    │
    └─ Task { }  ← MainActor 상속 (문제의 시작점)
         │
         ├─ CleanupPreviewService.analyze()     ← 호출자 executor 상속
         │    ├─ QualityAnalyzer.analyze()       ← MainActor에서 실행
         │    │    ├─ imageLoader.loadImage()       (async, 콜백 기반)
         │    │    ├─ exposureAnalyzer.analyze()    (동기, CPU)
         │    │    ├─ blurAnalyzer.analyze()        (동기, Metal/CPU)
         │    │    └─ detectTextScreenshot()        (동기 블로킹!)
         │    │         └─ VNImageRequestHandler.perform() ← 메인 스레드 차단
         │    │
         │    ├─ CleanupImageLoader.loadImage()  ← 중복 로드
         │    ├─ AestheticsAnalyzer.analyze()    ← ML 추론 (무거움)
         │    └─ detectTextScreenshot()          ← 동기 블로킹 (중복 호출!)
         │
         └─ CleanupService.startCleanup()       ← 같은 문제
              └─ QualityAnalyzer.analyzeBatch()  ← TaskGroup도 MainActor 상속
```

## 핵심 원인: `Task { }`가 MainActor를 상속하여 모든 분석이 메인 스레드에서 실행됨

`GridViewController+Cleanup.swift:353`에서 생성되는 `Task { }`가 문제의 시작점.

```swift
// GridViewController+Cleanup.swift:353
Task {
    let result = try await service.analyze(...)
}
```

`GridViewController`는 `UIViewController`의 하위 클래스이므로 `@MainActor`로 격리됨.
이 안에서 생성된 `Task { }`는 MainActor 컨텍스트를 상속받아, 모든 분석 코드가 메인 스레드에서 실행됨.

### 빌드 설정 주의: `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`

`project.pbxproj`에 Debug/Release 모두 아래 설정이 활성화되어 있음:
```
SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor;
SWIFT_APPROACHABLE_CONCURRENCY = YES;
```

이 설정은 **explicit actor isolation 어노테이션이 없는 모든 선언을 암묵적 `@MainActor`로 취급**함.
`CleanupPreviewService`, `CleanupService`, `QualityAnalyzer`, `SafeGuardChecker` 모두:
- PickPhoto 앱 타깃 소속 (AppCore 패키지 아님)
- explicit actor isolation 어노테이션 없음 (`@MainActor`, `nonisolated` 등 미사용)

따라서 이 클래스들의 모든 메서드가 **암묵적 `@MainActor`**가 됨.
`Task.detached { await service.analyze() }`를 해도 `analyze()`가 `@MainActor`이므로 Swift가 **MainActor로 다시 hop**함.

> **결론**: `Task.detached` 적용 **이전에** 분석 경로의 서비스 메서드에 `nonisolated` 명시가 선행되어야 함

## 처음 1회만 발생하는 이유: Vision 모델 Lazy Initialization

Vision Framework의 ML 모델은 **첫 호출 시에만 로드**되고, 이후 프로세스 메모리에 상주함:

| API | 사용 위치 | 1회차 (모델 로드) | 2회차 이후 |
|-----|----------|-----------------|---------|
| `CalculateImageAestheticsScoresRequest` | AestheticsAnalyzer.swift:73 | ~300-500ms | ~30-50ms |
| `VNRecognizeTextRequest` | CleanupPreviewService.swift:466, QualityAnalyzer.swift:379 | ~200-300ms | ~20-30ms |
| `VNDetectFaceCaptureQualityRequest` | SafeGuardChecker.swift:90 | ~200-300ms | ~20-30ms |

첫 사진 분석 시 모델 로드 비용 합계: **~700-1100ms** (메인 스레드에서 실행되므로 체감 프리즈)
2회차부터는 모델이 메모리에 남아 있어 분석만 수행 → 랙 없음

> **추가 악화 요인**: `detectTextScreenshot()`이 `CleanupPreviewService`와 `QualityAnalyzer`에서 **중복 호출**되어 모델 로드 비용도 중복 발생 가능

## iOS 26에서 특히 심한 이유

**iOS 16~17 경로** (가벼움):
- `QualityAnalyzer.analyze(asset)` → 이미지 로드 + 노출/블러 분석만

**iOS 18+ 경로** (iOS 26 포함, 매우 무거움):
- `QualityAnalyzer.analyze(asset)` → 이미지 로드 + 노출/블러 분석
- `CleanupImageLoader.loadImage(for: asset)` → 두 번째 이미지 로드 (중복!)
- `AestheticsAnalyzer.analyze(image)` → `CalculateImageAestheticsScoresRequest.perform(on:)` (무거운 ML 추론)
- `detectTextScreenshot(image)` → `VNImageRequestHandler.perform([request])` (동기 블로킹!)

사진 1장당 작업량이 3~4배 증가.

## 구체적인 블로킹 지점

### 1. `CleanupPreviewService.swift:484` - detectTextScreenshot 내부

```swift
let handler = VNImageRequestHandler(cgImage: image, options: [:])
try handler.perform([request])  // ← 동기 함수, 메인 스레드 완전 블로킹
```

`withCheckedContinuation` 안이지만, continuation 클로저 자체가 MainActor 스레드에서 동기 실행되므로 `perform()`이 메인 스레드를 차단.

### 2. `AestheticsAnalyzer.swift:77` - AestheticsScore 분석

```swift
let observation = try await request.perform(on: image)  // ← async지만 MainActor에서 resume
```

`async`이므로 완전 블로킹은 아니지만, resume이 MainActor에서 일어나 다른 UI 작업을 지연시킴.

### 3. `QualityAnalyzer.swift:429` - detectTextScreenshot (중복)

QualityAnalyzer 내부에도 동일한 `detectTextScreenshot`이 있어 동기 블로킹 발생.

### 4. `SafeGuardChecker.swift:93` - checkFaceQuality

```swift
let handler = VNImageRequestHandler(cgImage: image, options: [:])
try handler.perform([request])  // 동기 호출
```

### 추가 악화 요인

- `CleanupPreviewService.swift:212` - iOS 18+ 경로에서 `CleanupImageLoader.loadImage()`를 한 번 더 호출 (QualityAnalyzer가 이미 로드했는데 또 로드)
- 50장 루프 (`debugLightLimit`) 동안 누적 → 각 사진당 수백ms x 50장 = 수초~수십초 메인 스레드 점유
- `progressView.update(with: progress)` UI 업데이트도 분석이 메인 스레드를 차지하고 있어 프로그레스 UI도 갱신되지 않음 (화면 프리즈)

### 같은 문제가 있는 다른 코드

- `startCleanup` (`GridViewController+Cleanup.swift:172`)도 동일한 패턴의 `Task { }` 사용
- DEBUG 전용 `startIntegratedLogicTest` (`GridViewController+Cleanup.swift:444`)
- DEBUG 전용 `startModeComparisonTest` (`GridViewController+Cleanup.swift:540`)

## 해결 방향 비교

### 방향 A: 호출부에서 `Task.detached` 사용

```swift
// GridViewController+Cleanup.swift
Task.detached(priority: .userInitiated) {
    let result = try await service.analyze(...)
    await MainActor.run { /* UI 업데이트 */ }
}
```

| 항목 | 평가 |
|------|------|
| 수정 범위 | 호출부 3~4곳만 변경 |
| 즉효성 | **단독으로는 효과 없음** — `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` 때문에 호출 대상이 암묵적 `@MainActor`이면 다시 MainActor hop 발생. 서비스 메서드 `nonisolated` 전환이 선행되어야 함 |
| 근본적인가 | **아님** — 새로운 호출부에서 같은 실수 반복 가능 |
| 리스크 | `Task.detached`는 task-local values, priority 상속 안 됨 |

### 방향 B: 서비스 내부에서 백그라운드 보장 (자기 보호)

```swift
// detectTextScreenshot 내부
private func detectTextScreenshot(_ image: CGImage) async -> Bool {
    return await withCheckedContinuation { continuation in
        DispatchQueue.global(qos: .userInitiated).async {  // ← 항상 백그라운드
            let handler = VNImageRequestHandler(cgImage: image, options: [:])
            try handler.perform([request])
            ...
        }
    }
}
```

| 항목 | 평가 |
|------|------|
| 수정 범위 | 동기 블로킹 API 호출부 3~4곳 |
| 즉효성 | 동기 블로킹만 해결, async 작업(AestheticsScore)은 여전히 MainActor에서 resume |
| 근본적인가 | **부분적** — 블로킹은 해결하지만 전체 파이프라인이 MainActor를 점유하는 문제는 남음 |
| 리스크 | 낮음, 기존 동작 변경 최소 |

### 방향 C: A + B 조합 (2계층 방어) — 권장

**1계층** — 호출부에서 MainActor 분리:
```swift
// GridViewController 호출부
Task.detached(priority: .userInitiated) { ... }
```

**2계층** — 서비스 내부 자기 보호 (`VisionExecutor` 공통 유틸):
```swift
// VisionExecutor.swift (신규) — 동기 Vision 호출을 항상 백그라운드에서 실행
struct VisionExecutor {
    static func performTextRecognition(on image: CGImage) async -> [VNRecognizedTextObservation] { ... }
    static func performFaceQuality(on image: CGImage) async throws -> [VNFaceObservation] { ... }
    // 내부: DispatchQueue.global(qos: .userInitiated) + withCheckedContinuation
}
```

| 항목 | 평가 |
|------|------|
| 수정 범위 | **0계층(선행)** 서비스 nonisolated 전환 + 호출부 3~4곳 + 서비스 내부 3~4곳 + UI 콜백 보강 |
| 즉효성 | **0계층 전환 포함 시** 완전 해결. 0계층 없이는 1계층 `Task.detached`가 무력화됨 |
| 근본적인가 | **그렇다** — 호출부가 실수해도 서비스가 스스로를 보호 |
| 리스크 | 중간 — `nonisolated` 전환 시 가변 상태 동시 접근 보호(NSLock 등) 점검 필요 |

## 권장: 방향 C (2계층 방어)

**이유:**

0. **0계층 (선행 — 서비스 `nonisolated` 전환)**: `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` 설정 때문에, `nonisolated`를 명시하지 않으면 서비스 메서드가 암묵적 `@MainActor`가 되어 `Task.detached`에서 호출해도 MainActor로 다시 hop함. **이 단계가 없으면 1계층이 무력화됨**

1. **1계층 (호출부 `Task.detached`)**: 0계층 전환 후, 분석 파이프라인 전체가 MainActor에서 벗어남. `async` 메서드들의 resume도 cooperative pool에서 일어나므로 메인 스레드 점유 자체가 사라짐

2. **2계층 (서비스 내부 `DispatchQueue.global()`)**: 미래에 누군가 `Task { }` (MainActor 상속)로 호출하더라도 동기 블로킹 API가 메인 스레드를 차단하지 않음. **방어적 프로그래밍**

## 구현 순서 (의존성 기반 고정)

1. **공통 유틸 추가**: `VisionExecutor.swift`, `CleanupWarmupManager.swift` 신규 작성
2. **Vision 동기 호출 교체**: 3개 지점의 `handler.perform()`을 `VisionExecutor`로 교체
3. **서비스 nonisolated 전환**: 분석기/서비스의 entry 메서드에 `nonisolated` 명시 + 가변 상태 lock 보강
4. **호출부 전환**: Grid 호출부를 `Task.detached` + `await MainActor.run { }` UI 업데이트로 교체
5. **워밍업 연결**: `startPreviewCleanup`/`startCleanup` 시작 직전에 `CleanupWarmupManager.warmUpIfNeeded()` 호출
6. **중복 제거**: 이미지 재사용 + 텍스트 감지 결과 공유

| 리스크 | 대응 |
|--------|------|
| `nonisolated` 전환 중 컴파일 격리 오류 | 상태 접근을 lock/actor로 분리하고, UI 참조를 entry 바깥으로 이동 |
| 워밍업이 배터리/초기 부하 증가 | 1회 실행 + low priority + 앱 활성 상태에서만 수행 |
| 중복 제거로 판정 결과가 달라질 수 있음 | 기존/신규 결과 샘플 500장 비교 로그로 오차 검증 |

## 주의사항

- **`SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` 빌드 설정**: 서비스 메서드에 `nonisolated`를 명시하지 않으면 `Task.detached`가 무력화됨. 0계층 전환이 반드시 선행되어야 함
- `nonisolated` 전환 시 가변 상태 동시 접근 보호 점검 필요: `CleanupService`의 `currentSession`, `foundAssetIDs` 및 `QualityAnalyzer`의 `mode` 등이 기존 `NSLock` 범위로 충분히 보호되는지 확인
- `Task.detached` 내부에서 UI 업데이트는 반드시 `await MainActor.run { }`으로 감싸야 함 (`progressView.update(with:)`, 결과 표시 등)
- **Vision 모델 워밍업**: 0계층에서 `nonisolated` + `Task.detached`가 적용되면 모델 로드가 백그라운드에서 일어나므로, 별도 워밍업 없이도 메인 스레드 프리즈는 해소됨. 다만 "첫 클릭 시 분석 속도 자체가 느린" 체감을 줄이려면 워밍업으로 추가 개선 가능
