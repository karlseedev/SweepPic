# L1 일상 스크롤 Hitch 개선 계획 (2025-12-23)

## 문제 정의

### 목표
- **1차 목표**: Warning (< 10 ms/s)
- **최종 목표**: Good (< 5 ms/s)
- **기준 시나리오**: 일상 스크롤 (사진 찾는 정도, 15~25초 연속)

### Apple 공식 기준 (WWDC 2020)

| 등급 | 임계값 | 사용자 체감 |
|------|--------|-------------|
| Good | < 5 ms/s | 거의 인지 못함 |
| Warning | 5-10 ms/s | 인지하기 시작함 |
| Critical | > 10 ms/s | 크게 불편함 |

### 현재 상태 (P0 적용 전)

| 지표 | 수치 | 판정 |
|------|------|------|
| L1 hitch | 25.4~43.1 ms/s | Critical (목표의 2.5~4.3배) |
| dropped | 31~79 | 체감 버벅임 |
| preheat | 54~86회, 207~423개 에셋 | 스크롤 중 과도한 호출 |

## 근본 원인 분석

### 코드 분석 결과

**1차 용의자: prefetchItemsAt에서 스크롤 중 preheat** 🔴

```swift
// GridViewController.swift line 1254
imagePipeline.preheat(assetIDs: assetIDs, ...)

// ImagePipeline.swift - 내부에서 동기 호출!
public func preheat(assetIDs: [String], targetSize: CGSize) {
    let assets = assetIDs.compactMap { fetchAsset(for: $0) }  // 동기!
}

private func fetchAsset(for assetID: String) -> PHAsset? {
    // 캐시 미스 시 메인 스레드에서 동기 호출!
    let result = PHAsset.fetchAssets(withLocalIdentifiers: [assetID], options: nil)
}
```

**문제점:**
- prefetchItemsAt → preheat(assetIDs:) → fetchAsset() → PHAsset.fetchAssets() 동기 호출
- **메인 스레드에서** 실행되어 hitch 유발
- 스크롤 중 54~86회 호출 (200ms 스로틀에도 불구)

**기타 용의자:**

| 원인 | 영향도 | 비고 |
|------|--------|------|
| 콜백 폭발 (~156/s) | 중간 | 기여 요인 후보 |
| statsLock 경합 | 낮음 | P0 후 필요시 검토 |
| memMiss 100% | 구조적 | 요청 수 증가의 기반 |

## 개선 계획 (저위험 순)

### 계획 개요

| 순서 | 작업 | 설명 | 상태 |
|------|------|------|------|
| P0 | prefetchItemsAt preheat 완전 제거 | 동기 fetchAsset 비용 제거 | ✅ 완료 (f4dbefa) |
| P1 | preheatAfterScrollStop 검토 | assetIDs 경로 사용 여부 확인 | ✅ 이미 올바름 |
| P2 | degraded 처리 정책 실험 | 콜백 수 감소 (필요시) | ⏳ 대기 |

### P0: prefetchItemsAt preheat 완전 제거

**문제:**
- `prefetchItemsAt` → `preheat(assetIDs:)` → `fetchAsset()` → `PHAsset.fetchAssets()` 동기 호출
- 메인 스레드에서 Photos DB 쿼리 발생 → hitch 유발
- 스크롤 중 54~86회 호출됨

**해결:**
- `prefetchItemsAt`에서 preheat 호출 완전 제거 (early return)
- `preheatAfterScrollStop()`이 스크롤 정지 후 preheat 담당
- 회색 썸네일 문제 없음 (opportunistic + disk cache skip 조합)

**효과:** hitch 50~65% 개선 (43 → 14~16 ms/s)

### P1: preheatAfterScrollStop 검토

**목표:** `preheat(assetIDs:)` 경로 사용 여부 확인

**분석 결과:**
```swift
// preheatAfterScrollStop() - 이미 올바른 구조
let assets = extendedIndexPaths.compactMap { dataSourceDriver.asset(at: $0) }
// ↑ fetchResult.object(at:) = 인메모리 조회 (빠름)

DispatchQueue.global(qos: .userInitiated).async {
    ImagePipeline.shared.preheatAssets(assets, targetSize: targetSize)
    // ↑ 직접 호출 (문제의 preheat(assetIDs:) 경로 사용 안 함)
}
```

**비교:**
| 경로 | 호출 | 동작 |
|------|------|------|
| 문제의 경로 | `preheat(assetIDs:)` → `fetchAsset()` → `PHAsset.fetchAssets()` | **DB 쿼리** (느림) |
| preheatAfterScrollStop | `fetchResult.object(at:)` → `preheatAssets()` | **인메모리 조회** (빠름) |

**결론:** 이미 올바른 구조. 수정 불필요.

### P2: degraded 처리 정책 실험

**가설:** 스크롤 중 degraded/final 콜백이 과도하게 발생하여 hitch 기여

**실험 후보:**
1. 스크롤 중 final 적용 지연 (pendingFinalImage 저장 후 정지 시 적용)
2. degraded 콜백 무시 (final만 적용) - 권장하지 않음

---

### P2 Step1 실험 결과 (final 완전 무시)

**적용 내용:**
```swift
if !isDegraded {
    // final은 무시 (degraded만 적용)
} else {
    self.imageView.image = image  // degraded만 적용
}
```

**테스트 결과:**

| 테스트 | 스크롤 속도 | L1 hitch | 비고 |
|--------|------------|----------|------|
| 1차 | 53.9 req/s | 11.0 ms/s | 개선처럼 보임 |
| 2차 | 84.6 req/s | 22.4 ms/s | 오히려 악화 |

**부작용:**
- 썸네일 흐림 고정 (final 무시로 degraded만 표시)
- 스크롤 정지 후에도 선명도 복구 안 됨

**결론:**
- 효과 불확실 (결과 일관되지 않음)
- 품질 손해 큼 → **기각**
- final 적용이 hitch 주원인 아닐 가능성

---

### P2-1: final 지연 적용 (미구현)

**Step1과의 차이:**
- Step1: final 완전 무시 → 항상 흐림
- P2-1: final 저장 후 정지 시 적용 → 정지 후 선명

**구현 방향 (참고용):**
1. PhotoCell에 `pendingFinalImage` 추가
2. 스크롤 중 final → pending에 저장만
3. 스크롤 정지 시 visible 셀에 pending 적용

**상태:** Step1 효과 불확실로 P2-1 구현 보류

## 회색 셀 측정 기준

### 측정 방법

회색 셀(이미지 로드 전 빈 셀)을 객관적으로 측정하기 위한 카운터를 구현함.

**grayShown (화면에 회색 노출):**
- `willDisplay:forItemAt:`에서 `cell.imageView.image == nil`일 때 증가
- 의미: "화면에 회색이 실제로 노출된 순간" 카운트
- 한계: 1프레임만 스쳐도 카운트됨 (체감보다 클 수 있음)

**grayResolved (회색 해소):**
- `imageView.image` 할당 시, nil → non-nil 전환일 때만 증가
- 의미: 회색 상태에서 이미지가 로드된 횟수
- opportunistic의 degraded→final 2회 세팅에서도 1회만 카운트

**pending (미해소 회색):**
- `pending = grayShown - grayResolved`
- `pending > 0`: 회색 셀 문제 (해소되지 않음)
- `pending < 0`: offscreen에서 미리 로드 완료 (정상 동작)
- `pending ≈ 0`: 균형 상태

### 코드 위치

```swift
// PhotoCell.swift
static func incrementGrayShown()      // willDisplay에서 호출
static func incrementGrayResolved()   // nil→non-nil 전환 시 호출
static func getGrayCellStats() -> (shown: Int, resolved: Int)
static func logGrayCellStats(label: String)
static func resetGrayCellStats()

var isShowingGray: Bool { imageView.image == nil }

// GridViewController.swift (willDisplay)
if let photoCell = cell as? PhotoCell, photoCell.isShowingGray {
    PhotoCell.incrementGrayShown()
}
```

**핵심 발견:**
- P0 효과 재확인: hitch 63% 개선 (40.7 → 15.0)
- **회색 셀은 차이 없음** → preheat 제거해도 회색 안 늘어남
- preheat가 hitch의 주요 원인이었음 확정

### 분석

**Hitch:**
- L1 일상 스크롤: 15.0 ms/s → 목표(< 10)까지 5 ms/s 감소 필요
- L2 극한 스크롤: 29.8 ms/s → 극한이라 높은 게 정상

**회색 셀:**
- L1 일상 스크롤: grayShown = 90 → **회색 거의 안 보임** (양호)
- L2 극한 스크롤: grayShown = 7038 → 회색 많이 보임 (예상대로)
- pending 음수: offscreen에서 미리 로드 완료 (정상 동작)

**결론:**
- P0 적용 상태에서 **일상 스크롤은 회색 문제 없음**
- 극한 스크롤에서 회색이 보이는 건 정상 (어떤 앱이든 극한에선 발생)
- hitch 목표 미달 → P2 진행 필요
