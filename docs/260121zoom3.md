# 핀치줌/회전 모션 요구사항 정의

**작성일**: 2026-01-21
**상태**: 요구사항 정의
**버전**: v4 (v3 + fadeIn대상 명확화/클램프/레이아웃순서)

---

## 1. 핵심 개념

### 1.1 열 수 기준점
- **1열, 3열, 5열** 세 가지 기준점만 존재
- 핀치줌/회전 완료 시 반드시 이 중 하나로 snap

### 1.2 스케일 → 열 수 관계
```
scale 1.0 = 현재 열 수 유지
scale > 1.0 = 확대 → 열 수 감소 (3열 → 1열)
scale < 1.0 = 축소 → 열 수 증가 (3열 → 5열)
```

### 1.3 진행도 개념
```
0% = 현재 열 수 상태
100% = 목표 열 수 상태

예: 3열 → 1열 핀치
  0% = 3열 레이아웃
  50% = 중간 상태
  100% = 1열 레이아웃
```

### 1.4 앵커 셀
- **핀치**: 핀치 중심점에 있는 셀
- **회전**: 화면 중앙에 있는 셀
- 앵커 셀은 fadeIn 제외, 기존 이미지 유지

### 1.5 fadeIn
- **대상**: 기존에 visible이었던 셀 중 이미지 교체가 필요한 셀 (앵커 제외)
- **제외**: 새로 나타난 셀 (즉시 표시, fadeIn 금지)
- 현재 이미지 → 목표 레이아웃의 이미지로 부드럽게 전환
- 새 단계 시작 시 트리거

> **핵심 구분**: 축소 시 외곽에 새로 등장하는 셀은 "기존에 없던 셀"이므로
> fadeIn이 아니라 즉시 이미지가 표시되어야 함.

### 1.6 열 수 변경 규칙

#### 핀치줌
- 확대: 열 수 감소 (5 → 3 → 1)
- 축소: 열 수 증가 (1 → 3 → 5)

#### 회전
- 세로 → 가로: 1→3, 3→5, 5→5
- 가로 → 세로: 1→1, 3→3, 5→3

## 2. 요구사항

### 2.1 핵심 원칙

#### 금지 사항
- 셀 개별 이동 애니메이션 ❌
- 새로고침/끊김 ❌

#### 필수 사항
- 전체가 하나처럼 연속적인 모션 ✅
- fadeIn으로 부드러운 이미지 전환 ✅
- 핀치 진행도에 따라 실제 레이아웃이 점진적으로 변화 ✅

### 2.2 핀치줌

#### 핀치 시작 시
1. 앵커 셀 결정 (핀치 중심점의 셀)
2. **방향 결정 후** 나머지 모든 셀에 목표 이미지 fadeIn 시작
3. 레이아웃이 점진적으로 변하기 시작

> **주의**: `.began` 시점에서는 `gesture.scale`이 항상 1.0이므로 방향을 알 수 없음.
> fadeIn은 `.changed`의 첫 프레임에서 방향이 결정된 후 트리거해야 함.

#### 핀치 중
1. 핀치 진행도에 따라 셀 크기/위치가 **실시간으로** 변함
2. fadeIn 진행 중 (또는 이미 완료)
3. 외곽에 새 셀들이 나타남 (축소 시)
4. 앵커 셀: 기존 이미지 유지

#### 다단계 핀치 (예: 5열 → 3열 → 1열)
- 각 기준점(1/3/5열) 통과 시 = 새 단계 시작
- 새 단계 시작 시 fadeIn 트리거 (다음 목표 이미지로)

```
[1단계 핀치] 3열 → 1열
  방향 결정 → fadeIn (1열 이미지로)

[2단계 핀치] 5열 → 3열 → 1열
  방향 결정 → fadeIn (3열 이미지로)
  3열 기준점 통과 → fadeIn (1열 이미지로)
```

#### 손을 뗄 때
1. 핀치 100% 완료 상태 → 그대로 유지 (추가 애니메이션 없음)
2. 핀치 70% 상태 → 나머지 30%가 애니메이션으로 자연스럽게 완료
3. **끊김 없이** 현재 모션에서 최종 레이아웃으로 이어짐

### 2.3 회전

#### 핀치와 동일한 원리

| | 핀치 | 회전 |
|---|------|------|
| 앵커 | 핀치 중심점 셀 | 화면 중앙 셀 |
| fadeIn 시작 | 방향 결정 시 | 회전 시작 시 |
| 레이아웃 변화 | 핀치 진행도에 따라 점진적 | 회전 애니메이션과 동기화 |
| 완료 시 | 끊김 없이 snap | 끊김 없이 완료 |

#### 회전 시작 시
1. 앵커 셀 결정 (화면 중앙 셀)
2. **즉시** 나머지 모든 셀에 목표 이미지 fadeIn 시작
3. 회전 애니메이션과 함께 레이아웃이 자연스럽게 변화

#### 회전 완료 시
- 끊김/새로고침 없이 최종 레이아웃 상태
- 앵커 셀이 화면에 유지됨

### 2.4 기준점 판단 로직 (손을 뗄 때)

#### Case 1: 단방향 핀치 (10% 규칙)

한 방향으로만 핀치한 경우:

```
3열에서 축소 10% 이상 진행 → 손을 뗌
→ 의도 명확 → 5열로 완료
```

**10% 이상 진행하면 무조건 그 방향으로 진행**

#### Case 2: 방향 전환 (50% 규칙)

핀치 도중 방향이 바뀐 경우:

```
3열에서 축소 70% → 다시 확대 40%
→ 최종 위치: 30% (축소 방향)
→ 50% 미만 → 3열 유지

3열에서 축소 80% → 다시 확대 20%
→ 최종 위치: 60% (축소 방향)
→ 50% 이상 → 5열로 완료
```

**방향 전환 시 최종 위치 기준 50%로 판단**

#### 정리

| 상황 | 기준 | 결과 |
|-----|------|------|
| 단방향 10% 이상 | 10% | 그 방향으로 진행 |
| 방향 전환 | 50% | 최종 위치 기준 판단 |

## 3. 구현 방안

### 3.1 핵심 전략 (결론)
**연속 열 수(실수) 기반 커스텀 `UICollectionViewLayout`**으로 핀치 진행도에 따라
레이아웃을 실시간 재계산하고, 기준점 통과 시 fadeIn을 트리거한다.

핵심 이유:
- "점진적 레이아웃 변화" 요구를 만족하려면 열 수가 실수로 변해야 함
- 셀 개별 이동 애니메이션 없이 레이아웃 재계산만으로 연속 모션 제공 가능
- 다단계(5→3→1) 전환을 끊김 없이 처리 가능

---

### 3.2 아키텍처 구성

1) **ContinuousGridLayout (커스텀 레이아웃)**
- `virtualColumns: CGFloat`(예: 3.0 → 2.6 → 1.2)
- 값 변경 시 `invalidateLayout()`로 즉시 반영

2) **PinchController (제스처/상태 관리)**
- 핀치 진행도 → `virtualColumns` 변환
- 기준점 통과 감지 → fadeIn 트리거
- 앵커 셀 고정용 contentOffset 보정

3) **Cell FadeIn 레이어**
- 앵커 셀 제외 visible 셀에 목표 이미지 fadeIn
- 비동기 로딩 경쟁 조건 방지 (token/assetID 검증)

---

### 3.3 레이아웃 계산 원리

**핵심 아이디어**: 열 수를 정수로 끊지 말고 실수로 유지.

```swift
columnWidth = (bounds.width - spacing * (virtualColumns - 1)) / virtualColumns
```

- `virtualColumns`가 3.0 → 2.4 → 1.8처럼 연속적으로 변하며
  셀 크기/간격이 매 프레임 부드럽게 변화한다.
- 셀 배치는 "정수 열 수"가 아니라 **실수 열 수로 계산된 폭**에 맞춰
  `x = colIndex * (columnWidth + spacing)` 방식으로 배치한다.

---

### 3.4 앵커 셀 고정 (contentOffset 보정)

**목표**: 핀치 중심점 셀이 화면에서 고정된 것처럼 보이도록 유지.

```swift
// anchorPointInView: 핀치 시작 시 뷰 좌표 (bounds 기준)
// anchorCenterInContent: 레이아웃 갱신 후 앵커 셀의 center (content 좌표)
contentOffset = anchorCenterInContent - anchorPointInView
```

- `layer.anchorPoint` 변경 없이도 고정 가능
- 스크롤 가능한 범위로 clamp 필요

---

### 3.5 기준점 통과 감지 (다단계)

빠른 핀치로 5→1 이동 시 기준점 5,3,1 모두 통과 가능.
따라서 **통과한 기준점을 배열로 반환**하고 방향 순서대로 처리한다.

```swift
if cur < prev {
    crossed = thresholds.filter { $0 <= prev && $0 >= cur }.sorted(by: >)
} else {
    crossed = thresholds.filter { $0 >= prev && $0 <= cur }.sorted(by: <)
}
```

- 반환된 순서대로 fadeIn 트리거
- 동일 단계 중복 방지 (`currentTargetColumns` 체크)

---

### 3.6 fadeIn 트리거 규칙

**대상 구분**:
- **fadeIn 대상**: 트리거 시점에 이미 visible이었던 셀 (앵커 제외)
- **즉시 표시**: 핀치 중 새로 visible 되는 셀 (fadeIn 금지)

**트리거 시점**:
- **방향 결정 시**: 첫 목표 열 수 이미지로 fadeIn 시작 (`.changed` 첫 프레임)
- **기준점 통과 시**: 다음 단계 목표 이미지로 fadeIn 재시작

**안전 규칙**:
- **앵커 셀 제외**
- **비동기 로딩 검증**: 요청 시점의 열 수/assetID/token 확인 후 적용

---

### 3.7 손을 뗄 때 스냅 로직 (10% / 50% 규칙)

**단방향**: 10% 이상 진행 → 해당 방향으로 완료
**방향 전환**: 최종 위치 50% 이상 → 해당 방향으로 완료

완료 애니메이션은 `virtualColumns`를 목표 기준점까지 보간하며,
동일한 레이아웃 계산을 유지해 끊김을 방지한다.

---

### 3.8 회전 처리

회전도 동일 원리로 `virtualColumns`를 보간:
- 회전 시작 시 fadeIn 트리거
- `animate(alongsideTransition:)`에서 `virtualColumns` 즉시 변경
- 완료 후 앵커 복구 + 고해상도 재요청

---

### 3.9 후보 방안 비교 (참고)
1. `startInteractiveTransition`
   - 다단계 전환에서 끊김 가능성 높음
2. 스냅샷 오버레이
   - 점진적 레이아웃 변화 요구에 부적합
3. transform 방식 (폐기)
   - 축소 시 외곽 검정, 손 떼면 끊김, fadeIn 무효화

### 4.2 핀치 제스처 의사코드

```swift
// ZoomDirection enum
enum ZoomDirection {
    case zoomIn   // 확대 (열 수 감소)
    case zoomOut  // 축소 (열 수 증가)
}

// PinchState
struct PinchZoomState {
    var isActive = false
    var baseColumns: CGFloat = 3.0
    var previousVirtualColumns: CGFloat = 3.0
    var currentTargetColumns: GridColumnCount?
    var initialDirection: ZoomDirection?
    var didReverse = false
    var anchorAssetID: String?
    var anchorPointInView: CGPoint = .zero
    var hasFiredInitialFadeIn = false  // [수정] 첫 fadeIn 트리거 여부
}

@objc func handlePinch(_ gesture: UIPinchGestureRecognizer) {
    switch gesture.state {
    case .began:
        pinchState = PinchZoomState()
        pinchState.isActive = true
        pinchState.baseColumns = CGFloat(currentGridColumnCount.rawValue)
        pinchState.previousVirtualColumns = pinchState.baseColumns

        // 앵커 결정 (뷰 좌표)
        let location = gesture.location(in: collectionView)
        pinchState.anchorPointInView = location
        pinchState.anchorAssetID = resolveAnchorAssetID(at: location)

        // [수정] .began에서는 fadeIn 트리거하지 않음
        // gesture.scale이 항상 1.0이므로 방향을 알 수 없음

    case .changed:
        guard pinchState.isActive else { return }

        let scale = gesture.scale
        // [v4] virtualColumns 클램프 (극단 스케일 방지)
        let rawVirtualColumns = pinchState.baseColumns / scale
        let virtualColumns = min(max(rawVirtualColumns, 0.8), 5.2)  // 0.8~5.2 범위

        layout.virtualColumns = virtualColumns
        // [v4] 레이아웃 즉시 반영 후 앵커 보정 (순서 중요!)
        layout.invalidateLayout()
        collectionView.layoutIfNeeded()

        // 앵커 고정 (새 레이아웃 기준으로 보정)
        updateContentOffsetForAnchor(anchorAssetID: pinchState.anchorAssetID,
                                     anchorPointInView: pinchState.anchorPointInView)

        // [수정] 첫 프레임에서 방향 결정 후 fadeIn 트리거
        if !pinchState.hasFiredInitialFadeIn {
            // scale이 1.0이 아닐 때만 방향 결정 가능
            if abs(scale - 1.0) > 0.01 {
                pinchState.initialDirection = (scale > 1.0) ? .zoomIn : .zoomOut
                let initialTarget = inferTargetColumns(fromDirection: pinchState.initialDirection!,
                                                       baseColumns: currentGridColumnCount)
                pinchState.currentTargetColumns = initialTarget
                triggerFadeIn(target: initialTarget, anchorAssetID: pinchState.anchorAssetID)
                pinchState.hasFiredInitialFadeIn = true
            }
        }

        // 방향/전환 여부 추적
        updateDirectionState(virtualColumns: virtualColumns)

        // 기준점 통과 감지 (다단계)
        let crossed = detectCrossedThresholds(prev: pinchState.previousVirtualColumns,
                                             cur: virtualColumns,
                                             thresholds: [1, 3, 5])
        for threshold in crossed {
            if let target = GridColumnCount(rawValue: Int(threshold)) {
                if pinchState.currentTargetColumns != target {
                    pinchState.currentTargetColumns = target
                    triggerFadeIn(target: target, anchorAssetID: pinchState.anchorAssetID)
                }
            }
        }

        pinchState.previousVirtualColumns = virtualColumns

    case .ended, .cancelled:
        guard pinchState.isActive else { return }

        let finalTarget = decideSnapTarget(using: pinchState,
                                           currentVirtualColumns: layout.virtualColumns)
        animateToTargetColumns(finalTarget) {
            currentGridColumnCount = finalTarget
            didPerformZoom(to: finalTarget)
        }

        pinchState = PinchZoomState()

    default:
        break
    }
}

// [수정] 방향 기반으로 목표 열 수 추론
func inferTargetColumns(fromDirection direction: ZoomDirection,
                        baseColumns: GridColumnCount) -> GridColumnCount {
    let thresholds: [Int] = [1, 3, 5]
    guard let currentIndex = thresholds.firstIndex(of: baseColumns.rawValue) else {
        return baseColumns
    }

    switch direction {
    case .zoomIn:  // 확대 → 열 수 감소
        let targetIndex = max(0, currentIndex - 1)
        return GridColumnCount(rawValue: thresholds[targetIndex]) ?? baseColumns
    case .zoomOut: // 축소 → 열 수 증가
        let targetIndex = min(thresholds.count - 1, currentIndex + 1)
        return GridColumnCount(rawValue: thresholds[targetIndex]) ?? baseColumns
    }
}
```

### 4.3 앵커 고정 보정 의사코드

```swift
func updateContentOffsetForAnchor(anchorAssetID: String?, anchorPointInView: CGPoint) {
    guard let assetID = anchorAssetID,
          let indexPath = collectionIndexPath(for: assetID),
          let attributes = collectionView.collectionViewLayout
            .layoutAttributesForItem(at: indexPath) else { return }

    let anchorCenterInContent = attributes.center
    let newOffset = CGPoint(
        x: anchorCenterInContent.x - anchorPointInView.x,
        y: anchorCenterInContent.y - anchorPointInView.y
    )
    collectionView.contentOffset = clampOffset(newOffset)
}

// clampOffset 구현
func clampOffset(_ offset: CGPoint) -> CGPoint {
    let contentSize = collectionView.contentSize
    let boundsSize = collectionView.bounds.size
    let inset = collectionView.adjustedContentInset

    let maxX = max(0, contentSize.width - boundsSize.width + inset.right)
    let maxY = max(0, contentSize.height - boundsSize.height + inset.bottom)

    return CGPoint(
        x: min(max(offset.x, -inset.left), maxX),
        y: min(max(offset.y, -inset.top), maxY)
    )
}
```

### 4.6 마무리 애니메이션 (animateToTargetColumns) 구현

손을 뗄 때 `virtualColumns`를 목표값까지 부드럽게 보간하는 핵심 함수.

**왜 CADisplayLink인가?**
- `UIViewPropertyAnimator`는 커스텀 레이아웃 속성(`virtualColumns`)에 직접 적용 안 됨
- `CADisplayLink`는 핀치 중 사용하는 방식과 동일해서 끊김 없음

```swift
// 애니메이션 상태 저장
private var animationDisplayLink: CADisplayLink?
private var animationStartTime: CFTimeInterval = 0
private var animationStartValue: CGFloat = 0
private var animationTargetValue: CGFloat = 0
private var animationCompletion: (() -> Void)?

func animateToTargetColumns(_ target: GridColumnCount, completion: @escaping () -> Void) {
    let targetValue = CGFloat(target.rawValue)

    // 이미 목표에 도달했으면 즉시 완료
    if abs(layout.virtualColumns - targetValue) < 0.01 {
        completion()
        return
    }

    // 애니메이션 상태 저장
    animationStartValue = layout.virtualColumns
    animationTargetValue = targetValue
    animationStartTime = CACurrentMediaTime()
    animationCompletion = completion

    // 기존 애니메이션 중지 후 새로 시작
    animationDisplayLink?.invalidate()
    animationDisplayLink = CADisplayLink(target: self, selector: #selector(animationTick))
    animationDisplayLink?.add(to: .main, forMode: .common)
}

@objc private func animationTick() {
    let duration: CFTimeInterval = 0.25  // 마무리 애니메이션 길이
    let elapsed = CACurrentMediaTime() - animationStartTime
    let progress = min(elapsed / duration, 1.0)

    // easeOut 곡선: 처음 빠르고 끝에서 감속
    let easedProgress = 1 - pow(1 - progress, 3)

    // virtualColumns 보간
    let newValue = animationStartValue +
        (animationTargetValue - animationStartValue) * CGFloat(easedProgress)
    layout.virtualColumns = newValue
    layout.invalidateLayout()
    collectionView.layoutIfNeeded()

    // 앵커 보정 유지 (핀치 중 저장한 앵커 사용)
    updateContentOffsetForAnchor(anchorAssetID: pinchState.anchorAssetID,
                                  anchorPointInView: pinchState.anchorPointInView)

    // 완료 체크
    if progress >= 1.0 {
        animationDisplayLink?.invalidate()
        animationDisplayLink = nil
        animationCompletion?()
        animationCompletion = nil
    }
}

// 애니메이션 중단 (새 핀치 시작 시 호출)
func cancelSnapAnimation() {
    animationDisplayLink?.invalidate()
    animationDisplayLink = nil
    animationCompletion = nil
}
```

**주의사항**:
- 새 핀치가 시작되면 `cancelSnapAnimation()` 호출 필요
- `animationTick`에서도 앵커 보정을 계속 수행해야 부드러움

## 5. 레이아웃 계산식 상세 (실수 열 수 기반)

### 5.1 기본 계산식

- `virtualColumns`는 **실수** 값을 유지 (예: 3.0 → 2.6 → 1.4)
- 셀 크기/간격은 아래 공식으로 연속 변화

```swift
let columns = virtualColumns
let totalSpacing = spacing * (columns - 1)
let itemWidth = (bounds.width - totalSpacing) / columns
let itemHeight = itemWidth
```

### 5.2 row/col 매핑 규칙

**원칙**: "열 수 자체는 실수로 유지하되, 실제 배치 인덱스는 정수 열 기준으로 계산"
→ `effectiveColumns = Int(ceil(virtualColumns))`를 사용해 배치 인덱스를 만든다.

```swift
let effectiveColumns = Int(ceil(virtualColumns))
let row = index / effectiveColumns
let col = index % effectiveColumns

let x = CGFloat(col) * (itemWidth + spacing)
let y = CGFloat(row) * (itemHeight + spacing)
```

**효과**:
- 열 수가 실수로 변화하므로 `itemWidth`가 계속 변함
- 배치는 가장 가까운 정수 열로 유지되어 "셀 정렬 붕괴"를 방지

### 5.3 paddingCellCount 반영

패딩 셀을 앞에 둔다면 **레이아웃 계산은 전체 인덱스 기준**으로 수행하고,
실제 에셋 인덱스 변환은 별도로 처리한다.

```swift
// [수정] 핀치 중 paddingCount는 effectiveColumns 기준으로 계산
let effectiveColumns = Int(ceil(layout.virtualColumns))
let effectiveColumnCount = GridColumnCount(rawValue: effectiveColumns)
    ?? snapToNearestThreshold(effectiveColumns)
let paddingCount = paddingCellCount(for: effectiveColumnCount)

let collectionIndex = indexPath.item
let assetIndex = collectionIndex - paddingCount
if assetIndex >= 0 {
    let asset = gridDataSource.asset(at: assetIndex)
}

// 유효 기준점으로 스냅하는 헬퍼
func snapToNearestThreshold(_ columns: Int) -> GridColumnCount {
    let thresholds = [1, 3, 5]
    let nearest = thresholds.min(by: { abs($0 - columns) < abs($1 - columns) })!
    return GridColumnCount(rawValue: nearest)!
}
```

**주의**: 열 수 변경 시 paddingCount도 변하므로
**같은 indexPath라도 다른 asset**이 매핑될 수 있다.
따라서 fadeIn 시에는 "새 열 수 기준 paddingCount"를 사용해야 한다.

### 5.4 contentSize 계산

row 수는 `effectiveColumns` 기준으로 계산한다.

```swift
let itemCount = collectionView.numberOfItems(inSection: 0)
let effectiveColumns = Int(ceil(virtualColumns))
let rowCount = Int(ceil(Double(itemCount) / Double(effectiveColumns)))

let contentHeight = CGFloat(rowCount) * itemHeight +
    CGFloat(max(0, rowCount - 1)) * spacing
```

### 5.5 레이아웃 캐시/범위 최적화

`layoutAttributesForElements(in:)`는 전체 아이템을 돌지 않도록
**가시 영역 기반 row 범위**만 계산한다.

```swift
let minRow = max(0, Int(floor(rect.minY / (itemHeight + spacing))))
let maxRow = Int(ceil(rect.maxY / (itemHeight + spacing)))

for row in minRow...maxRow {
    for col in 0..<effectiveColumns {
        let index = row * effectiveColumns + col
        if index >= itemCount { break }
        // attributes 생성
    }
}
```

### 5.6 동작 예시 (3열 → 1열)

| virtualColumns | effectiveColumns | itemWidth 변화 | 시각 효과 |
|---------------|------------------|----------------|-----------|
| 3.0 | 3 | 기준 크기 | 3열 |
| 2.4 | 3 | 점진 확대 | 3열 느낌 유지 + 확대 |
| 1.8 | 2 | 더 큰 셀 | 2열 정렬 느낌 (중간 상태) |
| 1.2 | 2 | 거의 1열 | 1열 전환 직전 |
| 1.0 | 1 | 완료 | 1열 |

**중요 포인트**: `virtualColumns`는 연속으로 움직이고,
정렬은 `effectiveColumns`에 의해 안정적으로 유지된다.

> **설계 의도**: `effectiveColumns`는 중간 상태(2열, 4열)를 허용하여 부드러운 전환을 제공.
> 최종 스냅은 항상 유효 기준점(1, 3, 5)으로만 수행.

### 5.7 effectiveColumns 전환 구간 보정 (히스테리시스)

`virtualColumns`가 정수 경계(예: 2.0, 3.0)를 넘나들 때
`effectiveColumns`가 자주 바뀌면 셀 배치가 흔들릴 수 있다.
이를 방지하기 위해 **히스테리시스(완충 구간)**을 둔다.

```swift
// [수정] 유효 기준점 배열 사용
// effectiveColumns는 중간 상태(1~5 연속 정수)를 허용하지만,
// 스냅 기준점은 [1, 3, 5]만 사용

func stabilizedEffectiveColumns(_ virtualColumns: CGFloat,
                                current: Int) -> Int {
    let hysteresis: CGFloat = 0.2
    let downThreshold = CGFloat(current) - (1.0 - hysteresis)  // current - 0.8
    let upThreshold = CGFloat(current) + (1.0 - hysteresis)    // current + 0.8

    // effectiveColumns는 1~5 범위의 연속 정수 허용
    if virtualColumns <= downThreshold {
        return max(1, current - 1)
    }
    if virtualColumns >= upThreshold {
        return min(5, current + 1)
    }
    return current
}

// 최종 스냅 시에만 유효 기준점으로 변환
func snapToValidThreshold(_ columns: Int) -> GridColumnCount {
    // 1~2 → 1, 3~4 → 3, 5 → 5
    if columns <= 2 { return .one }
    if columns <= 4 { return .three }
    return .five
}
```

**추가 팁**
- `effectiveColumns` 변경 시 앵커 보정으로 점프를 숨긴다.
- 경계 전후로 `prefetch` 범위를 넓혀 외곽 셀 빈 영역을 줄인다.
- `effectiveColumns`는 1~5 범위로 clamp한다.

### 5.8 fadeIn 트리거/중복 방지 규칙

**목표**: 기준점 통과에 맞춰 fadeIn을 정확히 1회씩만 실행하고,
빠른 핀치/방향 전환/셀 재사용 상황에서도 잘못된 이미지 적용을 방지한다.

**트리거 규칙**
- 방향 결정 시 **첫 목표 열 수**로 즉시 1회 실행 (`.changed` 첫 프레임)
- 기준점 통과 시 **새 목표 열 수**로 1회 실행
- 같은 목표 열 수로는 중복 실행 금지

```swift
func shouldTriggerFadeIn(newTarget: GridColumnCount,
                         state: inout PinchZoomState) -> Bool {
    if state.currentTargetColumns == newTarget { return false }
    state.currentTargetColumns = newTarget
    return true
}
```

**방향 전환 처리**
- 방향이 바뀌면 `currentTargetColumns`를 nil로 리셋
- 다음 기준점 통과에서 새 목표로 다시 트리거

```swift
if state.didReverse {
    state.currentTargetColumns = nil
}
```

**경쟁 조건 방지**
- 셀마다 `fadeToken` 저장
- 요청 시점의 `targetColumns`를 캡처
- 완료 시 `token`과 `targetColumns` 둘 다 일치할 때만 적용

```swift
let token = UUID().uuidString
cell.fadeToken = token
let requestedColumns = targetColumns

loadThumbnail(...) { [weak cell] image in
    guard let cell = cell, cell.fadeToken == token else { return }
    guard state.currentTargetColumns == requestedColumns else { return }
    applyFadeIn(cell, image)
}
```

### 5.9 10% / 50% 규칙의 progress 계산 방식

**핵심 원칙**
- 진행도는 "현재 단계(stage)의 base → target" 구간에서 계산한다.
- 방향 전환이 없으면 **10%** 이상 진행 시 목표로 스냅.
- 방향 전환이 있으면 **최종 위치 기준 50%**로 판단한다.

#### 1) 단계 구간 정의

```swift
// stageBase: 현재 단계 시작 열 수 (예: 3)
// stageTarget: 현재 단계 목표 열 수 (예: 5)
// currentVirtual: 현재 가상 열 수 (실수)
```

#### 2) 진행도 계산

```swift
func stageProgress(stageBase: CGFloat,
                   stageTarget: CGFloat,
                   currentVirtual: CGFloat) -> CGFloat {
    let range = stageTarget - stageBase
    guard range != 0 else { return 0 }
    let raw = (currentVirtual - stageBase) / range
    return min(max(raw, 0), 1)
}
```

- 축소(열 수 증가): `stageTarget > stageBase` → progress가 0→1 증가
- 확대(열 수 감소): `stageTarget < stageBase` → 위 공식 그대로 사용 가능
  (range가 음수이므로 진행도는 0→1로 정상 증가)

#### 3) 방향 전환 처리

```swift
if didReverse {
    // 최종 위치 기준 50% 판단
    let p = stageProgress(stageBase, stageTarget, currentVirtual)
    snap = (p >= 0.5) ? stageTarget : stageBase
} else {
    // 단방향 10% 규칙
    let p = stageProgress(stageBase, stageTarget, currentVirtual)
    snap = (p >= 0.1) ? stageTarget : stageBase
}
```

#### 4) 다단계 진행 시 단계 갱신

기준점 통과 시 `stageBase`를 통과한 기준점으로 갱신하고,
`stageTarget`을 다음 기준점으로 설정한다.

```swift
// 예: 5 → 3 → 1
// 5→3 단계 완료 후: stageBase = 3, stageTarget = 1
```

**의도**: 진행도/스냅 판단은 **항상 "현재 단계" 기준**으로 수행한다.

### 5.10 앵커 셀 선택 실패 시 fallback 규칙

**문제**
- 핀치 시작 지점이 빈 공간/패딩 영역일 수 있음
- 이 경우 앵커 셀을 찾지 못하면 화면이 흔들릴 수 있음

**해결 원칙**
- 앵커 셀을 항상 확보한다 (nil 방지)
- 실패 시 "화면 중앙 셀"로 fallback
- 그래도 없으면 "가장 가까운 visible 셀"로 fallback

```swift
func resolveAnchorAssetID(at location: CGPoint) -> String? {
    // 1) 핀치 위치에서 직접 찾기
    if let indexPath = collectionView.indexPathForItem(at: location),
       indexPath.item >= paddingCellCount {
        return assetIDForCollectionIndexPath(indexPath)
    }

    // 2) 화면 중앙 셀로 fallback
    let centerPoint = CGPoint(
        x: collectionView.bounds.midX,
        y: collectionView.bounds.midY + collectionView.contentOffset.y
    )
    if let centerIndexPath = collectionView.indexPathForItem(at: centerPoint) {
        return assetIDForCollectionIndexPath(centerIndexPath)
    }

    // 3) visible 셀 중 가장 가까운 셀로 fallback
    let visible = collectionView.indexPathsForVisibleItems
    guard let nearest = visible.min(by: { a, b in
        let pa = collectionView.layoutAttributesForItem(at: a)?.center ?? .zero
        let pb = collectionView.layoutAttributesForItem(at: b)?.center ?? .zero
        return distance(pa, centerPoint) < distance(pb, centerPoint)
    }) else { return nil }

    return assetIDForCollectionIndexPath(nearest)
}

// 거리 계산 헬퍼
func distance(_ a: CGPoint, _ b: CGPoint) -> CGFloat {
    let dx = a.x - b.x
    let dy = a.y - b.y
    return sqrt(dx * dx + dy * dy)
}
```

**주의**: fallback으로 선택한 앵커는 `anchorPointInView`도 동일한 기준으로 계산해야 한다.
예: 화면 중앙 셀을 앵커로 삼았다면 `anchorPointInView = bounds.center`.

### 5.11 회전 시 앵커 유지 규칙

**목표**: 회전 중에도 동일 에셋이 화면 중심에 유지되도록 보정.

**규칙**
- 회전 시작 시 화면 중앙 셀의 assetID를 저장
- 회전 애니메이션 동안에도 해당 assetID를 기준으로 보정
- 회전 완료 후 해당 assetID가 화면 중앙에 오도록 scroll 복구

```swift
override func viewWillTransition(to size: CGSize,
                                 with coordinator: UIViewControllerTransitionCoordinator) {
    super.viewWillTransition(to: size, with: coordinator)

    // 1) 회전 시작: 중앙 셀 저장
    saveScrollAnchorAssetID()
    let anchorID = scrollAnchorAssetID

    // 2) 새 열 수 계산
    let newColumns = (size.width > size.height)
        ? currentGridColumnCount.landscapeColumnCount
        : currentGridColumnCount.portraitColumnCount

    // 3) 회전 시작 시 fadeIn 트리거 (목표 열 수 기준)
    triggerFadeIn(target: newColumns, anchorAssetID: anchorID)

    // 4) 회전과 동기화하여 virtualColumns 즉시 변경
    // [수정] coordinator.animate 사용 (UIViewPropertyAnimator 불필요)
    coordinator.animate(alongsideTransition: { [weak self] _ in
        guard let self = self else { return }
        self.layout.virtualColumns = CGFloat(newColumns.rawValue)
        self.layout.invalidateLayout()
        self.collectionView.layoutIfNeeded()
        self.updateCellSize()
        self.updateContentInset()
    }, completion: { [weak self] _ in
        // 5) 회전 완료: 앵커 복구
        if anchorID != nil {
            self?.restoreScrollAnchorAssetID()
        }
        self?.currentGridColumnCount = newColumns
        self?.didPerformZoom(to: newColumns)
    })
}
```

**설계 결정**:
- `coordinator.animate(alongsideTransition:)` 사용 (권장)
- 회전 애니메이션과 자동 동기화되어 별도 타이밍 관리 불필요
- `UIViewPropertyAnimator`는 회전과 별개로 동작하여 동기화 어려움

### 5.13 축소 시 외곽 셀 Prefetch 전략

**문제**:
- 축소(3열 → 5열) 시 화면에 더 많은 셀이 표시됨
- 화면 밖 셀이 아직 로드되지 않으면 빈 셀로 보임

**해결 원칙**:
핀치 방향에 따라 prefetch 범위를 동적으로 확장한다.

```swift
// 핀치 시작 시 prefetch 범위 확장
func expandPrefetchForPinch(direction: ZoomDirection) {
    guard direction == .zoomOut else { return }  // 축소 시에만

    // 현재 visible rect보다 1.5배 넓은 영역 prefetch
    let visibleRect = collectionView.bounds
    let expandedRect = visibleRect.insetBy(
        dx: -visibleRect.width * 0.5,
        dy: -visibleRect.height * 0.5
    )

    // 확장된 영역의 indexPath들 prefetch
    let indexPaths = layout.indexPathsForElements(in: expandedRect)
    prefetchAssets(for: indexPaths)
}

// 핀치 종료 시 prefetch 범위 복원
func restoreNormalPrefetch() {
```

### 5.14 fadeIn 이미지 로딩 지연 시 Fallback

**문제**:
```
1. 3열 → 1열 핀치 (확대)
2. fadeIn(1열 고해상도 이미지) 트리거
3. 레이아웃 빠르게 변경 → 셀 크기 확대
4. 1열 이미지 아직 로딩 중
5. 결과: 3열 저해상도 이미지가 크게 표시 → 픽셀 깨짐
```

**해결 원칙**:
"없는 것보다 낮은 해상도라도 보여주는 게 낫다"

**전략 1: 계층적 캐시 활용**
```swift
func loadThumbnailWithFallback(assetID: String,
                                targetSize: CGSize,
                                completion: @escaping (UIImage?) -> Void) {
    // 1) 목표 해상도 캐시 확인
    if let cached = imageCache.image(for: assetID, size: targetSize) {
        completion(cached)
        return
    }

    // 2) 더 큰 해상도 캐시 확인 (다운샘플링)
    if let largerCached = imageCache.largestCachedImage(for: assetID, minSize: targetSize) {
        let downsampled = downsample(largerCached, to: targetSize)
        completion(downsampled)
        // 백그라운드에서 정확한 크기 로드 (선택적)
        return
    }

    // 3) 더 작은 해상도 캐시 확인 (일단 표시)
    if let smallerCached = imageCache.anyCachedImage(for: assetID) {
        completion(smallerCached)  // 일단 표시
        // 목표 해상도 비동기 로드
        loadAsync(assetID: assetID, targetSize: targetSize) { image in
            // 로드 완료 시 교체 (fadeIn)
        }
        return
    }

    // 4) 캐시 없음 → 비동기 로드
    loadAsync(assetID: assetID, targetSize: targetSize, completion: completion)
}
```

**전략 2: 셀에서 처리**
```swift
// PhotoCell
func applyFadeInImage(_ newImage: UIImage, for assetID: String) {
    guard self.assetID == assetID else { return }

    // 새 이미지가 현재 이미지보다 해상도가 높을 때만 교체
    if shouldReplaceImage(current: imageView.image, with: newImage) {
        UIView.transition(with: imageView, duration: 0.15, options: .transitionCrossDissolve) {
            self.imageView.image = newImage
        }
    }
}

func shouldReplaceImage(current: UIImage?, with new: UIImage) -> Bool {
    guard let current = current else { return true }
    // 새 이미지가 더 크거나, 현재 이미지가 placeholder인 경우
    return new.size.width * new.size.height > current.size.width * current.size.height
}
```

### v4 수정 사항

1. **1.5, 3.6 fadeIn 대상 명확화**
   - 문제: 축소 시 새로 나타나는 셀에 fadeIn 적용 여부 불명확
   - 수정: fadeIn 대상 = "기존에 visible이었던 셀" (앵커 제외)

   - 문제: 극단 스케일에서 columnWidth가 음수/과대 가능

   - 문제: `layout.virtualColumns` 변경 후 앵커 보정 전에 레이아웃 반영 미보장
   - 수정: `invalidateLayout()` → `layoutIfNeeded()` → `updateContentOffsetForAnchor()` 순서 명시
