# 자녀 휴대폰 제어 앱 — 기술 및 시장 조사

> 조사일: 2026-03-27

---

## 요약 결론

| 항목 | Android | iOS |
|------|---------|-----|
| **자녀 폰 제어 앱 개발** | 충분히 가능 | 제한적으로 가능 |
| **일정 시간 후 폰 잠금** | 가능 (여러 방법) | 직접 잠금 불가, 전체 앱 Shield로 우회적 구현 |
| **앱별 시간 제한** | 가능 | 가능 |
| **우회 방지** | 강력하게 가능 | 아이가 설정에서 해제 가능 (치명적 약점) |
| **앱 삭제 방지** | Device Admin으로 가능 | Family Sharing `.child` 모드에서만 가능 |
| **개발 난이도** | 중간 | 높음 (API 버그 다수, 문서 부족) |

**결론: Android가 압도적으로 유리. iOS는 Apple이 서드파티 앱의 기기 수준 제어를 구조적으로 막아놓은 상태.**

## 1. Android — 사용 가능한 API 및 기술

### 1-1. 핵심 API 일람

| API | 역할 | 핵심 기능 |
|-----|------|----------|
| `UsageStatsManager` | 사용 시간 추적 | 앱별 사용 시간 조회, 앱 전환 이벤트 감지 |
| `SYSTEM_ALERT_WINDOW` | 차단 화면 표시 | 전체 화면 오버레이로 폰 사용 차단 |
| `DevicePolicyManager` | 기기 관리 | `lockNow()`로 즉시 잠금, 앱 삭제 방지 |
| `AccessibilityService` | 실시간 앱 감지 | 포그라운드 앱 실시간 감지 + 우회 방지 |
| `VpnService` | 웹 필터링 | 로컬 VPN으로 유해 사이트/앱 네트워크 차단 |
| `Lock Task Mode` | 키오스크 모드 | 허용 앱만 실행 가능 (Device Owner 필요) |

### 1-2. Device Admin API / Device Policy Controller (DPC)

**가능한 것:**
- `DevicePolicyManager.lockNow()` — 즉시 기기 잠금
- 비밀번호 정책 설정 (최소 길이, 복잡도)
- 카메라 비활성화
- 앱 설치/제거 제한 (Device Owner 모드)
- `setLockTaskPackages()` — 특정 앱만 실행 허용 (키오스크 모드)

**제한:**
- Android 9(API 28)부터 Device Admin의 많은 정책이 deprecated
- Android 10(API 29) 이상 타겟 앱에서 deprecated 정책 호출 시 `SecurityException` 발생
- Google은 Device Admin에서 Android Enterprise(Device Owner/Profile Owner)로의 마이그레이션 강력 권고

### 1-3. Device Owner vs Profile Owner

| 항목 | Device Owner | Profile Owner |
|------|-------------|---------------|
| 제어 범위 | 기기 전체 완전 제어 | 업무 프로필 내부만 |
| 설정 방법 | 초기 셋업 또는 팩토리 리셋 후 (또는 ADB) | 런타임에서 설정 가능 |
| 계정 제한 | 설정 전 모든 계정 삭제 필요 | 기존 계정 유지 가능 |
| Wi-Fi/Bluetooth 제어 | 가능 | 불가 |
| 기기 초기화 | 가능 | 불가 |
| Lock Task Mode | 가능 | 불가 |
| 자녀 제어 적합성 | 높음 (모든 제어 가능) | 낮음 |

- Device Owner 설정 시 모든 계정을 삭제해야 하므로 일반 사용자에겐 부담
- ADB 명령어(`adb shell dpm set-device-owner`)로 팩토리 리셋 없이 설정 가능하나 PC 필요
- **Dhizuku** 같은 도우미 앱을 통해 ADB로 Device Owner 권한 위임 가능

### 1-4. UsageStatsManager — 앱 사용 시간 추적

- `queryUsageStats()` — 지정 기간의 앱별 사용 시간 조회 (일/주/월/년)
- `queryEvents()` — 앱 전환, 포그라운드/백그라운드 이벤트 조회
- 권한: `PACKAGE_USAGE_STATS` — 사용자가 설정에서 수동 허용 필요
- **실시간 데이터가 아님** — 지연이 있어 즉각적 차단에는 부적합
- 폴링 주기를 너무 짧게 하면 배터리 소모 심각

### 1-5. AccessibilityService로 앱 차단

**기술적으로 가능한 것:**
- 포그라운드 앱 실시간 감지 (`AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED`)
- 차단 앱 감지 시 홈 화면으로 강제 이동
- UsageStatsManager보다 훨씬 빠른 실시간 감지

**Google Play 정책 (2026.01.28 시행):**
- Accessibility API는 장애인 보조 목적으로 설계됨
- 단, **자녀 제어 앱은 예외 인정**: "authorized by a parent or guardian through a parental control app"
- Google Play 설명에 Accessibility API 사용 목적 명시 필수
- 데이터 수집 범위, 용도, 공유 방법 공개 필수

### 1-6. 일정 시간 이후 폰 잠금 구현 방법

| 방법 | 설명 | 장단점 |
|------|------|--------|
| **A: Overlay Window** | `SYSTEM_ALERT_WINDOW`로 전체 화면 잠금 UI | 가장 현실적, 홈/뒤로가기 우회 가능 → Accessibility로 보완 |
| **B: lockNow()** | DevicePolicyManager로 즉시 기기 잠금 | Device Admin 필요, 비밀번호 모르면 해제 불가 |
| **C: Lock Task Mode** | 허용 앱만 실행 가능한 완전 잠금 | 가장 강력, Device Owner 필요 |
| **D: VPN 차단** | 로컬 VPN으로 네트워크 차단 | 앱은 실행되지만 인터넷 사용 불가 |

**현실적 권장 조합:**
`UsageStatsManager`(시간 추적) + `ForegroundService`(모니터링) + `SYSTEM_ALERT_WINDOW`(차단 UI) + 선택적 `AccessibilityService`(우회 방지)

### 1-7. Android 13/14/15 API 변경 및 제한

| Android 버전 | 주요 변경 | 영향 |
|-------------|---------|------|
| **13 (API 33)** | Restricted Settings — 사이드로드 앱의 Accessibility/Device Admin 권한 차단 | Play Store 배포 필수화 |
| **13** | 알림 권한(`POST_NOTIFICATIONS`) 런타임 요청 필요 | 추가 권한 요청 필요 |
| **14 (API 34)** | ForegroundService 타입 선언 필수 | 매니페스트에 서비스 타입 명시 |
| **14** | 백그라운드에서 ForegroundService 시작 제한 강화 | 포그라운드에서만 서비스 시작 |
| **14** | 정확한 알람(`SCHEDULE_EXACT_ALARM`) 제한 | 시간 기반 잠금 구현 어려워짐 |
| **15 (API 35)** | Enhanced Confirmation Mode — 사이드로드 앱 권한 추가 제한 | Overlay, UsageStats, Accessibility 모두 제한 |
| **15** | `BOOT_COMPLETED`에서 일부 ForegroundService 시작 금지 | 부팅 시 자동 모니터링 시작 제약 |

### 1-8. 권장 아키텍처

```
┌─────────────────────────────────────────┐
│          자녀 제어 앱 (Parent App)        │
├─────────────────────────────────────────┤
│ 1. UsageStatsManager                    │  ← 앱 사용 시간 추적
│ 2. ForegroundService                    │  ← 지속적 모니터링
│ 3. SYSTEM_ALERT_WINDOW (Overlay)        │  ← 앱 차단/잠금 화면
│ 4. AccessibilityService (선택)          │  ← 실시간 앱 감지 + 우회 방지
│ 5. VpnService (선택)                    │  ← 콘텐츠/웹 필터링
│ 6. DeviceAdmin (선택)                   │  ← lockNow(), 삭제 방지
│ 7. Device Owner via ADB (고급)          │  ← Lock Task Mode, 완전 제어
└─────────────────────────────────────────┘
```

**단계별 구현 전략:**

1. **MVP**: UsageStatsManager + Overlay Window + ForegroundService
   - Google Play 정책 부담 가장 낮음
2. **중급**: + AccessibilityService + DeviceAdmin
   - 실시간 앱 감지, 앱 삭제 방지
3. **고급**: + Device Owner(ADB 설정 가이드) + VpnService
   - Lock Task Mode, DNS 필터링

## 2. iOS — 사용 가능한 API 및 기술

### 2-1. Screen Time API (iOS 15+)

Apple이 WWDC 2021에서 발표한 3개 프레임워크:

#### FamilyControls
- 인증 게이트웨이, iOS 16+에서 `.individual` 모드 추가
- **두 가지 인증 모드:**
  - `.individual` (iOS 16+): 자기 자신의 기기를 스스로 제어 (디지털 웰빙 앱용)
  - `.child`: 부모가 자녀 기기를 제어 (iCloud Family Sharing 필요)
- `FamilyActivityPicker`: 차단할 앱/카테고리 선택 SwiftUI 컴포넌트
- `.child` 인증 시 부모 승인 없이 앱 삭제 불가
- **프라이버시 제한**: 앱 이름/번들ID 대신 암호화된 토큰(opaque token)으로만 앱 식별

#### ManagedSettings
- **앱 차단 (Shield)**: `store.shield.applications`에 토큰 집합을 할당
- **카테고리 단위 차단**: `store.shield.applicationCategories = .all()`로 전체 차단 가능
- **웹 도메인 차단**: Safari에서 특정 도메인 접근 차단
- **최대 50개 Named Store**: Extension 간 자동 공유

**할 수 없는 것:**
- **기기 잠금 불가**
- **설정 앱 접근 차단 불가** — 사용자가 권한 쉽게 해제 가능
- **패스코드 보호 불가** — Apple 기본 Screen Time은 가능하나 서드파티 앱은 불가

#### DeviceActivity
- 스케줄 설정: 시작/종료 시간을 `DateComponents`로 정의
- 사용 시간 임계값 설정
- 콜백: `intervalDidStart`, `intervalDidEnd`, `eventDidReachThreshold`
- `DeviceActivityReport` (iOS 16+): 커스텀 사용량 리포트

**할 수 없는 것:**
- Extension 메모리 제한 6MB — 초과 시 크래시
- Report 데이터를 메인 앱으로 가져올 수 없음 (표시 전용)
- iOS 17 안정성 이슈 다수 보고

#### ShieldConfiguration / ShieldAction
- Shield 화면 커스터마이징 (제목, 본문, 아이콘, 버튼)
- **메인 앱 열기 불가**, **앱 실행 불가** — Shield에서 할 수 있는 것이 극히 제한적
- 토큰 불일치 버그, UI 갱신 실패 버그 존재

### 2-2. MDM (Mobile Device Management)

**할 수 있는 것 (Screen Time API보다 훨씬 강력):**
- 앱 블랙리스트/화이트리스트
- 시스템 앱 비활성화 (카메라, Safari, Siri 등)
- App Store 설치 자체 차단
- 기기 설정 변경 차단
- 원격 관리
- MDM 프로필 삭제 방지

**제약:**
- **기기 초기화 필요** — 비감독 기기 → 감독 모드 전환 시 전체 데이터 삭제
- App Store 심사 엄격 (상업 기업, 교육기관, 정부기관, 자녀 제어 회사만 허용)
- 데이터 판매/제3자 공유 절대 금지
- 일반 소비자에게 설정 과정이 비현실적

### 2-3. iOS에서 "일정 시간 후 폰 잠금" 구현

| 방법 | 가능 여부 | 설명 |
|------|----------|------|
| 전체 기기 잠금 | **불가능** | Screen Time API로 기기 자체를 잠글 수 없음 |
| 전체 앱 Shield | **우회적 가능** | `shield.applicationCategories = .all()`로 모든 앱에 Shield |
| Downtime 스타일 | **가능** | DeviceActivitySchedule로 시간대별 차단 |
| MDM 기기 잠금 | **가능** | 감독 모드 필요 (기기 초기화 필수) |

### 2-4. iOS의 치명적 약점

1. **기기 잠금 불가** — 전체 앱 Shield 우회 방법만 가능
2. **권한 해제 용이** — 아이가 설정에서 서드파티 앱 권한 쉽게 해제 가능 (패스코드 보호 불가)
3. **API 버그 다수** — 토큰 무작위 변경, 임계값 오발화
4. **Family Sharing 필수** — `.child` 모드는 iCloud Family Sharing 선행 필요
5. **MDM은 기기 초기화 필요** — 소비자에게 비현실적
6. **FamilyControls 엔타이틀먼트 승인 31~33일 소요**

### 2-5. iOS 버전별 변경

| iOS 버전 | 주요 변경 |
|----------|----------|
| **iOS 15** | Screen Time API 최초 도입, `.child` 인증만 지원 |
| **iOS 16** | `.individual` 인증 추가, Named ManagedSettingsStore (최대 50개), DeviceActivityReport |
| **iOS 17** | 안정성 이슈 다수 — 스케줄 기반 차단 미작동 사례 |
| **iOS 18** | 기존 API 안정성 개선 위주 |
| **iOS 26** | PermissionKit 신규 도입 — 자녀가 부모에게 권한 요청, 앱스토어 연령 등급 강화 |

### 2-6. 2019년 앱스토어 제거 논란

- 2019년 초: Apple이 17개 인기 자녀 제어 앱 중 11개를 앱스토어에서 제거
- Apple 주장: MDM이 프라이버시/보안 위협
- 개발사 반론: 자사 Screen Time과의 경쟁 제거 목적의 반경쟁 행위
- Qustodio, Kidslox가 EU 경쟁 감시기관에 공식 제소
- 2019.06: 반독점 조사 압박 속 Apple이 정책 철회, MDM을 "제한된 경우"에 허용

**현재 정책 (2024~2026):**
- App Store Review Guideline 5.5에서 MDM 사용 조건 명시
- 상업 기업, 교육기관, 정부기관 + 자녀 제어 서비스에 한해 허용
- 데이터를 제3자에게 판매/공유 절대 금지

## 4. 사업적 권장사항

### 4-1. 플랫폼 전략

- **Android 우선 개발** → MVP를 Android에서 검증 후 iOS 확장
- iOS는 Screen Time API + VPN 필터링 조합이 현실적 최선
- iOS의 구조적 한계(우회 방지 불가)를 인지하고 기대치 관리 필요

### 4-2. 한국 시장 진입

- 통신사 무료 앱(ZEM)과 직접 경쟁 피해야 함
- 차별화 포인트: AI 콘텐츠 분석, 크로스 플랫폼 지원, 가족 소통 기능
- ZEM의 약점(배터리, 버그, 우회)을 보완하는 포지셔닝 유효

### 4-3. 수익 모델

- 연 $50~$80 가족 플랜 구독 (한화 약 6.5만~10만원)
- 7일 무료 체험으로 전환율 확보
- 기기 수 기반 티어링(1~3대 vs 가족 무제한)

### 4-4. 주의사항

- "감시"가 아닌 **"신뢰 기반 관리"** 메시지가 트렌드 (Bark 모델 참고)
- 과도한 감시는 인권 이슈 — ZEM 헌법소원 사례 존재
- 기술적 우회가 커뮤니티에 공유되는 문제는 모든 자녀 제어 앱의 공통 과제

- `ScreenTimeWorker.java`: WorkManager로 백그라운드 복원력 확보
- 크레딧 기반 시간 관리 (일일 120분 + 연장 크레딧)
