## 0. v6 핵심 목표: 6개 문제 완전 해결

### 0.1 v5에서 발생한 6개 문제 (재발 방지 대상)

| # | 문제 | 근본 원인 | v6 해결 전략 |
|---|------|----------|-------------|
| 1 | X축 좌측 정렬 | 레이아웃에 xOffset 미적용 | **레이아웃 내부에서 xOffset 적용** |
| 2 | 스크롤 점프 + fadeIn 미동작 | offset 보정이 레이아웃 갱신과 분리됨 | **레이아웃 내부에서 offset 계산, atomic 갱신** |
| 3 | 바깥 셀 클리핑 | rect 확장 없음 + clipsToBounds | **핀치 중 rect 확장 + clipsToBounds=false** |
| 4 | 10% 규칙 미적용 | 진행도 계산이 현재 위치 기준 | **핀치 시작 시 base/target 고정** |
| 5 | 취소 시 스크롤 점프 | 문제 2와 동일 | 문제 2와 동일 해결 |
| 6 | 좌표계 혼용 | bounds/content 혼용 | v5에서 해결됨 ✅ |

### 0.2 v6 핵심 설계 원칙

#### 원칙 1: 레이아웃이 앵커를 안다 (Layout-Aware Anchor)

```
[v5 방식 - 문제 발생]
handlePinchChanged:
  1. layout.virtualColumns = 2.5
  2. layout.invalidateLayout()
  3. collectionView.layoutIfNeeded()  ← 화면 갱신 (offset 틀어진 상태!)
  4. updateContentOffsetForAnchor()   ← 뒤늦게 보정

[v6 방식 - 해결]
handlePinchChanged:
  1. layout.anchorInfo = (assetID, pointInView, xOffset)  ← 앵커 정보 전달
  2. layout.virtualColumns = 2.5
  3. layout.invalidateLayout()
  4. collectionView.layoutIfNeeded()  ← 레이아웃이 앵커 기준으로 계산 완료
  (외부 offset 보정 불필요!)
```

**핵심**: 레이아웃이 앵커 정보를 알고, `prepare()`에서 **모든 셀 위치를 앵커 기준으로 계산**.

#### 원칙 2: X축은 xOffset으로, Y축은 contentOffset으로

```
[X축 처리]
- 그리드는 항상 x=0부터 시작 (contentOffset.x로는 중앙 고정 불가)
- 레이아웃에서 모든 셀의 x 좌표에 xOffset 적용
- xOffset = anchorPointInView.x - anchorCellNormalX
- 손 뗄 때: xOffset → 0 애니메이션

[Y축 처리]
- contentOffset.y로 보정 (기존 방식 유지)
- 레이아웃의 contentSize는 정상 계산
```

#### 원칙 3: base/target 이중 구조 (스냅용 vs 단계용)

**핵심**: base/target은 **두 가지 용도**로 분리하여 관리한다.

```swift
struct PinchZoomState {
    // ===== 1. 스냅 판단용 (핀치 전체, 절대 변경 금지!) =====
    // - 10%/50% 규칙 적용 시 사용
    // - 핀치 시작 시 설정, 핀치 종료까지 불변
    let snapBaseColumns: CGFloat           // 핀치 시작 열 수 (예: 3.0)
    var snapTargetColumns: GridColumnCount // 최종 목표 열 수 (예: 1열) - 방향 결정 시 설정

    // ===== 2. 단계별 fadeIn용 (다단계 핀치에서 변경됨) =====
    // - fadeIn 트리거 시 사용
    // - 기준점 통과할 때마다 재설정
    var stageBaseColumns: CGFloat          // 현재 단계 시작점 (예: 3.0 → 1.0)
    var stageTargetColumns: GridColumnCount // 현재 단계 목표 (예: 1열)

    // ===== 3. 기타 고정값 =====
    let anchorAssetID: String
    let anchorPointInView: CGPoint
    let frozenPaddingCellCount: Int
    let frozenItemCount: Int
}
```

**용도 구분**:
| 용도 | 사용하는 값 | 언제 변경? |
|-----|-----------|----------|
| **스냅 판단 (10%/50%)** | `snapBaseColumns`, `snapTargetColumns` | 절대 변경 안 함 |
| **fadeIn 트리거** | `stageBaseColumns`, `stageTargetColumns` | 기준점 통과 시 재설정 |

**예시: 5열 → 3열 → 1열 다단계 핀치**
```
[핀치 시작]
snapBase=5, snapTarget=1 (고정!)
stageBase=5, stageTarget=3

[3열 기준점 통과]
snapBase=5, snapTarget=1 (여전히 고정!)
stageBase=3, stageTarget=1 (재설정)
→ fadeIn 트리거 (1열 이미지로)

[손을 뗄 때 - 진행도 80%]
스냅 판단: (현재 - snapBase) / (snapTarget - snapBase) = 80%
→ 10% 초과이므로 snapTarget(1열)으로 스냅
```

#### 원칙 4: 가시 영역 확장

```
[핀치 중 가시 영역]
- layoutAttributesForElements(in rect:)에서 rect를 확장
- 확장량 = 현재 셀 크기 × 2 (상하좌우)
- clipsToBounds = false (핀치 중에만)
- 바깥 셀도 미리 로딩 (prefetch 확장)
```

#### 원칙 5: 핀치 중 dataSource 변경 차단

**문제**: 핀치 중 사진 삭제/추가 시 레이아웃과 데이터 불일치로 점프 또는 크래시 발생 가능

**규칙**:
```
[핀치 시작]
1. frozenItemCount = collectionView.numberOfItems(inSection: 0)
2. PHPhotoLibraryChangeObserver 업데이트 지연 플래그 ON
3. isPinching = true

[핀치 중 - 데이터 변경 발생 시]
- 변경 사항을 큐에 저장 (즉시 적용 안 함)
- 레이아웃은 frozenItemCount 기준으로 계속 동작

[핀치 종료]
1. isPinching = false
2. 지연 플래그 OFF
3. 큐에 저장된 변경 사항 일괄 적용
4. collectionView.reloadData()
```

**구현 위치**:
- `PHPhotoLibraryChangeObserver.photoLibraryDidChange(_:)`에서 `isPinching` 체크
- 핀치 중이면 변경 적용 지연

## 1. 핵심 개념

### 1.1 열 수 기준점
- **1열, 3열, 5열** 세 가지 기준점만 존재
- 핀치줌/회전 완료 시 반드시 이 중 하나로 snap

### 1.2 스케일 → 열 수 관계
```
scale 1.0 = 현재 열 수 유지
scale > 1.0 = 확대 → 열 수 감소 (3열 → 1열)
scale < 1.0 = 축소 → 열 수 증가 (3열 → 5열)
```

### 1.3 진행도 개념 (v6 수정)

```
[v5 문제점]
- base/target이 현재 virtualColumns 기준으로 계속 변함
- 진행도가 불안정하여 10% 규칙이 제대로 동작 안 함

[v6 해결]
- 핀치 시작 시 base/target 고정
- 진행도 = (현재 - base) / (target - base)
- 선형적이고 예측 가능한 진행도
```

**예시: 3열에서 축소 시작**
```
base = 3.0 (고정)
target = 5.0 (고정)

virtualColumns = 3.2 → 진행도 = (3.2-3)/(5-3) = 10%
virtualColumns = 4.0 → 진행도 = (4-3)/(5-3) = 50%
virtualColumns = 4.8 → 진행도 = (4.8-3)/(5-3) = 90%
```

### 1.4 앵커 셀
- **핀치**: 핀치 중심점에 있는 셀
- **회전**: 화면 중앙에 있는 셀
- 앵커 셀은 fadeIn 제외, 기존 이미지 유지
- **v6 추가**: 앵커 셀의 화면 위치(X, Y 모두)가 고정되어야 함

### 1.5 fadeIn
- **대상**: 기존에 visible이었던 셀 중 이미지 교체가 필요한 셀 (앵커 제외)
- **제외**: 새로 나타난 셀 (즉시 표시, fadeIn 금지)
- 현재 이미지 → 목표 레이아웃의 이미지로 부드럽게 전환
- 새 단계 시작 시 트리거

### 1.6 열 수 변경 규칙

#### 핀치줌
- 확대: 열 수 감소 (5 → 3 → 1)
- 축소: 열 수 증가 (1 → 3 → 5)

#### 회전
- 세로 → 가로: 1→3, 3→5, 5→5
- 가로 → 세로: 1→1, 3→3, 5→3

## 2. 요구사항

### 2.1 핵심 원칙

#### 금지 사항
- 셀 개별 이동 애니메이션 ❌
- 새로고침/끊김 ❌
- **스크롤 점프 ❌** (v6 강조)

#### 필수 사항
- 전체가 하나처럼 연속적인 모션 ✅
- fadeIn으로 부드러운 이미지 전환 ✅
- 핀치 진행도에 따라 실제 레이아웃이 점진적으로 변화 ✅
- **앵커 셀이 화면상 X, Y 모두 고정 ✅** (v6 추가)
- **바깥 셀이 잘린 채로 나타남 ✅** (v6 추가)

### 2.2 핀치줌 상세 동작

#### 핀치 시작 시 (.began)
1. 앵커 셀 결정 (핀치 중심점의 셀)
2. **앵커의 화면 위치(bounds 좌표) 저장**
3. **base/target 열 수 결정 및 고정**
4. **paddingCellCount, itemCount 고정**

> **주의**: `.began` 시점에서는 방향을 알 수 없음. target은 `.changed` 첫 프레임에서 결정.

#### 핀치 첫 프레임 (.changed 첫 번째)
1. 방향 결정 (scale > 1.0 = 확대, scale < 1.0 = 축소)
2. target 열 수 결정 및 고정
3. fadeIn 트리거 (목표 이미지로)
4. 레이아웃 갱신 시작

#### 핀치 중 (.changed)
1. virtualColumns 계산: `baseColumns / scale`
2. **레이아웃이 앵커 정보를 기반으로 모든 셀 위치 계산**
3. X축: xOffset 적용, Y축: contentOffset 보정
4. 외곽에 새 셀들이 **잘린 채로** 나타남 (축소 시)
5. 앵커 셀: 기존 이미지 유지, 화면 위치 고정

#### 다단계 핀치 (예: 5열 → 3열 → 1열)
- 각 기준점(1/3/5열) 통과 시 = 새 단계 시작
- 새 단계 시작 시:
  - fadeIn 재트리거 (다음 목표 이미지로)
  - **stageBase/stageTarget 재설정** (fadeIn용, 단계별)
  - **snapBase/snapTarget은 절대 변경 안 함!** (스냅 판단용)

#### 손을 뗄 때 (.ended)
1. **10%/50% 규칙으로 스냅 방향 결정** (snapBase/snapTarget 기준)
2. 현재 상태 → 목표 상태로 애니메이션
3. **xOffset → 0 애니메이션** (anchorPointInView 보간 방식)
4. 스냅 완료 후 paddingCellCount 업데이트

**xOffset 애니메이션 구현 방식**:
```
[핀치 중]
anchorPointInView = 핀치 시작 위치 (예: x=100)
xOffset = anchorPointInView.x - anchorCellNormalX

[스냅 애니메이션]
anchorPointInView를 "목표 열 수에서 앵커 셀이 위치할 정상 X좌표"로 보간
→ 애니메이션 진행에 따라 xOffset이 자연스럽게 0으로 수렴

[애니메이션 완료]
xOffset = 0 (정상 레이아웃)
```

### 2.3 원하는 시각적 동작 (v6 명확화)

#### 확대 시 (E가 핀치 중심축인 경우)
```
[시작 - 3열]
┌─────────────────────┐
│  A    B    C        │
│  D   [E]   F        │  ← E가 앵커
│  G    H    I        │
└─────────────────────┘

[확대 중]
┌─────────────────────┐
│ A│   B    │C        │  ← A, C 일부 잘림 (화면 밖으로)
│ D│  [E]   │F        │  ← E는 화면상 같은 위치에 고정!
│ G│   H    │I        │  ← G, I 일부 잘림
└─────────────────────┘
→ E 기준 사방으로 확장
→ 스크린샷이 확대되는 느낌

[완료 - 1열]
┌─────────────────────┐
│                     │
│        [E]          │  ← E만 보임
│                     │
└─────────────────────┘
```

#### 축소 시 (E가 핀치 중심축인 경우)
```
[시작 - 3열]
┌─────────────────────┐
│  A    B    C        │
│  D   [E]   F        │  ← E가 앵커
│  G    H    I        │
└─────────────────────┘

[축소 중]
┌─────────────────────┐
│J│ A  B  C │K        │  ← J, K가 잘린 채로 등장!
│L│ D [E] F │M        │  ← E는 화면상 같은 위치에 고정!
│N│ G  H  I │O        │  ← N, O가 잘린 채로 등장!
└─────────────────────┘
→ E 기준 사방으로 축소
→ 새 셀(J,K,L,M,N,O)이 가장자리에서 잘린 채로 나타남
→ clipsToBounds = false 필수!

[완료 - 5열]
┌─────────────────────┐
│  J  A  B  C  K      │
│  L  D [E] F  M      │
│  N  G  H  I  O      │
└─────────────────────┘
```

### 2.4 기준점 판단 로직 (손을 뗄 때)

#### Case 1: 단방향 핀치 (10% 규칙)

한 방향으로만 핀치한 경우:

```
3열에서 축소 방향으로 10% 이상 진행 → 손을 뗌
진행도 = (virtualColumns - 3) / (5 - 3) ≥ 0.1
→ 의도 명확 → 5열로 완료
```

**10% 이상 진행하면 무조건 그 방향으로 진행**

#### Case 2: 방향 전환 (50% 규칙)

핀치 도중 방향이 바뀐 경우:

```
3열에서 축소 70% → 다시 확대 40%
→ 최종 위치: 30% (축소 방향)
→ 50% 미만 → 3열 유지

3열에서 축소 80% → 다시 확대 20%
→ 최종 위치: 60% (축소 방향)
→ 50% 이상 → 5열로 완료
```

**방향 전환 시 최종 위치 기준 50%로 판단**

#### v6 추가: 방향 전환 감지 임계값

```swift
// 방향 전환 감지 시 임계값 적용 (미세한 떨림 무시)
let directionChangeThreshold: CGFloat = 0.05  // 5%

// 이전 프레임 대비 5% 이상 반대 방향으로 움직여야 방향 전환으로 인식
```

### 2.5 회전

#### 핀치와 동일한 원리

| | 핀치 | 회전 |
|---|------|------|
| 앵커 | 핀치 중심점 셀 | 화면 중앙 셀 |
| fadeIn 시작 | 방향 결정 시 | 회전 시작 시 |
| 레이아웃 변화 | 핀치 진행도에 따라 점진적 | 회전 애니메이션과 동기화 |
| 완료 시 | 끊김 없이 snap | 끊김 없이 완료 |

## 3. 구현 방안

### 3.1 핵심 전략 (v6)

**Layout-Aware Anchor 기반 커스텀 `UICollectionViewLayout`**

- 레이아웃이 앵커 정보(assetID, pointInView)를 직접 보유
- `prepare()`에서 앵커 기준으로 모든 셀 위치 계산
- X축은 xOffset으로, Y축은 contentOffset 계산값 제공
- 외부에서 별도 offset 보정 불필요 → 스크롤 점프 원천 차단

### 3.2 아키텍처 구성

#### 1) ContinuousGridLayout (커스텀 레이아웃)

```swift
class ContinuousGridLayout: UICollectionViewLayout {
    // 핀치 상태
    var virtualColumns: CGFloat = 3.0
    var isPinching: Bool = false

    // 앵커 정보 (v6 핵심!)
    var anchorIndexPath: IndexPath?
    var anchorPointInView: CGPoint = .zero  // bounds 좌표

    // xOffset (v6 핵심!)
    var xOffset: CGFloat = 0  // X축 앵커 보정용

    // 계산된 Y offset (외부에서 contentOffset.y에 적용)
    var calculatedContentOffsetY: CGFloat = 0

    // 고정값
    var frozenPaddingCellCount: Int = 0
    var frozenItemCount: Int = 0
}
```

#### 2) PinchZoomState (상태 관리)

```swift
struct PinchZoomState {
    var isActive = false

    // 고정값 (핀치 시작 시 설정, 변경 금지)
    var baseColumns: CGFloat = 3.0
    var targetColumns: GridColumnCount = .three
    var anchorAssetID: String?
    var anchorPointInView: CGPoint = .zero
    var frozenPaddingCellCount: Int = 0
    var frozenItemCount: Int = 0

    // 진행 상태
    var initialDirection: ZoomDirection?
    var didReverse = false
    var hasFiredInitialFadeIn = false

}
```

```

2. anchorAssetID, anchorPointInView 결정
3. frozenPaddingCellCount, frozenItemCount 고정

[.changed 첫 프레임]

6. fadeIn 트리거

[.changed 매 프레임]

11. collectionView.layoutIfNeeded()
   → prepare()에서 앵커 기준 계산
   → xOffset, calculatedContentOffsetY 계산

[.ended]

14. 애니메이션: virtualColumns → target, xOffset → 0
```

### 3.3 레이아웃 계산 원리 (v6)

#### 기본 셀 위치 계산

```swift
// 정상 위치 (xOffset 적용 전)
let normalX = CGFloat(col) * (itemWidth + spacing)
let normalY = CGFloat(row) * (itemHeight + spacing)

// 앵커 기준 보정
if isPinching, let anchorIP = anchorIndexPath {
    let anchorNormalX = CGFloat(anchorCol) * (itemWidth + spacing) + itemWidth/2
    let anchorNormalY = CGFloat(anchorRow) * (itemHeight + spacing) + itemHeight/2

    // xOffset 계산 (앵커가 화면상 같은 X 위치에 고정되도록)
    xOffset = anchorPointInView.x - anchorNormalX

    // Y offset 계산 (contentOffset.y로 적용)
    calculatedContentOffsetY = anchorNormalY - anchorPointInView.y
}

// 최종 셀 위치 (xOffset 적용)
let finalX = normalX + xOffset
let finalY = normalY
```

#### contentSize 계산

```swift
// Y축 contentSize는 정상 계산 (xOffset 무관)
let rowCount = ceil(Double(itemCount) / Double(effectiveColumns))
let contentHeight = CGFloat(rowCount) * itemHeight + CGFloat(rowCount - 1) * spacing

// X축 contentSize도 정상 계산 (xOffset은 시각적 보정일 뿐)
let contentWidth = bounds.width
```

### 3.5 진행도 계산 (문제 4 해결)

```swift
func computeProgress() -> CGFloat {
    // base와 target은 핀치 시작 시 고정된 값 사용!
    let base = pinchState.baseColumns
    let target = CGFloat(pinchState.targetColumns.rawValue)
    let current = layout.virtualColumns

    let range = target - base
    guard abs(range) > 0.001 else { return 0 }

    return (current - base) / range
}
```

**핵심**: `stageBase()`, `stageTarget()` 같은 동적 계산 함수 제거. 고정된 값만 사용.

### 3.6 방향 전환 감지 (문제 4 해결)

```swift
let directionChangeThreshold: CGFloat = 0.05  // 5%

func updateDirectionState() {
    let currentProgress = computeProgress()
    let delta = currentProgress - pinchState.lastProgressForDirectionCheck

    // 5% 이상 반대 방향으로 움직여야 방향 전환 인식
    if pinchState.initialDirection == .zoomOut && delta < -directionChangeThreshold {
        pinchState.didReverse = true
    } else if pinchState.initialDirection == .zoomIn && delta > directionChangeThreshold {
        pinchState.didReverse = true
    }

    pinchState.lastProgressForDirectionCheck = currentProgress
}
```

### 3.7 스냅 로직

```swift
func decideSnapTarget() -> GridColumnCount {
    let progress = computeProgress()

    if pinchState.didReverse {
        // 방향 전환: 50% 규칙
        return progress >= 0.5 ? pinchState.targetColumns : currentGridColumnCount
    } else {
        // 단방향: 10% 규칙
        return progress >= 0.1 ? pinchState.targetColumns : currentGridColumnCount
    }
}
```

## 5. 수정 이력

### v6 수정 사항 (v5 문제 완전 해결)

1. **Layout-Aware Anchor 설계**
   - 레이아웃이 앵커 정보를 직접 보유
   - prepare()에서 앵커 기준 계산
   - 외부 offset 보정 제거 → 스크롤 점프 원천 차단

2. **xOffset 도입**
   - X축 앵커 고정을 위한 레이아웃 내부 offset
   - 손 뗄 때 xOffset → 0 애니메이션

3. **진행도 계산 단순화**
   - 핀치 시작 시 base/target 고정
   - 동적 계산 함수(stageBase, stageTarget) 제거
   - 선형적이고 예측 가능한 진행도

4. **방향 전환 감지 임계값**
   - 5% 이상 반대 방향 이동 시에만 방향 전환 인식
   - 미세한 떨림으로 인한 오감지 방지

5. **가시 영역 확장**
   - 핀치 중 rect 확장 (셀 크기 × 2)
   - clipsToBounds = false (핀치 중에만)

6. **검증 체크리스트 추가**
   - 각 문제별 검증 항목 명시
