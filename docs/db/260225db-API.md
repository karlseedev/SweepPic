## 1. TelemetryDeck Query API

### 1.1 API 개요

| 항목 | 내용 |
|------|------|
| 호스트 | `api.telemetrydeck.com` |
| 인증 | Bearer Token (이메일+비밀번호 → 토큰 발급) |
| 쿼리 언어 | TQL (TelemetryDeck Query Language, Druid 기반) |
| 응답 형식 | JSON |
| 실행 방식 | 비동기 3단계 (제출 → 상태 폴링 → 결과 조회) |

### 1.2 인증 흐름

```
1. 토큰 발급
   POST https://api.telemetrydeck.com/api/v3/users/login
   Authorization: Basic <base64(email:password)>
   → { "value": "<bearer_token>", "expiresAt": "..." }

2. API 호출 시
   Authorization: Bearer <bearer_token>
```

### 1.3 쿼리 실행 3단계

```
Step 1: 쿼리 제출
  POST /api/v3/query/calculate-async/
  Body: TQL 쿼리 (JSON)
  → { "queryTaskID": "55b3487da8018369" }

Step 2: 상태 확인 (폴링)
  GET /api/v3/task/{taskID}/status/
  → { "status": "running" } 또는 { "status": "successful" }

Step 3: 결과 조회
  GET /api/v3/task/{taskID}/value/
  → { "calculationDuration": 0.218, "result": { "rows": [...] } }
```

동기 엔드포인트 (`POST /api/v3/query/calculate/`)도 사용 가능 (빠른 테스트, 단건 조회).

### 1.4 지원 쿼리 타입

| 쿼리 타입 | 용도 | 활용 예시 |
|----------|------|----------|
| **timeseries** | 시간별 추이 | 일별 앱 실행 수, 주간 삭제 추이 |
| **topN** | 상위 N개 값 | 가장 많은 오류 TOP 5 |
| **groupBy** | 차원별 그룹화 | OS 버전별 사진 열람, 정리 방식별 비율 |
| **scan** | 개별 이벤트 목록 | 특정 날짜 정리 이벤트 전체 |
| **funnel** | 퍼널 분석 | 정리 도달 단계별 이탈률 |
| **retention** | 리텐션 분석 | 주간 재방문율 |

### 1.5 실측 메모 (쿼리 작성 시 참고)

**필수 설정:**

| 항목 | 값 | 비고 |
|------|-----|------|
| dataSource | `"com.simon"` | 조직 namespace. `"telemetry-signals"` 사용하면 0건 반환 |
| appID 필터 | `.env`의 `TELEMETRYDECK_APP_ID` | 필터에 포함 필수 |
| aggregation type (건수) | `"count"` | Druid 행 수. ~~`eventCount`~~는 null 반환하므로 사용 금지 |
| aggregation type (합산) | `"longSum"` | 파라미터 값(문자열 숫자) 세션 간 합산 |

**isTestMode 분리:**

TelemetryDeck SDK가 빌드 환경에 따라 자동 설정:
- Xcode 디버그 빌드 / 시뮬레이터 → `isTestMode = "true"`
- 릴리스(앱스토어) 빌드 → `isTestMode = "false"`

```bash
./td-report.sh --test-mode   # 디버그 데이터 조회
./td-report.sh               # 릴리스 데이터만 (기본)
```

**시그널 이름 규칙:**

코드에서 `sendEvent("app.launched")` → 저장 시 **`PickPhoto.app.launched`** (앱 이름 접두사 자동 추가)

쿼리 필터에는 접두사 포함 전체 이름 사용:
```json
{ "type": "selector", "dimension": "type", "value": "PickPhoto.app.launched" }
```

**확인된 시그널 (2026-02-14 기준):**

| signalType | 출처 |
|---|---|
| PickPhoto.app.launched | 이벤트 1 |
| PickPhoto.session.photoViewing | 이벤트 3 |
| PickPhoto.session.deleteRestore | 이벤트 4-1 |
| PickPhoto.session.trashViewer | 이벤트 4-2 |
| PickPhoto.session.similarAnalysis | 이벤트 5-1 |
| PickPhoto.similar.groupClosed | 이벤트 5-2 |
| PickPhoto.cleanup.completed | 이벤트 7 |
| TelemetryDeck.Session.started | SDK 자동 |
| TelemetryDeck.Acquisition.newInstallDetected | SDK 자동 |

**타임스탬프:**

서버 응답 타임스탬프는 **UTC+0**. KST 변환 시 +9시간.
세션 시그널 타임스탬프 = **백그라운드 진입(flush) 시점**. 개별 액션 시점이 아님.

**longSum aggregation:**

TelemetryDeck SDK는 파라미터를 **문자열**로 전송하지만, `longSum` aggregation이 숫자 파싱/합산 정상 작동:
```json
{ "type": "longSum", "name": "viewerTrashButton", "fieldName": "viewerTrashButton" }
```
- `count`는 시그널 건수(세션 수), `longSum`은 파라미터 값 합산 — 용도 구분 필수

### 1.6 삽질 기록

- `"dataSource": "telemetry-signals"` → 모든 쿼리 0건. namespace 모드 조직은 `"com.simon"` 필수
- `eventCount` vs `count` 혼동 주의: `eventCount`=원본 이벤트 수(rollup 전)이지만 **null 반환**. `count`=Druid 행 수로 정상 작동
- isTestMode 필터 빠뜨리면 디버그+릴리스 합산됨
- dimension을 문자열로 넣으면 에러 → `{"type": "default", "dimension": "...", "outputName": "..."}` 객체 형태 필요
- 수집 지연: 초기 테스트 시 2~6시간 지연 관찰. 이후 정상화됨 (수분 내 도착)

## 2. TD 스크립트

### 2.1 파일 구조

```
scripts/analytics/
├── .env.example           # 환경변수 템플릿 (git 추적)
├── .env                   # 실제 credentials (git 제외)
├── td-auth.sh             # 토큰 발급 + 캐싱
├── td-query.sh            # 비동기 3단계 쿼리 실행 (범용)
├── td-report.sh           # 주간 리포트 (전체 시그널 일괄 조회)
└── queries/               # 시그널별 TQL 쿼리 템플릿
    ├── app-launched.json
    ├── permission.json
    ├── photo-viewing.json
    ├── delete-restore.json
    ├── trash-viewer.json
    ├── similar-analysis.json
    ├── similar-group.json
    ├── errors.json
    ├── cleanup.json
    └── preview-cleanup.json
```

### 2.2 Credential 관리

**.env 파일:**
```bash
# TelemetryDeck
TELEMETRYDECK_EMAIL=user@example.com
TELEMETRYDECK_PASSWORD=****
TELEMETRYDECK_APP_ID=<App ID>
```

- `.env`는 절대 git에 커밋하지 않음
- Bearer Token은 `/tmp/td-token.json`에 캐싱 (만료 시간 기반 자동 갱신)

### 2.3 스크립트별 동작

#### td-auth.sh — 토큰 발급 + 캐싱 (~40줄)

```
1. /tmp/td-token.json 확인
   ├─ 존재 + 만료 전 → 캐시된 토큰 출력
   └─ 없음 or 만료 → 2로
2. .env에서 credentials 로드
3. POST /api/v3/users/login (Basic Auth)
4. 응답 → /tmp/td-token.json에 캐싱
5. 토큰 값 stdout 출력
```

#### td-query.sh — 비동기 3단계 쿼리 (~90줄)

```bash
# 파일에서 쿼리 읽기
./td-query.sh queries/app-launched.json
# stdin으로 쿼리 전달
echo '{"queryType":"timeseries",...}' | ./td-query.sh
# 기간 오버라이드
./td-query.sh queries/app-launched.json --days 30
# 테스트 모드
./td-query.sh queries/app-launched.json --test-mode
```

자동 주입: `--days` → `relativeIntervals`, appID 필터, isTestMode 필터

#### td-report.sh — 주간 리포트 (~60줄)

```bash
./td-report.sh              # 기본 7일
./td-report.sh --days 30    # 30일
./td-report.sh --test-mode  # 테스트 데이터 포함
```

queries/ 디렉토리의 10개 쿼리를 순차 실행 → 마크다운 리포트 stdout 출력.

### 2.4 쿼리 템플릿 (10개)

| 파일 | queryType | 시그널 | aggregation | 핵심 데이터 |
|------|-----------|--------|------------|-----------|
| `app-launched.json` | timeseries | `app.launched` | count | 일별 실행 수 |
| `permission.json` | groupBy | `permission.result` | count | result×timing별 건수 |
| `photo-viewing.json` | timeseries | `session.photoViewing` | count + longSum | 세션 수 + 총 열람 수 |
| `delete-restore.json` | timeseries | `session.deleteRestore` | count + longSum | 세션 수 + 삭제/복구 합산 |
| `trash-viewer.json` | timeseries | `session.trashViewer` | count + longSum | 세션 수 + 완전삭제/복구 합산 |
| `similar-analysis.json` | timeseries | `session.similarAnalysis` | count + longSum | 세션 수 + 완료/취소/그룹 합산 |
| `similar-group.json` | groupBy | `similar.groupClosed` | count | deletedCount 분포 |
| `errors.json` | timeseries | `session.errors` | count + longSum×13 | 에러 항목별 합산 |
| `cleanup.json` | groupBy | `cleanup.completed` | count | reachedStage별 퍼널 |
| `preview-cleanup.json` | groupBy | `cleanup.previewCompleted` | count | finalAction별 분포 |

## 3. Supabase Query

### 3.1 PostgREST REST API

Supabase는 PostgREST를 통해 표준 REST API로 데이터를 조회할 수 있다.

**기본 패턴:**
```bash
# service_role key 사용 (RLS 우회하여 SELECT 가능)
# anon key는 INSERT-only RLS이므로 조회 불가
curl -s "$SUPABASE_URL/rest/v1/events?select=*&limit=10" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY"
```

**필터 예시:**
```bash
# 특정 이벤트만 조회
?event_name=eq.session.deleteRestore

# 날짜 범위
?created_at=gte.2026-02-20

# 프로덕션 데이터만 (디버그 제외)
?is_test=eq.false

# 구독 tier별 필터
?subscription_tier=eq.free

# 특정 유저의 이벤트 (device_id 기반)
?device_id=eq.C8735166-FF88-467C-9945-58428C2919DD

# BM 이벤트만
?event_name=like.bm.*

# 복합 조건
?event_name=eq.session.errors&created_at=gte.2026-02-20&is_test=eq.false&limit=20&order=created_at.desc
```

**RPC 함수 호출:**
```bash
# 일별 이벤트 요약
curl -s "$SUPABASE_URL/rest/v1/rpc/daily_summary" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"p_days": 7}'

# BM 게이트 퍼널 요약
curl -s "$SUPABASE_URL/rest/v1/rpc/gate_funnel_summary" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"p_days": 30}'

# tier별 이벤트 요약
curl -s "$SUPABASE_URL/rest/v1/rpc/tier_summary" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"p_days": 30}'

# 유저 단위 게이트→구독 전환율
curl -s "$SUPABASE_URL/rest/v1/rpc/gate_conversion_rate" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"p_days": 90}'

# DAU (일별 고유 사용자 수)
curl -s "$SUPABASE_URL/rest/v1/rpc/daily_active_users" \
  -H "apikey: $SUPABASE_SERVICE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"p_days": 30}'
```

### 3.2 스크립트

#### sb-query.sh — PostgREST 조회 + RPC 호출 (~80줄)

```bash
# 테이블 직접 조회
./sb-query.sh --table events --filter "event_name=eq.session.errors" --limit 20

# RPC 함수 호출
./sb-query.sh --rpc daily_summary '{"p_days": 30}'
./sb-query.sh --rpc delete_restore_summary '{"p_days": 7}'

# 최근 N건
./sb-query.sh --table events --limit 5 --order "created_at.desc"
```

#### sb-report.sh — 리포트 생성 (~40줄)

```bash
./sb-report.sh --days 7    # 7일 요약 리포트
./sb-report.sh --days 30   # 30일 요약 리포트
```

내부적으로 `daily_summary` RPC를 호출하여 마크다운 리포트 생성.

### 3.3 Credential 관리

**.env 파일 (TD 변수 + Supabase 변수):**
```bash
# TelemetryDeck
TELEMETRYDECK_EMAIL=
TELEMETRYDECK_PASSWORD=
TELEMETRYDECK_APP_ID=

# Supabase
SUPABASE_URL=https://xxx.supabase.co
SUPABASE_ANON_KEY=sb_publishable_xxx
SUPABASE_SERVICE_KEY=sb_secret_xxx
```

**sb-query.sh는 `SUPABASE_SERVICE_KEY` (service_role key) 사용:**
- service_role key는 RLS를 우회하여 SELECT 가능
- anon key는 INSERT-only RLS이므로 조회 불가

### 3.4 service_role key 보안

- `.env` 파일에만 보관 (git 제외)
- 앱 코드(Swift)에 **절대 포함 금지**
- 서버 측 스크립트/Claude Code에서만 사용
- Supabase Dashboard에서 명칭: **Secret Key**

### 3.5 사용 예시

```bash
# 일별 이벤트 요약 (지난 7일)
./scripts/analytics/sb-query.sh --rpc daily_summary '{"p_days": 7}'

# 삭제/복원 상세 (지난 30일)
./scripts/analytics/sb-query.sh --rpc delete_restore_summary '{"p_days": 30}'

# 특정 이벤트 최근 20건 (params 포함)
./scripts/analytics/sb-query.sh --table events \
```

## 5. 주의사항

### 5.1 TelemetryDeck

| 항목 | 상세 |
|------|------|
| API 접근 | 현재 무료 사용 가능. **Tier 2 이상 유료 플랜에서 공식 지원** 예고 |
| Bearer Token | 만료 시간 확인 필수 (보통 24시간~). `/tmp/td-token.json`에 캐싱 |
| BSD date | macOS `date`와 GNU `date` 문법 차이 → td-auth.sh에서 호환 코드 필요 |
| isTestMode | 개발 중엔 `--test-mode`, 프로덕션에선 기본 제외 |
| longSum + 문자열 | SDK가 `String(count)`로 보낸 값이 Druid `longSum`으로 합산 가능 (실측 검증 완료) |

### 5.2 Supabase

| 항목 | 상세 |
|------|------|
| is_test 분리 | `#if DEBUG` → `is_test=true`, Release → `false`. 조회 시 `?is_test=eq.false`로 프로덕션만 필터 |
| 무료 티어 pause | 일정 기간 비활성 시 자동 일시정지. Dashboard에서 수동 Resume 필요 |
| pg_cron | pause 중에는 중단됨. DAU 1,000+ 이후 설정 |
| service_role key | RLS 우회 — 스크립트/Claude Code 전용. 앱 코드 포함 금지. `.env`에만 보관 |
| subscription_tier | `free` 또는 `plus`. BM 이전 데이터는 NULL. 앱 시작 직후 이벤트는 SubscriptionStore 초기화 전이라 `free`로 기록될 수 있음 |
| device_id | IDFV UUID. BM 이전 데이터는 NULL. 기기 잠금 상태 시작 시 `unknown` (극히 드묾) |
| 500MB 한도 | 90일 보존 기준 DAU 1,000 → ~500MB (BM 9종 포함). 한도 접근 시 purge_old_events 또는 pg_cron 설정 |

### 5.3 의존성

| 도구 | 용도 | macOS 기본 |
|------|------|-----------|
| `curl` | HTTP 요청 | ✅ |
| `jq` | JSON 파싱 | ❌ (`brew install jq`) |
| `date` | 만료 시간 비교 | ✅ (BSD date) |
| `base64` | Basic Auth 인코딩 | ✅ |

> `jq`만 추가 설치 필요. 없으면 스크립트가 안내 메시지 출력.

### 5.4 미검증 항목

- **파라미터 키의 점(.) 포함 이름**: `"photoLoad.gridThumbnail"` 같은 키가 TQL dimension/fieldName으로 정상 작동하는지 데이터 축적 후 확인 필요. 문제 시 밑줄(`_`)로 변경
- **Insight API**: 대시보드에서 저장한 쿼리를 `POST /api/v3/insights/{insightID}/query/`로 가져와 실행 가능 — 활용 시 검증 필요
