## 1. 전략적 프레임워크

### 1.1 현재 상태

PickPhoto는 현재 **완전히 로컬 기반** 앱이다.

| 항목 | 상태 |
|------|------|
| 네트워크/서버 통신 | 없음 |
| Analytics / 분석 | 없음 |
| 크래시 리포팅 | 없음 |
| 사용자 데이터 수집 | 없음 |
| 로깅 | 로컬만 (`Log.swift`, `FileLogger.swift`) |

### 1.2 North Star Metric

> **주간 정리 사진 수 (Weekly Photos Organized)**

이 지표가 상승하면:
- 사용자가 앱의 핵심 가치를 체감하고 있다는 증거
- 리텐션이 자연스럽게 따라올 가능성이 높음
- 저장 공간 확보라는 실질적 결과와 직결
- 구독 전환의 선행 지표 역할

### 1.3 Aha Moment

> **"첫 세션에서 20~50장 정리 + 확보된 저장 공간 확인"**

- Facebook의 Aha Moment = "7일 내 10명 친구 추가"
- PickPhoto의 Aha Moment = 첫 정리 세션에서 가치 체감
- 이 순간까지의 **Time to Value**를 줄이는 것이 가장 높은 ROI

### 1.4 핵심 퍼널

```
앱 설치 → 권한 허용 → 첫 그리드 로드 → 첫 사진 열기 → 첫 스와이프 삭제 → "OO장 정리 완료" → D7 재방문
```

### 1.5 AARRR 지표 체계

| 단계 | 의미 | PickPhoto 핵심 지표 | 선행/후행 |
|------|------|---------------------|----------|
| **Acquisition** | 사용자 획득 | 설치 수, 스토어 전환율 | 후행 |
| **Activation** | 첫 가치 경험 | 첫 삭제까지 시간, 권한 부여율, 온보딩 완료율 | **선행** |
| **Retention** | 재방문 | D1/D7/D30, 세션당 삭제 수 | **선행**/후행 |
| **Revenue** | 수익화 | 프리미엄 전환율, ARPU, LTV | 후행 |
| **Referral** | 추천 | 공유 횟수, 초대 전환율 | 후행 |

### 1.6 리텐션 벤치마크 (2025 기준)

| 지표 | iOS 앱 평균 | 양호 | 우수 |

## 2. 경쟁사 분석

### 2.1 경쟁앱 데이터 수집 현황

| 앱 | 프라이버시 라벨 | 사진 서버 전송 | 수집 항목 | 비고 |
|----|--------------|-------------|----------|------|
| **Gemini (CleanMyPhone)** | Not Linked to Identity | 없음 | 사용 빈도, 기능 사용, 크래시 | 구독 모델 |
| **Slidebox** | Not Linked to Identity | 없음 | 익명 사용 패턴 | 프라이버시 강조 마케팅 |
| **Google Photos** | Linked + Tracking | 있음 (클라우드) | 위치, 콘텐츠, 얼굴, 검색 | AI 차별화 |
| **Cleanup** | Linked + Tracking | 불명 | 추적 포함 다수 | 광고+구독 혼합 |
| **FlicPic** | **Data Not Collected** | 없음 | 없음 | 가장 깨끗 |

### 2.2 핵심 시사점

- 인디 앱일수록 프라이버시 라벨의 영향이 **더 크다**
- "추적 데이터 수집" 앱은 다운로드 **2.2배 더 하락** (UNSW 연구)
- **94%의 사용자**가 프라이버시 라벨이 다운로드 결정에 영향 (9to5Mac 설문)
- **"Data Not Collected" 라벨** 자체가 경쟁 우위이자 마케팅 자산

### 2.3 인디 앱 성공 사례

| 앱 | 분석 도구 | 핵심 성과 |
|----|----------|----------|
| **Blackbox** (퍼즐, Apple Design Award) | TelemetryDeck | 퍼널 분석으로 온보딩 이탈 지점 발견 → 순서 조정으로 개선 |
| **InTolerApp** (알레르기) | TelemetryDeck | 퍼널 시각화 → 정보 순서 재배치 → **이탈률 22% 감소** |
| **Widgetsmith** (위젯, 1.3억 DL) | RevenueCat | 구독 전환 퍼널 실시간 모니터링으로 5년간 지속 성장 |

**공통 패턴:**
- 퍼널 분석이 가장 ROI 높음
- 핵심 3~5개 지표에 집중
- 프라이버시 = 경쟁 우위

## 3. 솔루션 선정

### 3.4 왜 Mixpanel/Amplitude가 아닌가?

| 비교 항목 | TelemetryDeck | Mixpanel | Amplitude |
|----------|--------------|----------|-----------|
| 프라이버시 라벨 | Not Linked | Linked 가능 | Linked 가능 |
| ATT 팝업 필요 | **불필요** | 조건부 | 조건부 |
| GDPR 동의 배너 | **조건부 불필요** (8.2절 참조) | 필요할 수 있음 | 필요할 수 있음 |
| 무료 한도 | 100K/월 | 1M/월 (2026-02 기준, 변동 가능) | 제한적 |
| DAU 10K 비용 | **29유로/월** | ~$168/월 | $49+/월 |
| DAU 100K 비용 | **99유로/월** | ~$3,948/월 | 커스텀 |
| SDK 크기 | 경량 | 중간 | 큼 |
| 사용자 식별 | 불가 (의도적) | 가능 | 가능 |

→ 사진 앱에서 프라이버시가 핵심이고, 장기 비용 효율성이 중요하므로 TelemetryDeck이 최적

### 5.2 Protocol 기반 추상화

```swift
// Sources/AppCore/Services/Analytics/AnalyticsProvider.swift

/// 분석 이벤트
public struct AnalyticsEvent {
    public let name: String
    public let parameters: [String: String]
}

/// 분석 제공자 프로토콜 — 도구 교체 시 이것만 새로 구현
public protocol AnalyticsProvider {
    var name: String { get }
    func configure()
    func track(event: AnalyticsEvent)
    func setUserProperty(key: String, value: String)
    func flush()
}
```

### 5.3 AnalyticsManager 핵심 구조

```swift
// Sources/AppCore/Services/Analytics/AnalyticsManager.swift

public final class AnalyticsManager {
    public static let shared = AnalyticsManager()

    private var providers: [AnalyticsProvider] = []
    private var offlineQueue: [AnalyticsEvent] = []   // 오프라인 큐 (최대 500)

    // 제공자 등록 (AppDelegate에서 호출)
    public func register(provider: AnalyticsProvider)

    // 이벤트 전송 (opt-out 체크 → 디버그 출력 → 제공자 라우팅)
    public func track(_ eventName: String, parameters: [String: String] = [:])

    // 백그라운드 진입 시 보류 이벤트 즉시 전송
    public func flush()
}
```

### 5.4 TelemetryDeck Provider

```swift
// PickPhoto/PickPhoto/Services/TelemetryDeckProvider.swift

import TelemetryDeck

final class TelemetryDeckProvider: AnalyticsProvider {
    let name = "TelemetryDeck"
    private let appID: String

    func configure() {
        let config = TelemetryDeck.Config(appID: appID)
        config.defaultSignalPrefix = "PickPhoto."
        config.defaultParameters = {[
            "photoLibraryBucket": PhotoLibraryService.shared.sizeBucket
        ]}
        TelemetryDeck.initialize(config: config)
    }

    func track(event: AnalyticsEvent) {
        TelemetryDeck.signal(event.name, parameters: event.parameters)
    }
}
```

### 5.5 AppDelegate 초기화

```swift
// AppDelegate.swift

func application(_:didFinishLaunchingWithOptions:) -> Bool {
    // 1. Firebase 먼저 (크래시 핸들러가 빠를수록 좋음)
    FirebaseApp.configure()

    // 2. Analytics 제공자 등록
    AnalyticsManager.shared.register(provider: TelemetryDeckProvider(appID: "APP-ID"))
    AnalyticsManager.shared.register(provider: CrashlyticsProvider())

    // 3. 앱 실행 이벤트
    AnalyticsManager.shared.track("App.Session.started")

    return true
}
```

### 5.7 Opt-out 지원

```swift
struct AnalyticsSettings {
    // 기본값 true (opt-out 방식)
    static var isEnabled: Bool {
        get { UserDefaults.standard.object(forKey: "analytics_enabled") as? Bool ?? true }
        set { UserDefaults.standard.set(newValue, forKey: "analytics_enabled") }
    }
}

// AnalyticsManager.track() 내부에서:
guard AnalyticsSettings.isEnabled else { return }
```

## 7. 비용 시뮬레이션

### 7.1 사용자당 시그널 추정

**활성 사용자 (DAU) 기준:**

| 항목 | 수치 |
|------|------|
| 세션당 평균 이벤트 | ~5~10개 (Phase 1 이벤트 8종 기준) |

| **DAU 사용자 일간 시그널** | **~10~20개** |
| **DAU 사용자 월간 시그널** | **~300~600개** |

**비활성 MAU (월 1~3회 사용) 기준:**

| 항목 | 수치 |
|------|------|
| 월간 세션 수 | ~2~3회 |
| **비활성 MAU 월간 시그널** | **~15~30개** |

**블렌딩 추정 (DAU/MAU 비율 1:3 가정):**

| 사용자 유형 | 비율 | 월간 시그널 | 가중 기여 |
|-----------|------|-----------|----------|
| 활성 (DAU) | 1/3 | ~450 | ~150 |
| 비활성 | 2/3 | ~20 | ~13 |
| **MAU 평균** | — | — | **~100** |

> ⚠️ TelemetryDeck 공식 FAQ는 "사용자당 월 약 30 시그널"로 안내하나, 이는 이벤트가 적은 단순 앱 기준이다. PickPhoto는 Phase 1에서 8종 이벤트를 추적하므로 **MAU 평균 ~100 시그널/월**을 비용 산정 기준으로 사용한다.

### 8.2 ATT / GDPR 대응

| 규제 | 대응 |
|------|------|
| **ATT (App Tracking Transparency)** | **팝업 불필요** — 크로스 앱 추적 안 함 |
| **GDPR** | **조건부** — TelemetryDeck 단독 사용 시 동의 배너 불필요 (공식 FAQ 근거: 개인 비식별, IP 미수집, EU 서버). 단, Firebase Crashlytics 병행 시 Google 데이터 처리 약관 고려하여 **법률 검토 권장** |
| **Apple Privacy Manifest** | TelemetryDeck SDK에 PrivacyInfo.xcprivacy **내장** |

### 8.3 사진 앱 특수 규칙

| 규칙 | 내용 |
|------|------|
| 사진 메타데이터 (위치, 얼굴) | **절대 서버 전송 금지** |
| 라이브러리 크기 | 구간(bucket)으로만 전송 (1K/5K/10K/50K+) |
| 얼굴 인식 데이터 | 100% 온디바이스 처리, 전송 없음 |
| 개별 사진 정보 | 어떤 형태로도 전송 금지 |

### 8.4 개인정보 처리방침 필수 항목

App Store 제출 시 개인정보 처리방침 URL 필수. 포함 내용:

```
1. 수집하는 정보
   - 비식별 사용 데이터: 기능 사용 빈도, 세션 시간
   - 기기 정보: OS 버전, 기기 모델, 화면 크기
   - 성능 데이터: 오류 횟수, 앱 충돌 정보
   - 사진 라이브러리 규모: 구간 정보만 (정확한 수 아님)
   ※ 사진 내용, 얼굴, 위치, 개인 식별 정보 수집하지 않음

2. 데이터 처리 도구
   - TelemetryDeck (독일 소재, GDPR 준수): 비식별 앱 사용 분석
   - Firebase Crashlytics (Google): 앱 충돌 분석

3. 사용자 권리
   - 설정 > 분석 데이터 공유에서 비활성화 가능
```

### 8.5 Opt-out 제공

강제 동의 배너 대신, 앱 설정 화면에 토글 배치:

```
[설정]
  └─ 분석 데이터 공유  [ON/OFF]
     "앱 개선을 위해 익명 사용 데이터를 공유합니다.
      사진 내용은 전송되지 않습니다."
```

## 9. 데이터 → 행동 변환 프레임워크

### 9.1 의사결정 시나리오

#### 시나리오 A: 스와이프 삭제 사용률 < 20%

```
[심층 분석]
├─ 온보딩에서 기능을 안내했는가?
│   └─ NO → 온보딩에 스와이프 튜토리얼 추가
├─ 기능을 발견했지만 사용하지 않는가?
│   └─ YES → 사용자 인터뷰 (제스처가 어려운가?)
└─ 삭제 후 복구율이 높은가?
    └─ YES → Undo UI를 더 명확하게 개선
```

#### 시나리오 B: Day 7 리텐션 < 15%

```
[코호트 분석]
├─ Activation 완료 그룹 vs 미완료 그룹 비교
│   └─ 완료 그룹 D7이 훨씬 높다면 → Activation 전환율 개선에 집중
├─ 사진 수 적은 사용자(<100장) 이탈률 높다면
│   └─ 소규모 라이브러리 사용자를 위한 가치 제안 재설계
└─ Day 3→4 급락?
    └─ 반복 사용 이유 부재 → "오늘의 정리 제안" 기능 검토
```

#### 시나리오 C: 유사 사진 열지만 정리 미완료 (완료율 < 30%)

```
[퍼널 분석]
├─ 그룹 열자마자 닫음 → UI 복잡 또는 로딩 느림
├─ 1~2장 선택 후 중단 → 어떤 사진을 삭제할지 결정 어려움
└─ 삭제 버튼까지 갔지만 취소 → 영구 삭제 불안감
    └─ A/B 테스트: AI 자동 추천 삭제 vs 수동 선택
```

### 9.2 행동 변환 체크리스트

데이터 분석 후 행동으로 옮기기 전 검증:

```
[ ] 데이터가 충분한가? (통계적으로 유의미한 표본 크기)
[ ] 상관관계가 아닌 인과관계인가? (A/B 테스트 또는 정성 데이터로 확인)
[ ] 하나의 변수만 변경하는가?
[ ] 성공 기준이 사전에 정의되어 있는가?
[ ] 롤백 계획이 있는가? (Feature Flag)
```

### 9.3 분석 주기

| 주기 | 활동 | 대상 지표 |
|------|------|----------|
| **매일** | 대시보드 확인, 이상 징후 탐지 | DAU, 크래시율, 핵심 이벤트 수 |
| **매주** | Leading 지표 트렌드 분석 | 세션당 삭제 수, 기능별 사용률 |
| **격주** | 사용자 인터뷰 1~2건 | 정성적 인사이트 |
| **매월** | Lagging 지표 리뷰, 실험 결과 정리 | 리텐션, MRR, NPS |
| **분기** | 전략 리뷰, AARRR 전체 점검 | 전 단계 종합 |

### 9.4 A/B 테스트 가이드

**최소 표본 크기 (기본 전환율 10% 기준):**

| 감지 효과 (MDE) | 필요한 총 사용자 |
|----------------|----------------|
| 5% (작은 변화) | ~6,000명 |
| 10% (중간 변화) | ~3,000명 |
| 20% (큰 변화) | ~800명 |

**소규모 앱 현실적 대안:**
- MDE를 15~20%로 높여 적은 사용자로 테스트
- 테스트 기간 4~6주로 연장하여 누적 표본 확보
- TelemetryDeck 자체 Experiment 기능 활용 (코호트 + z-score/p-value)

**점진적 롤아웃:**

```
내부 테스터 (0.1%) → 얼리 어답터 (5%) → 확대 (25%) → 대다수 (75%) → 전체 (100%)
```

### 9.5 정성적 피드백 수집

| 방법 | 타이밍 | 빈도 |
|------|--------|------|
| **마이크로 서베이** | 유사 사진 정리 완료 직후 | 90일에 1회 |
| **NPS** | "이 앱을 추천하시겠습니까?" | 분기 1회 |
| **App Store 리뷰 분석** | 50개 이상 쌓이면 키워드 분석 시작 | 매주 |
| **사용자 인터뷰** | 인앱 배너로 모집 | 격주 2~3명 |
