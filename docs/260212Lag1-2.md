## v1 실패 분석: 왜 VisionExecutor 단독으로 효과가 없었나

### v1 전제 (틀렸음)
> "동기 Vision 호출 3곳만 MainActor를 블로킹하고, 나머지는 await 지점에서 UI 갱신 가능"

### 실제 동작
`SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` 설정으로 **분석 루프 전체가 MainActor에서 실행**.
Vision 호출을 백그라운드로 옮겨도 **루프 자체가 MainActor를 계속 점유**하여 UI 갱신 불가.

```
[MainActor]  분석(사진1) → Vision(백그라운드) → resume → 분석(사진2) → Vision → resume → ...
                                                ↑
                                    continuation이 UI보다 먼저 스케줄됨
                                    → RunLoop가 화면을 렌더링할 틈 없음
```

Swift 런타임의 MainActor cooperative executor는 **continuation 재개를 UI 이벤트보다 우선** 처리.
100장 배치 동안 `Task.yield()` 없이 연속 실행 → **8-20초 UI 프리즈**.

### 핵심 깨달음
VisionExecutor로 개별 호출을 백그라운드로 보내도, **루프의 다음 iteration이 즉시 MainActor를 다시 점유**.
`await` 지점이 있어도 executor가 분석 continuation을 우선 재개하므로 UI 갱신 기회가 없음.

**→ 루프 자체를 MainActor에서 분리해야 함.**

## 왜 `nonisolated`만 해결책인가

### `Task.detached` / 전용 Actor가 안 되는 이유

```swift
Task.detached {
    await service.analyze(...)  // analyze()가 @MainActor → 다시 MainActor로 hop
}
```

어떤 래퍼를 써도 **호출되는 함수 자체가 @MainActor면 MainActor로 hop**함.
유일한 해법: **호출되는 함수를 `nonisolated`로 선언**하여 cooperative pool에서 실행.

### `nonisolated` 적용 시 동작 변화

```
[Before — 현재]
[MainActor]  루프 전체 실행 (수초~수십초 연속 점유)
             └─ 사진1 분석 (~80ms) → 사진2 분석 → ... → 100장 후 yield

[After — nonisolated]
[Coop Pool]  루프 실행 → QA.analyze(사진1) 호출 → suspend
[MainActor]  ← FREE (UI 업데이트 가능) ←
[MainActor]  QA.analyze(사진1) 실행 (~13ms) → 반환
[Coop Pool]  resume → progress → QA.analyze(사진2) 호출 → suspend
[MainActor]  ← FREE (UI 업데이트 가능) ←
[MainActor]  QA.analyze(사진2) 실행 (~13ms) → 반환
...
```

MainActor 점유: **연속 수초 → 사진당 ~13ms hop**으로 분산.
hop 사이에 MainActor가 해방되어 **UI 렌더링 가능**.

### Swift 5 모드의 이점

`SWIFT_VERSION = 5.0`이므로:
- `nonisolated` 메서드에서 @MainActor stored property 접근 = **경고 (에러 아님)**
- 실제 thread safety는 이미 NSLock으로 보호되어 있으므로 안전
- Swift 6 모드였다면 컴파일 에러로 대규모 리팩토링 필요했음

## 수정 대상

### 1. `CleanupPreviewService.analyze()` — `nonisolated` 전환

**파일**: `CleanupPreviewService.swift:121`

```swift
// Before
func analyze(
    method: CleanupMethod,
    progressHandler: @escaping (CleanupProgress) -> Void
) async throws -> PreviewResult {

// After
nonisolated func analyze(
    method: CleanupMethod,
    progressHandler: @escaping @Sendable (CleanupProgress) -> Void
) async throws -> PreviewResult {
```

**analyze() 내부에서 접근하는 stored property:**

| 속성 | 종류 | nonisolated 접근 | 안전성 |
|------|------|-----------------|--------|
| `path1AgreeThreshold` 등 5개 | `private let` (Float/Int/CGFloat) | 경고 | 불변, 안전 |
| `maxScanCount` | `private let Int` | 경고 | 불변, 안전 |
| `extremeAspectRatioThreshold` | `private let CGFloat` | 경고 | 불변, 안전 |
| `lock` | `private let NSLock` | 경고 | NSLock은 Sendable, 안전 |
| `isCancelled` | `private var Bool` (lock 보호) | 경고 | `cancelled` 계산 프로퍼티가 NSLock으로 보호, 안전 |

모든 접근이 **경고**(Swift 5 모드)이며 실제 thread safety 문제 없음.

**analyze() 내부에서 호출하는 @MainActor 메서드:**

| 호출 | 동작 | MainActor hop 시간 |

| `Self.clearSession()` | 짧은 hop | < 1ms |

| `QualityAnalyzer.shared.analyze(asset)` | 분석 실행 hop | ~13ms (VisionExecutor 적용 후) |
| `CleanupImageLoader.shared.loadImage(for:)` | async hop | ~10-50ms (async 해제) |

| 리스크 | 심각도 | 대응 |
|--------|--------|------|

## 실패 기록

**구현 커밋**: `217dd7e`
**롤백 포인트**: `c3afd55`
**테스트 결과**: 전혀 차이 없음 — UI 프리즈 동일

### 실패 원인 분석

v2의 핵심 전제는 "`nonisolated`로 분석 루프를 cooperative pool로 이동시키면, @MainActor 메서드 호출이 짧은 hop이 되어 hop 사이에 UI 갱신 가능"이었다.

**가능한 실패 원인:**

1. **`nonisolated`가 실제로 cooperative pool에서 실행되지 않음**
   - `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` + `SWIFT_VERSION = 5.0` 조합에서 `nonisolated` 명시가 런타임에 실제로 cooperative pool로 이동시키는지 불확실
   - Swift 5 모드에서 `nonisolated`은 컴파일러 경고/에러 제어일 뿐, 런타임 executor 선택에 영향을 안 줄 수 있음

2. **호출되는 @MainActor 메서드들이 hop이 아닌 연속 실행**
   - `QualityAnalyzer.shared.analyze()`, `CleanupImageLoader`, `AestheticsAnalyzer` 등 분석 루프 내 거의 모든 호출이 @MainActor
   - 사진당 hop 횟수가 많아 cooperative executor가 사실상 MainActor에서 대부분 시간을 보냄
   - hop 사이의 "해방" 시간이 UI 렌더링에 충분하지 않을 수 있음

3. **cooperative executor 스케줄링 한계**
   - cooperative pool → MainActor hop 시, executor가 UI 이벤트보다 hop을 우선 처리할 수 있음
   - v1과 동일한 문제: await 지점이 있어도 executor가 분석 continuation을 우선 재개

**결론**: `nonisolated` + VisionExecutor 조합으로는 불충분. **분석 루프 자체를 GCD/OperationQueue 등 cooperative executor 외부에서 실행**하거나, **MainActor 의존성 자체를 제거**하는 근본적 접근이 필요할 수 있음.

## v3: 근본 원인 재특정 (2026-02-13)

### v1/v2가 실패한 진짜 이유: 랙 시점을 잘못 파악

v1, v2 모두 **분석 루프(analyze)** 를 최적화했다. 그러나 실제 랙은 **분석이 시작되기 한참 전, 정리 버튼을 누르는 순간**에 발생한다.

**실제 증상**: iOS 26에서 정리 버튼을 누르면, Liquid Glass 확장 애니메이션이 **1초 이상 멈춘 상태**로 유지된 후 다음 단계(방식 선택 시트)로 진행된다. 2회째부터는 즉시 반응.

**정리 버튼 탭 시 실제 흐름**:
```
[정리 버튼 탭] → cleanupButtonTapped()
  ├─ CleanupService.shared.isTrashEmpty()  ← 휴지통 확인
  └─ showCleanupMethodSheet()              ← 방식 선택 시트
      ↓ (사용자가 방식 선택한 후에야)
      startPreviewCleanup() → progressView → Task { analyze() }
```

분석(analyze)은 사용자가 방식을 선택한 **이후**에 시작된다. 버튼 탭 시점에는 분석과 무관한 코드만 실행된다. v1/v2는 원인과 다른 곳을 고치고 있었다.

### 근본 원인: 싱글톤 캐스케이드 초기화

`cleanupButtonTapped()`에서 `CleanupService.shared`에 **처음 접근**하면, 싱글톤 init이 연쇄적으로 발생:

```
CleanupService.shared  ← 첫 접근
  └─ CleanupService.init()
      ├─ CleanupSessionStore.shared  ← 싱글톤 init + loadFromFiles(queue.async)
      ├─ QualityAnalyzer.shared      ← 싱글톤 init
      │    ├─ BlurAnalyzer.shared    ← ⚠️ MTLCreateSystemDefaultDevice() + MPS 커널 생성
      │    ├─ ExposureAnalyzer.shared
      │    ├─ SafeGuardChecker.shared
      │    ├─ CleanupImageLoader.shared
      │    └─ VideoFrameExtractor.shared
      └─ TrashStore.shared           ← 파일 I/O (이미 초기화됨)
```

**핵심**: `isTrashEmpty()` 하나를 확인하려고 `CleanupService.shared`에 접근하는데, 그 init이 `QualityAnalyzer.shared`를 초기화하고, 그게 `BlurAnalyzer.shared`를 초기화하면서 **Metal 디바이스를 생성**한다.

### iOS 26에서 특히 심한 이유 — FirstLoading 문서와 동일 패턴

`docs/complete/260131FirstLoading1.md`, `docs/complete/260203FirstLoading2.md` 참조.

```
iOS 25:
[앱 시작] → FloatingOverlay 생성 → LiquidGlassKit → Metal 이미 초기화됨
          → GlassIconButton → feedbackGenerator.prepare() → dyld 워밍업 완료
          → (수초 후) 정리 버튼 → 프레임워크 이미 warm → 빠름

iOS 26:
[앱 시작] → useFloatingUI = false → FloatingOverlay 미생성 → Metal cold ❌
          → feedbackGenerator.prepare() 추가됨 → dyld 워밍업만 ✅
          → Metal/MPS 프레임워크 cold ❌
          → 정리 버튼 → CleanupService.shared init → Metal cold-start 발생!
```

iOS 26에서는 FloatingOverlay(LiquidGlassKit)가 없어서 **앱 전체에서 Metal이 한 번도 로딩된 적 없는 상태**. 정리 버튼이 Metal 최초 호출 트리거가 된다.

추가로 `showCleanupMethodSheet()`에서:
```swift
sessionStore.latestSession   // → queue.sync (loadFromFiles 완료 대기)
sessionStore.byYearSession   // → queue.sync
```
`CleanupSessionStore.init()`이 `loadFromFiles()`를 `queue.async`로 시작하고, 직후에 같은 serial queue에 `queue.sync`를 걸어서 파일 I/O 완료까지 MainActor가 추가 블로킹된다.

### 수정 1: `CleanupService.shared`를 버튼 탭 경로에서 제거

**파일**: `GridViewController+Cleanup.swift:118`

```swift
// Before

// After
if trashStore.trashedCount > 0 {
```

- `BaseGridViewController`에 주입된 `trashStore` 프로퍼티 사용 (DI 준수, `TrashStore.shared` 직접 참조 회피)

- `trashedCount`는 메모리 내 `state.trashedCount` 반환 (파일 I/O 없음, O(1))
- `isTrashEmpty()`의 구현이 정확히 `trashStore.trashedCount == 0`이므로 로직 동일

### 수정 2: `CleanupSessionStore` 초기화 시 파일 동기 로드

**파일**: `CleanupSessionStore.swift:87`

```swift
// Before
private init() {
    // ...
    loadFromFiles()      // queue.async → 백그라운드 파일 I/O 시작
}

var latestSession: CleanupSession? {
    return queue.sync { cachedLatestSession }  // 파일 I/O 완료 대기 → MainActor 블로킹
}

// After
private init() {
    // ...
    loadFromFilesSync()  // 동기 로드 (< 10ms, 작은 JSON 2개)
}

// queue.sync는 유지 (save/update/clear가 queue.async로 쓰기하므로 data race 방지)
```

- `CleanupSessionStore.shared`는 정리 버튼 탭 전까지 어디서도 접근되지 않음 (조기 접근 없음 확인)
- `loadFromFilesSync()`는 이미 테스트 인스턴스용으로 구현되어 있음 — 검증된 코드
- init에서 동기 로드하면, 이후 `queue.sync` 호출 시 큐에 대기 작업이 없으므로 즉시 반환

> **참고**: `showCleanupMethodSheet()`에서 `latestSession`/`byYearSession`을 전달하지만, 시트 초기 화면(`showMainActionSheet`)에서 `latestSession`은 사용되지 않고, `byYearSession`은 "연도별 정리" 선택 후에만 사용됨. 현재 v3에서는 `loadFromFilesSync()`로 blocking 패턴만 제거하며, 세션 지연 로드는 향후 개선 여지로 남김 (init < 10ms이므로 실질 영향 미미).

### 부작용 검토

| 항목 | 결과 |
|------|------|
| `trashStore` (주입 인스턴스) 초기화 시점 | ✅ 앱 시작 시 이미 초기화됨 (TabBarController → GridViewController) |
| `trashedCount` 동작 차이 | ✅ 없음 (`isTrashEmpty()`가 정확히 `trashedCount == 0` 반환) |
| `loadFromFilesSync()` 검증 여부 | ✅ 테스트 인스턴스에서 이미 사용 중 |
| `queue.sync` 제거 가능성 | ⚠️ 불가 — `save()/update()/clear()`가 `queue.async`로 쓰기하므로 유지 필요 |
| CleanupService lazy 전환 필요성 | ❌ 불필요 — 버튼 탭 경로에서 CleanupService 자체가 빠지므로 |
| 연도별 정리 첫 실행 시 CleanupService 초기화 지연 | ⚠️ 가능 — `startCleanup()` 내 `Task { }` 안에서 `CleanupService.shared` 첫 접근 시 Metal 초기화. progress view가 먼저 표시되므로 UX 영향 낮음 |
