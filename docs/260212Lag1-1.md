# iOS 26 정리버튼 최초 1회 랙 해결 계획 (수정본)

**날짜**: 2026-02-12  
**목표**: `docs/260212Lag1.md` 보완. "첫 클릭 프리즈"를 실제로 없애고, 2회차만 빨라지는 편차도 줄인다.

---

## 1) 해결 목표 (완료 기준)

1. 최초 클릭 후 0.5초 내 진행 UI가 반드시 1회 이상 갱신된다.
2. 최초 클릭 구간에서 Main Thread 100ms 이상 stall이 0회다.
3. 최초 1회 총 분석시간이 기존 대비 최소 30% 감소한다.
4. 2회차 이후와 1회차 간 체감 편차(모델 초기화 지연)가 크게 줄어든다.

---

## 3) 최종 해결 전략 (이번 구현 권장안)

### A. 실행 컨텍스트 분리 (필수)

1. 호출부 `Task {}`를 `Task.detached`로 변경한다.
2. 단, 이것만으로는 불충분하므로 분석 체인 entry를 명시적으로 non-main으로 만든다.
3. 분석 체인 메서드에 `nonisolated`를 명시하고, 가변 상태 접근은 락/actor로 분리한다.

핵심: `Task.detached` + 분석 entry non-main 보장 + 상태 동기화가 동시에 필요하다.

### B. 동기 Vision 호출 백그라운드 강제 (필수)

`VNImageRequestHandler.perform()`를 공통 헬퍼로 감싸 항상 백그라운드 큐에서 수행한다.  
직접 호출 금지 규칙을 만든다.

### C. 최초 1회 모델 워밍업 (필수)

정리 버튼 클릭 직전(또는 권한 획득 직후) 1회 워밍업을 백그라운드로 실행한다.

대상:
1. `CalculateImageAestheticsScoresRequest`
2. `VNRecognizeTextRequest`
3. `VNDetectFaceCaptureQualityRequest`

워밍업은 idempotent(1회만 실행)로 만든다.

### D. 중복 연산 제거 (강력 권장)

1. Preview 경로에서 `QualityAnalyzer`가 로드한 이미지를 재사용한다.
2. 텍스트 감지 결과를 path1/path2에서 1회 계산 후 공유한다.

## 6) 검증 계획

## 6-1. 기능 검증

1. 최초 클릭 시 진행 UI가 즉시 갱신되는지
2. 취소 동작이 정상인지 (부분 이동 없음)
3. DEBUG 테스트 경로 2개가 동일하게 동작하는지

## 6-2. 성능 검증

1. Instruments Time Profiler로 최초 클릭 구간 측정
2. Main Thread call tree에서 `VNImageRequestHandler.perform`가 사라졌는지 확인
3. `os_signpost`로 아래 지표 수집
- tap -> first progress update
- tap -> result
- first run vs second run 비교

## 6-3. 회귀 검증

1. iOS 16~17 경로(비 Aesthetics) 정상 동작
2. iOS 18+ 경로에서 분류 결과 개수 차이 과다 여부 확인
3. Thread Sanitizer로 `CleanupService`/`QualityAnalyzer` 데이터 레이스 검사

---

## 7) 리스크와 대응

1. `nonisolated` 전환 중 컴파일 격리 오류 가능  
대응: 상태 접근을 lock/actor로 분리하고, UI 참조를 entry 바깥으로 이동

2. 워밍업이 배터리/초기 부하를 키울 수 있음  
대응: 1회 실행 + low priority + 앱 활성 상태에서만 수행

3. 중복 제거로 판정 결과가 달라질 수 있음  
대응: 기존/신규 결과 샘플 500장 비교 로그로 오차 검증

---

## 8) 이번 문서와 이전 문서의 차이

1. `Task.detached` 단독 효과를 전제로 하지 않는다.
2. 최초 1회 문제를 별도 워밍업 단계로 명시한다.
3. 데이터 레이스 방지 계획(lock/actor)을 구현 범위에 포함한다.
4. 중복 로드/중복 텍스트 감지를 같은 작업 묶음으로 처리한다.
