# iOS 26 뷰어 삭제 최초 1회 랙 분석

**날짜**: 2026-03-01
**증상**: iOS 26에서 뷰어의 삭제 버튼 탭 또는 위로 스와이프 삭제 시, **최초 1회에만** UI 랙 발생. 2회차부터 정상.
**관련 이슈**: 정리 버튼 랙과 유사한 "첫 1회" 패턴 (`260212Lag1-1~3.md` 참조)

## iOS 26 뷰어 삭제의 특수성

### iOS 26에서의 뷰어 구조

| 항목 | iOS 16~25 | iOS 26 |
|------|-----------|--------|
| 뷰어 전환 방식 | Modal (custom transition) | Navigation Push |
| 하단 버튼 | GlassTextButton (커스텀) | 시스템 UIToolbar (Liquid Glass) |
| 삭제 버튼 | GlassTextButton(lazy var) → setupActionButtons()에서 addSubview | UIBarButtonItem(title:) → setupSystemToolbar()에서 설정 |
| `useSystemUI` | false | true |
| GlassTextButton lazy init | viewDidLoad → setupActionButtons()에서 트리거 | **트리거 안 됨** (시스템 툴바 사용) |

### iOS 26에서 setupUI() 분기

```swift

    setupActionButtons()  // iOS 16~25: GlassTextButton 생성 + addSubview
```

## 유력 원인: 뷰어 삭제에만 존재하는 동기 호출

### 후보 3개 (우선순위순)

#### 1. `snapshotView(afterScreenUpdates: false)` — **가장 유력**

| 항목 | 내용 |
|------|------|
| 뷰어에서 첫 호출 시점 | **삭제 시** (초기 표시, 스와이프 탐색에서는 미사용) |
| 삭제 버튼에서 사용 | O (`deleteButtonTapped` 직접 호출) |
| 스와이프 삭제에서 사용 | O (`handleSwipeDelete` 직접 호출) |
| 그리드 삭제에서 사용 | **✗** |
| 추정 원인 | GPU 렌더 서버(backboardd) 측 스냅샷 파이프라인 첫 초기화 비용 |

두 뷰어 삭제 경로 **모두에서 호출**되고, 그리드 삭제에는 **없는** 유일한 무거운 동기 호출.

#### 2. `pageViewController.setViewControllers(animated: false)`

| 항목 | 내용 |
|------|------|
| 뷰어 초기 표시 | `displayInitialPhoto()`에서 1회 호출됨 (이미 실행된 코드 경로) |
| 사용자 스와이프 탐색 | dataSource 경유 (다른 코드 경로) |
| 추정 원인 | 초기 표시에서 이미 호출되었으므로 가능성 낮음. 단, `animated: false` + 기존 VC 교체 조합이 다른 내부 경로일 수 있음 |

#### 3. `updateSimilarPhotoOverlay()` → `checkAndShowFaceButtons()`

| 항목 | 내용 |
|------|------|
| 실행 방식 | `Task { @MainActor in ... }` 비동기 |
| 추정 원인 | Task 내부이므로 동기 블로킹은 아님. 단, Task 생성 + SimilarityCache actor hop 오버헤드 가능 |

## 코드 분석 결론

**`snapshotView(afterScreenUpdates: false)`가 첫 1회 랙의 주 원인으로 추정됨.**

근거:
1. 뷰어 삭제(버튼 + 스와이프) 양쪽 모두에서 호출 — 증상 일치
2. 그리드 삭제에서는 미사용 — 랙 없음과 일치
3. 뷰어 라이프사이클에서 삭제 시점이 첫 snapshotView 호출 — "최초 1회" 패턴과 일치
4. iOS 26 렌더 서버 측 파이프라인 첫 초기화 비용 — 시스템 레벨 first-use 패턴 (Lag1 v4와 동일 유형)

### 정리 버튼 랙(Lag1)과의 유사성

`260212Lag1-3.md` v4 진단에서 확인된 패턴:
- 우리 코드 실행: **43ms**
- 시스템 Hang: **2.6초**
- **코드 밖에서 2.56초 발생** (UIAlertController + UIVisualEffectView 첫 사용)

이번에도 코드 자체는 경량이지만, `snapshotView`의 내부 구현(렌더 서버 IPC, GPU 파이프라인)에서 시스템 레벨 first-use 비용이 발생하는 것으로 추정.

## 미확인 사항

1. **정확한 랙 시간**: Hang detected 로그 또는 CACurrentMediaTime 측정 미수행
2. **코드 실행 시간 vs 시스템 비용 분리**: v4처럼 타이밍 측정으로 분리 필요
3. **iOS 25에서 동일 증상 여부**: iOS 25에서도 발생하는지 비교 미수행
4. **스와이프 삭제 먼저 실행 후 버튼 삭제가 빨라지는지 (또는 그 반대)**: 두 경로가 같은 first-use 자원(`snapshotView`)을 공유하는지 확인 필요
5. **snapshotView를 제거한 대체 모션에서 랙이 사라지는지**: 원인 확정을 위한 A/B 테스트

## 다음 단계: 타이밍 측정

v4와 동일한 방법으로 `CACurrentMediaTime()` 삽입하여 병목 특정:

```swift
// deleteButtonTapped / handleSwipeDelete 양쪽에 삽입
let t0 = CACurrentMediaTime()
// 햅틱
let t1 = CACurrentMediaTime()
// snapshotView
let t2 = CACurrentMediaTime()
// delegate?.delete
let t3 = CACurrentMediaTime()
// moveToNextAfterDeleteNoAnimation
let t4 = CACurrentMediaTime()
let ms = { String(format: "%.1f", ($0 as Double) * 1000) }
Logger.viewer.debug("[DeleteLag] haptic=\(ms(t1-t0)) snap=\(ms(t2-t1)) delete=\(ms(t3-t2)) move=\(ms(t4-t3)) total=\(ms(t4-t0))ms")
```

### 기대 결과 시나리오

| 시나리오 | 의미 | 대응 |
|----------|------|------|
| total < 100ms + Hang detected 2초 | 시스템 레벨 비용 (Lag1 v4와 동일) | snapshotView 제거 또는 지연 생성 테스트 |
| snap 구간 500ms+ | snapshotView 첫 사용 비용 | 스냅샷 워밍업 또는 스냅샷 없는 대체 모션 |
| move 구간 500ms+ | setViewControllers 첫 사용 비용 | 페이지 전환 방식 변경 |
| haptic 구간 200ms+ | 햅틱 잔여 워밍업 비용 | prepare() 시점 앞당김 |
