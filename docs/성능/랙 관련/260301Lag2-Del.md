# iOS 26 뷰어 삭제 최초 1회 랙 분석

**날짜**: 2026-03-01
**증상**: iOS 26에서 뷰어의 삭제 버튼 탭 또는 위로 스와이프 삭제 시, **최초 1회에만** UI 랙 발생. 2회차부터 정상.
**관련 이슈**: 정리 버튼 랙과 유사한 "첫 1회" 패턴 (`260212Lag1-1~3.md` 참조)

## iOS 26 뷰어 삭제의 특수성

### iOS 26에서의 뷰어 구조

| 항목 | iOS 16~25 | iOS 26 |
|------|-----------|--------|
| 뷰어 전환 방식 | Modal (custom transition) | Navigation Push |
| 하단 버튼 | GlassTextButton (커스텀) | 시스템 UIToolbar (Liquid Glass) |
| 삭제 버튼 | GlassTextButton(lazy var) → setupActionButtons()에서 addSubview | UIBarButtonItem(title:) → setupSystemToolbar()에서 설정 |
| `useSystemUI` | false | true |
| GlassTextButton lazy init | viewDidLoad → setupActionButtons()에서 트리거 | **트리거 안 됨** (시스템 툴바 사용) |

### iOS 26에서 setupUI() 분기

```swift
// ViewerViewController.swift:512
if !useSystemUI {
    setupActionButtons()  // iOS 16~25: GlassTextButton 생성 + addSubview
    setupBackButton()
}
// iOS 26: setupActionButtons() 호출 안 됨
// → deleteButton, previousPhotoButton lazy var 초기화되지 않음
// → 대신 viewWillAppear → setupSystemUIIfNeeded() → setupSystemToolbar()
```

## 코드 레벨 원인 후보 분석

### 동기 호출 중 첫 1회 비용이 높은 후보

| 후보 | 첫 1회 비용? | 근거 |
|------|------------|------|
| `UIImpactFeedbackGenerator.impactOccurred()` | △ | TabBarController.viewDidLoad에서 `prepare()` 이미 호출. 잔여 비용 가능 |
| `AnalyticsService.shared` | ✗ | `private init() {}` 빈 초기화. viewDidAppear에서 이미 접근됨 |
| `snapshotView(afterScreenUpdates: false)` | △ | 사진은 이미 렌더링된 상태이나, 뷰어에서 첫 스냅샷 캡처 시 GPU 파이프라인 셋업 가능 |
| `trashStore.moveToTrash()` | ✗ | saveState()는 async. 상태 변경 자체는 메모리 내 Set 조작 |
| `collectionView.reloadItems(at:)` | ✗ | 매번 동일 비용 |
| `coordinator.refreshFilteredIndices()` | ✗ | .normal 모드에서 O(1) identity 매핑 |
| `createPageViewController(at:)` | ✗ | 경량 (PhotoPageVC 생성만) |
| `setViewControllers(animated: false)` | △ | 첫 프로그래밍 방식 페이지 전환의 내부 비용 가능 |

### 코드 분석 결론

**명확한 동기 블로킹 지점을 발견하지 못함.**

모든 동기 호출이 매번 동일하거나, 이미 워밍업되어 있거나, 비동기로 실행됨.
→ **코드 밖(iOS 26 시스템 레벨)에서 발생하는 first-use 비용일 가능성이 높음.**

## 유력 원인: iOS 26 시스템 레벨 first-use 비용

### 정리 버튼 랙(Lag1)과의 유사성

`260212Lag1-3.md` v4 진단에서 확인된 패턴:
- 우리 코드 실행: **43ms**
- 시스템 Hang: **2.6초**
- **코드 밖에서 2.56초 발생** (UIAlertController + UIVisualEffectView 첫 사용)

### iOS 26 뷰어 삭제에서 추정되는 시스템 비용 후보

1. **시스템 Liquid Glass 툴바 버튼의 첫 터치 반응**
   - iOS 26 UIToolbar는 Liquid Glass 렌더링을 사용
   - 버튼 터치 시 내부 렌더링 파이프라인 첫 초기화 비용 가능
   - 단, 스와이프 삭제에서도 랙이 있으므로 이것만이 원인은 아님

2. **`snapshotView()` GPU 파이프라인 첫 초기화**
   - 뷰어에서 첫 snapshotView 호출 시 렌더 서버(backboardd) 측 파이프라인 셋업
   - 삭제 버튼 / 스와이프 삭제 **모두 snapshotView 사용** → 공통 원인으로 유력

3. **UIPageViewController `setViewControllers` 첫 프로그래밍 전환**
   - 뷰어 초기 표시 시 `setViewControllers`는 호출되지만, 사용자 스와이프는 dataSource 경유
   - 삭제 후 `moveToNextAfterDeleteNoAnimation()`에서 호출하는 것이 **첫 프로그래밍 방식 전환**일 수 있음
   - 내부적으로 다른 코드 경로를 타며 초기화 비용 발생 가능

## 미확인 사항

1. **정확한 랙 시간**: Hang detected 로그 또는 CACurrentMediaTime 측정 미수행
2. **코드 실행 시간 vs 시스템 비용 분리**: v4처럼 타이밍 측정으로 분리 필요
3. **iOS 25에서 동일 증상 여부**: iOS 25에서도 발생하는지 비교 미수행
4. **스와이프 삭제 먼저 실행 후 버튼 삭제가 빨라지는지 (또는 그 반대)**: 두 경로가 같은 first-use 자원을 공유하는지 확인 필요

## 다음 단계: 타이밍 측정

v4와 동일한 방법으로 `CACurrentMediaTime()` 삽입하여 병목 특정:

```swift
// deleteButtonTapped / handleSwipeDelete 양쪽에 삽입
let t0 = CACurrentMediaTime()
// 햅틱
let t1 = CACurrentMediaTime()
// snapshotView
let t2 = CACurrentMediaTime()
// delegate?.delete
let t3 = CACurrentMediaTime()
// moveToNextAfterDeleteNoAnimation
let t4 = CACurrentMediaTime()
let ms = { String(format: "%.1f", ($0 as Double) * 1000) }
Logger.viewer.debug("[DeleteLag] haptic=\(ms(t1-t0)) snap=\(ms(t2-t1)) delete=\(ms(t3-t2)) move=\(ms(t4-t3)) total=\(ms(t4-t0))ms")
```

### 기대 결과 시나리오

| 시나리오 | 의미 | 대응 |
|----------|------|------|
| total < 100ms + Hang detected 2초 | 시스템 레벨 비용 (Lag1 v4와 동일) | snapshotView 제거 또는 지연 생성 테스트 |
| snap 구간 500ms+ | snapshotView 첫 사용 비용 | 스냅샷 워밍업 또는 스냅샷 없는 대체 모션 |
| move 구간 500ms+ | setViewControllers 첫 사용 비용 | 페이지 전환 방식 변경 |
| haptic 구간 200ms+ | 햅틱 잔여 워밍업 비용 | prepare() 시점 앞당김 |
