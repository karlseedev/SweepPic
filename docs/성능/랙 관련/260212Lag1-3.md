# 정리버튼 첫 클릭 랙 — v4 원인 추적

**날짜**: 2026-02-12
**이전 문서**: `docs/260212Lag1-2.md` (v1~v3 실패)
**증상**: 정리버튼 첫 클릭 시 UI 프리즈 (iOS 18에서 ~2.4초 확인, iOS 26에서도 체감 랙 있음)

---

## 완료 기준

1. 최초 클릭 후 **0.5초 내** alert/시트가 표시된다
2. iOS hang detector에 hang이 보고되지 않는다
3. 2회차 이후와 체감 차이가 없다

---

## v1~v3 실패 요약

| 버전 | 접근 | 결과 |
|------|------|------|
| v1 | VisionExecutor로 Vision 호출 백그라운드화 | 효과 없음 |
| v2 | nonisolated + VisionExecutor + yield | 효과 없음 |
| v3 | CleanupService.shared 우회 + SessionStore 동기화 | 효과 없음 |

**공통 실패 원인**: 모두 `cleanupButtonTapped()` 내부 코드가 느리다고 가정했으나, 실제 코드 실행은 ~40ms에 불과. 랙은 우리 코드 밖에서 발생.

---

## v4 진단 과정

### 1단계: 코드 타이밍 측정 (iOS 18)

`cleanupButtonTapped()` 내부에 `CACurrentMediaTime` 타이밍 삽입.

**결과**:
```
[CleanupLag] isTrashEmpty: 24.3ms, methodSheet: 19.0ms, total: 43.3ms
Hang detected: 2.60s (debugger attached, not reporting)
```

우리 코드 **43.3ms** vs 시스템 행 **2.60초** → 2.56초는 우리 코드 밖에서 발생.

### 2단계: 첫 번째 present() 가설 검증 (iOS 18)

"선택" 버튼에 더미 UIAlertController를 추가하여 "정리" 보다 먼저 present() 호출.

**결과**:
- "선택" (첫 번째 present): `Hang detected: 2.42s`
- "정리" (두 번째 present): 38.3ms, 랙 없음

→ **어떤 버튼이든 첫 번째 `present(alert)` 에서 랙 발생** 확인.

### 3단계: 원인 분리 테스트 A/B (iOS 18)

| 테스트 | 내용 | Hang 로그 |
|--------|------|-----------|
| 원래 | UIAlertController, `animated: true` | `Hang detected: 2.42s` |
| A | UIAlertController, `animated: false` | `Hang detected: 0.87s` |
| B | 빈 UIViewController, `animated: true` | 없음 (빠름) |

**결론**:
- `present()` 자체는 문제 없음 (B가 증명)
- 애니메이션 자체도 문제 없음 (B가 증명)
- **UIAlertController 고유의 문제** — `animated: false`에서도 0.87초

### 4단계: 뷰 계층 격리 테스트 C/D (iOS 18)

| 테스트 | 내용 | Hang 로그 |
|--------|------|-----------|
| C | 빈 흰색 VC를 먼저 present, 그 위에서 alert | `Hang detected: 2.09s` |
| D | MTKView 3개를 `isHidden=true` 후 alert | `Hang detected: 0.92s` |

**핵심 발견**:
- Test C: 빈 배경 위에서도 2.09s → 배경의 **보이는 내용**과 무관
- Test C에서 MTKView가 여전히 window 레이어 트리에 존재 → UIVisualEffectView 블러가 `isHidden`이 아닌 레이어는 가려져 있어도 캡처
- Test D: MTKView를 `isHidden=true`로 렌더링에서 제외하면 0.92s → **MTKView 3개가 ~1.5초를 차지**

### 5단계: iOS 26 확인

| 테스트 | 내용 | 결과 |
|--------|------|------|
| D (iOS 26) | MTKView 숨김 시도 | `MTKView hidden: 0개` (iOS 26에서는 MTKView 미생성) |

- iOS 26에서 MTKView는 0개 (LiquidGlassKit이 `isNative: true`로 네이티브 UIVisualEffectView 사용)
- 체감 랙 있음 ("약 1초?") — **단, Hang 로그 미확인. 정확한 수치 미검증.**

## 미확인 사항

1. **iOS 26 정확한 행 시간**: 체감 "약 1초"만 확인, `Hang detected` 로그 미수집. 정확한 수치 검증 필요.
2. **UIAlertController 기본 ~0.9s가 우리 앱 고유인지 범용인지**: 빈 프로젝트에서 동일 테스트 필요.
3. **UIVisualEffectView 단독 비용**: UIAlertController를 거치지 않고 UIVisualEffectView만 추가해도 동일 비용인지 미확인.

## 후일 확인: 랙 자연 해소 (2026-02-13)

다른 수정 작업을 진행하는 과정에서 초기 랙이 체감상 사라짐. **단, Hang detected 로그로 재검증하지 않았으므로 확정이 아닌 추론임.**

### 유력 원인: `.actionSheet` → `.alert` 스타일 변경

**커밋 `9c579a7`** (2026-02-12): `CleanupMethodSheet`의 `preferredStyle`을 `.actionSheet` → `.alert`로 변경.

`.actionSheet`는 present 시 내부적으로 전체 화면 frosted glass dimming(UIVisualEffectView 기반 블러)을 생성하며, 이것이 v4 진단에서 확인된 비용의 핵심 트리거:
- **iOS 18**: `.actionSheet` 블러가 window 레이어 트리의 MTKView 3개를 캡처 → ~1.5s. `.alert`는 전체 화면 블러 없이 단순 dimming만 사용하므로 이 비용 제거.
- **iOS 26**: MTKView 0개이므로 캡처 비용은 없지만, `.actionSheet` 고유의 블러 렌더링 파이프라인 첫 초기화 비용이 ~1초 체감 랙의 원인이었을 가능성.

### 보조 원인: `CleanupSessionStore.queue.sync` x2 제거

같은 시기 `showCleanupMethodSheet()` 구조가 변경됨:

```
// 이전: utility QoS 직렬 큐에서 동기 대기 (메인 스레드 블로킹 가능)
sessionStore.latestSession  → queue.sync { cachedLatestSession }
sessionStore.byYearSession  → queue.sync { cachedByYearSession }

// 현재: UserDefaults 직접 읽기 (즉각 반환)
CleanupPreviewService.canContinue → UserDefaults.standard.object(forKey:)
```

큐가 파일 저장 등으로 점유 중이었다면 메인 스레드가 대기했을 수 있음.

### 불확실한 점

1. **v4 Test A/B/D에서 사용한 UIAlertController가 `.alert`인지 `.actionSheet`인지 미확인** — `.alert`였다면 ~0.9s 기본 비용은 `.actionSheet`와 무관하게 존재하는 것이므로 해소 설명이 불완전
2. **여러 변경이 동시에 적용**되어 어떤 것이 실제 효과인지 격리 안 됨
3. **"사라졌다"는 것 자체가 체감 기반** — Hang detected 로그로 검증하지 않음
4. **iOS 26은 원래 정확한 수치가 없었음** — 비교 기준 부재
