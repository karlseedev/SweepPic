# 앨범 탭 랙 근본 해결 계획 ✅ 완료

## Context

앨범 탭 진입 시 심한 랙이 발생합니다. 원인은 `AlbumsViewController.loadData()`가 메인 스레드에서 **50회 이상의 동기 PhotoKit 호출**을 수행하기 때문입니다:
- `fetchSmartAlbums()`: 10종 × 2회 = 20회 동기 호출 (각 앨범 전체 에셋 fetch)
- `fetchUserAlbums()`: N개 앨범 × 1회 = N회 추가 호출
- `AlbumCell.loadThumbnail()`: 셀마다 `PHAsset.fetchAssets` 동기 호출
- `viewWillAppear`에서 매번 무조건 전체 재로드
- iOS 18/26 모두 동일한 코드 경로 → 동일한 문제

### 1차 구현 후 테스트 피드백 (반영 완료)

1. **탭 전환 랙은 해소됨** — 비동기 fetch로 메인 스레드 블로킹 제거 성공
2. **앨범 목록이 처음 나타날 때 여전히 랙이 심함** — `updateEmptyState()`에서 `collectionView.isHidden = true → false` 토글로 인해, 데이터 로드 완료 시 컬렉션뷰가 갑자기 나타나며 많은 셀을 한꺼번에 생성/레이아웃하는 부하 발생
3. **0.1초 이내 여러 번 깜빡임** — `viewDidLoad` → `loadData()` + `viewWillAppear` → `loadData()` 이중 호출로 `reloadData()`가 2회 연속 실행됨. 셀 destroy/recreate 반복으로 깜빡임 발생

## 수정 파일 및 변경 내용

### 1. `Sources/AppCore/Services/AlbumService.swift`

**변경 내용:**

**(A) 기존 동기 메서드 내부 최적화 — fetchLimit = 1**

`fetchSmartAlbum(type:)` 및 `fetchUserAlbums()` 내부에서:
- keyAsset용 fetch와 count용 fetch를 분리
- keyAsset: `fetchLimit = 1` + `sortDescriptors = [creationDate descending]` → 1개만 로드
- count: 기존 `PHAsset.fetchAssets(in:options:).count` 유지 (PHFetchResult.count는 내부 최적화됨)
- `estimatedAssetCount == 0`인 스마트 앨범은 fetch 없이 건너뛰기 (빠른 필터)
  - 주의: `NSNotFound` 반환 시에는 스킵하지 않고 fetch로 확인

**(B) 비동기 일괄 fetch 메서드 추가**

```swift
/// 비동기 앨범 전체 로드
/// - 내부: DispatchQueue.global(qos: .userInitiated)에서 PhotoKit fetch 실행
/// - completion: 메인 스레드에서 호출됨
/// - keyAssets: 앨범ID → PHAsset 딕셔너리 (셀에서 재 fetch 방지)
public func fetchAllAlbumsAsync(
    completion: @escaping (
        _ smartAlbums: [SmartAlbum],
        _ userAlbums: [Album],
        _ keyAssets: [String: PHAsset]
    ) -> Void
)
```

- 내부에서 `DispatchQueue.global(qos: .userInitiated).async`로 fetch 실행
- fetch 완료 후 `DispatchQueue.main.async`로 completion 호출
- fetchLimit=1로 가져온 PHAsset을 `keyAssets` 딕셔너리에 수집
- TrashAlbumViewController의 `loadTrashedAssets()` 패턴과 동일한 구조

**(C) AlbumServiceProtocol에 비동기 메서드 선언 추가**

### 2. `PickPhoto/PickPhoto/Features/Albums/AlbumCell.swift`

**변경 내용:**

**(A) PHAsset 직접 전달 configure 오버로드 추가**

```swift
func configure(album: Album, keyAsset: PHAsset?, targetSize: CGSize)
func configure(smartAlbum: SmartAlbum, keyAsset: PHAsset?, targetSize: CGSize)
```

**(B) loadThumbnail을 PHAsset 기반으로 변경**

- 기존: `loadThumbnail(assetID:)` → `PHAsset.fetchAssets(withLocalIdentifiers:)` 동기 호출 후 `PHImageManager.default().requestImage()`
- 변경: `loadThumbnail(asset:)` → `ImagePipeline.shared.requestImage(for:targetSize:)` 직접 호출
  - `ImagePipeline`은 `PHCachingImageManager` + 백그라운드 OperationQueue 사용 (이미 구현됨)
  - `Cancellable` 리턴 → `prepareForReuse`에서 취소
  - **iCloud 주의**: 현재 AlbumCell은 `isNetworkAccessAllowed = true`이나, ImagePipeline 기본값은 `false`. 앨범 셀 전용으로 `isNetworkAccessAllowed = true` 옵션을 적용하거나, ImagePipeline에 quality 파라미터로 분기 필요

**(C) 기존 assetID 기반 configure는 하위 호환을 위해 유지** (TrashAlbum 등에서 사용)

**(D) keyAsset cache miss 시 fallback**

- `keyAsset == nil && keyAssetIdentifier != nil`인 경우, 기존 assetID 기반 `loadThumbnail(assetID:)` 경로로 fallback
- 정상 흐름에서는 발생하지 않으나 edge case(iCloud 동기화 지연 등) 대비

### 3. `PickPhoto/PickPhoto/Features/Albums/AlbumsViewController.swift`

**변경 내용:**

**(A) loadData() 비동기화 + 응답 역전 방지**

```swift
private var loadGeneration: UInt = 0

private func loadData() {
    guard !isLoading else {
        // 로딩 중 요청 → 완료 후 재로드 필요
        needsReload = true
        return
    }
    isLoading = true
    needsReload = false  // 플래그 소비
    loadGeneration += 1
    let currentGeneration = loadGeneration

    albumService.fetchAllAlbumsAsync { [weak self] smartAlbums, userAlbums, keyAssets in
        guard let self = self else { return }
        // stale 응답 무시 (더 새로운 요청이 발행된 경우)
        guard currentGeneration == self.loadGeneration else { return }

        self.isLoading = false
        self.lastLoadTime = CFAbsoluteTimeGetCurrent()
        self.smartAlbums = smartAlbums
        self.userAlbums = userAlbums
        self.keyAssetCache = keyAssets
        self.updateEmptyState()
        self.collectionView.reloadData()

        // ⚠️ 깜빡임 방지: completion 내 즉시 재호출 금지
        // needsReload은 다음 viewWillAppear 또는 photoLibraryDidChange(visible)에서 처리
    }
}
```

새 프로퍼티:
- `isLoading: Bool` — 중복 로딩 방지
- `loadGeneration: UInt` — 비동기 응답 역전(stale overwrite) 방지
- `lastLoadTime: CFAbsoluteTime` — 재로딩 간격 제어
- `needsReload: Bool` — 로딩 중 또는 백그라운드 변경 플래그
- `keyAssetCache: [String: PHAsset]` — 셀에 전달할 PHAsset 캐시
- `hasLoadedOnce: Bool` — 최초 로드 완료 여부 (viewWillAppear 이중 호출 방지)

**(B) viewDidLoad + viewWillAppear 이중 로드 방지 (깜빡임 해결 핵심)**

기존 문제:
```
viewDidLoad → loadData() (async fetch 시작, gen=1)
viewWillAppear → loadData() → isLoading → needsReload = true
fetch 완료 → reloadData() → needsReload → loadData() 재호출 (gen=2)
두 번째 fetch 완료 → reloadData() 또 호출
= reloadData() 2회 연속 → 셀 destroy/recreate → 깜빡임!
```

수정:
```swift
override func viewDidLoad() {
    super.viewDidLoad()
    setupUI()
    setupObservers()
    loadData()  // 최초 로드 (유일한 초기 loadData 호출)
    // ...
}

override func viewWillAppear(_ animated: Bool) {
    super.viewWillAppear(animated)

    // ⚠️ 최초 진입 시 viewDidLoad에서 이미 loadData() 호출됨 → 이중 호출 방지
    if hasLoadedOnce {
        let now = CFAbsoluteTimeGetCurrent()
        if needsReload || (now - lastLoadTime > 1.0) {
            loadData()
        }
    }

    configureFloatingOverlayForAlbums()
}
```

- `hasLoadedOnce`는 loadData() completion에서 `true`로 설정
- 최초 진입 시: viewDidLoad → loadData() 1회만 실행, viewWillAppear는 스킵
- 이후 진입 시: viewWillAppear에서 조건부 로드 (needsReload 또는 1초 경과)

**(C) photoLibraryDidChange 개선**

```swift
nonisolated func photoLibraryDidChange(_ changeInstance: PHChange) {
    DispatchQueue.main.async { [weak self] in
        guard let self = self else { return }

        if self.view.window != nil {
            // 화면 보이는 중: 비동기 로드 (블로킹 없음)
            self.loadData()
        } else {
            // 화면 안 보이는 중: 플래그만 설정 (viewWillAppear에서 처리)
            self.needsReload = true
        }
    }
}
```

**(D) updateEmptyState() — collectionView.isHidden 토글 제거 (초기 진입 랙 해결 핵심)**

기존 문제:
```swift
// 기존: 데이터 없으면 collectionView 숨김 → 데이터 도착 시 한꺼번에 나타남
collectionView.isHidden = isEmpty  // true → false 전환 시 갑작스러운 셀 생성 부하
```

수정:
```swift
private func updateEmptyState() {
    let isEmpty = smartAlbums.isEmpty && userAlbums.isEmpty
    emptyStateView.isHidden = !isEmpty
    // ⚠️ collectionView.isHidden 토글 제거
    // 빈 컬렉션뷰는 시각적으로 문제없고,
    // hidden → visible 전환 시 모든 셀이 한꺼번에 생성되는 레이아웃 부하를 방지
}
```

- 빈 상태에서 collectionView는 항상 visible (빈 화면 = 자연스러움)
- emptyStateView만 토글 (가벼운 뷰 1개)
- 데이터 도착 후 reloadData()가 셀을 점진적으로 생성 (hidden→visible 전환 없음)

**(E) cellForItemAt에서 keyAssetCache 활용 + fallback**

## 데이터 플로우 (수정 후)

```
[최초 진입]
viewDidLoad → loadData()
  ├→ isLoading = true, loadGeneration = 1
  └→ albumService.fetchAllAlbumsAsync { ... }

viewWillAppear → hasLoadedOnce = false → 스킵 (이중 호출 방지)

fetchAllAlbumsAsync 완료 (백그라운드 → 메인):
  ├→ guard gen == loadGeneration
  ├→ smartAlbums, userAlbums, keyAssetCache 설정
  ├→ updateEmptyState()  (emptyStateView만 토글, collectionView는 항상 visible)
  ├→ collectionView.reloadData()  ← 1회만 실행!
  └→ hasLoadedOnce = true

[이후 진입 (앨범 상세 → 뒤로가기)]
viewWillAppear → hasLoadedOnce = true
  └→ needsReload || (now - lastLoadTime > 1.0) → 조건부 loadData()

[포토 라이브러리 변경]
photoLibraryDidChange
  ├→ 화면 visible → loadData()
  └→ 화면 hidden → needsReload = true

cellForItemAt
  └→ cell.configure(album:, keyAsset: keyAssetCache[id], targetSize:)
       ├→ keyAsset != nil → ImagePipeline.requestImage(for: asset, ...)
       └→ keyAsset == nil → fallback: loadThumbnailLegacy(assetID:)
```

### Phase 2 깜빡임 해결 (9d870df)

**원인**: Phase 1(estimatedAssetCount, 22개 사용자 앨범) vs Phase 2(mediaType=image 필터, 21개) 앨범 수 불일치 → sameStructure=false → `reloadData()` → 전체 셀 재생성 → 깜빡임

**해결**:
1. **sameStructure 체크**: Phase 1/2 앨범 ID 목록 비교. 동일하면 `reloadData()` 스킵
2. **updateVisibleCellCounts()**: sameStructure=true 시 보이는 셀의 숫자만 갱신
3. **셀 독립 썸네일 로딩 (loadThumbnailFromCollection)**: Phase 1에서 keyAsset 없이도 셀이 collectionID로 백그라운드 fetchLimit=1 → ImagePipeline으로 즉시 썸네일 로딩. Phase 2 완료를 기다리지 않음

### 사용자 앨범 count 분리 fetch (4b79cdb)

**원인**: 사용자 앨범 fetch에 `mediaType=image` 필터 → 동영상 전용 앨범 count=0 표시

**해결** — AlbumService `fetchAllAlbumsAsync` 사용자 앨범 섹션:
1. **count**: 필터 없이 전체 에셋 수 → 동영상 포함 실제 개수
2. **keyAsset**: `mediaType=image` + `fetchLimit=1` → 이미지 우선 썸네일
3. **동영상 전용 앨범 fallback**: 이미지 keyAsset이 nil이면 필터 없이 재검색 → 동영상 에셋으로 썸네일 표시

AlbumCell `loadThumbnailFromCollection`에도 동일한 fallback 적용 (Phase 1 대응).

### 스마트 앨범 빈 폴더 표시 수정

**원인**: 9d870df에서 sameStructure 보장을 위해 스마트 앨범의 `assetCount == 0` 체크를 제거 → 빈 버스트/타임랩스/슬로모션 앨범이 표시됨

**해결**: 스마트 앨범은 종류가 10종 고정이므로 `assetCount == 0` 체크 복원. Phase 1/2 불일치 시 reloadData가 실행되지만, 빈 앨범 제거는 정당한 구조 변경.

### count 업데이트 애니메이션

Phase 2 완료 시 숫자 변경이 눈에 띄는 문제 → `updateCount()`에 0.25초 `crossDissolve` 애니메이션 적용. 값이 동일하면 애니메이션 스킵.

## 최종 데이터 플로우

```
[Phase 1 — 즉시 표시]
viewDidLoad → loadDataLightweight()
  → fetchAlbumMetadataSync() (메인 스레드, ~11회 collection 호출)
  → estimatedAssetCount + placeholder 썸네일
  → collectionView.reloadData()
  → 셀 각자 loadThumbnailFromCollection() (백그라운드 fetchLimit=1 → ImagePipeline)

[Phase 2 — 백그라운드 정확한 데이터]
viewDidLoad → loadData()
  → fetchAllAlbumsAsync() (백그라운드 스레드)
    → 스마트 앨범: mediaType 필터 + assetCount==0 체크
    → 사용자 앨범: count(전체) + keyAsset(이미지 우선, 동영상 fallback)
  → 완료 시 메인 스레드:
    → sameStructure 체크 (앨범 ID 비교)
    → true: updateVisibleCellCounts() (crossDissolve 애니메이션)
    → false: collectionView.reloadData() (구조 변경 시에만)
```
