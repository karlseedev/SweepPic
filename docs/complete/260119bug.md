# +버튼 딜레이 분석 (260119)

## 해결 상태: ✅ 완료

**원인**: `asset(for:)` 메서드의 O(N) 전수탐색이 핵심 병목이었음

**해결**: `ee1d81b fix(002-similar-photo): iOS 26 스와이프 랙 수정`
- `assetCache` 딕셔너리 도입으로 O(N) → O(1) 개선
- `fetchResult` 파라미터 제거, 필요한 asset만 캐싱

**비고**: 스와이프 랙과 +버튼 딜레이가 동일한 `asset(for:)` 병목으로 인한 문제였음

---

## 현상 (해결됨)
- iOS 26에서 +버튼 누를 때 딜레이가 느껴짐
- iOS 25 이하에서는 체감되지 않던 지연이 iOS 26에서 두드러짐

## 현재 +버튼 탭 시 처리 순서

1. `SimilarityCache.getState()` - async
2. `SimilarityCache.getGroup()` - async
3. `ComparisonGroup.create()` - sync
4. `FaceComparisonViewController` 생성 및 `present()`
5. `viewDidLoad` → `loadPhotoFaces()`, `loadValidPersonIndices()` (둘 다 async)
6. `cellForItemAt` → face 정보가 있을 때 `asset(for:)` → `loadCroppedFaceImage()` (원본 이미지 요청 + 크롭)

## 핵심 병목

### 1. 원본 크기 고품질 이미지 요청 (가장 무거움)

`loadCroppedFaceImage(for:boundingBox:completion:)`:
```swift
let options = PHImageRequestOptions()
options.deliveryMode = .highQualityFormat
options.resizeMode = .exact

imageManager.requestImage(
    for: asset,
    targetSize: CGSize(width: asset.pixelWidth, height: asset.pixelHeight),
    ...
)
```
- 12MP 사진 = 4032x3024 픽셀 전체 로드
- I/O + HEIF 디코딩 비용이 셀당 수십~수백ms

### 2. asset(for:) O(N) 전수탐색 (전체 라이브러리 기준)

`asset(for:)`:
```swift
for i in 0..<fetchResult.count {
    let asset = fetchResult.object(at: i)
    if asset.localIdentifier == assetID { return asset }
}
```
- fetchResult는 전체 라이브러리 스코프 (`fetchAllPhotos()`)
- 10만 장 라이브러리 × 8장 비교 = 최대 80만 회 반복

### 3. 인라인 크롭 CPU 부담

`loadCroppedFaceImage` completion 내부:
```swift
let croppedImage = try FaceCropper.cropFace(from: image, boundingBox: boundingBox)
```
- PHImageManager completion의 실행 스레드는 메인 보장이 없음
- 콜백 스레드가 불명확하고 크롭이 인라인으로 실행돼 CPU 부담이 큼
- 메인 스레드에서 실행될 경우 UI 지연 유발

### 4. loadPhotoFaces() 직렬 await (영향 낮음)

`loadPhotoFaces()`:
```swift
for assetID in comparisonGroup.selectedAssetIDs {
    let faces = await SimilarityCache.shared.getFaces(for: assetID)
}
```
- 최대 8장이라 체감 영향은 작음

## iOS 26 추가 병목 (의심)

iOS 26에서 특히 느린 이유로 다음 요소들이 의심됨:

### S1. viewWillAppear에서 동기적 UI 설정

`FaceComparisonViewController.viewWillAppear`:
```swift
override func viewWillAppear(_ animated: Bool) {
    super.viewWillAppear(animated)
    if #available(iOS 26.0, *) {
        setupSystemNavigationBar()  // present 애니메이션 중에 호출
    }
}
```
- present 애니메이션 도중에 `setupSystemNavigationBar()` 실행
- 3개의 SF Symbol 이미지 로드 필요
- iOS 26에서는 Liquid Glass 스타일로 렌더링되어 비용 증가 가능

### S2. Liquid Glass 초기화 비용

```swift
modalPresentationStyle = .fullScreen
```
- iOS 26에서 fullScreen modal present 시 Liquid Glass 배경 효과 준비
- `setupPageViewControllerWithSystemNav()`에서 safeArea 기반 레이아웃 계산

### S3. 기존 병목과의 복합 효과

iOS 26 추가 오버헤드가 기존 병목들과 직렬로 연결:
```
+버튼 탭
→ SimilarityCache 조회 (async)
→ FaceComparisonVC present 시작
→ [iOS 26] Liquid Glass 초기화 ← 추가
→ viewWillAppear → setupSystemNavigationBar() ← 추가
→ viewDidLoad → setupUI()
→ cellForItemAt → loadCroppedFaceImage()
```

### iOS 16~25와의 차이

| 항목 | iOS 16~25 | iOS 26 |
|------|-----------|--------|
| 네비게이션바 | 커스텀 FloatingTitleBar | 시스템 네비바 + Liquid Glass |
| SF Symbol | 일반 렌더링 | Liquid Glass 스타일 렌더링 |
| viewWillAppear | 추가 작업 없음 | setupSystemNavigationBar() 호출 |
| 레이아웃 기준 | customTitleBar 기준 | safeArea 기준 |

## 현재 디버그 코드 현황

### 측정 가능한 구간 (ViewerViewController+SimilarPhoto.swift)

| 구간 | 로그 태그 | 비고 |
|------|----------|------|
| 캐시 체크 → +버튼 표시 | `[ViewerPerf] Cache HIT/MISS` | 버튼 표시까지 시간 |
| +버튼 탭 → ComparisonGroup 생성 | `[ViewerPerf] +Button → ComparisonGroup` | present 직전까지 |
| 통계 리포트 | `VIEWER PERFORMANCE` | 3회마다 출력 |

### 측정 불가능한 구간 (추가 필요)

| 구간 | 필요성 | 우선순위 |
|------|--------|----------|
| present() → viewDidLoad | Liquid Glass 초기화 비용 | 높음 |
| viewDidLoad → viewWillAppear | 기본 UI 설정 비용 | 중간 |
| viewWillAppear 내 setupSystemNavigationBar() | **iOS 26 핵심 의심 구간** | 높음 |
| viewDidAppear | 전체 present 완료 시간 | 높음 |
| 첫 번째 셀 이미지 로드 완료 | 이미지 병목 측정 | 높음 |

### 추가할 측정 포인트

```
[T0] +버튼 탭 ─────────────────── ✅ 있음
[T1] ComparisonGroup 생성 완료 ── ✅ 있음
[T2] present() 호출 직전 ──────── ❌ 없음
[T3] viewDidLoad 시작 ─────────── ❌ 없음
[T4] viewWillAppear 시작 ──────── ❌ 없음
[T5] setupSystemNavigationBar() ─ ❌ 없음 (iOS 26)
[T6] viewDidAppear ────────────── ❌ 없음
[T7] 첫 번째 셀 로드 완료 ──────── ❌ 없음
```

**결론**: present() 이후 구간 측정이 없어서 iOS 26에서 정확히 어디가 느린지 파악 불가
