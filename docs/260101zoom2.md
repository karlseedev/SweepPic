# Visible Cells Transform 기반 핀치 줌 계획 (260101)

## 목표
- CompositionalLayout 유지
- 핀치 중 연속적인 시각 피드백 제공(사진앱 체감 근접)
- layout invalidation 최소화 및 60fps 안정성 확보
- Grid/Album/TrashAlbum 동작 동일화

## 현 상태 요약
- 현재는 임계값(1.15/0.85) 도달 시 즉시 레이아웃 전환만 수행
- 핀치 중에는 시각적 피드백이 거의 없음
- `UICollectionViewTransitionLayout`은 CompositionalLayout과 호환성 이슈가 있음

## 설계 방향 (Visible Cells Transform)
- 핀치 중에는 visible 셀에만 `CGAffineTransform` 적용
- 핀치 종료 시점에만 레이아웃 전환(`setCollectionViewLayout`)
- 앵커는 현재 코드의 assetID 기반 로직 유지(padding 보정 포함)
- 셀 재사용/선택/삭제 상태와 충돌하지 않도록 transform 합성 규칙 정의

## 세부 설계

### 1) Pinch 상태 모델 추가
- 공통 상태 구조체(또는 private 프로퍼티) 추가
  - `isPinching: Bool`
  - `anchorPoint: CGPoint`
  - `currentScale: CGFloat`
  - `lastAppliedScale: CGFloat`
  - `lastUpdateTime: CFTimeInterval`
  - `baseColumnCount: ColumnCount`

### 2) 핀치 제스처 흐름
- `.began`
  - `isPinching = true`
  - `anchorPoint = gesture.location(in: collectionView)`
  - `currentScale = 1.0` / `lastAppliedScale = 1.0`
  - `collectionView.isScrollEnabled = false` (이전 값 저장 후 복구)
  - 스와이프 삭제/투핑거 제스처 활성 상태면 취소(선택)
- `.changed`
  - `scale = clamp(gesture.scale, 0.85...1.15)` (임계값 근접 범위)
  - `anchorPoint`는 핀치 시작 위치로 고정(핀치 중 업데이트 없음)
  - `abs(scale - lastAppliedScale)`가 작으면 스킵(스로틀링)
  - visible 셀 transform 적용
- `.ended` / `.cancelled`
  - `isPinching = false`
  - 셀 transform 초기화 + 스냅백 애니메이션
  - 임계값 기준으로 열 수 변경 여부 결정 후 `performZoom(to:)` 호출
  - `collectionView.isScrollEnabled = true` 복구

### 3) Visible Cells Transform 공식
- 앵커 기준으로 셀 중심 이동 + 스케일
- 이동량: `tx = (cell.center.x - anchor.x) * (scale - 1)`
- 이동량: `ty = (cell.center.y - anchor.y) * (scale - 1)`
- 적용: `CGAffineTransform(translationX: tx, y: ty).scaledBy(x: scale, y: scale)`

```swift
private func applyPinchTransform(scale: CGFloat, anchor: CGPoint) {
    for cell in collectionView.visibleCells {
        guard let cell = cell as? PhotoCell else { continue }
        let center = cell.center
        let dx = center.x - anchor.x
        let dy = center.y - anchor.y
        let tx = dx * (scale - 1)
        let ty = dy * (scale - 1)
        let transform = CGAffineTransform(translationX: tx, y: ty)
            .scaledBy(x: scale, y: scale)
        cell.setPinchTransform(transform)
    }
}
```

### 4) 셀 재사용/선택 상태와 transform 충돌 방지
- 현재 `PhotoCell.updateSelectionUI`에서 `self.transform`을 사용 중
- 선택 스케일과 핀치 스케일이 충돌하지 않도록 분리 적용
- 선택 스케일은 `contentView.transform`, 핀치 스케일은 `cell.transform`에 적용
- `prepareForReuse()`에서 `transform = .identity` 및 `contentView.transform = .identity` 초기화

### 5) 새로 나타나는 셀 처리
- 핀치 중 스크롤을 비활성화하므로 신규 셀 처리 로직은 기본적으로 불필요
- 스크롤 허용으로 전환할 경우 `collectionView(_:willDisplay:forItemAt:)`에서 현재 transform 적용 필요

### 6) 앵커 assetID 유지
- 기존 로직 유지(패딩 보정 포함)
  - Grid: `assetIDForCollectionIndexPath` / `collectionIndexPath(for:)`
  - Album/TrashAlbum: `fetchResult.object(at:)` + padding 보정
- 핀치 종료 후 `performZoom`에서 앵커 indexPath 계산은 동일

### 7) 성능 보호 장치
- 스로틀링: `abs(scale - lastAppliedScale) < 0.01`이면 스킵
- 또는 `lastUpdateTime` 기준 16ms 이하 업데이트 금지
- `invalidateLayout()` 호출 금지

## 결정 사항 (비교 분석)

### A) 핀치 중 스크롤 허용 여부
- 옵션 1: 스크롤 비활성화 — 장점: 앵커 안정성 최고, 신규 셀 처리 불필요, 핀치 성능 예측 가능; 단점: 핀치 중 스크롤 불가
- 옵션 2: 스크롤 허용 + willDisplay 처리 — 장점: 핀치와 스크롤 동시 UX; 단점: 신규 셀 transform 동기화 필요, 시각적 일관성 깨짐 가능
- 결정: **스크롤 비활성화**

### B) PhotoCell transform 합성 방식
- 옵션 1: 단일 transform 합성 — 장점: 구조 변경 최소, 셀 내부 일원화; 단점: 합성 순서 이슈, 상태 관리 증가
- 옵션 2: transform 분리(핀치=cell, 선택=contentView) — 장점: 충돌 감소, 앵커 안정성 높음, 구현 단순; 단점: reuse 초기화 2곳 필요
- 결정: **transform 분리**

### C) 앵커 동작
- 옵션 1: 시작 위치 고정 — 장점: 흔들림 최소, 앵커 안정성 높음; 단점: 손가락 이동 시 중심이 따라오지 않음
- 옵션 2: 손가락 위치 추적 — 장점: 손가락을 따라오는 느낌; 단점: 중심 흔들림 가능, 제어 난이도 증가
- 결정: **고정**

### D) 스케일 클램핑 범위
- 옵션 1: 0.7~1.3 — 장점: 안전 여유 큼; 단점: 임계값(0.85/1.15)과 괴리
- 옵션 2: 0.85~1.15 근접 — 장점: 임계값과 일관, 과도한 변형 방지; 단점: 여유 범위 작음
- 결정: **0.85~1.15 근접**

## 리스크 및 대응
- **선택 스케일과 충돌**: 핀치/선택 transform 분리 적용으로 해결
- **핀치 중 스크롤**: scroll 비활성화
- **앵커 고정**: 손가락 이동 시 중심이 따라오지 않음 → 핀치 체감 확인 필요
- **클램핑 좁은 범위**: 강한 핀치 입력에서 저항감 발생 가능 → 체감 테스트 필요
- **셀 재사용 시 잔류 transform**: prepareForReuse에서 초기화
- **성능 저하**: visibleCells만 처리 + 스로틀링
