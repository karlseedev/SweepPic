# iOS 26 System UI 모션 분석 방법 조사 정리

**작성일**: 2026-01-22
**목표**: iOS 26 시스템 UI(Photos 앱 등)에서 자동으로 적용되는 모션(스프링/애니메이션 파라미터)을 추출하거나 재현하기 위한 방법 조사 및 비교.

---

## 0. 현재 도구 상태
- `SystemUIInspector`는 뷰/레이어 **정적 속성 덤프**만 수행합니다.
- `animationKeys()`, `presentationLayer()`, `CAAnimation` 캡처 같은 **모션 수집 로직이 없음**.
- 따라서 현재 도구만으로는 **모션 파라미터를 가져올 수 없습니다**.

---

## 1. 조사한 방법 목록 (장단점 포함)

### 1) `CALayer.add(_:forKey:)` 후킹(스위즐링)
**개념**: 애니메이션이 레이어에 추가되는 순간 `CAAnimation` 객체를 직접 덤프.

**장점**
- 스프링이면 `mass / stiffness / damping / initialVelocity / settlingDuration`까지 **정확히 추출 가능**.
- 애니메이션 시작 시점을 **100% 포착** 가능.

**단점**
- 런타임 스위즐링은 **침습적**이며 충돌 위험이 있음.
- **디버그 전용**으로만 사용 권장 (배포 불가 위험).

**활용 포인트**
- 시스템 UI 애니메이션이 **CAAnimation 기반**일 때 가장 강력함.

---

### 2) `UIViewPropertyAnimator` / `UISpringTimingParameters` 캡처
**개념**: `UIViewPropertyAnimator` 생성 시점에 `timingParameters`를 읽어서 스프링 파라미터 확보.

**장점**
- UIKit 스프링 애니메이션 파라미터를 **공식 API로 확인** 가능.
- 비교적 안전.

**단점**
- 시스템 UI가 내부에서 만든 animator는 외부에서 접근 불가.
- 마찬가지로 **생성 시점 후킹**이 필요할 수 있음.

---

### 3) `CADisplayLink` + `presentationLayer()` 샘플링
**개념**: 매 프레임 `presentationLayer()` 값을 샘플링해 모션 곡선을 재구성(피팅).

**장점**
- 스위즐링 없이 **안전하게 동작**.
- 실제 화면에서 보이는 값으로 곡선 추정 가능.

**단점**
- 계산 비용이 크고, 짧은 애니메이션은 놓칠 수 있음.
- 스프링 파라미터는 **역추정**이므로 오차 발생 가능.

---

### 4) 지연 캡처(Delayed Capture) + `animationKeys()` 스캔
**개념**: 사용자가 동작을 수행한 뒤 특정 시점에 레이어 트리를 순회해 활성 애니메이션을 덤프.

**장점**
- 구현이 간단하고 안전함.
- 일정 타이밍에 **실행 중인 CAAnimation**을 확보 가능.

**단점**
- 타이밍을 놓치면 애니메이션이 이미 종료됨.
- 짧은 애니메이션에는 취약.

---

### 5) `CAAnimationDelegate` 활용
**개념**: 애니메이션 시작/종료 타이밍을 기록해서 기간과 시퀀스를 추적.

**장점**
- 생명주기(시작/종료) 파악에 유용.
- 모션 타이밍 분석에 보조적 역할.

**단점**
- 실제 스프링 파라미터는 얻기 어려움.
- 결국 1) 또는 3)과 결합 필요.

---

## 2. 현실적 제약
- **다른 앱(Photos)**의 시스템 UI 모션은 샌드박스 때문에 직접 캡처 불가.
- 따라서 **우리 앱 내에서 동일한 UI를 만들고** 분석하는 방식이 현실적.
- 디버그 전용 기능으로만 구현해야 안전.

---

## 3. 추천 전략
1. **스위즐링으로 CAAnimation 캡처** (정확도 최우선)
2. **지연 캡처 + CADisplayLink 샘플링**으로 누락 보완
3. 필요한 경우 `UIViewPropertyAnimator` 생성 시점 로깅 추가
