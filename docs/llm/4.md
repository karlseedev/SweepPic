## 상세 분석

### 1. Precision 모드와 Strong 신호 논리 모순 (심각도: 높음)

**원본 확인**:
- 39-46행: `렌즈 가림`, `주머니 샷`, `극단 단색`이 **Strong** 신호로 정의
- 89-96행: Precision 모드에서 이들을 **"의도적으로 제외"**
- 126-137행: Stage 2 파이프라인에서 모두 `LOW_QUALITY (Strong)`으로 판정

**모순**:
Strong 신호의 정의는 "단일 항목만으로 즉시 저품질 확정"인데, Precision 모드에서 제외한다면 **Strong이 아님**.
Stage 2 파이프라인에는 모드 분기 없이 모두 Strong으로 처리하도록 되어 있어 명세와 구현이 불일치.

**해결 제안**:
신호를 3단계로 재분류하고, 파이프라인에 모드별 분기 명시:

```
| 구분 | 항목 | Precision | Recall |
|:---:|------|:---------:|:------:|
| Strong | 극단 노출, 심각 블러 | O | O |
| Conditional | 렌즈 가림, 주머니 샷, 극단 단색 | X | O |
| Weak | 일반 블러, 일반 노출, 저해상도 등 | X | 2+조합 |
```

Stage 2 파이프라인 수정:
```
판정:
  • 극단 노출 → LOW_QUALITY (Strong - 모든 모드)
  • 극단 단색/렌즈 가림/주머니 샷 → LOW_QUALITY (Recall 모드만)
```

### 2. iOS 18 AestheticsScore와 파이프라인 관계 불명확 (심각도: 높음)

**원본 확인**:
- 289행: "iOS 18+ → `CalculateImageAestheticsScoresRequest` 우선 사용"
- 296-307행: 코드 예시에서 `overallScore < 0 → 저품질`
- 기존 4단계 파이프라인과의 관계 미정의

**질문**:
1. AestheticsScore가 기존 파이프라인을 **완전 대체**하는가?
2. `isUtility == true`일 때 "별도 처리"의 구체적 동작?
3. AestheticsScore 실패 시 기존 파이프라인으로 Fallback?

**해결 제안**:
섹션 6.1을 확장하여 분기 로직 명시:

```
## iOS 버전별 전략 (상세)

iOS 18+:
  1. AestheticsScore 분석 시도
  2. 성공 시:
     - overallScore < 0 → LOW_QUALITY (기존 파이프라인 대체)
     - isUtility == true → UTILITY 카테고리 (스크린샷과 동일 취급)
  3. 실패 시 (시뮬레이터 등):
     - 기존 4단계 파이프라인으로 Fallback

iOS 16-17:
  - 기존 4단계 파이프라인 사용
```

### 3. Safe Guard 적용 위치/설명 혼란 (심각도: 중간)

**원본 확인**:
- 57-64행: Safe Guard 표에서 "전체 무효: 즐겨찾기, 편집됨" + "적용 위치: Stage 4"
- 119-122행: Stage 1에서 즐겨찾기/편집됨/스크린샷 처리 (SKIP)
- 161-162행: "노출 판정인 경우: Safe Guard 적용 안함 (이미 Stage 1에서 처리)"

**혼란 지점**:
- Safe Guard 표에 "적용 위치: Stage 4"라고 했는데, 즐겨찾기/편집됨은 Stage 1에서 처리
- 162행 "노출 판정은 Stage 1에서 처리"는 잘못된 설명. Stage 1은 메타데이터(즐겨찾기 등)만 처리하지 노출 판정을 하지 않음

**해결 제안**:
Safe Guard를 시점별로 분리하고, Stage 4 설명 수정:

```
### C. 안전장치 (Safe Guard)

| 적용 시점 | 조건 | Stage | 효과 |
|----------|------|:-----:|------|
| 조기 필터 | 즐겨찾기, 편집됨 | 1 | 전체 분석 건너뜀 |
| 후처리 필터 | 심도 효과, 선명한 얼굴 | 4 | 블러 판정만 무효화 |

Stage 4 설명 수정:
  • 블러 판정인 경우:
    - depthEffect 확인 → SKIP
    - FaceQuality >= 0.4 확인 → SKIP
  • 노출/단색/렌즈가림 판정인 경우:
    - 후처리 Safe Guard 해당 없음 (그대로 LOW_QUALITY)
```

### 6. iCloud 제외 시 UX 안내 부재 (심각도: 중간)

│  로컬에 저장된 사진만 분석됩니다      │

**후보군 선택적 다운로드** (선택적 구현):
- 메타데이터로 1차 필터링 후
- 후보군에 대해서만 iCloud 다운로드 수행

### 9. 비네팅 수치 계산 방법 미정의 (심각도: 중간)

**원본 확인**:
- 44행: "비네팅<0.05" 조건이 주머니 샷 판별에 사용
- 계산 방법/공식 없음

**문제점**:
구현마다 결과가 달라질 수 있음.

**해결 제안**:
섹션 6에 비네팅 계산 추가:

```
### 6.6 비네팅 계산

비네팅 = 1 - (모서리 평균 휘도 / 중앙 평균 휘도)

- 3×3 그리드 기준
- 중앙: (1,1) 셀
- 모서리: (0,0), (0,2), (2,0), (2,2) 셀의 평균
- 결과: 0.0 (비네팅 없음) ~ 1.0 (완전 어두운 모서리)
- 주머니 샷 기준: < 0.05 (즉, 모서리와 중앙 차이 거의 없음)
```
