# 260121zoom2 요구사항 구현 조사 정리

**작성일**: 2026-01-21
**목표**: `docs/260121zoom2.md` 요구사항을 충족하는 구현 방법 조사 및 정리

---

## 1. 요구사항 핵심 요약

- 핀치/회전 동안 **레이아웃이 점진적으로 변화**해야 함 (진행도 기반).
- **셀 개별 이동 애니메이션 금지** (예: batchUpdates/performBatchUpdates 기반의 이동 애니메이션 금지).
- **전체가 하나처럼 연속적인 모션**.
- **fadeIn**: 앵커 셀 제외 모든 visible 셀에 목표 이미지로 전환.
- 기준점은 **1/3/5열**만 사용, 완료 시 반드시 스냅.
- 핀치/회전 모두 동일한 원리 적용.
- 손을 뗄 때: 현재 모션을 이어받아 자연스럽게 마무리(끊김/점프 금지).

이 요구사항을 만족하려면 **레이아웃 계산을 “연속적인 열 수(실수)” 기반으로 구현**하는 것이 핵심이다.

---

## 2. 구현 후보와 적합성 평가

### 2.1 `UICollectionViewTransitionLayout` (startInteractiveTransition)

**개념**
- 두 레이아웃 사이를 `transitionProgress`로 선형 보간.

**장점**
- 진행도 기반 인터랙티브 전환에 적합.
- `finishInteractiveTransition()`으로 현재 진행도에서 자연스럽게 마무리 가능.

**한계**
- 항상 “A ↔ B 두 레이아웃”만 보간.
- 5 → 3 → 1 같이 **다단계**는 **중간에서 전환 레이아웃을 재시작**해야 함.
- CompositionalLayout에서는 호환성 이슈가 자주 발생.
- 기준점 통과 시 전환 재시작은 **미세한 끊김 위험**.

**요구사항 적합성**
- 단일 단계(3↔1, 3↔5)는 가능.
- 다단계/연속 전환 요구에는 불리.

---

### 2.2 커스텀 `UICollectionViewLayout` (권장)

**개념**
- `virtualColumns`(실수)를 레이아웃 속성으로 두고,
  **열 수가 실시간으로 변하는 grid 레이아웃**을 직접 계산.

**핵심 아이디어**
- 1/3/5열은 “완료 스냅 기준점”일 뿐,
  핀치 중에는 **2.1열, 4.2열처럼 실수 값**으로 레이아웃을 계산.
- 이 방식은 **셀 이동 애니메이션이 아닌, 단순 레이아웃 재계산**이므로
  금지 조건을 피하면서도 점진적 변화를 제공.

**장점**
- 핀치 진행도에 따른 **진짜 레이아웃 변화** 구현 가능.
- 다단계(5→3→1)를 끊김 없이 처리 가능.
- 회전 시에도 동일 로직 재사용 가능.

**한계**
- Layout 계산 비용 증가 (캐싱/최적화 필요).
- 앵커 고정을 위해 `contentOffset` 보정 로직 필요.

**요구사항 적합성**
- 가장 높은 적합성.

---

### 2.3 스냅샷 오버레이 방식

**개념**
- 핀치 시작 시 `collectionView` 스냅샷을 만든 뒤 이를 스케일.
- 실제 레이아웃은 뒤에서 별도로 전환.

**장점**
- “스크린샷이 확대/축소되는 느낌” 구현에 유리.

**한계**
- 요구사항의 “레이아웃이 점진적으로 변화”와 충돌.
- fadeIn이 스냅샷 레이어와 충돌 가능.

**요구사항 적합성**
- 단독으로는 부족. 보조 효과로만 사용 가능.

---

## 3. 권장 접근: “연속 열 수 커스텀 레이아웃”

### 3.1 구조 개요

1. **Custom Layout**에 `virtualColumns` 속성을 둔다.
2. 핀치 진행도에 따라 `virtualColumns`를 실시간 업데이트.
3. Layout은 이 값으로 **셀 크기/위치**를 계산한다.
4. 기준점(1/3/5)을 통과하면 **fadeIn 시작**.
5. 손을 떼면 `virtualColumns`를 목표 열 수로 **애니메이션**해서 마무리.

이 구조는 “점진적 레이아웃 변화”, “연속 모션”, “끊김 없는 완료”를 동시에 만족한다.

---

## 4. 상세 구현 설계

### 4.1 Custom Layout 핵심 계산

```swift
final class ContinuousGridLayout: UICollectionViewLayout {
    var virtualColumns: CGFloat = 3.0 { didSet { invalidateLayout() } }
    var itemSpacing: CGFloat = 2.0

    override func prepare() { /* 캐싱 준비 */ }

    override func layoutAttributesForElements(in rect: CGRect) -> [UICollectionViewLayoutAttributes]? {
        // virtualColumns 기준으로 셀 프레임 계산
    }

    override func layoutAttributesForItem(at indexPath: IndexPath) -> UICollectionViewLayoutAttributes? {
        // 동일한 계산 로직
    }

    override var collectionViewContentSize: CGSize {
        // virtualColumns 기준으로 전체 contentSize 계산
    }
}
```

**계산 방식 예시**
- `columnWidth = (bounds.width - spacing*(virtualColumns-1)) / virtualColumns`
- 인덱스 → row/col 계산:
  - `col = index % Int(ceil(virtualColumns))` 대신,
  - **실수 열 수**에서는 실제 간격을 일정하게 유지하고
    `x = colIndex * (columnWidth + spacing)` 방식으로 계산.
- 중요한 포인트: `virtualColumns`가 실수일 때 **열 수는 그대로 실수 값으로 사용**하여
  `columnWidth`만 자연스럽게 변화하게 한다.

> 결과적으로 핀치 진행에 따라 셀 사이 간격/크기가 연속적으로 변한다.

---

### 4.2 앵커 셀 고정 (콘텐츠 오프셋 보정)

**목표**: 핀치 중심점(혹은 화면 중앙)에 있는 셀이 화면에서 고정되어 보이게 한다.

**원리**
1. 핀치 시작 시 앵커의 **화면 좌표**(`anchorPointInView`)를 저장.
2. 매 프레임마다 앵커 셀의 **콘텐츠 좌표**(`anchorCenterInContent`)를 계산.
3. `contentOffset = anchorCenterInContent - anchorPointInView`로 설정.

```swift
func updateContentOffsetForAnchor(anchorIndexPath: IndexPath, anchorPointInView: CGPoint) {
    guard let attributes = layoutAttributesForItem(at: anchorIndexPath) else { return }
    let anchorCenter = attributes.center
    let newOffset = CGPoint(
        x: anchorCenter.x - anchorPointInView.x,
        y: anchorCenter.y - anchorPointInView.y
    )
    collectionView.contentOffset = clamped(newOffset)
}
```

이 방식은 layer.transform이나 anchorPoint 변경 없이도 **앵커 고정**을 달성한다.

---

### 4.3 다단계 기준점 통과 처리

**문제**: 빠른 핀치로 한 번에 5→1 이동 시 기준점 5,3,1 모두 통과.

**해결**: 기준점 통과를 “배열”로 반환하고, **방향에 맞는 순서로 처리**.

```swift
func detectCrossedThresholds(prev: CGFloat, cur: CGFloat, thresholds: [CGFloat]) -> [CGFloat] {
    if cur < prev {
        // 축소 방향: 높은 값 -> 낮은 값
        return thresholds.filter { $0 <= prev && $0 >= cur }.sorted(by: >)
    } else {
        // 확대 방향: 낮은 값 -> 높은 값
        return thresholds.filter { $0 >= prev && $0 <= cur }.sorted(by: <)
    }
}
```

- 반환된 배열 순서대로 fadeIn을 **연속 트리거**.
- 동일 단계 중복을 방지하기 위해 `currentTargetColumns` 체크 필요.

---

### 4.4 손을 뗄 때 스냅 로직 (10% / 50% 규칙)

**필요 상태**
- `initialDirection` (처음 감지된 방향)
- `didReverse` (방향 전환 여부)
- `progressInCurrentStage` (0~1)

**판단 로직**
- 방향 전환이 없으면 `progress >= 0.1` 시 **그 방향 완료**.
- 방향 전환이 있으면 `progress >= 0.5` 기준으로 결정.

**완료 애니메이션**
- 현재 `virtualColumns` → 목표 기준점까지 `UIViewPropertyAnimator`로 보간.
- 진행 중 끊김 없도록 **같은 레이아웃 계산식 유지**.

---

### 4.5 fadeIn 처리

**요구사항 반영 원칙**
- 앵커 셀은 제외.
- 기준점 통과 시마다 새 목표 열 수 이미지로 fadeIn.
- 빠른 핀치 시 중복 트리거 방지.

**권장 방식**
- `cellReuseToken` 또는 `assetID`로 완료 시 검증.
- `currentTargetColumns`와 요청 시점의 columns를 비교.

```swift
imagePipeline.loadThumbnail(..., token: requestToken) { image, token in
    guard token == cell.currentToken else { return }
    guard pinchState.currentTargetColumns == requestedColumns else { return }
    fadeIn(...)
}
```

## 5. 회전 적용

회전은 핀치와 동일하게 `virtualColumns`를 목표 열 수로 **애니메이션 보간**하여
시스템 회전 애니메이션과 동기화한다.

**핵심 포인트**
- 회전 시작 시 fadeIn 트리거.
- `UIViewControllerTransitionCoordinator`의 `animate(alongsideTransition:)`에서
  `virtualColumns`를 보간.
- 회전 완료 후 앵커 스크롤 복구.

---

## 6. 결론: 요구사항 달성 가능성

**요구사항을 “정확히” 맞추려면**
- 단순 transform 방식은 부족.
- `UICollectionViewTransitionLayout`만으로는 다단계가 불안정.
- **연속 열 수 커스텀 레이아웃 + 앵커 보정 + 단계별 fadeIn**이 가장 안정적.

이 방식은 아래 요구를 모두 충족한다:
- 셀 이동 애니메이션 없음 (레이아웃 재계산만 수행).
- 핀치 진행도에 따른 점진적 레이아웃 변화.
- 다단계(5→3→1)에서 단계별 fadeIn.
- 손을 뗄 때 끊김 없는 자연스러운 완료.
