# iOS 시스템 UI 모션 분석 방법 조사 보고서

**작성일**: 2026-01-22
**목표**: iOS 26 시스템 UI(Photos 앱 등)에서 사용되는 애니메이션의 물리적 파라미터(Damping, Stiffness, Mass, Duration 등)를 추출하여 커스텀 구현에 반영하기 위한 기술 조사.

---

## 1. 개요

iOS 시스템 UI의 "쫀득한" 느낌은 `CASpringAnimation` 또는 `UIViewPropertyAnimator`의 정교한 물리 상수 설정에서 나옵니다. 이를 똑같이 구현하려면 단순한 관측이 아닌, 실제 실행 중인 애니메이션 객체의 내부 값을 추출해야 합니다.

---

## 2. 분석 방법 비교

### 2.1. 정적 분석 (Current SystemUIInspector)
현재 구현된 방식입니다. 화면의 UI 계층 구조를 스냅샷 찍어 분석합니다.

*   **원리**: `UIWindow`부터 시작하여 모든 하위 뷰/레이어를 순회하며 속성(`frame`, `color`, `cornerRadius` 등)을 덤프합니다.
*   **장점**: 안전하며, 정적인 레이아웃 구조와 스타일(색상, 블러 등)을 파악하는 데 탁월합니다.
*   **단점**: **애니메이션 정보를 얻을 수 없습니다.** 애니메이션은 "과정"이므로 정적 스냅샷에는 최종 상태(`model layer`)나 현재 보이는 상태(`presentation layer`)의 값만 남고, "어떻게 움직이는지"(물리 상수)는 남지 않습니다.

### 2.2. 실시간 폴링 (CADisplayLink)
화면 주사율에 맞춰 매 프레임마다 레이어의 상태를 확인하는 방식입니다.

*   **원리**: `CADisplayLink`를 사용하여 매 프레임마다 관심 있는 레이어의 `animationKeys()`를 확인하고, 애니메이션이 감지되면 정보를 추출합니다.
*   **장점**: 애니메이션의 시작과 끝을 실시간에 가깝게 포착할 수 있습니다.
*   **단점**:
    *   **오버헤드**: 매 프레임 모든 뷰를 검사하는 것은 성능에 큰 부담을 줍니다.
    *   **타이밍 이슈**: 애니메이션이 추가되고 1프레임 내에 끝나거나 제거되는 경우 놓칠 수 있습니다.

### 2.3. 메서드 스위즐링 (Method Swizzling) - 가장 강력함
런타임에 시스템 메서드를 가로채는 방식입니다.

*   **원리**: `CALayer.add(_:forKey:)` 또는 `UIView.animate` 메서드를 후킹(Hooking)하여, 애니메이션 객체가 시스템에 등록되는 **그 순간**을 포착합니다. 이때 전달되는 `CAAnimation` 객체의 속성을 낱낱이 기록합니다.
*   **장점**:
    *   **완전한 포착**: 모든 시스템 애니메이션의 초기 설정값(Initial Settings)을 100% 정확하게 얻을 수 있습니다.
    *   **타이밍 무관**: 애니메이션이 아무리 짧아도 등록되는 순간을 잡기 때문에 놓치지 않습니다.
*   **단점**:
    *   **위험성**: 런타임 조작은 앱의 안정성을 해칠 수 있으며, 잘못 구현 시 충돌(Crash)의 원인이 됩니다.
    *   **구현 복잡도**: Objective-C 런타임 API를 다워야 하므로 구현이 까다롭습니다.
    *   **배포 불가**: 디버깅 용도로만 사용해야 하며, 앱스토어 심사에서 거절될 수 있습니다.

### 2.4. 지연 캡처 (Delayed Capture) - 권장 방식
사용자가 동작을 수행하는 동안 기다렸다가, 특정 시점에 실행 중인 애니메이션을 조회하는 방식입니다.

*   **원리**: "3초 후 캡처" 버튼을 누른 뒤, 사용자가 UI 동작(탭 전환 등)을 수행합니다. 3초 뒤 도구가 모든 레이어의 `layer.animation(forKey:)`를 조회하여 현재 실행 중인(Active) 애니메이션 객체를 가져옵니다.
*   **장점**:
    *   **안전함**: 시스템을 변조하지 않고 Public API만 사용하므로 안전합니다.
    *   **정확함**: 실제 렌더링에 쓰이고 있는 `CASpringAnimation` 객체를 직접 열어볼 수 있어 물리 상수를 정확히 얻을 수 있습니다.
    *   **구현 용이**: Swift 코드로 깔끔하게 구현 가능합니다.
*   **단점**: 캡처 시점에 애니메이션이 이미 끝나버렸다면 정보를 얻을 수 없습니다. (타이밍을 잘 맞춰야 함)

## 4. 결론 및 제안

iOS 26 시스템 UI의 애니메이션을 가장 안전하고 정확하게 분석하기 위해 **"방법 4: 지연 캡처(Delayed Capture)"** 방식을 우선적으로 적용하는 것을 제안합니다.

이 방식은 `SystemUIInspector`에 "3초 타이머" 기능을 추가하는 것만으로 구현 가능하며, 사용자가 3초 안에 탭 전환이나 제스처를 수행하면 그 순간의 `damping`, `stiffness` 값을 파일로 저장해줍니다. 만약 이 방식이 타이밍 문제로 데이터를 놓친다면, 차선책으로 "방법 3: 메서드 스위즐링"을 제한적으로 도입할 수 있습니다.
