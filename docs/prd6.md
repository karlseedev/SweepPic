# PickPhoto PRD v6

## 0. 문서 정보

| 항목 | 내용 |
|------|------|
| 제품명 | PickPhoto |
| 목적 | iOS 기본 사진 앱과 유사한 "매우 빠른" 사진 브라우징 경험 제공 |
| 문서 성격 | 제품 요구사항 (PRD) + 성능/품질 수용 기준 + UI/UX 상세 스펙 |

## 1. 배경/문제 정의

iOS 기본 사진 앱은 대규모 라이브러리에서도 다음을 제공합니다:
- 빠른 실행
- 즉시성 있는 썸네일 채움
- 매우 부드러운 스크롤 (특히 ProMotion)
- 자연스러운 전환/제스처

**PickPhoto는 가능한 한 동일한 공개 API(주로 PhotoKit) 기반으로 이 체감 품질을 달성하는 것을 목표로 합니다.**

추가로, 위 스와이프 삭제 등 "사진 정리에 특화된" UX를 제공하여 기본 앱 대비 차별점을 만듭니다.

## 2. 목표 (Goals)

### 2.1 핵심 UX 목표

- All Photos 그리드가 "즉시 열리고", 스크롤이 "매우 부드럽다"고 느껴지게 합니다.
- 핀치 제스처로 그리드 밀도가 자연스럽게 변화합니다 (Photos와 유사한 감성).
- 앨범 리스트/앨범 진입에서도 그리드 브라우징을 제공합니다. (앨범 핀치 줌은 M2 이후)
- 그리드와 뷰어에서 모두 삭제가 가능하며, 삭제 후 흐름이 끊기지 않습니다.

### 2.2 성능/품질 목표 (측정 가능한 KPI)

아래 수치는 **초기 목표치**이며, 스파이크(실험) 및 계측 결과로 조정될 수 있습니다.

| 항목 | 목표 (Target) | 허용 (Acceptable) | 출처 |
|------|--------------|-------------------|------|
| 스크롤 hitch time ratio | < 5 ms/s | < 5 ms/s | [Apple WWDC 2020](https://developer.apple.com/videos/play/wwdc2020/10077/) Good 기준 |
| 첫 그리드 유효 썸네일 | < 300ms | < 300ms | [NNg 지각 임계](https://www.nngroup.com/articles/response-times-3-important-limits/) |
| 썸네일 응답 시간 (p50) | < 100ms | < 100ms | [NNg 즉각 반응](https://www.nngroup.com/articles/response-times-3-important-limits/) |
| 오표시 (정확성) | 0 | 0 | 품질 필수 요건 |
| 메모리 상한 | < 400MB | < 500MB | iPhone 11+ (4GB RAM) 기준, Jetsam 임계값 ~2,000MB의 20~25% |
| 삭제 반영 | < 100ms | < 250ms | [NNg 직접 조작](https://www.nngroup.com/articles/response-times-3-important-limits/) / [Apple hang 임계](https://developer.apple.com/videos/play/wwdc2020/10077/) |
| 콜드 스타트 | < 400ms | < 1s | [Apple WWDC 2019](https://developer.apple.com/videos/play/wwdc2019/423/) / [NNg 사고 흐름 한계](https://www.nngroup.com/articles/response-times-3-important-limits/) |

**지표 정의 보충:**
- **hitch time ratio**: 프레임 예산(120Hz: 8.3ms / 60Hz: 16.7ms) 초과분을 누적하여 시간당 ms로 표현. Good: < 5 ms/s, Warning: 5-10 ms/s, Critical: > 10 ms/s.
- **첫 그리드 유효 썸네일**: 앱 진입부터 최초 화면 셀 N개(예: 12개)가 실 이미지로 채워질 때까지의 시간. N은 스파이크에서 확정.
- **썸네일 응답 시간**: 셀 요청부터 표시까지의 시간.
- **오표시**: 셀 재사용 시 이전 이미지가 잠깐 보이는 현상. 0이 아니면 품질 실패.
- **메모리 상한**: 초과 시 iOS Jetsam이 앱을 강제 종료. iPhone 11+ (4GB) 기준 크래시 임계값 ~2,000MB.
- **삭제 반영**: 목표 100ms는 "직접 조작 느낌", 허용 250ms는 "UI hang 임계점".
- **콜드 스타트**: 목표 400ms는 Apple 권장, 허용 1s는 "사용자 사고 흐름이 끊기지 않는 한계".

### 5.3 네비게이션 구조

**그리드 화면은 edge-to-edge이며, 상/하단 UI는 콘텐츠 위에 떠 있는 플로팅 오버레이로 표시한다.**

```
┌─────────────────────────────────────┐
│ [상단 플로팅 오버레이 - 블러+딤]      │
│   "Photos"              [Select]    │
│   (타이틀은 통과, Select만 터치)      │
├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
│                                     │
│      Edge-to-Edge Content           │
│   (콘텐츠가 화면 전체, safe area 무시) │
│                                     │
├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
│ [하단 플로팅 오버레이 - 블러+딤]      │
│   [📷 Library]  [📁 Albums]         │
│   캡슐 형태 컨테이너 (iOS 18 Photos)  │
└─────────────────────────────────────┘
```

**iOS 버전별 정책:**

| iOS 버전 | 정책 |
|----------|------|
| iOS 16~25 | 시스템 바(네비/탭) 숨김 + 커스텀 플로팅 오버레이 사용 (상단 타이틀+Select버튼 + 하단 캡슐 탭) |
| iOS 26+ | 커스텀 오버레이 비활성화, 시스템 기본 외관 사용 |

**iOS 버전별 조합 매트릭스:**

| iOS 버전 | UI 스타일 | 전환 애니메이션 | 비고 |
|----------|-----------|-----------------|------|
| 26+ | 시스템 기본 UI | 네이티브 줌 (`preferredTransition`) | 최신 환경 |
| 18~25 | 커스텀 FloatingUI | 네이티브 줌 (`preferredTransition`) | 현재 주력 타겟 |
| 16~17 | 커스텀 FloatingUI | 커스텀 또는 기본 전환 | 출시 전 fallback 구현 필요 |

> 상세 구현은 [ios18version.md](./ios18version.md) 참조

**탭 구성:**

| 탭 | 아이콘 | 화면 |
|----|--------|------|
| Library | `photo.on.rectangle` (SF Symbol) | All Photos 그리드 |
| Albums | `rectangle.stack` (SF Symbol) | 앨범 리스트 |

### 5.4 공통 상호작용

**핀치 연속 줌**
- MVP: **All Photos 내부에서만** 밀도 변화 제공
- 앨범 그리드는 MVP에서 3열 고정 (핀치 줌 M2 이후)
- **그리드 줌 단계: 1/3/5열** (iOS Photos 앱 동일, 자체 측정)
- 목표 동작: Photos처럼 부드럽게 변화하는 감성
- Years/Months/Days 모드 스냅은 MVP 이후

**앵커 유지 (핵심)**
- 핀치 제스처 중심점(두 손가락 중간) 아래 콘텐츠를 앵커로 캡처
- 줌 변화 중/후에도 해당 콘텐츠가 튀지 않게 유지 (오프셋 보정)

**멀티 선택**
- Select 모드로 여러 장 선택 후 일괄 삭제
- **Select All 없음** - 드래그 선택만 제공 (실수 방지)

### 5.5 삭제 정책

| 항목 | 정책 |
|------|------|
| 삭제 정책 | 항상 "사진 삭제"로 단순화 (앨범에서 "앨범 제거" 없음) |
| 삭제 동작 | **앱 내 휴지통으로 이동** (2단계 삭제) |
| 확인 팝업 | 앱 자체 확인 없음, **완전삭제 시 iOS 시스템 팝업만** (필수) |
| 삭제 위치 | 그리드 (단일 + Select 모드) + 뷰어 (단일) |
| 뷰어 삭제 제스처 | **위 스와이프 = 앱 내 휴지통 이동** (Tinder 스타일) |
| 삭제 후 이동 (뷰어) | **이전 사진** → 없으면 다음 → 없으면 그리드 복귀 |
| 휴지통 사진 표시 | 그리드에서 **딤드(어둡게) 처리**하여 표시 |
| 복구 | **앱 내 휴지통에서 복구** (탭 → 뷰어 → 복구 버튼) |
| 완전 삭제 | 휴지통 비우기 또는 개별 완전삭제 시 iOS "최근 삭제됨"으로 이동 |

**삭제 안전장치 (필수 요구사항)**

| 조건 | 요구사항 |
|------|----------|
| 삭제 가능 조건 | `.authorized` 또는 `.limited` 일 때 삭제 가능 |
| .limited 정책 | 선택된 사진은 삭제 허용 (Photos 앱과 동일) |
| 권한 부족 시 | 삭제 UI 비활성화 또는 권한 요청 안내 표시 |
| 연속/멀티 삭제 | 삭제 후 UI가 안정적으로 수렴해야 함 (선택/스크롤/뷰어 이동 규칙 준수) |
| 뷰어 삭제 후 이동 | 반드시 "이전 사진 우선" 규칙 준수 |
| **외부 변경 시** | 동일 규칙 적용 (외부에서 삭제된 사진도 "이전 사진 우선" 이동) |

### 5.6 의도된 차별점 (Photos 앱과 다른 점)

> PickPhoto는 "기본 Photos 앱과 유사한 경험"을 목표로 하지만, **사진 정리 특화**를 위해 다음 UX는 의도적으로 다르게 설계되었습니다.

| 항목 | 기본 Photos 앱 | PickPhoto | 이유 |
|------|---------------|-----------|------|
| 삭제 동작 | 즉시 iOS 휴지통 + 확인 팝업 | **앱 내 휴지통 → 완전삭제 시 시스템 팝업만** | 빠른 정리 (앱 자체 확인 없음) |
| 뷰어 삭제 제스처 | 하단 삭제 버튼만 | **위 스와이프 = 삭제** | Tinder 스타일 빠른 분류 |
| 삭제 후 표시 | 완전 제거 | **딤드 표시 유지** | 삭제한 것 시각적 확인/복구 용이 |
| 복구 | iOS "최근 삭제됨" | **앱 내 휴지통에서 즉시 복구** | 빠른 실수 복구 |
| 즐겨찾기 | 하트 버튼 지원 | MVP 제외 | 정리 특화 (분류보다 삭제 중심) |

## 6. UI/UX 상세 스펙

### 6.1 그리드 레이아웃

**기본 구조:**

| 항목 | 값 | 비고 |
|------|-----|------|
| 기본 열 수 | 3열 | 앱 시작 시 기본값 (iOS Photos 앱 동일) |
| 줌 단계 | 1열 / 3열 / 5열 | 핀치로 전환 (iOS Photos 앱 동일, 자체 측정) |
| 셀 간격 | 2px | 모든 방향 동일 (iOS Photos 앱 참조, 자체 측정) |
| 셀 비율 | 1:1 (정사각형) | |
| 섹션 헤더 | 없음 (MVP) | M2에서 Days 모드와 함께 추가 |
| 스크롤 인디케이터 | 기본 (시스템) | 커스텀 스크럽바는 M3 |

**시스템 바 정책 (Edge-to-Edge + 플로팅 UI):**

| 항목 | 정책 |
|------|------|
| 구현 방식 | 시스템 바(네비/탭) 숨김 + 커스텀 플로팅 오버레이 |
| 콘텐츠 영역 | safe area 무시, 화면 전체(edge-to-edge) |
| 상단 오버레이 | 플로팅 타이틀바 (타이틀 + Select 버튼), 44pt + safe area, 블러+딤, **Select만 터치, 나머지 통과** |
| 하단 오버레이 | 플로팅 캡슐 탭바 (iOS 18 Photos 스타일), 블러+딤, **버튼만 터치** |
| contentInset | 플로팅 UI 높이만큼 설정 (첫/마지막 줄 가림 방지) |
| indicatorInsets | contentInset과 동일 |
| 대비 | 밝은 사진 위에서도 아이콘/텍스트 가독성 보장 (블러+딤 알파 0.3~0.4) |

**셀 크기 계산:**
```
cellWidth = (screenWidth - (columns - 1) * spacing) / columns

예시 (iPhone 15 Pro, 393pt 너비, 3열):
cellWidth = (393 - 2 * 2) / 3 = 129.67pt
```

**핀치 줌 파라미터 (Gate 3 확정, [spiketest.md](./spiketest.md#gate-3-결과-pinch-zoom) 참조):**

| 파라미터 | 값 | 설명 |
|----------|-----|------|
| zoomInThreshold | 0.85 | scale < 0.85 → 확대 (열 수 감소) |
| zoomOutThreshold | 1.15 | scale > 1.15 → 축소 (열 수 증가) |
| cooldown | 200ms | 전환 간 최소 간격 |
| 앵커 drift 허용 | 0px (Auto 테스트 기준) | 전환 후 앵커 셀 위치 오차 (Gate 3 결과) |

### 6.2 선택 모드 (Select Mode)

**선택 UI:**

## 7. 핵심 요구사항 (불변 규칙)

> 본 섹션은 **"반드시 지켜야 하는 요구사항"**만 정의합니다.
> 구현 방식/기술 선택은 8장 또는 [TechSpec.md](./TechSpec.md)에서 다룹니다.

### 7.1 대용량 (5만 장) 요구사항

- 5만 장에서도 스크롤 hitch 비율이 목표치 이내여야 함
- "전체 에셋 배열" 또는 "전체 스냅샷 반복 생성"으로 인한 메모리 급증이 없어야 함
- 안정적인 식별자 기반으로만 이동/선택/앵커 유지

### 7.2 스크롤 품질 요구사항

- 빠른 플릭/역방향 스크롤에서도 **오표시 0**
- 화면에 보이지 않는 셀의 로딩이 보이는 셀의 로딩을 방해하면 안 됨
- 스크롤 중 프레임 예산(8.3ms/16.7ms, 1000ms ÷ 주사율)을 초과하는 메인 스레드 작업 금지 ([Apple WWDC 2020](https://developer.apple.com/videos/play/wwdc2020/10077/))

**ProMotion (120Hz) 요구사항 (잠정 결정 - Mock 테스트 기준, 실사진 테스트 보류)**
- ProMotion은 가변 주사율(10~120Hz)이며, 앱이 120Hz를 강제할 수는 없음 ([Apple ProMotion 스펙](https://www.apple.com/iphone-15-pro/))
- **앱은 시스템이 120Hz를 선택할 수 있도록 프레임 버짓(8.3ms) 준수를 보장한다**
- **ProMotion 정책: 시스템 자동 관리** (Mock 테스트 관찰 결과, 실사진+120Hz 조합 테스트 후 재검토 필요)
- 이미지 디코딩 등 무거운 작업은 메인 스레드 외부에서 처리

### 7.3 핀치 줌 요구사항

- **줌 단계: 1/3/5열** (iOS Photos 앱 동일, 자체 측정)
- 핀치 중심점 아래 콘텐츠가 **앵커로 유지**되어야 함 (점프/떨림 불가)
- 핀치 제스처 중 입력 지연이 체감되면 안 됨
- 밀도 변화 후에도 동일 콘텐츠가 화면에 유지

### 7.4 삭제 요구사항

- 멀티 삭제/연속 삭제 후에도 **UI가 안정적으로 수렴**
- 삭제 반영 시간이 목표치(100ms/250ms) 이내
- 뷰어에서 삭제 시 "이전 사진 우선" 이동 규칙 준수

### 7.5 메모리 요구사항

- 장시간 스크롤/뷰어 왕복 후에도 메모리 상한(400MB, 허용 500MB) 이내 유지
- 메모리 경고 발생 시 캐시 즉시 해제

#### 7.5.1 iOS Jetsam 메모리 제한 근거

iOS는 swap 메모리를 지원하지 않아, 메모리 부족 시 **Jetsam**이 앱을 강제 종료한다.

**디바이스별 크래시 임계값 (커뮤니티 테스트 기준):**

## 8. 기술 방향 (확정 항목)

> 스파이크/게이트 결과로 확정된 기술 결정입니다.

### 8.1 확정된 기술 스택

| 항목 | 결정 | 근거 |
|------|------|------|
| UI 프레임워크 | **UIKit 기반** | 셀 재사용 완전 제어, 대용량 메모리 관리, 통일성 우선 |
| 데이터 소스 | **performBatchUpdates** | 50k 기준 일정한 비용, p95 5ms ([spiketest.md Spike 1](./spiketest.md#spike-1-결론)) |
| 핀치 줌 | **0.85/1.15 threshold, 200ms cooldown** | [spiketest.md Gate 3](./spiketest.md#gate-3-결과-pinch-zoom) |
| ProMotion | **시스템 자동 관리 (잠정)** | Mock 테스트 관찰 결과, 실사진 테스트 보류 ([spiketest.md](./spiketest.md#gate-4-결과-120hz-performance)) |

**UI 프레임워크 정책**:
- 기본: UIKit으로 구현
- 예외: SwiftUI가 현저히 유리한 경우, 사용자 확인 후 부분적 하이브리드 허용
- 판단 기준: "UIKit 대비 개발 효율 또는 UX 품질이 현저히 높을 때"

### 8.2 이미지 캐싱

- **PHCachingImageManager**: PhotoKit 공식 API, 시스템 최적화
- 가시 영역 기반 프리히트 정책 필요
- 세부 정책(윈도우 크기, 속도 적응)은 Gate 2 실기기 테스트에서 확정됨

### 8.3 이미지 로딩 정책 (Gate 2 실기기 테스트 완료)

**썸네일 옵션 후보:**

| 후보 | 설명 | 장단점 |
|------|------|--------|
| A) 빠른 표시 우선 | `opportunistic` + `fast` + 셀 픽셀 크기 | 첫 채움↑, 스크롤↑, 저→고 교체감 있을 수 있음 |
| B) 품질 우선 | `highQualityFormat` + targetSize 여유 | 교체감↓, 첫 채움↓, 메모리↑ |

**preheat 후보:**
- A) 고정 윈도우: ±N (N ∈ {1, 2, 3, 4})
- B) 속도 적응형: 느림 ±1, 보통 ±2, 빠름 ±3

## 9. 아키텍처 경계 (모듈 책임)

| 모듈 | 책임 |
|------|------|
| **라이브러리 저장소** | PhotoKit fetch/권한/변경 감지, 정렬/필터 정책 단일 소스 |

### 10.1 개발 전 스파이크 (완료)

#### ~~Spike 0: UI 기술 스택~~ → **확정: UIKit**

> 대용량 그리드에서 UIKit UICollectionView가 SwiftUI LazyVGrid보다 셀 재사용/메모리 관리에서 유리함이 명확하므로 스파이크 없이 확정.

#### Spike 1: 데이터 소스 패턴 결정 → **확정: performBatchUpdates**

**결과 (2025-12-16)**

| 후보 | 50k 단일 삭제 p95 | Hitch Time Ratio | 스케일링 | 판정 |
|------|-------------------|------------------|----------|------|
| Plan A (DiffableDataSource) | 52ms | 22 ms/s ❌ Critical | 선형 증가 | 불합격 |
| **Plan B (performBatchUpdates)** | **5ms** | **0 ms/s** ✅ Good | **일정** | **합격** |

> 상세: [spiketest.md](./spiketest.md)

## 11. 계측/검증 (Definition of Done)

### 11.1 계측 도구

- **Instruments**: Time Profiler, Core Animation (hitch), Allocations/Leaks
- **MetricKit**: 앱 성능 메트릭 수집 (릴리즈 품질 확인)
- **XCTest**: 자동화된 성능 테스트

### 11.2 수용 기준 (MVP Gate)

MVP는 다음 조건을 **모두** 만족해야 합니다:

| 기준 | 조건 |
|------|------|
| 스크롤 품질 | 5만 장에서 hitch 비율이 목표치 이내 |
| 정확성 | 빠른 플릭/역방향 스크롤에서 오표시 0 |
| 핀치 줌 | 앵커가 튀지 않고, 입력 지연 체감 없음 |
| 삭제 안정성 | 멀티/연속 삭제 후 UI 안정 수렴 |
| 메모리 | 장시간 사용 후 400MB 이내 유지 (허용 500MB) |
| **플로팅 UI** | iOS 18에서 그리드가 노치까지 edge-to-edge로 표시되고, 상/하단은 블러+딤의 플로팅 UI로 보임 |
| **터치 충돌** | 상단 오버레이에서 스크롤 제스처 충돌 없음 (터치 통과), 하단은 버튼만 터치 반응 |
