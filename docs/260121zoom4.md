## 0. v5 핵심 수정 사항 (롤백 피드백 반영)

### 0.1 발견된 문제점 (6개)

구현 후 테스트에서 발견된 6가지 문제:

| # | 현상 | 근본 원인 | 상태 |
|---|------|----------|------|
| 1 | X축 좌측 정렬 (Y축은 OK) | X축 앵커 보정 로직 미구현 | 미해결 |
| 2 | 스크롤 점프 + fadeIn 안됨 | 앵커 오류 + fadeIn 로직 오류 | 미해결 |
| 3 | 바깥 셀이 잘린 상태로 안 보임 | 클리핑 + prefetch 미확장 | 미해결 |
| 4 | 10% 규칙 안 먹힘 (70% 필요) | 진행도 계산 오류 | 미해결 |
| 5 | 취소 시 스크롤 100장 위로 점프 | 문제 2와 동일 | 미해결 |
| 6 | 좌표계 혼용 | bounds/content 좌표 혼용 | **해결됨** |

#### 문제 1: X축 좌측 정렬

**증상**: 핀치줌으로 줄어들때 가운데가 기준이 아니라 좌측이 기준이 됨.

**현재 동작 (문제):**
- 핀치 중심축 위치와 **무관하게** 항상 좌측이 고정점
- Y축: contentOffset.y로 보정되어 앵커 셀이 화면상 Y 위치에 고정됨 ✅
- X축: 보정이 안 되어서 좌측이 화면에 붙은 채로 확대/축소됨 ❌

```
[확대 중 - 현재 문제 상태] (E가 핀치 중심축인 경우)
┌─────────────────────┐
│A  B   │C            │  ← C 일부 잘림, A,B 잘보임
│D [E]  │F            │  ← F 일부 잘림, D,E 잘보임
│G  H   │I            │  ← I 일부 잘림, G,H 잘보임
└─────────────────────┘
→ 고정점이 E가 아니라 좌측(D쪽)인 느낌
→ X축이 좌측 정렬되어 있음
```

**원하는 동작:**
- 핀치 중심축 위치가 **고정점**이 되어야 함
- 앵커 셀이 화면상 **X, Y 모두** 같은 위치에 고정
- 주변 셀들이 앵커 셀을 중심으로 **사방으로** 확장/축소
- 기준: Photos 앱 동작 ("스크린샷이 확대/축소되는 느낌")

```
[확대 중 - 원하는 동작] (E가 핀치 중심축인 경우)
┌─────────────────────┐
│A│  B   │C           │  ← A, C 일부 잘림
│D│ [E]  │F           │  ← E는 고정, D, F 일부 잘림
│G│  H   │I           │  ← G, I 일부 잘림
└─────────────────────┘
→ 고정점이 E (핀치 중심)
→ 좌우 균등하게 잘림
```

```
[축소 중 - 원하는 동작] (E가 핀치 중심축인 경우, 3열→5열)
┌─────────────────────┐
│J│ A  B  C │K        │  ← J, K가 화면 가장자리에서 일부 나타남
│L│ D [E] F │M        │  ← E는 고정, L, M도 가장자리에서 일부 나타남
│N│ G  H  I │O        │  ← N, O가 화면 가장자리에서 일부 나타남
└─────────────────────┘
→ 새 셀들이 가장자리에서 잘린 채로 등장
```

**참고**: 핀치 중심축이 D라면 좌측 고정이 정상 동작임. 문제는 중심축 위치와 무관하게 항상 좌측이 고정된다는 것.

**구현 방향:**
- Y축: contentOffset.y로 보정 (기존 방식 유지)
- X축: 레이아웃 계산 시 xOffset 적용
  - `xOffset = anchorPointInView.x - anchorCellNewCenter.x`
  - 모든 셀의 x 좌표에 xOffset 추가
- 손 뗄 때: xOffset을 0으로 애니메이션하며 정상 레이아웃으로 스냅

**근본 원인**: X축 앵커 보정 로직 미구현

#### 문제 2: 스크롤 점프 + fadeIn 미동작

**증상 1: 스크롤 점프**
- 핀치 시작할 때 스크롤이 위로 갔다가 돌아옴 (한 번 발생)
- 앵커가 틀어지면서 발생하는 것으로 추정

**증상 2: fadeIn 미동작**
- fadeIn 애니메이션 없이 즉시 교체됨
- fadeIn이 발생해야 하는 시점 (문서 정의):
  - 방향 결정 시 (`.changed` 첫 프레임)
  - 기준점(1, 3, 5) 통과 시

**원하는 동작:**
1. 핀치 시작/진행 중 **앵커가 틀어지지 않고 유지**됨 (스크롤 점프 방지)
2. 방향 결정 시 + 기준점 통과 시 **fadeIn 애니메이션**으로 부드럽게 이미지 전환

**근본 원인 (추정)**:
- 앵커 해결 타이밍 또는 좌표 계산 오류
- fadeIn 트리거 조건 또는 애니메이션 로직 오류

#### 문제 3: 바깥 셀 클리핑

**증상**: 핀치줌으로 줄어들때 바깥의 셀이 일부분부터 바로 보이기 시작해야하는데, 셀이 다 화면에 들어오는 크기의 시점이 되어야 나타남.

**현재 동작 (문제):**
```
[축소 중 - 현재 문제]
┌─────────────────────┐
│  A  B  C            │  ← 새 셀(J,K,L,M,N,O)이 안 보임
│  D [E] F            │
│  G  H  I            │
└─────────────────────┘
→ 새 셀이 완전히 화면에 들어와야 보임
```

**원하는 동작:**
```
[축소 중 - 원하는 동작]
┌─────────────────────┐
│J│ A  B  C │K        │  ← J, K 일부 보임 (이미지 로딩 완료 상태)
│L│ D [E] F │M        │  ← L, M 일부 보임 (이미지 로딩 완료 상태)
│N│ G  H  I │O        │  ← N, O 일부 보임 (이미지 로딩 완료 상태)
└─────────────────────┘
→ 새 셀이 잘린 채로 등장
→ 새 셀의 이미지도 미리 로딩되어 있어야 함
```

**근본 원인 (추정):**
- `clipsToBounds = true` 설정
- 또는 `layoutAttributesForElements(in rect:)`에서 rect 바깥 셀 미반환
- prefetch 확장 필요 (화면 바깥 셀도 미리 로딩)

#### 문제 4: 10% 규칙 미적용

**증상**: 핀치줌 동작 실행 및 취소에 대한 10%/50% 규칙이 전혀 적용되지 않음. 70% 이상 진행해야 스냅됨.

**문서에 정의된 규칙:**
- 단방향 핀치: **10% 이상** 진행 → 그 방향으로 스냅
- 방향 전환: **50% 이상** → 그 방향으로 스냅

**현재 동작 (문제):**
- 10% 진행해도 스냅 안 됨
- 70% 이상 진행해야 스냅됨

**원하는 동작:**
- 문서에 명시된 10%/50% 규칙을 정확하게 구현

**근본 원인 (추정):**
- 진행도 계산 오류
- 또는 방향 전환 감지가 너무 민감하여 항상 50% 규칙이 적용됨

#### 문제 5: 취소 시 스크롤 점프

**증상**: 핀치줌 제스처 중 취소할 경우 스크롤이 확 위로 올라가버림 (100장 정도).

**원하는 동작:**
- 핀치 취소 시 원래 열 수로 부드럽게 스냅
- 스크롤 위치 유지 (앵커 기반)

**근본 원인**: 문제 2와 동일 (앵커 오류)

#### 문제 6: 좌표계 혼용 (Critical) - 해결됨

**증상**: 핀치줌 시작 시 앵커가 마지막 사진쪽으로 갑자기 이동

**근본 원인 (커밋 40700c4에서 분석):**
- `gesture.location(in: collectionView)`가 **content 좌표**를 반환함 (스크롤 포함된 절대 위치)
- 스크롤 상태에서 Y=151163 같은 큰 값이 나옴
- 이를 bounds 좌표로 착각하고 사용하여 앵커가 잘못 선택됨

**해결 (근본 원인 수정):**
- `locationInContent`: 앵커 검색용 (`indexPathForItem`은 content 좌표 기대)
- `locationInBounds`: offset 계산용 (content - contentOffset으로 변환)
- `anchorPointInView`에 bounds 좌표 저장

**상태**: 해결됨 (커밋 40700c4)

## 1. 핵심 개념

### 1.1 열 수 기준점
- **1열, 3열, 5열** 세 가지 기준점만 존재
- 핀치줌/회전 완료 시 반드시 이 중 하나로 snap

### 1.2 스케일 → 열 수 관계
```
scale 1.0 = 현재 열 수 유지
scale > 1.0 = 확대 → 열 수 감소 (3열 → 1열)
scale < 1.0 = 축소 → 열 수 증가 (3열 → 5열)
```

### 1.3 진행도 개념
```
0% = 현재 열 수 상태
100% = 목표 열 수 상태

예: 3열 → 1열 핀치
  0% = 3열 레이아웃
  50% = 중간 상태
  100% = 1열 레이아웃
```

### 1.4 앵커 셀
- **핀치**: 핀치 중심점에 있는 셀
- **회전**: 화면 중앙에 있는 셀
- 앵커 셀은 fadeIn 제외, 기존 이미지 유지

### 1.5 fadeIn
- **대상**: 기존에 visible이었던 셀 중 이미지 교체가 필요한 셀 (앵커 제외)
- **제외**: 새로 나타난 셀 (즉시 표시, fadeIn 금지)
- 현재 이미지 → 목표 레이아웃의 이미지로 부드럽게 전환
- 새 단계 시작 시 트리거

> **핵심 구분**: 축소 시 외곽에 새로 등장하는 셀은 "기존에 없던 셀"이므로
> fadeIn이 아니라 즉시 이미지가 표시되어야 함.

### 1.6 열 수 변경 규칙

#### 핀치줌
- 확대: 열 수 감소 (5 → 3 → 1)
- 축소: 열 수 증가 (1 → 3 → 5)

#### 회전
- 세로 → 가로: 1→3, 3→5, 5→5
- 가로 → 세로: 1→1, 3→3, 5→3

## 2. 요구사항

### 2.1 핵심 원칙

#### 금지 사항
- 셀 개별 이동 애니메이션 ❌
- 새로고침/끊김 ❌

#### 필수 사항
- 전체가 하나처럼 연속적인 모션 ✅
- fadeIn으로 부드러운 이미지 전환 ✅
- 핀치 진행도에 따라 실제 레이아웃이 점진적으로 변화 ✅

### 2.2 핀치줌

#### 핀치 시작 시
1. 앵커 셀 결정 (핀치 중심점의 셀)
2. **방향 결정 후** 나머지 모든 셀에 목표 이미지 fadeIn 시작
3. 레이아웃이 점진적으로 변하기 시작

> **주의**: `.began` 시점에서는 `gesture.scale`이 항상 1.0이므로 방향을 알 수 없음.
> fadeIn은 `.changed`의 첫 프레임에서 방향이 결정된 후 트리거해야 함.

#### 핀치 중
1. 핀치 진행도에 따라 셀 크기/위치가 **실시간으로** 변함
2. fadeIn 진행 중 (또는 이미 완료)
3. 외곽에 새 셀들이 나타남 (축소 시)
4. 앵커 셀: 기존 이미지 유지

#### 다단계 핀치 (예: 5열 → 3열 → 1열)
- 각 기준점(1/3/5열) 통과 시 = 새 단계 시작
- 새 단계 시작 시 fadeIn 트리거 (다음 목표 이미지로)

```
[1단계 핀치] 3열 → 1열
  방향 결정 → fadeIn (1열 이미지로)

[2단계 핀치] 5열 → 3열 → 1열
  방향 결정 → fadeIn (3열 이미지로)
  3열 기준점 통과 → fadeIn (1열 이미지로)
```

#### 손을 뗄 때
1. 핀치 100% 완료 상태 → 그대로 유지 (추가 애니메이션 없음)
2. 핀치 70% 상태 → 나머지 30%가 애니메이션으로 자연스럽게 완료
3. **끊김 없이** 현재 모션에서 최종 레이아웃으로 이어짐

### 2.3 회전

#### 핀치와 동일한 원리

| | 핀치 | 회전 |
|---|------|------|
| 앵커 | 핀치 중심점 셀 | 화면 중앙 셀 |
| fadeIn 시작 | 방향 결정 시 | 회전 시작 시 |
| 레이아웃 변화 | 핀치 진행도에 따라 점진적 | 회전 애니메이션과 동기화 |
| 완료 시 | 끊김 없이 snap | 끊김 없이 완료 |

#### 회전 시작 시
1. 앵커 셀 결정 (화면 중앙 셀)
2. **즉시** 나머지 모든 셀에 목표 이미지 fadeIn 시작
3. 회전 애니메이션과 함께 레이아웃이 자연스럽게 변화

#### 회전 완료 시
- 끊김/새로고침 없이 최종 레이아웃 상태
- 앵커 셀이 화면에 유지됨

### 2.4 기준점 판단 로직 (손을 뗄 때)

#### Case 1: 단방향 핀치 (10% 규칙)

한 방향으로만 핀치한 경우:

```
3열에서 축소 10% 이상 진행 → 손을 뗌
→ 의도 명확 → 5열로 완료
```

**10% 이상 진행하면 무조건 그 방향으로 진행**

#### Case 2: 방향 전환 (50% 규칙)

핀치 도중 방향이 바뀐 경우:

```
3열에서 축소 70% → 다시 확대 40%
→ 최종 위치: 30% (축소 방향)
→ 50% 미만 → 3열 유지

3열에서 축소 80% → 다시 확대 20%
→ 최종 위치: 60% (축소 방향)
→ 50% 이상 → 5열로 완료
```

**방향 전환 시 최종 위치 기준 50%로 판단**

#### 정리

| 상황 | 기준 | 결과 |
|-----|------|------|
| 단방향 10% 이상 | 10% | 그 방향으로 진행 |
| 방향 전환 | 50% | 최종 위치 기준 판단 |

## 3. 구현 방안

### 3.1 핵심 전략 (결론)
**연속 열 수(실수) 기반 커스텀 `UICollectionViewLayout`**으로 핀치 진행도에 따라
레이아웃을 실시간 재계산하고, 기준점 통과 시 fadeIn을 트리거한다.

핵심 이유:
- "점진적 레이아웃 변화" 요구를 만족하려면 열 수가 실수로 변해야 함
- 셀 개별 이동 애니메이션 없이 레이아웃 재계산만으로 연속 모션 제공 가능
- 다단계(5→3→1) 전환을 끊김 없이 처리 가능

---

### 3.2 아키텍처 구성

1) **ContinuousGridLayout (커스텀 레이아웃)**
- `virtualColumns: CGFloat`(예: 3.0 → 2.6 → 1.2)
- 값 변경 시 `invalidateLayout()`로 즉시 반영

2) **PinchController (제스처/상태 관리)**
- 핀치 진행도 → `virtualColumns` 변환
- 기준점 통과 감지 → fadeIn 트리거
- 앵커 셀 고정용 contentOffset 보정

3) **Cell FadeIn 레이어**
- 앵커 셀 제외 visible 셀에 목표 이미지 fadeIn
- 비동기 로딩 경쟁 조건 방지 (token/assetID 검증)

---

### 3.3 레이아웃 계산 원리

**핵심 아이디어**: 열 수를 정수로 끊지 말고 실수로 유지.

```swift
columnWidth = (bounds.width - spacing * (virtualColumns - 1)) / virtualColumns
```

- `virtualColumns`가 3.0 → 2.4 → 1.8처럼 연속적으로 변하며
  셀 크기/간격이 매 프레임 부드럽게 변화한다.
- 셀 배치는 "정수 열 수"가 아니라 **실수 열 수로 계산된 폭**에 맞춰
  `x = colIndex * (columnWidth + spacing)` 방식으로 배치한다.

---

### 3.4 앵커 셀 고정 (contentOffset 보정)

**목표**: 핀치 중심점 셀이 화면에서 고정된 것처럼 보이도록 유지.

```swift
// anchorPointInView: 핀치 시작 시 뷰 좌표 (bounds 기준)
// anchorCenterInContent: 레이아웃 갱신 후 앵커 셀의 center (content 좌표)
contentOffset = anchorCenterInContent - anchorPointInView
```

- `layer.anchorPoint` 변경 없이도 고정 가능
- 스크롤 가능한 범위로 clamp 필요

---

### 3.5 기준점 통과 감지 (다단계)

빠른 핀치로 5→1 이동 시 기준점 5,3,1 모두 통과 가능.
따라서 **통과한 기준점을 배열로 반환**하고 방향 순서대로 처리한다.

- 반환된 순서대로 fadeIn 트리거
- 동일 단계 중복 방지 (`currentTargetColumns` 체크)

---

### 3.6 fadeIn 트리거 규칙

**대상 구분**:
- **fadeIn 대상**: 트리거 시점에 이미 visible이었던 셀 (앵커 제외)
- **즉시 표시**: 핀치 중 새로 visible 되는 셀 (fadeIn 금지)

**트리거 시점**:
- **방향 결정 시**: 첫 목표 열 수 이미지로 fadeIn 시작 (`.changed` 첫 프레임)
- **기준점 통과 시**: 다음 단계 목표 이미지로 fadeIn 재시작

**안전 규칙**:
- **앵커 셀 제외**
- **비동기 로딩 검증**: 요청 시점의 열 수/assetID/token 확인 후 적용

---

### 3.7 손을 뗄 때 스냅 로직 (10% / 50% 규칙)

**단방향**: 10% 이상 진행 → 해당 방향으로 완료
**방향 전환**: 최종 위치 50% 이상 → 해당 방향으로 완료

완료 애니메이션은 `virtualColumns`를 목표 기준점까지 보간하며,
동일한 레이아웃 계산을 유지해 끊김을 방지한다.

---

### 3.8 회전 처리

회전도 동일 원리로 `virtualColumns`를 보간:
- 회전 시작 시 fadeIn 트리거
- `animate(alongsideTransition:)`에서 `virtualColumns` 즉시 변경
- 완료 후 앵커 복구 + 고해상도 재요청

---

### 3.9 후보 방안 비교 (참고)
1. `startInteractiveTransition`
   - 다단계 전환에서 끊김 가능성 높음
2. 스냅샷 오버레이
   - 점진적 레이아웃 변화 요구에 부적합
3. transform 방식 (폐기)
   - 축소 시 외곽 검정, 손 떼면 끊김, fadeIn 무효화
