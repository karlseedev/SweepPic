# iCloud 사진 대응 구현 계획

> 작성일: 2026-02-13
> 상태: 계획 수립 중

---

## 목표

### 문제

현재 앱은 `isNetworkAccessAllowed = false`로 iCloud 사진 다운로드를 차단하고 있다.
iOS 사용자 대다수가 iCloud 사진을 사용하며, "저장 공간 최적화" 옵션을 켠 사용자는 기기에 저해상도 썸네일만 보관한다.
이 상태에서 우리 앱을 사용하면:

- 그리드에서 일부 사진 썸네일이 표시되지 않거나 저해상도만 표시
- 뷰어에서 원본 이미지를 볼 수 없음 (줌 시 흐릿한 이미지)
- 다운로드 진행 상황이나 실패 원인을 알 수 없음

→ iCloud 사용자에게 **앱이 정상 작동하지 않는 것처럼 보인다**.

### 달성 목표

| # | 목표 | 측정 기준 |
|---|------|----------|
| G1 | iCloud 사진이 그리드/뷰어에서 정상 표시 | "저장 공간 최적화" 기기에서 모든 사진 열람 가능 |
| G2 | iCloud 다운로드 중 사용자에게 진행 상황 피드백 | 뷰어에서 프로그레스 인디케이터 표시 |
| G3 | 네트워크 실패 시 명확한 안내와 재시도 경로 제공 | 오프라인/에러 시 안내 UI + 재시도 동작 |
| G4 | 기존 로컬 사진 사용자의 성능 저하 없음 | 로컬 전용 기기에서 동일 FPS 유지 |

### 스코프 외 (이번에 하지 않는 것)

- iCloud 동기화 엔진 자체 구현 (PhotoKit이 처리)
- 셀룰러/Wi-Fi 구분 다운로드 정책 UI (향후 설정 화면에서)
- 공유 앨범 전용 UI (아이콘, 삭제 불가 안내 등)
- 백그라운드 다운로드 관리

## Phase 1. 기본 네트워크 활성화

### 1-1. ImagePipeline isNetworkAccessAllowed 활성화

**대상 파일**: `Sources/AppCore/Services/ImagePipeline.swift`

**변경 내용**:

```swift
// 라인 323: thumbnailOptions
options.isNetworkAccessAllowed = true  // false → true

// 라인 333: fullSizeOptions
options.isNetworkAccessAllowed = true  // false → true
```

**주의사항**:
- `thumbnailOptions`에서 `true`로 바꾸면 그리드 스크롤 시 iCloud 요청 다량 발생 가능
- 그러나 `deliveryMode: .opportunistic`이 이를 완화함:
  - PhotoKit이 로컬 저해상도 썸네일을 먼저 반환 (degraded=true)
  - 네트워크에서 고해상도 도착 시 두 번째 콜백 (degraded=false)
- `PHCachingImageManager`의 프리히트(`startCachingImages`)도 동일 옵션 사용 → 프리히트 시 iCloud 다운로드 시도할 수 있음

**결론**: 단순 `true` 전환으로 기본 작동. 성능 이슈는 1-5에서 대응.

---

### 1-2. opportunistic + networkAccessAllowed 동작 모델

`deliveryMode: .opportunistic` + `isNetworkAccessAllowed: true` 조합 시 PhotoKit 콜백 흐름:

```
requestImage() 호출
    │
    ├─ [로컬에 사진 있음]
    │   ├── 1차 콜백: 저해상도 (isDegraded=true) — 즉시
    │   └── 2차 콜백: 고해상도 (isDegraded=false) — 수~수백ms
    │
    ├─ [로컬에 썸네일만 있음 (저장공간 최적화)]
    │   ├── 1차 콜백: 저해상도 썸네일 (isDegraded=true) — 즉시
    │   ├── progressHandler: 0.0 → ... → 1.0 — 네트워크 다운로드
    │   └── 2차 콜백: 고해상도 원본 (isDegraded=false) — 수초
    │
    └─ [로컬에 아무것도 없음 (신규 동기화)]
        ├── 1차 콜백: nil + info[PHImageResultIsInCloudKey]=true
        ├── progressHandler: 0.0 → ... → 1.0 — 네트워크 다운로드
        └── 2차 콜백: 이미지 (isDegraded=false) — 수초~수십초
```

**현재 코드와의 호환성**:
- `completion: (UIImage?, Bool isDegraded)` 시그니처로 이미 degraded 플래그 전달 중
- PhotoCell은 `isDegraded`를 받아 셀 업데이트 가능 (기존 로직 유지)
- 추가 필요: `image == nil` + iCloud 상태일 때의 분기 (Phase 3에서 처리)

### 1-4. PHCachingImageManager + iCloud 검증

`PHCachingImageManager.startCachingImages()`는 내부적으로 `PHImageRequestOptions`를 사용.

**검증 포인트**:
- `isNetworkAccessAllowed = true` 상태에서 프리히트가 iCloud 다운로드를 트리거하는지
- Apple 문서: "프리히트는 백그라운드에서 저해상도 버전을 미리 로드" → 네트워크 사용 최소화
- 하지만 명시적 보장은 없음

**대응 방안**:
- 프리히트용 옵션을 별도로 분리하여 `isNetworkAccessAllowed = false` 유지 가능
- 또는 현재 옵션 공유 구조 유지 (PHCachingImageManager가 자체적으로 최적화)

```swift
// 안전한 방식: 프리히트 전용 옵션 분리
private lazy var preheatOptions: PHImageRequestOptions = {
    let options = PHImageRequestOptions()
    options.deliveryMode = .opportunistic
    options.resizeMode = .fast
    options.isNetworkAccessAllowed = false  // 프리히트는 로컬만
    options.isSynchronous = false
    return options
}()
```

**수정 대상**: `ImagePipeline.swift:499-504` (`preheatAssets` 메서드)

### 1-5. 그리드 스크롤 시 화면 밖 요청 취소 전략

**현재 구현 상태** (이미 양호):
- `PhotoCell.prepareForReuse()` → `currentCancellable?.cancel()` (라인 340-341)
- `CancellableToken.cancel()` → `imageManager.cancelImageRequest(requestID)` (라인 462-463)

> **참고**: `GridViewController`는 스크롤 중 prefetch를 비활성화(override → `return`)하고,
> 스크롤 정지 후 `preheatAfterScrollStop()`에서 preheat를 수행하는 구조.
> `BaseGridViewController`의 기본 prefetch/cancelPrefetch 로직은 GridVC에서 오버라이드됨.
> (`GridViewController.swift:914-923`)

**iCloud 활성화 시 추가 고려사항**:
- iCloud 다운로드 중인 요청도 `cancelImageRequest`로 즉시 취소 가능 (Apple 보장)
- 빠른 스크롤 시 다운로드가 시작되었다가 바로 취소되는 패턴 발생 → 네트워크 낭비 가능
- 그러나 `deliveryMode: .opportunistic`에서 PhotoKit이 자체적으로 로컬 캐시 우선 반환하므로 실제 네트워크 요청은 최소화됨
- `prepareForReuse`의 즉시 취소가 핵심 방어선 역할

**결론**: 기존 취소 로직(`prepareForReuse` + `CancellableToken`)이 충분.
추가 최적화(스크롤 중 네트워크 억제 등)는 실제 성능 테스트 후 결정.

### 1-6. 공유 앨범 사진 처리

**문제점**:

**향후 고려** (이번 스코프 외):

## Phase 2. 뷰어 다운로드 프로그레스 UI

### 2-1. progressHandler 기반 진행률 추적

**변경 대상**: `ImagePipeline.swift`

ImagePipeline의 `requestImage` 메서드에 progressHandler 파라미터 추가:

```swift
// ImagePipeline.swift — requestImage 시그니처 확장
public func requestImage(
    for asset: PHAsset,
    targetSize: CGSize,
    contentMode: PHImageContentMode = .aspectFill,
    quality: ImageQuality = .fast,
    progressHandler: ((Double) -> Void)? = nil,  // ← 추가
    completion: @escaping (UIImage?, Bool) -> Void
) -> Cancellable
```

**구현 방식** (VideoPipeline.swift:76-82 참고):

```swift
// ImagePipeline 내부에서 PHImageRequestOptions에 progressHandler 연결
// progressHandler가 전달된 경우에만 설정 (nil이면 기존 동작)

if let progressHandler = progressHandler {
    // 옵션을 복사해서 progressHandler 설정 (lazy var 원본 변경 방지)
    let optionsWithProgress = options.copy() as! PHImageRequestOptions
    optionsWithProgress.progressHandler = { progress, error, stop, info in
        DispatchQueue.main.async {
            progressHandler(progress)
        }
    }
    // 이 요청에만 optionsWithProgress 사용
}
```

**주의**: `PHImageRequestOptions`은 `NSCopying` 프로토콜 채택 → `.copy()` 가능.
기존 `thumbnailOptions`/`fullSizeOptions` lazy var를 직접 수정하면 안 됨 (공유 인스턴스).

**프로토콜 업데이트**: `ImagePipelineProtocol`에도 progressHandler 파라미터 추가 필요.

### 2-2. 프로그레스 인디케이터 UI

**대상 파일**: `PhotoPageViewController.swift`

**디자인 방향** (Apple HIG 참고):

- 사진 중앙에 반투명 배경 + 원형 프로그레스
- 프로그레스 값 0.0~1.0 → 원형 게이지 채워짐

```
┌──────────────────────────────┐
│                              │
│        [저해상도 사진]         │
│                              │
│           ╭───╮              │
│           │ ◐ │  42%         │
│           ╰───╯              │
│                              │
└──────────────────────────────┘
```

**구현 계획**:

1. `iCloudProgressView` 컴포넌트 생성 (CAShapeLayer 기반 원형 프로그레스)
2. `PhotoPageViewController`에서 LOD1 요청 시 progressHandler 전달
3. 진행률 0.0 수신 시 프로그레스 뷰 표시
4. 진행률 1.0 도달 또는 이미지 수신 시 프로그레스 뷰 숨김

```swift
// PhotoPageViewController — LOD1 요청 수정

```

**그리드 셀에서는**: 프로그레스 UI 불필요 (썸네일은 빠르게 로드되므로).

### 2-3. 다운로드 취소 처리 (PHImageRequestID)

**현재 상태**: 부분 구현 — **버그 수정 필요**

- `CancellableToken.cancel()` → `imageManager.cancelImageRequest(requestID)` ✅
- 뷰어에서 사진 넘길 때: `requestCancellable?.cancel()` 호출 ✅
- 새 LOD1 요청 전: `fullSizeRequestCancellable?.cancel()` 호출 ✅

**버그**: `PhotoPageViewController.deinit`(라인 221-222)에서 `requestCancellable`만 취소하고
`fullSizeRequestCancellable`은 취소하지 않음 → iCloud 다운로드가 VC 해제 후에도 계속됨.

```swift
// 현재 (버그)
deinit {
    requestCancellable?.cancel()
}

// 수정
deinit {
    requestCancellable?.cancel()
    fullSizeRequestCancellable?.cancel()  // ← 추가
}
```

**프로그레스 UI와 연계**:
- 프로그레스 뷰에 "X" 취소 버튼 추가 (선택적)
- 탭 시 `fullSizeRequestCancellable?.cancel()` + 저해상도 유지

**"저장 공간 최적화"란**:
- 설정 > 사진 > "iPhone 저장 공간 최적화" 활성화 시
- 기기에 저해상도 썸네일만 보관, 원본은 iCloud에만 존재
- 사진 앱 사용자의 상당수가 이 옵션 사용 중

**영향**:
- 그리드: 로컬 썸네일은 대부분 존재 → **큰 영향 없음**
- 뷰어: 원본 요청 시 거의 항상 iCloud 다운로드 필요 → **프로그레스 필수**
- 줌: 고해상도가 iCloud에만 있으므로 다운로드 후 줌 가능

**대응 방식**:
- Phase 1의 `isNetworkAccessAllowed = true`로 자동 대응
- Phase 2의 프로그레스 UI가 사용자 경험 보장
- 별도 "최적화 모드" 감지 로직 불필요 (PhotoKit이 투명하게 처리)

**확인 방법** (런타임):
```swift
// 저장 공간 최적화 여부는 직접 확인 불가
// 대신 requestImage의 결과로 간접 판단:
// - image != nil + isDegraded == true → 저해상도만 로컬에 있음
// - image == nil + isInCloud == true → 로컬에 아무것도 없음
```

## Phase 3. 에러 핸들링 & 재시도

### 3-1. PHImageErrorKey 기반 에러 감지

**현재 상태**: info dictionary에서 `isDegraded`만 추출 (라인 417), 에러 정보 무시

**변경 대상**: `ImagePipeline.swift` completion 콜백 내부

**추출해야 할 키**:

| 키 | 타입 | 의미 |
|---|------|------|
| `PHImageResultIsDegradedKey` | Bool | 저해상도 임시 이미지 (이미 사용 중) |
| `PHImageResultIsInCloudKey` | Bool | 원본이 iCloud에 있음 |
| `PHImageErrorKey` | Error? | 다운로드 실패 에러 |
| `PHImageCancelledKey` | Bool | 요청이 취소됨 |

**구현 방향**:

ImagePipeline의 completion 시그니처를 바꾸지 않되, 에러 발생 시 내부적으로 처리:

```swift
// ImagePipeline.swift 콜백 내부 (라인 410~ 수정)
) { [weak self] image, info in
    // 에러 체크
    if let error = info?[PHImageErrorKey] as? Error {
        Log.print("[Pipeline] iCloud download error: \(error.localizedDescription)")
        // 에러 콜백 (image=nil, isDegraded=false)
        DispatchQueue.main.async {
            guard !cancellable.isCancelled else { return }
            completion(nil, false)
        }
        return
    }

    // 취소 체크
    if info?[PHImageCancelledKey] as? Bool == true {
        return  // 조용히 무시
    }

    // iCloud 상태 로깅
    if info?[PHImageResultIsInCloudKey] as? Bool == true && image == nil {
        Log.print("[Pipeline] Asset in iCloud, awaiting download...")
    }

    // 기존 로직 유지
    let isDegraded = (info?[PHImageResultIsDegradedKey] as? Bool) ?? false
    // ...
}
```

### 3-2. 재시도 로직

**정책**: UI 계층에서 수동 재시도 (뷰어 전용)

> **설계 결정**: 파이프라인 내부 자동 재시도는 채택하지 않음.
> 이유: 그리드/뷰어/앨범 모든 호출에 동일 적용되어 네트워크 낭비, 중복 요청,
> 통계 왜곡 가능성이 있음. 재시도 판단은 UI 계층이 컨텍스트를 알고 있으므로
> UI에서 명시적으로 제어하는 것이 안전함.

**뷰어 재시도** (`PhotoPageViewController`):
- 에러 발생 시 (completion에서 `image == nil` + progressHandler 미호출)
  → "다운로드 실패 — 탭하여 재시도" 오버레이 표시
- 탭 제스처로 LOD1 재요청

```swift
// PhotoPageViewController
func showRetryOverlay() {
    retryLabel.text = "iCloud에서 다운로드할 수 없습니다\n탭하여 재시도"
    retryOverlay.isHidden = false
}

@objc func retryTapped() {
    retryOverlay.isHidden = true
    requestHighQualityImage()  // 기존 LOD1 요청 재실행
}
```

**그리드에서는**: 재시도 UI 없음. 스크롤 시 자연스럽게 재요청됨
(셀이 다시 화면에 나타나면 `configure` → `requestImage` 재호출).

### 3-3. 오프라인 상태 안내 UI

**감지 방법**: `NWPathMonitor` 사용 (Network 프레임워크)

```swift
import Network

class NetworkMonitor {
    static let shared = NetworkMonitor()
    private let monitor = NWPathMonitor()
    private(set) var isConnected = true

    func start() {
        monitor.pathUpdateHandler = { [weak self] path in
            self?.isConnected = (path.status == .satisfied)
        }
        monitor.start(queue: DispatchQueue.global(qos: .utility))
    }
}
```

**UI 대응**:
- 오프라인 + iCloud 사진 요청 실패 시:
  - 뷰어: "오프라인 상태입니다. Wi-Fi 연결 후 다시 시도해주세요" 토스트
  - 그리드: 로컬 썸네일은 정상 표시, 원본만 못 가져오는 것이므로 영향 제한적

**주의**: 오프라인이어도 그리드 썸네일은 대부분 로컬에 존재 → 그리드는 정상 작동.
뷰어에서 고해상도 요청만 실패.

### 3-4. 그리드 셀 플레이스홀더

**시나리오**: iCloud 전용 사진인데 네트워크 실패 → 썸네일도 없음

**대상 파일**: `PhotoCell.swift`

**현재 상태**: `image == nil` 시 빈 셀 (회색 배경)

**개선 방안**:
```swift
// PhotoCell — 이미지 로딩 실패 시
if image == nil {
    // 클라우드 아이콘 + 회색 배경 표시
    cloudPlaceholderView.isHidden = false
} else {
    cloudPlaceholderView.isHidden = true
    imageView.image = image
}
```

**cloudPlaceholderView 디자인**:
- 회색 배경 + 중앙에 `icloud.slash` SF Symbol
- 최소한의 시각적 피드백으로 "사진은 있지만 로딩 불가" 상태 전달

### 3-5. 셀룰러/Wi-Fi 데이터 정책

**방향**: 첫 버전에서는 네트워크 종류 구분 없이 허용

**이유**:
- Apple 기본 사진 앱도 셀룰러에서 iCloud 다운로드 허용
- `PHImageRequestOptions`에는 `allowsCellularAccess` 같은 속성이 없음
- 네트워크 레벨에서 제어하려면 커스텀 URLSession이 필요한데, PhotoKit은 내부적으로 네트워크를 관리

**향후 확장** (이번 스코프 외):
- 설정 화면에 "셀룰러 데이터에서 iCloud 사진 다운로드" 토글 추가
- 셀룰러 감지 시 `isNetworkAccessAllowed = false`로 전환
- NWPathMonitor에서 `path.usesInterfaceType(.cellular)` 확인

```swift
// 향후 구현 시 참고
let isCellular = monitor.currentPath.usesInterfaceType(.cellular)
let userAllowsCellular = UserDefaults.standard.bool(forKey: "allowCellularICloud")
thumbnailOptions.isNetworkAccessAllowed = !isCellular || userAllowsCellular
```

## 부록

### A. 핵심 API 레퍼런스

#### info dictionary 키 정리

```swift
// PHImageManager requestImage 콜백의 info: [AnyHashable: Any]?

// 1. 저해상도 여부
let isDegraded = info?[PHImageResultIsDegradedKey] as? Bool ?? false

// 2. iCloud에 있는지 (로컬에 없음)
let isInCloud = info?[PHImageResultIsInCloudKey] as? Bool ?? false

// 3. 에러 (네트워크 실패 등)
let error = info?[PHImageErrorKey] as? Error

// 4. 취소됨
let isCancelled = info?[PHImageCancelledKey] as? Bool ?? false
```

#### PHImageRequestOptions 주요 속성

```swift
let options = PHImageRequestOptions()

// 이미지 품질/속도 균형
options.deliveryMode = .opportunistic  // 저해상도 먼저 → 고해상도 나중
options.deliveryMode = .highQualityFormat  // 고해상도만 (뷰어용)
options.deliveryMode = .fastFormat  // 저해상도만 (빠른 표시)

// 네트워크 접근
options.isNetworkAccessAllowed = true  // iCloud 다운로드 허용

// 다운로드 진행률
options.progressHandler = { (progress: Double, error: Error?, stop: UnsafeMutablePointer<ObjCBool>, info: [AnyHashable: Any]?) in
    // progress: 0.0 ~ 1.0
    // error: 다운로드 에러 (nil이면 정상)
    // stop: stop.pointee = true로 설정하면 다운로드 중단
    // 주의: 임의의 직렬 큐에서 호출됨 → UI 업데이트 시 DispatchQueue.main 필요
}
```

#### 요청 취소

```swift
// 요청 시 반환된 ID로 취소
let requestID: PHImageRequestID = imageManager.requestImage(...)
imageManager.cancelImageRequest(requestID)
// → iCloud 다운로드 중이어도 즉시 취소
```

### B. 테스트 시나리오

| # | 시나리오 | 검증 포인트 |
|---|---------|------------|
| 1 | 로컬 사진만 있는 기기 | 기존 동작과 동일한지 (성능 저하 없음) |
| 2 | "저장 공간 최적화" 켠 기기 | 그리드 썸네일 정상 표시, 뷰어 원본 다운로드 |
| 3 | 비행기 모드에서 iCloud 사진 열기 | 저해상도 표시 + 에러 안내 |
| 4 | 느린 네트워크 (Network Link Conditioner) | 프로그레스 UI 정상 작동 |
| 5 | 빠른 스크롤 후 정지 | 불필요한 네트워크 요청 취소, visible cell만 로딩 |
| 6 | 뷰어에서 사진 빠르게 넘기기 | 이전 다운로드 취소, 현재 사진만 다운로드 |
| 7 | 공유 앨범 사진 표시 | 썸네일/원본 정상 로딩 |
| 8 | iCloud 다운로드 중 앱 백그라운드 전환 | 크래시 없음, 복귀 후 재개/재시도 |
| 9 | 사진 접근 권한 `.limited` 상태 | 허용된 사진만 표시, iCloud 다운로드 정상 |
| 10 | iCloud 비활성 / 로그아웃 상태 | 로컬 사진만 정상 표시, 에러 없음 |
| 11 | 사진 접근 권한 거부 상태 | 권한 요청 흐름 정상, 크래시 없음 |

**테스트 도구**:
- Xcode Network Link Conditioner (Developer Settings)
- iCloud 계정에 사진 업로드 후 "저장 공간 최적화" 활성화
