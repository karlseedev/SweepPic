# 가로 모드(Landscape) 대응 분석 및 전략

> 작성일: 2026-02-19

---

## 1. 앱 레벨 회전 설정 현황

- **iPhone**: Portrait + LandscapeLeft + LandscapeRight (가로 모드 허용)
- **iPad**: 모든 방향 허용
- `supportedInterfaceOrientations` / `shouldAutorotate` 오버라이드: **어디에도 없음**
- **회전을 막는 화면은 없음** — Info.plist 설정에 전적으로 의존

---

## 2. 화면별 가로 모드 대응 상태

### 2-1. 심각 (가로 전환 시 즉시 눈에 보이는 문제)

| 화면/컴포넌트 | 문제 |
|-------------|------|
| **LiquidGlassTabBar** | `setupConstraints()`에서 `UIScreen.main.bounds.width` (Portrait 기준)로 Platter 너비 고정. 가로 모드에서 Platter가 좁은 채로 유지됨 |
| **SwipeDeleteHandler** | `screenHeight`가 초기화 시 Portrait 높이로 고정. 가로 모드에서 삭제 임계값이 화면 높이의 ~43%가 되어 스와이프 삭제가 매우 어려워짐 |
| **CoachMarkOverlayView** | 모든 자식 뷰(메시지, 버튼, 화살표)가 생성 시점 frame 기반 절대 좌표. `layoutSubviews` 없음. 코치마크 표시 중 회전 시 UI 완전 깨짐 |

### 2-2. 중간 (기능적 문제지만 사후 보정됨)

| 화면/컴포넌트 | 문제 |
|-------------|------|
| **AlbumsViewController** | `viewWillTransition` 없음. 회전 애니메이션 중 순간적으로 레이아웃이 어색할 수 있음 (CompositionalLayout이 사후 보정) |
| **FaceComparisonViewController** | `viewWillTransition` 없음. Auto Layout 기반이라 사후 보정되나 회전 중 동기화 없음 |
| **PreviewGridViewController** | `viewWillTransition` 없음. CompositionalLayout + safe area 기반으로 사후 보정 |

### 2-3. 양호 (적절히 대응됨)

| 화면/컴포넌트 | 상태 |
|-------------|------|
| **BaseGridViewController** (GridVC, AlbumGridVC, TrashAlbumVC) | `viewWillTransition`에서 열 수 전환(3↔5), 셀 크기 재계산, 스크롤 위치 복원 구현됨 |
| **ViewerViewController** | `viewWillTransition`에서 FaceButton 숨기기→재배치, Auto Layout 기반 |
| **PhotoPageViewController** | `viewDidLayoutSubviews`에서 회전 감지 후 aspect fit 재계산, 줌 리셋 |
| **TabBarController** | FloatingOverlay Auto Layout, `viewDidLayoutSubviews`에서 safe area 갱신 |
| **FloatingOverlayContainer / FloatingTitleBar** | Auto Layout + safe area 기반 |
| **FaceButtonOverlay** | 회전 완료 후 `layoutButtons(for:)` 재호출 |
| **PermissionViewController** | 완전 Auto Layout + StackView |

### 2-4. 잠재적 주의 사항

- **ZoomAnimator**: 뷰어가 열린 상태에서 회전 후 닫을 때, `zoomSourceFrame`이 회전 전 그리드 셀 위치를 참조하여 줌 아웃 좌표가 어긋날 가능성
- **PhotoPageViewController LOD0**: 가로 모드 최초 진입 시 Portrait 해상도로 이미지 요청 → 이후 보정됨 (체감 차이 미미)

## 3. 앱 네비게이션 구조

```
SceneDelegate
 └── TabBarController (UITabBarController)
      ├── UINavigationController → GridViewController
      ├── UINavigationController → AlbumsViewController → AlbumGridViewController
      └── UINavigationController → TrashAlbumViewController

뷰어 표시 방식:
 iOS 16~25: Modal (.custom) + ZoomTransitionController
 iOS 26+:   Navigation Push
```

- `UINavigationController`는 서브클래스 없이 표준 클래스 사용 중
- `ZoomPresentationController`: `shouldRemovePresentersView = false` (presenting VC 유지)
- `ViewerViewController`: `modalPresentationStyle = .custom`

## 4. "그리드 세로 고정 + 뷰어만 가로 허용" 전략 분석

### 4-1. 구현 방법 (가장 안정적)

AppDelegate 레벨 제어:

```swift
// AppDelegate
static var isLandscapeAllowed = false

func application(_ app: UIApplication,
                 supportedInterfaceOrientationsFor window: UIWindow?) -> UIInterfaceOrientationMask {
    return Self.isLandscapeAllowed ? .allButUpsideDown : .portrait
}
```

```swift
// ViewerViewController
override func viewWillAppear(_ animated: Bool) {
    AppDelegate.isLandscapeAllowed = true
}
override func viewWillDisappear(_ animated: Bool) {
    AppDelegate.isLandscapeAllowed = false
}
```

### 4-2. 해결되는 문제

그리드가 세로 고정이므로 심각 3개 + 중간 3개 = **가로 관련 문제 전부 일괄 해결**

### 4-3. 핵심 걸림돌: 가로 뷰어 → 닫기 시 ZoomTransition

```
[가로 뷰어] → dismiss → [세로 그리드]
                 ↑
    줌 애니메이션 + 회전이 동시 발생 → 좌표계 불일치
```

해결책 (난이도순):

| 방법 | 난이도 | UX |
|------|--------|-----|
| **A. 가로일 때 Fade 전환** | 낮음 | 깔끔. 가로에서만 줌 대신 페이드아웃 |
| **B. 닫기 전 세로 강제 회전** | 중간 | `requestGeometryUpdate()`로 세로 전환 후 줌 |
| **C. 크로스 방향 줌 계산** | 높음 | 가장 완성도 높지만 복잡 |

### 4-4. iOS 26+ Navigation Push 특이사항

iOS 26에서는 뷰어가 Push로 열림. Pop 시 회전이 함께 일어나야 하는데:
- 시스템이 pop 완료 후 자동으로 세로로 회전시킴
- Navigation transition과 rotation이 순차 발생하여 약간의 어색함 가능
- 방법 A(가로일 때 커스텀 전환) 적용 시 자연스러움

## 5. 레퍼런스 앱 비교 (실제 확인)

### 사진/미디어 뷰어에서 가로 허용하지 않는 앱

| 앱 | 피드/그리드 | 미디어 뷰어 | 비고 |
|----|-----------|------------|------|
| **Instagram** | 세로 고정 | 세로 고정 | 앱 전체 portrait-locked |
| **X (Twitter)** | 세로 고정 | 세로 고정 (iPhone) | iPad만 가로 지원 |

### "피드 세로 + 미디어 가로 허용" 패턴 앱

| 앱 | 피드/그리드 | 미디어/재생 | 비고 |
|----|-----------|------------|------|
| **YouTube** | 세로 고정 | 전체화면 시 가로 허용 | 동영상 특화 |
| **Netflix** | 세로 고정 | 재생 시 가로 허용 | 동영상 특화 |

### 모든 화면에서 가로 허용하는 앱

| 앱 | 그리드 | 뷰어 | 비고 |
|----|-------|------|------|
| **Apple 사진** | 모든 방향 | 모든 방향 | 사진 앱의 표준 |

> **참고**: "그리드 세로 + 사진 뷰어 가로" 패턴의 직접적 레퍼런스 사진 앱은 확인되지 않음.
> 동영상 앱(YouTube, Netflix)에서는 일반적인 패턴.

## 6. 가능한 전략 옵션

### A. 앱 전체 세로 고정 (Instagram/X 방식)

- **장점**: 가장 간단. Info.plist에서 Portrait만 허용. 가로 모드 문제 전부 사라짐
- **단점**: 뷰어에서 가로 사진/영상 볼 때 불편할 수 있음

### B. 그리드 세로 고정 + 뷰어만 가로 허용 (YouTube/Netflix식)

- **장점**: 뷰어에서 가로 콘텐츠 최적 표시. 그리드 가로 문제 전부 해결
- **단점**: ZoomTransition 처리 필요 (방법 A: Fade 전환으로 해결 가능). 사진 앱 레퍼런스 부족
- **구현 난이도**: 낮음~중간

### C. 전체 화면 가로 모드 대응 (Apple 사진 방식)

- **장점**: 가장 완성도 높음. Apple 사진 앱과 동일 UX
- **단점**: 심각한 문제 3개 + 중간 문제 3개 모두 개별 수정 필요. 작업량 가장 많음
- **구현 난이도**: 높음
