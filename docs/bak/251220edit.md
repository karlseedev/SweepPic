## 원인 추적 과정

### 1차: scrollToItem 의심 → 무죄

- E1→E2가 0.2ms로 scrollToItem 자체는 문제 없음
- E0→E1이 1.8초 = **메인큐 backlog(블로킹)**

### 2차: prefetch/preheat 의심 → 부분 기여

- prefetch 비활성화해도 E0→E1이 1.3초로 여전히 큼
- prefetch는 범인이 아니거나 기여 요인 중 하나

### 3차: viewWillAppear 중복 reloadData → 수정했으나 효과 없음

- viewWillAppear에서 초기 진입 시 reloadData 스킵하도록 수정
- E0→E1 여전히 2초 (효과 없음)

### 4차: cellForItemAt 분석 → configure가 97% 차지

## 최종 결론

### 범인: `ImagePipeline.requestImage()` 호출 자체

- 첫 10회 호출에서 평균 **72.17ms/회** 소요
- 이후 점점 감소 (캐싱 효과)
- `imageView.image = image` 적용은 0.03ms로 무죄

### 원인 추정

`ImagePipeline.requestImage()` 내부에서 `PHCachingImageManager.requestImage()` 호출 시:

1. **첫 호출 시 초기화 비용** - PHCachingImageManager 워밍업
2. **deliveryMode = .opportunistic** - 저해상도 이미지를 동기 반환 시도 가능성
3. **thumbnailOptions 설정** - isSynchronous=false이지만 내부적으로 동기 작업 발생 가능

## 상세 분석

### A/B 테스트 결과 (결정적)

| 측정 | 결과 | 의미 |
|------|------|------|
| B) imageApply | 평균 0.03ms | `imageView.image = image` 자체는 전혀 병목 아님 |
| A) requestImage 호출 | 첫 10회 평균 72ms | **요청을 걸기 위한 동기 작업**이 병목 |

```
requestImage 호출: 10회 평균 72.17ms → 20회 평균 36.45ms → 30회 평균 24.55ms
```

- 첫 10회가 특히 느림 = **캐시 워밍/내부 초기화 비용 패턴**
- 병목은 "디코딩/이미지 적용"이 아니라 **"요청을 걸기 위해 준비하는 동기 작업"**

### requestImage 호출이 느린 이유 (코드 기반 추정)

현재 `PhotoCell.configure(asset:)` → `ImagePipeline.shared.requestImage(for: asset:)` 호출

ImagePipeline 내부:
```swift
let requestID = imageManager.requestImage(
    for: asset,
    targetSize: targetSize,
    contentMode: contentMode,
    options: thumbnailOptions  // deliveryMode = .opportunistic
) { ... }
```

호출 전후가 수십 ms라면, **PhotoKit이 하는 동기 작업**이 원인:

1. **캐시 미스 시 즉시 썸네일 생성** - targetSize/옵션에 따라 준비 비용이 호출 스레드(메인)에서 발생
2. **연속 호출 시 내부 락/큐 대기** - 동일 run에서 짧은 시간에 연속 호출되며 동기 비용 누적
3. **deliveryMode = .opportunistic** - 저해상도 이미지를 즉시 동기적으로 반환 시도 가능

### E0→E1(1.88초)과의 정합성

- 메인큐 backlog(1.88초) = **많은 셀들이 생성되면서 requestImage를 메인에서 연속 호출**하는 동안 발생
- cellForItemAt 총 17회/751ms, requestImage 누적 700ms대 = 같은 방향
- 구조: **셀 생성 구간에서 메인스레드가 requestImage 호출 비용으로 길게 점유** → main.async(최신 점프)가 뒤로 밀림

// ImagePipeline.requestImage 내부
let phStart = CACurrentMediaTime()
let requestID = imageManager.requestImage(...)  // ← 이 호출 전후

| 결과 | 의미 |
|------|------|

| 이 구간은 짧음 | 함수 내부 다른 작업 또는 측정 범위 문제 |

## 해결 방향 (파이프라인 변경 전 방향)

### 확정된 사실

> "메인스레드에서 requestImage를 연속 호출하면 초반 프리즈를 만든다"

### 가능한 해결책

| 방안 | 설명 | 적용 난이도 |
|------|------|-------------|
| **초기 표시 셀 수 제한** | 초기엔 일부만 요청, 나머지는 다음 프레임/idle에 분산 | 쉬움 |
| **요청 분산 스케줄링** | 한 프레임에 몰리지 않게 우선순위/배치 처리 | 중간 |
| **요청 큐 분리** | 메인에서 연속 호출하지 않도록 구조 변경 | 어려움 (PhotoKit 메인스레드 요구 확인 필요) |
| **deliveryMode 변경** | `.opportunistic` → `.fastFormat` 등 | 쉬움 (부작용 확인 필요) |
| **PHCachingImageManager 사전 워밍업** | 앱 시작 시 미리 캐시 준비 | 중간 |

### 우선 시도 순서

1. `deliveryMode` 변경 테스트 (가장 빠름)
2. 초기 요청 수 제한 (효과 확실)
3. 요청 분산 스케줄링 (근본 해결)
