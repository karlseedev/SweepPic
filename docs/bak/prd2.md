# PickPhoto PRD v2 (초안)

## 0. 문서 정보

- 제품명: PickPhoto
- 목표: iOS 기본 사진(Photos) 앱에 최대한 근접한 성능/사용감의 사진 브라우저
- 상태: 기획 확정(구현 전)
- 최소 지원: iOS 16+
- 타깃 디바이스: iPhone 17(최고 성능), iPhone 12(최저 성능 보장)

---

## 1. 목표(Goals)

### 1.1 성능 목표(체감 품질)

- 스크롤: iPhone 17에서 ProMotion 120Hz 체감으로 “끊김이 거의 없는” 수준
- 첫 진입: All Photos 그리드에서 유효 썸네일이 빠르게 채워짐(로딩 티가 적음)
- 정확성: 빠른 스크롤/플릭/역방향에서도 “썸네일 오표시(다른 사진이 잠깐 보임)” 0
- 메모리: 장시간 스크롤/뷰어 왕복 후에도 메모리 사용이 안정적으로 수렴

### 1.2 제품 목표(UX)

- Photos와 유사한 그리드 기반 브라우징
- 핀치 제스처로 “연속 줌” 감성 제공(Photos와 동일한 느낌 목표)
- 앨범 리스트/앨범 진입(그리드/줌 동작 동일)

---

## 2. 비목표(Non-Goals, MVP 기준)

- iCloud 사진 보관함(네트워크 에셋) 지원: 제외
- 의미 검색(사람/장소/사물): 장기 옵션(별도 로드맵), MVP 제외
- “최근 삭제된 항목(복구)” 화면: MVP 제외(시스템 Photos 앱에서 복구)
- 즐겨찾기(하트): MVP 제외
- 삭제 Undo(실행 취소): MVP 제외(Undo 토스트/스낵바 미제공)
- 앱 내부 휴지통(추후 삭제/복구용 soft delete): MVP 제외(추후 여부 결정)

---

## 3. 타깃/전제

### 3.1 환경

- iOS 16+
- 사진 개수: **5만 장** 기준으로 성능 저하 없이 동작하도록 설계(“전체 배열/전체 스냅샷 물질화” 금지 원칙)

### 3.2 데이터 범위

- 로컬(기기) 내장 사진 중심
- 이미지 요청 시 네트워크 접근은 기본적으로 비허용(`isNetworkAccessAllowed = false` 정책)

---

## 4. MVP 범위(Scope)

### 4.1 화면

- 라이브러리: `All Photos` 그리드
- 앨범 탭: 사용자 앨범 리스트 + 앨범 진입(앨범 내 그리드)
- 스마트 앨범(일부): 예) `Screenshots`(필수), 필요 시 `Videos`/`Selfies` 추가
- 뷰어: 전체화면 사진 뷰어(그리드에서 진입)

### 4.2 공통 상호작용

- 핀치 연속 줌(그리드 밀도 변화)
  - MVP에서는 `All Photos` 내부에서만 밀도 변화 제공(Years/Months/Days 모드는 MVP 이후)
  - 목표 동작: Photos처럼 “부드럽게 변화하다가(인터랙션) 특정 임계에서 스냅되는” 감성(단, MVP에서는 모드 스냅은 비활성화 가능)
- 앵커 유지(핀치 중 위치 고정)
  - 핀치 제스처 중심점(두 손가락 중간) 아래의 콘텐츠를 앵커로 삼아, 줌 전환 중/후에도 해당 콘텐츠가 튀지 않게 유지

### 4.3 삭제(필수, 읽기 전용 아님)

- 삭제는 “항상 사진 삭제”로 단순화(앨범에서도 ‘앨범에서 제거’ 없음)
- 확인 팝업 없이 즉시 삭제
  - 단일 삭제: 즉시
  - 멀티 삭제: 즉시(Select 모드)
- 삭제 가능 위치
  - 그리드(All/앨범/스마트 앨범) + 뷰어(단일)
- 뷰어 삭제 제스처
  - 위 스와이프 = 삭제
- 뷰어 삭제 후 이동 규칙
  - 기본: 이전 사진으로 이동
  - 폴백: 이전이 없으면 다음, 남은 사진이 0이면 그리드로 복귀
- 복구 UX
  - MVP: 앱 내 복구/휴지통 화면은 제공하지 않음(시스템 Photos 앱에서 복구)

---

## 5. 핵심 설계 원칙(구현 독립 명세)

### 5.1 대용량(5만장 기준) 원칙

- “전체 에셋 배열 생성/유지” 및 “전체 스냅샷 반복 생성”을 설계적으로 피한다.
- 안정적인 식별자: `PHAsset.localIdentifier` 기반으로만 이동/선택/앵커를 유지한다.

### 5.2 스크롤 품질 원칙

- 보이는 것만 요청 + 프리패치 윈도우 기반 선제 캐싱 + 셀 재사용 시 요청 취소를 강제한다.
- 그리드용 썸네일은 작은 타깃 사이즈(디코딩/메모리 절약), 뷰어는 단계적 품질 향상(저→고)을 원칙으로 한다.

### 5.3 UI 기술 선택(성능 우선)

- 핵심 스크롤 영역(그리드/뷰어)은 UIKit 기반(`UICollectionView`)을 기본으로 한다.
- SwiftUI는 셸/설정/상태 화면 등 비핵심에 하이브리드로 사용 가능(핵심 성능 경로는 UIKit 고정).

## 6. 아키텍처 경계(모듈 정의)

아래 경계는 “빠른 이동(Years/Months/Days)”과 “검색(옵션)”을 나중에 얹어도 코어를 흔들지 않기 위한 고정 규칙이다.

- `LibraryStore`
  - PhotoKit fetch/권한/변경 감지(Change Observer)
  - 정렬/필터 정책의 단일 소스
- `TimelineIndex` (MVP에서는 All-only로 최소 구현, v1~v2에서 확장)
  - 줌 레벨(Years/Months/Days/All)별 섹션/버킷 경계 및 ID↔indexPath 매핑
  - “앵커(특정 assetID)”를 기준으로 빠른 점프/전환을 지원
- `GridDataSource`
  - UI 전용 어댑터(프리패치, 재사용, 멀티선택)
  - 실제 이미지 요청은 `ImagePipeline`에만 위임
- `ImagePipeline`
  - 요청/취소/코얼레싱/캐시 정책을 단일화
  - `PHCachingImageManager` 기반 preheat 정책 포함
- `ViewerCoordinator`
  - 뷰어 내비게이션(이전/다음), 삭제 후 이동, 전환 애니메이션 정책
- `SearchService` (옵션, MVP 제외)
  - 쿼리 → `assetID[]` 반환 인터페이스만 고정
  - 인덱싱/저장소는 사이드카(없어도 앱 동작)로 분리

## 7. 조사/스파이크 계획(구현 전 검증)

목표: “Photos급 체감”을 좌우하는 리스크를 짧은 실험으로 판정하고, 결과로 세부 정책(프리히트 윈도우/스냅샷 전략/앵커 보정)을 고정한다.

- S1. 대용량 데이터 소스 전략
  - All Photos **5만장** 가정에서 데이터 소스/업데이트 전략이 메인 스레드 프레임을 깨지 않는지 계측
- S2. 프리히트(preheat) 정책
  - 스크롤 속도에 적응하는 preheat 윈도우가 hitch를 얼마나 줄이는지, 메모리/CPU 트레이드오프 산정
- S3. 요청 취소/정확성
  - 셀 재사용/빠른 플릭에서 오표시 0을 보장하는 취소+검증(토큰) 패턴 확정
- S4. 핀치 앵커 보정
  - 핀치 중심 앵커 유지가 자연스럽게 동작하는지(점프/떨림/지연) 계측 및 UX 시나리오 정의
- S5. 삭제/변경 감지
  - 멀티삭제/연속삭제 후에도 그리드/뷰어 상태(선택/스크롤/현재 항목)가 안정적으로 유지되는지 검증
