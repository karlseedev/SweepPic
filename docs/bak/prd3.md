# PickPhoto PRD v3 (통합본)

## 0. 문서 정보

- 제품명: PickPhoto
- 목적: iOS 기본 사진(Photos) 앱과 유사한 “매우 빠른” 사진 브라우징 경험 제공
- 본 문서의 성격: 제품 요구사항(PRD) + 성능/품질 수용 기준(측정 가능)
- 최소 지원: iOS 16+
- 타깃 디바이스
  - 최고 성능 기준: iPhone 17 (ProMotion 120Hz)
  - 최저 성능 보장: iPhone 12

---

## 1. 배경/문제 정의

iOS 기본 사진 앱은 대규모 라이브러리에서도 빠른 실행, 즉시성 있는 썸네일 채움, 매우 부드러운 스크롤(특히 ProMotion), 자연스러운 전환/제스처를 제공합니다. PickPhoto는 가능한 한 동일한 공개 API(주로 PhotoKit) 기반으로 이 체감 품질을 달성하는 것을 목표로 합니다.

---

## 2. 목표(Goals)

### 2.1 핵심 UX 목표

- All Photos 그리드가 “즉시 열리고”, 스크롤이 “매우 부드럽다”고 느껴지게 합니다.
- 핀치 제스처로 그리드 밀도(한 화면에 보이는 사진 수)가 자연스럽게 변화합니다(Photos와 유사한 감성).
- 앨범 리스트/앨범 진입에서도 동일한 그리드/줌 경험을 제공합니다.
- 그리드와 뷰어에서 모두 삭제가 가능하며, 삭제 후 흐름이 끊기지 않습니다.

### 2.2 성능/품질 목표(측정 가능한 KPI, 초기 목표치)

아래 수치는 “초기 목표치”이며, 스파이크(실험) 및 계측 결과로 조정 가능합니다.

| 항목 | iPhone 17 목표 | iPhone 12 목표 | 비고(정의) |
|---|---:|---:|---|
| 스크롤 체감 | 120Hz급(끊김 거의 없음) | 60Hz급(끊김 최소) | hitch 비율로 관리 |
| 스크롤 hitch 비율 | < 1% | < 2% | 스크롤 중 프레임 예산 초과 비율 |
| 첫 그리드 “유효 썸네일” 표시 | < 300ms | < 600ms | 초기 화면에서 placeholder가 실 이미지로 대체되는 시점 |
| 썸네일 응답 시간(p50) | < 100ms | < 150ms | 화면 내 셀 단위(요청→표시) |
| 오표시(정확성) | 0 | 0 | 빠른 플릭에서도 다른 사진이 잠깐 보이면 실패 |
| 메모리 상한(장시간 스크롤/뷰어 왕복) | < 400MB | < 400MB | 초기 목표치(계측 후 조정) |
| 삭제 반영 | < 500ms | < 700ms | UI에서 사라지고 상태가 안정화되는 시점 |
| 콜드 스타트 | < 2s | < 2.5s | 첫 화면 인터랙션 가능 시점(초기 목표치) |

#### 2.2.1 KPI 측정 정의(초기)

- 스크롤 hitch 비율
  - 정의: 스크롤 중 “프레임 예산(120Hz: 8.3ms / 60Hz: 16.7ms)을 초과한 프레임”의 비율
  - 측정: Instruments(Core Animation) 중심으로 계측하며, 산출 방식(샘플링 구간/스크롤 시나리오/집계)은 스파이크에서 확정합니다.
- 첫 그리드 “유효 썸네일” 표시
  - 정의: 최초 화면에서 placeholder가 실 이미지로 대체되어 “그리드가 채워졌다”고 느껴지는 시점
  - 측정: 앱 진입 시점부터 최초 화면 셀 N개(예: 12개)가 실 이미지로 채워질 때까지의 시간(스파이크에서 N 확정)

## 3. 비목표(Non-Goals, MVP 기준)

- iCloud 사진 보관함(네트워크 에셋) 본격 지원: 제외
  - 정책: 기본적으로 `isNetworkAccessAllowed = false`를 전제로 설계합니다.
- 의미 검색(사람/장소/사물): 장기 옵션(별도 로드맵), MVP 제외
- 즐겨찾기(하트): MVP 제외
- 앱 내 “최근 삭제됨(복구)” 화면: MVP 제외
- 삭제 실행 취소 토스트: MVP 제외
- 앱 내부 휴지통(soft delete/복구를 위한 별도 보관함): 후속 옵션(별도 로드맵), MVP 제외

## 4. 타깃/전제

### 4.1 대용량 기준

- 성능 보장 기준: **5만 장** 라이브러리에서도 성능 저하 없이 동작하도록 설계합니다.
- 원칙: “전체 에셋 배열 생성/유지” 및 “전체 스냅샷 반복 생성”을 설계적으로 피합니다.

### 4.2 데이터 범위(로컬 중심)

- 로컬(기기) 사진 중심으로 최적화합니다.
- iCloud-only 에셋(기기에 원본 없음)이 섞여 있을 수 있으므로, MVP에서는 “다운로드를 시도하지 않는다” 정책 하에 사용자에게 명확한 상태(placeholder/안내)를 제공하는 방향으로 정의합니다(세부 UX는 스파이크에서 확정).

## 5. MVP 범위(Scope)

### 5.1 화면

- 라이브러리: `All Photos` 그리드
- 앨범 탭
  - 사용자 앨범 리스트
  - 앨범 진입(앨범 내 그리드)
  - 스마트 앨범 일부: `Screenshots`(필수), 필요 시 `Videos`/`Selfies` 추가
- 상세 뷰어: 전체화면 사진 뷰어(그리드에서 진입)

### 5.2 그리드/줌(Photos-like)

- 핀치 연속 줌(그리드 밀도 변화)
  - MVP: `All Photos` 내부에서 “연속 밀도 변화” 제공
  - 후속: Years/Months/Days/All 4단 모드 스냅은 v2~v3 로드맵으로 확장
- 앵커 유지(핵심)
  - 핀치 제스처 중심점(두 손가락 중간) 아래의 콘텐츠를 앵커로 캡처하고,
  - 줌 변화 중/후에도 해당 콘텐츠가 튀지 않게 유지합니다(오프셋 보정).

### 5.3 삭제(필수, 읽기 전용 아님)

- 삭제 정책: **항상 “사진 삭제”**로 단순화합니다(앨범에서 “앨범에서 제거”는 제공하지 않습니다).
- 삭제 확인: 제공하지 않습니다(즉시 삭제).
- 삭제 위치
  - 그리드(All/앨범/스마트 앨범): 단일 삭제 + 멀티선택 삭제(Select 모드)
  - 뷰어(단일): 제스처로 삭제
- 뷰어 삭제 제스처: **위 스와이프 = 삭제**
- 삭제 후 이동(뷰어)
  - 기본: 이전 사진으로 이동
  - 폴백: 이전이 없으면 다음, 남은 사진이 0이면 그리드로 복귀
- 복구: MVP에서는 앱 내 복구/휴지통 UI를 제공하지 않습니다(시스템 Photos 앱의 “최근 삭제됨”을 통해서만 복구 가능).

## 6. 기술 스택/핵심 결정(제품 요구사항 관점)

### 6.1 공개 API 기반 “네이티브에 가까운” 스택

- 라이브러리 접근/변경 감지: PhotoKit(`Photos`)
- 이미지 요청/캐싱: `PHCachingImageManager`
- 그리드 UI: UIKit `UICollectionView` (성능 경로는 UIKit을 기본으로 고정)
- SwiftUI: 설정/셸 UI 등 비핵심에 한해 하이브리드 적용 가능

#### 6.1.1 `UICollectionView` 선택 근거(비교표, PRD 레벨)

| 기준 | SwiftUI `LazyVGrid` | UIKit `UICollectionView` | 결론 |
|---|---|---|---|
| 셀 재사용/수명 제어 | 간접적(패턴 의존) | 직접적(재사용 풀) | 대용량에서 유리 |
| Prefetch/취소 | 제약이 있음 | 세밀한 제어 가능 | 오표시 0에 유리 |
| 스크롤 성능(5만장) | 편차가 커질 수 있음 | 검증된 패턴이 많음 | 예측 가능성 높음 |
| 핀치/앵커/전환 | 구현 가능하나 우회가 늘 수 있음 | 직접 제어 가능 | Photos-like에 유리 |
| PRD 결론 | 비핵심 UI에 적합 | 핵심 그리드/뷰어에 적합 | 그리드는 `UICollectionView` 고정 |

### 6.2 이미지 로딩 정책(요구사항)

- 그리드: 작은 타깃 사이즈 + 빠른 포맷 우선(스크롤 품질 우선)
- 뷰어: 단계적 품질 향상(저→고)을 허용하되, 전환/제스처 중 프레임 예산을 침범하지 않도록 제한합니다.
- 프리패치/프리히트: 가시 영역 기반 + 스크롤 속도에 적응하는 preheat 윈도우를 운영합니다.

## 7. 아키텍처 경계(모듈 정의, PRD 레벨)

후속 기능(Years/Months/Days, 빠른 이동, 검색 옵션)을 추가하더라도 코어 성능 경로를 흔들지 않기 위한 고정 규칙입니다.

- `LibraryStore`: PhotoKit fetch/권한/변경 감지, 정렬/필터 정책 단일 소스
- `TimelineIndex`: (MVP는 All-only 최소) 줌 레벨/버킷 경계/ID↔indexPath 매핑, 앵커 점프 지원
- `AlbumIndex`: 앨범/스마트 앨범 목록, 앨범 내 fetch/정렬 정책
- `ImagePipeline`: 요청/취소/코얼레싱/캐시 정책 단일화, `PHCachingImageManager` preheat 포함
- `GridController`: 프리패치/재사용/멀티선택 등 UI 어댑터(이미지는 `ImagePipeline`만 호출)
- `ViewerCoordinator`: 뷰어 내비게이션, 삭제 제스처/삭제 후 이동, 전환 정책
- `SearchService`(옵션): 쿼리 → `assetID[]` 반환 인터페이스만 고정(인덱싱/저장소는 사이드카)

## 8. 계측/검증(Definition of Done)

### 8.1 계측 도구(권장)

- Instruments: Time Profiler, Core Animation, Allocations/Leaks
- MetricKit: 앱 성능 메트릭 수집(릴리즈 품질 확인)

### 8.2 수용 기준(게이트)

MVP는 다음 조건을 모두 만족해야 합니다.

- 5만 장 라이브러리 기준으로도 스크롤 hitch 비율이 목표치 이내
- 빠른 플릭/역방향 스크롤에서도 오표시 0
- 핀치 줌 중/후에 앵커가 튀지 않고, 입력 지연이 체감되지 않음
- 멀티선택 삭제/연속 삭제 후에도 UI가 안정적으로 수렴(선택/스크롤/뷰어 이동 규칙 준수)

## 9. 스파이크(조사/실험) 계획

목표: 핵심 리스크를 짧게 실험해 정책을 확정하고, KPI를 현실적으로 보정합니다.

- S1. 대용량 데이터 소스 전략(5만장)
  - “전체 배열/전체 스냅샷” 없이도 그리드가 안정적으로 동작하는 전략을 확정합니다.
- S2. 프리히트(preheat) 정책
  - ±N 화면 프리캐시, 속도 적응형 윈도우의 hitch/메모리 트레이드오프를 수치화합니다.
- S3. 요청 취소/정확성
  - 셀 재사용/프리패치 취소에서 오표시 0을 보장하는 패턴(취소 + 토큰 검증)을 확정합니다.
- S4. 핀치 앵커 보정
  - 핀치 중심 앵커 유지가 자연스럽게 동작하는지(점프/떨림/지연) 검증하고, 폴백 규칙을 확정합니다.
- S5. 삭제/변경 감지
  - 멀티삭제/연속삭제 후에도 그리드/뷰어 상태가 안정적으로 유지되는지 검증합니다.
- S6. ProMotion(120Hz) 달성 검증
  - ProMotion 디바이스에서 스크롤/핀치/전환 중 hitch 비율이 목표치 이내인지 계측하고, 병목(메인 스레드/디코딩/레이아웃)을 분류해 개선 포인트를 도출합니다.

## 10. 로드맵(마일스톤)

- M0. 명세 고정: 본 PRD 기준으로 “불변 규칙” 1페이지 요약본 작성
- M1. MVP: All + Albums + Viewer + Delete(즉시) + 핀치 줌(밀도 변화)
- M2. Zoom 확장: Days 추가(All↔Days 스냅)
- M3. Full Timeline: Months/Years + 4단 스냅 + 빠른 이동(스크럽/인덱스) UI
- M4. 옵션: 의미 검색(사람/장소/사물), 앱 내부 휴지통(soft delete/복구)
