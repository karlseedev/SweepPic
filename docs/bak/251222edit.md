### Phase 3: 필요 시 추가 최적화

| # | 작업 | 조건 | 방식 |
|---|------|------|------|
| 3-1 | 50%→100% 캐시 fallback | Phase 1 후에도 L2 스파이크 지속 시 | 리사이즈 없이 100% 그대로 반환 |
| 3-2 | ImagePipeline dedupe | req/cancel churn 확인 시 | assetID+targetSize 단위 중복 방지 |
| 3-3 | maxInFlight 제한 | Pipeline 폭주 확인 시 (Phase 2 계측 후) | 8~12개 제한 |

**3-1 구현 (안전한 방식):**
```swift
// PhotoCell.swift - 메모리 캐시 조회 시
// 1. 50% 캐시 조회
if let cached50 = cache.get(assetID, size: halfSize) {
    return cached50  // ✅ 정확한 크기
}

// 2. 100% fallback (리사이즈 없이 그대로)
if let cached100 = cache.get(assetID, size: fullSize) {
    return cached100  // ✅ UIImageView가 GPU 스케일링
    // Pipeline 요청 생략 → 요청 폭주 감소
}

// 3. 둘 다 없으면 Pipeline 요청
requestFromPipeline(...)
```

**주의:**
- ❌ 100% → resize → 50% (매번 CPU 리사이즈) → hitch 위험
- ✅ 100% 그대로 UIImageView에 표시 → GPU가 스케일링 (CPU 무부하)
- opportunistic의 degraded가 "빠른 표시" 담당 → fallback은 "요청 줄이기" 목적

## 교훈

### 후보 A에서 배운 것

1. **fastFormat은 전 구간 적용 금지**
   - 품질 저하 (degraded 없음 → 흐릿함)
   - opportunistic의 degraded first-paint가 더 효과적

2. **didEndDisplaying은 완주율 하락 유발**
   - 과도한 취소 → 회색 셀 증가
   - prepareForReuse만으로 충분

3. **테스트 환경 차이 주의**
   - gate2-clean-start (캐시 없음) ≠ 실제 환경 (캐시 있음)
   - 같은 설정도 다른 결과 가능

### 일반 원칙

1. **증거 기반 판단**
   - 로그로 인과관계 확인 후 조치
   - 추측으로 강한 레버 조작 금지

2. **리스크 최소화**
   - 쉽고 안전한 것부터 적용
   - 단계별 효과 확인 후 다음 진행

3. **정책 일관성**
   - "디스크 캐시 = 프리로드 전용" 정책 지키기
   - 스크롤 중 저장은 정책 위반

## Phase 3-1 결과 (롤백됨)

### 시도한 구현

**목표:** 50% 캐시 미스 시 100% 캐시 fallback으로 memHit 증가

```swift
// PhotoCell.swift - 50%→100% fallback 시도

    // 50% 히트
} else if let fallback = MemoryThumbnailCache.shared.get(assetID: assetID, size: fullPixelSize) {
    // 100% fallback 히트 → Pipeline 요청 생략
}
```

### 테스트 결과

| 지표 | 이전 | 이후 | 변화 |
|------|------|------|------|
| memHit | 0 | 0 | ❌ 변화 없음 |
| memMiss | ~2500 | ~5000 | ❌ 2배 증가 |

### 실패 원인 분석

**Bug 1: 키 계산 문제**
```swift
// thumbnailSize(forScrolling: false)가 isScrolling=true 상태에서도 50% 반환
// GridViewController 내부에서 isScrolling 상태 체크가 발생
// → fullPixelSize가 실제 100% 크기가 아님
```

**Bug 2: MemoryThumbnailCache 채워지지 않음**
```swift
// MemoryThumbnailCache는 프리로드 시에만 채워짐 (12개)
// Pipeline 완료 시 MemoryThumbnailCache에 저장하지 않음
// → 스크롤 중 100% fallback이 항상 미스
```

### 결론

3-1은 **"헛조회 1회 추가"**만 발생:
- 50% 캐시 조회 (미스) → 100% 캐시 조회 (미스) → Pipeline 요청
- memMiss가 2배로 증가하면서 오버헤드만 발생
