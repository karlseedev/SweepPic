| 항목 | 내용 |
|------|------|
| 제품명 | PickPhoto |

## 1. 배경/문제 정의

iOS 기본 사진 앱은 대규모 라이브러리에서도 다음을 제공합니다:
- 빠른 실행
- 즉시성 있는 썸네일 채움
- 매우 부드러운 스크롤 (특히 ProMotion)
- 자연스러운 전환/제스처

**PickPhoto는 가능한 한 동일한 공개 API(주로 PhotoKit) 기반으로 이 체감 품질을 달성하는 것을 목표로 합니다.**

추가로, 위 스와이프 삭제 등 "사진 정리에 특화된" UX를 제공하여 기본 앱 대비 차별점을 만듭니다.

## 2. 목표 (Goals)

### 2.1 핵심 UX 목표

- All Photos 그리드가 "즉시 열리고", 스크롤이 "매우 부드럽다"고 느껴지게 합니다.
- 핀치 제스처로 그리드 밀도가 자연스럽게 변화합니다 (Photos와 유사한 감성).
- 앨범 리스트/앨범 진입에서도 동일한 그리드/줌 경험을 제공합니다.
- 그리드와 뷰어에서 모두 삭제가 가능하며, 삭제 후 흐름이 끊기지 않습니다.

### 2.2 성능/품질 목표 (측정 가능한 KPI)

아래 수치는 **초기 목표치**이며, 스파이크(실험) 및 계측 결과로 조정될 수 있습니다.

| 항목 | iPhone 17 Pro | iPhone 12 | 정의/측정 방법 |
|------|---------------|-----------|----------------|
| 스크롤 체감 | 120Hz급 (끊김 거의 없음) | 60Hz급 (끊김 최소) | 사용자 체감 + hitch 비율 |
| 스크롤 hitch 비율 | < 1% | < 2% | Instruments Core Animation으로 측정 |
| 첫 그리드 유효 썸네일 | < 300ms | < 600ms | placeholder → 실 이미지 대체 시점 |
| 썸네일 응답 시간 (p50) | < 100ms | < 150ms | 셀 요청 → 표시 (Instruments 측정) |
| 오표시 (정확성) | 0 | 0 | 빠른 플릭에서 다른 사진이 잠깐 보이면 실패 |
| 메모리 상한 | < 400MB | < 400MB | 장시간 스크롤/뷰어 왕복 후 (Allocations) |
| 삭제 반영 | < 500ms | < 700ms | UI에서 사라지고 상태 안정화 시점 |
| 콜드 스타트 | < 2s | < 2.5s | 첫 화면 인터랙션 가능 시점 |

**지표 정의 보충:**
- **hitch 비율**: 스크롤 중 프레임 예산(120Hz: 8.3ms / 60Hz: 16.7ms)을 초과한 프레임의 비율. 사용자가 "끊김"으로 체감. Instruments Core Animation으로 계측하며, 산출 방식(샘플링 구간/스크롤 시나리오/집계)은 스파이크에서 확정.
- **첫 그리드 유효 썸네일**: 최초 화면에서 placeholder가 실 이미지로 대체되어 "그리드가 채워졌다"고 느껴지는 시점. 앱 진입부터 최초 화면 셀 N개(예: 12개)가 실 이미지로 채워질 때까지의 시간. N은 스파이크에서 확정.
- **오표시**: 셀 재사용 시 이전 이미지가 잠깐 보이는 현상. 0이 아니면 품질 실패.
- **메모리 상한**: 이 수치를 초과하면 시스템 메모리 경고/앱 종료 위험 증가.

## 5. MVP 범위 (Scope)

### 5.1 화면

| 화면 | 설명 |
|------|------|
| 라이브러리 | All Photos 그리드 |
| 앨범 탭 | 사용자 앨범 리스트 + 앨범 진입 (앨범 내 그리드) |
| 스마트 앨범 | Screenshots (필수), Videos/Selfies (선택) |
| 뷰어 | 전체화면 사진 뷰어 (그리드에서 진입) |

### 5.2 공통 상호작용

**핀치 연속 줌**
- MVP: All Photos 내부에서 밀도 변화 제공
- **그리드 줌 단계: 1/3/5열** (Photos와 동일)
- 목표 동작: Photos처럼 부드럽게 변화하는 감성
- Years/Months/Days 모드 스냅은 MVP 이후

**앵커 유지 (핵심)**
- 핀치 제스처 중심점(두 손가락 중간) 아래 콘텐츠를 앵커로 캡처
- 줌 변화 중/후에도 해당 콘텐츠가 튀지 않게 유지 (오프셋 보정)

**멀티 선택**
- Select 모드로 여러 장 선택 후 일괄 삭제

### 5.3 삭제 정책

| 항목 | 정책 |
|------|------|
| 삭제 정책 | 항상 "사진 삭제"로 단순화 (앨범에서 "앨범 제거" 없음) |
| 확인 팝업 | 없음 (즉시 삭제) |
| 삭제 위치 | 그리드 (단일 + Select 모드) + 뷰어 (단일) |
| 뷰어 삭제 제스처 | **위 스와이프 = 삭제** |
| 삭제 후 이동 (뷰어) | **이전 사진** → 없으면 다음 → 없으면 그리드 복귀 |
| 복구 | 시스템 Photos '최근 삭제됨'에서 (MVP) |
| 앱 내 휴지통 | 후속 버전 (M4) |

**삭제 안전장치 (필수 요구사항)**

| 조건 | 요구사항 |
|------|----------|
| 삭제 가능 조건 | `PHAuthorizationStatus` = `.authorized` (readWrite) 일 때만 삭제 가능 |
| 권한 부족 시 | 삭제 UI 비활성화 또는 권한 요청 안내 표시 |
| 연속/멀티 삭제 | 삭제 후 UI가 안정적으로 수렴해야 함 (선택/스크롤/뷰어 이동 규칙 준수) |
| 뷰어 삭제 후 이동 | 반드시 "이전 사진 우선" 규칙 준수 |

## 6. 핵심 요구사항 (불변 규칙)

> 본 섹션은 **"반드시 지켜야 하는 요구사항"**만 정의합니다.
> 구현 방식/기술 선택은 7장 또는 [TechSpec.md](./TechSpec.md)에서 다룹니다.

### 6.1 대용량 (5만 장) 요구사항

- 5만 장에서도 스크롤 hitch 비율이 목표치 이내여야 함
- "전체 에셋 배열" 또는 "전체 스냅샷 반복 생성"으로 인한 메모리 급증이 없어야 함
- 안정적인 식별자 기반으로만 이동/선택/앵커 유지

### 6.2 스크롤 품질 요구사항

- 빠른 플릭/역방향 스크롤에서도 **오표시 0**
- 화면에 보이지 않는 셀의 로딩이 보이는 셀의 로딩을 방해하면 안 됨
- 스크롤 중 프레임 예산(8.3ms/16.7ms)을 초과하는 메인 스레드 작업 금지

**ProMotion (120Hz) 요구사항**
- ProMotion은 가변 주사율(10~120Hz)이며, 앱이 120Hz를 강제할 수는 없음
- **앱은 시스템이 120Hz를 선택할 수 있도록 프레임 버짓(8.3ms) 준수를 보장한다**
- 이미지 디코딩 등 무거운 작업은 메인 스레드 외부에서 처리

### 6.3 핀치 줌 요구사항

- **줌 단계: 1/3/5열** (Photos와 동일)
- 핀치 중심점 아래 콘텐츠가 **앵커로 유지**되어야 함 (점프/떨림 불가)
- 핀치 제스처 중 입력 지연이 체감되면 안 됨
- 밀도 변화 후에도 동일 콘텐츠가 화면에 유지

### 6.4 삭제 요구사항

- 멀티 삭제/연속 삭제 후에도 **UI가 안정적으로 수렴**
- 삭제 반영 시간이 목표치(500ms/700ms) 이내
- 뷰어에서 삭제 시 "이전 사진 우선" 이동 규칙 준수

### 6.5 메모리 요구사항

- 장시간 스크롤/뷰어 왕복 후에도 메모리 상한(400MB) 이내 유지
- 메모리 경고 발생 시 캐시 즉시 해제

## 7. 기술 방향 (후보 솔루션, 스파이크에서 검증)

> 아래는 PRD 목표 달성을 위한 **후보 방향**입니다.
> 스파이크 결과에 따라 변경될 수 있으며, 최종 결정은 Tech Spec에서 확정합니다.

### 7.1 그리드 UI 기술 후보

| 후보 | 장점 | 단점 | 검증 스파이크 |
|------|------|------|---------------|
| UIKit UICollectionView | 셀 재사용 완전 제어, 프리패치 지원 | 코드량 증가 | S1 |
| SwiftUI LazyVGrid | 선언적 UI, 빠른 개발 | 대용량 메모리 누적 위험 | S1 |

**권장 방향**: UIKit 기반 (S1에서 5만 장 테스트로 검증)

### 7.2 이미지 캐싱 후보

- **PHCachingImageManager**: PhotoKit 공식 API, 시스템 최적화
- 가시 영역 기반 프리히트 정책 필요
- 세부 정책(윈도우 크기, 속도 적응)은 **S2에서 확정**

### 7.3 이미지 로딩 정책 후보

**썸네일 옵션** (Photos 내부 옵션은 비공개이므로 A/B 테스트로 "유사 체감" 확보)

| 후보 | 설명 | 장단점 | 검증 스파이크 |
|------|------|--------|---------------|
| A) 빠른 표시 우선 | `opportunistic` + `fast` + 셀 픽셀 크기 | 첫 채움↑, 스크롤↑, 저→고 교체감 있을 수 있음 | S3 |
| B) 품질 우선 | `highQualityFormat` + targetSize 여유 | 교체감↓, 첫 채움↓, 메모리↑ | S3 |

- 1/3/5열 모드별로 다른 옵션 적용 가능 (스파이크로 확정)

**요청 취소 패턴**
- 셀 재사용 시 이전 요청 취소 + 토큰 검증 패턴
- 세부 패턴은 **S3에서 확정**

## 9. 검증 계획

### 9.0 공통 전제: 120Hz 계측 규약

모든 검증 단계에서 공통으로 적용되는 성능 측정 규약입니다.

| 항목 | 규약 |
|------|------|
| 측정 도구 | Instruments Core Animation (hitch 비율) |
| 측정 디바이스 | iPhone 12 (60Hz) + ProMotion 디바이스 (120Hz) |
| 측정 시나리오 | 빠른 스크롤 30초 (왕복 포함), 정지→스크롤 반복 |
| 통과 기준 | hitch 비율 목표치 이내 (2.2절 KPI 참조) |

### 9.1 개발 전 스파이크 (1~2일, 필수)

> **기준**: 결과에 따라 아키텍처가 바뀌고, 나중에 교체 비용이 큰 결정

#### ~~Spike 0: UI 기술 스택~~ → **확정: UIKit**

> 대용량 그리드에서 UIKit UICollectionView가 SwiftUI LazyVGrid보다 셀 재사용/메모리 관리에서 유리함이 명확하므로 스파이크 없이 확정.
> SwiftUI는 설정/셸 등 비핵심 UI에만 사용.

#### Spike 1: 데이터 소스 패턴 결정 → **확정: performBatchUpdates**

| 항목 | 내용 |
|------|------|
| 목적 | DiffableDataSource가 5만 장 + 대량 삭제에서 버티는지 검증 |
| 방법 | UICollectionView + DiffableDataSource + 5만 개 더미 데이터 |
| 시나리오 | 100개 삭제 + 연속 삭제 20회 + 스크롤 중 삭제 |
| 판정 기준 | Apple Hitch 기준 < 5 ms/s (Good) |
| 후보 | A) DiffableDataSource, B) 수동 배치 업데이트 (`performBatchUpdates`) |

**Spike 1 결과 (2025-12-16)**

| 후보 | 50k 단일 삭제 p95 | Hitch Time Ratio | 스케일링 | 판정 |
|------|-------------------|------------------|----------|------|
| Plan A (DiffableDataSource) | 52ms | 22 ms/s ❌ Critical | O(N) | 불합격 |
| **Plan B (performBatchUpdates)** | **5ms** | **0 ms/s** ✅ Good | **O(1)** | **합격** |

> **채택**: Plan B (performBatchUpdates + 수동 배열)
> - `apply()` 비용이 O(N)이므로 10k 이상에서 DiffableDataSource 구조적 한계
> - 상세: [TechSpec.md](./TechSpec.md) Spike 1 결과, [spiketest.md](./spiketest.md)

### 9.2 개발 단계별 검증 게이트

> 각 단계 완료 시 통과해야 하는 검증 항목. 실패 시 해당 단계에서 해결 후 진행.

#### Gate 1: 그리드 기반 (이미지 로딩 전)

| 검증 항목 | 통과 조건 |
|-----------|-----------|
| 5만 장 스크롤 | hitch 비율 목표 이내 (더미 셀 또는 placeholder) |
| 데이터 소스 안정성 | 삭제/변경 시 상태 꼬임 없음 |

#### Gate 2: 이미지 로딩

| 검증 항목 | 통과 조건 |
|-----------|-----------|
| 썸네일 로딩 | 첫 그리드 유효 썸네일 목표 이내 |
| 오표시 | 빠른 플릭/역방향 스크롤에서 0 |
| preheat 정책 확정 | 고정(±N) vs 속도 적응형 선택, N 값 확정 |
| 썸네일 옵션 확정 | 빠른 표시 vs 품질 우선 선택, 1/3/5열 모드별 테이블 |

**preheat 후보:**
- A) 고정 윈도우: ±N (N ∈ {1, 2, 3, 4})
- B) 속도 적응형: 느림 ±1, 보통 ±2, 빠름 ±3

**썸네일 옵션 후보:** (Photos 내부 옵션은 비공개이므로 A/B 테스트로 "유사 체감" 확보)
- A) 빠른 표시 우선: `opportunistic` + `fast` + 셀 픽셀 크기
- B) 품질 우선: `highQualityFormat` + targetSize 여유

#### Gate 3: 핀치 줌 / 삭제

| 검증 항목 | 통과 조건 |
|-----------|-----------|
| 핀치 앵커 | 중심점 콘텐츠가 점프/떨림 없이 유지 |
| 핀치 입력 지연 | 체감 지연 없음 |
| 삭제 안정성 | 멀티 삭제/연속 삭제 후 UI 안정 수렴 |
| 변경 감지 | 부분 업데이트 vs 전체 리로드 규칙 확정 |
| 뷰어 삭제 후 이동 | "이전 사진 우선" 규칙 준수 |

#### Gate 4: 성능 튜닝 (MVP 완성 직전)

| 검증 항목 | 통과 조건 |
|-----------|-----------|
| 120Hz 프레임 버짓 | ProMotion에서 8.3ms 버짓 준수 (hitch 비율 목표 이내) |
| 메모리 | 장시간 스크롤/뷰어 왕복 후 400MB 이내 |
| 병목 분류 | 메인스레드/디코딩/레이아웃 중 병목 식별 및 개선 |
| 콜드 스타트 | 첫 화면 인터랙션 가능 시점 목표 이내 |

## 10. 계측/검증 (Definition of Done)

### 10.1 계측 도구

- **Instruments**: Time Profiler, Core Animation (hitch), Allocations/Leaks
- **MetricKit**: 앱 성능 메트릭 수집 (릴리즈 품질 확인)
- **XCTest**: 자동화된 성능 테스트

### 10.2 수용 기준 (MVP Gate)

MVP는 다음 조건을 **모두** 만족해야 합니다:

| 기준 | 조건 |
|------|------|
| 스크롤 품질 | 5만 장에서 hitch 비율이 목표치 이내 |
| 정확성 | 빠른 플릭/역방향 스크롤에서 오표시 0 |
| 핀치 줌 | 앵커가 튀지 않고, 입력 지연 체감 없음 |
| 삭제 안정성 | 멀티/연속 삭제 후 UI 안정 수렴 |
| 메모리 | 장시간 사용 후 400MB 이내 유지 |
