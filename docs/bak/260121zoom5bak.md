## 0. v6 핵심 수정 사항 (v5 테스트 피드백 반영)

### 0.2 문제별 상세 분석 및 해결 방안

#### 문제 1: X축 좌측 정렬

**증상**: 핀치줌으로 줄어들때 가운데가 기준이 아니라 좌측이 기준이 됨. 천천히 핀치줌할때 4열 상태가 되었다가 5열로 됨.

**원인 분석**:
- 현재 `updateContentOffsetForAnchor`에서 X, Y 모두 계산하지만
- collectionView가 가로 스크롤이 없어서 (width == contentWidth) X 보정이 무의미
- **핵심**: 앵커 보정은 Y축만 유효, X축은 레이아웃 자체가 좌측 정렬됨

**해결 방안**:
```
현재 레이아웃 계산:
  x = col * (itemWidth + spacing)  // 항상 좌측부터 시작

필요한 변경:
  핀치 중심점 X를 기준으로 전체 그리드가 확대/축소되도록
  → 방법 A: transform 사용 (이전에 폐기됨)
  → 방법 B: X축 offset 보정 (새 방안)

방법 B 상세:
  핀치 시작 시 anchorPointInView.x를 저장
  레이아웃 변경 후, 앵커 셀의 새 center.x와 anchorPointInView.x의 차이만큼 contentInset.left 조정
  (contentOffset.x는 항상 0이므로 inset으로 보정)
```

#### 문제 2: 스크롤 점프 + fadeIn 미동작

**증상**: 핀치 줌 시작하면 각 셀이 바뀌어야 할 이미지로 페이드인 되어야 하는데 안됨. 셀 기준 좌표 축이 틀어지면서 위로 스크롤이 되었다가 다시 돌아옴.

**원인 분석**:
1. **스크롤 점프**:
   - `.began`에서 앵커 결정 시 좌표 변환 문제
   - 또는 첫 `.changed`에서 레이아웃 갱신 시 앵커 보정 타이밍 불일치

2. **fadeIn 미동작**:
   - `triggerFadeIn` 조건: `abs(scale - 1.0) > 0.01`
   - 조건 통과해도 `cell.fadeInImage`가 호출되지 않거나
   - `fadeToken` 검증 실패
   - 또는 `isDegraded == true`로 early return

**해결 방안**:
```
1. 스크롤 점프 해결:
   - .began에서 앵커 확정 후 contentOffset 저장
   - .changed 첫 프레임에서 저장된 offset 복원 후 레이아웃 갱신
   - 레이아웃 갱신 후 앵커 기반 offset 재계산

2. fadeIn 해결:
   - isDegraded 조건 제거 (일단 표시 후 고해상도로 교체)
   - fadeToken 검증 로직 단순화
   - 디버그 로그 추가로 실패 원인 파악
```

#### 문제 3: 바깥 셀 클리핑

**증상**: 핀치줌으로 줄어들때 4열 바깥의 셀이 일부분부터 바로 보이기 시작해야하는데, 셀이 다 화면에 들어오는 크기의 시점이 되어야 나타남.

**원인 분석**:
- collectionView.clipsToBounds = true (기본값)
- `layoutAttributesForElements(in rect:)`가 rect 내부 셀만 반환

**해결 방안**:
```swift
// 1. clipsToBounds 비활성화
collectionView.clipsToBounds = false

// 2. layoutAttributesForElements에서 rect 확장
func layoutAttributesForElements(in rect: CGRect) -> [UICollectionViewLayoutAttributes]? {
    let expandedRect = rect.insetBy(dx: -rect.width * 0.5, dy: -rect.height * 0.5)
    return cachedAttributes.filter { $0.frame.intersects(expandedRect) }
}
```

#### 문제 4: 10% 규칙 미적용

**증상**: 핀치줌 동작 실행 및 취소에 대한 10%/50% 규칙이 전혀 적용되지 않음. 70% 이상 진행해야 적용됨.

**원인 분석**:
- `computeStageProgress()` 계산 오류
- `stageBase()`와 `stageTarget()` 로직 문제

현재 로직:
```swift
func stageBase() -> GridColumnCount {
    switch direction {
    case .zoomIn:  // 확대 (열 수 감소)
        // 현재 위치보다 큰 기준점 중 가장 작은 것
        thresholds.filter({ CGFloat($0) >= current }).min()
    case .zoomOut:
        thresholds.filter({ CGFloat($0) <= current }).max()
    }
}
```

문제:
- 3열에서 축소 시작 → virtualColumns = 3.2
- stageBase = 3 (3 <= 3.2이므로)
- stageTarget = 5 (3.2 < 5이므로)
- progress = (3.2 - 3) / (5 - 3) = 0.1 ← 10% 맞음

하지만 실제로는 70% 필요하다면, `decideSnapTarget`의 조건이 문제일 수 있음.

**실제 원인 추정**:
- `didReverse` 플래그가 너무 쉽게 true가 됨
- 미세한 scale 변화에도 방향 전환으로 인식
- 결과적으로 50% 규칙이 적용되어 70%쯤 되어야 스냅

**해결 방안**:
```swift
// 방향 전환 감지에 threshold 추가
func updateDirectionState(virtualColumns: CGFloat, state: inout PinchZoomState) {
    guard let initialDirection = state.initialDirection else { return }

    let diff = virtualColumns - state.previousVirtualColumns
    // 미세한 변화는 무시 (threshold 추가)
    guard abs(diff) > 0.05 else { return }

    let current = currentDirection(virtualColumns: virtualColumns, state: state)

    if current != initialDirection && !state.didReverse {
        state.didReverse = true
    }
}
```

#### 문제 5: 취소 시 스크롤 점프

**증상**: 핀치줌 제스처 중 취소할 경우 스크롤이 확 위로 올라가버림 (100장 정도).

**원인 분석**: 문제 2와 동일 - 앵커 해결 오류

**해결 방안**:
- `.cancelled` 상태에서도 앵커 기반 복원 수행
- 취소 시 원래 열 수로 부드럽게 스냅

```swift
case .cancelled:
    // 취소 시 원래 열 수로 복원
    let originalColumns = GridColumnCount(rawValue: Int(state.baseColumns)) ?? currentGridColumnCount
    animateToTargetColumns(originalColumns, state: state) { ... }
```

#### 문제 6: 좌표계 혼용

**증상**: 핀치줌 시작 시 앵커가 마지막 사진쪽으로 갑자기 이동

**원인 분석 (추정)**:
현재 코드에서는 변환이 적용되어 있지만, 다른 곳에서 혼용 가능:
- `layoutAttributesForItem(at:).center` → content 좌표
- `collectionView.bounds` → bounds 좌표
- 이들을 직접 비교하면 오류

**해결 방안**:
- 모든 좌표 계산 위치에 주석으로 좌표계 명시
- 변환 함수 분리:
```swift
func boundsToContent(_ point: CGPoint) -> CGPoint {
    CGPoint(x: point.x + contentOffset.x, y: point.y + contentOffset.y)
}

func contentToBounds(_ point: CGPoint) -> CGPoint {
    CGPoint(x: point.x - contentOffset.x, y: point.y - contentOffset.y)
}
```

## 1. 핵심 개념

### 1.1 열 수 기준점
- **1열, 3열, 5열** 세 가지 기준점만 존재
- 핀치줌/회전 완료 시 반드시 이 중 하나로 snap

### 1.2 스케일 → 열 수 관계
```
scale 1.0 = 현재 열 수 유지
scale > 1.0 = 확대 → 열 수 감소 (3열 → 1열)
scale < 1.0 = 축소 → 열 수 증가 (3열 → 5열)
```

### 1.3 진행도 개념
```
0% = 현재 열 수 상태
100% = 목표 열 수 상태

예: 3열 → 1열 핀치
  0% = 3열 레이아웃
  50% = 중간 상태
  100% = 1열 레이아웃
```

### 1.4 앵커 셀
- **핀치**: 핀치 중심점에 있는 셀
- **회전**: 화면 중앙에 있는 셀
- 앵커 셀은 fadeIn 제외, 기존 이미지 유지
- **v6 추가**: 앵커는 X, Y 모두 기준점 역할

### 1.5 fadeIn
- **대상**: 기존에 visible이었던 셀 중 이미지 교체가 필요한 셀 (앵커 제외)
- **제외**: 새로 나타난 셀 (즉시 표시, fadeIn 금지)
- 현재 이미지 → 목표 레이아웃의 이미지로 부드럽게 전환
- 새 단계 시작 시 트리거
- **v6 수정**: isDegraded 조건 제거, 일단 표시 후 고해상도 교체

> **핵심 구분**: 축소 시 외곽에 새로 등장하는 셀은 "기존에 없던 셀"이므로
> fadeIn이 아니라 즉시 이미지가 표시되어야 함.

### 1.6 열 수 변경 규칙

#### 핀치줌
- 확대: 열 수 감소 (5 → 3 → 1)
- 축소: 열 수 증가 (1 → 3 → 5)

#### 회전
- 세로 → 가로: 1→3, 3→5, 5→5
- 가로 → 세로: 1→1, 3→3, 5→3

## 2. 요구사항

### 2.1 핵심 원칙

#### 금지 사항
- 셀 개별 이동 애니메이션 ❌
- 새로고침/끊김 ❌

#### 필수 사항
- 전체가 하나처럼 연속적인 모션 ✅
- fadeIn으로 부드러운 이미지 전환 ✅
- 핀치 진행도에 따라 실제 레이아웃이 점진적으로 변화 ✅
- **v6 추가**: 핀치 중심점이 X, Y 모두 고정 ✅

### 2.2 핀치줌

#### 핀치 시작 시
1. 앵커 셀 결정 (핀치 중심점의 셀)
2. **v6 추가**: 현재 contentOffset 저장
3. **방향 결정 후** 나머지 모든 셀에 목표 이미지 fadeIn 시작
4. 레이아웃이 점진적으로 변하기 시작

> **주의**: `.began` 시점에서는 `gesture.scale`이 항상 1.0이므로 방향을 알 수 없음.
> fadeIn은 `.changed`의 첫 프레임에서 방향이 결정된 후 트리거해야 함.

#### 핀치 중
1. 핀치 진행도에 따라 셀 크기/위치가 **실시간으로** 변함
2. fadeIn 진행 중 (또는 이미 완료)
3. **v6 수정**: 외곽에 새 셀들이 **즉시** 나타남 (clipsToBounds=false)
4. 앵커 셀: 기존 이미지 유지, **X/Y 위치 고정**

#### 다단계 핀치 (예: 5열 → 3열 → 1열)
- 각 기준점(1/3/5열) 통과 시 = 새 단계 시작
- 새 단계 시작 시 fadeIn 트리거 (다음 목표 이미지로)

```
[1단계 핀치] 3열 → 1열
  방향 결정 → fadeIn (1열 이미지로)

[2단계 핀치] 5열 → 3열 → 1열
  방향 결정 → fadeIn (3열 이미지로)
  3열 기준점 통과 → fadeIn (1열 이미지로)
```

#### 손을 뗄 때
1. 핀치 100% 완료 상태 → 그대로 유지 (추가 애니메이션 없음)
2. 핀치 70% 상태 → 나머지 30%가 애니메이션으로 자연스럽게 완료
3. **끊김 없이** 현재 모션에서 최종 레이아웃으로 이어짐

#### 취소 시 (v6 추가)
- 원래 열 수로 부드럽게 스냅
- 스크롤 위치 유지 (점프 방지)

### 2.3 회전

#### 핀치와 동일한 원리

| | 핀치 | 회전 |
|---|------|------|
| 앵커 | 핀치 중심점 셀 | 화면 중앙 셀 |
| fadeIn 시작 | 방향 결정 시 | 회전 시작 시 |
| 레이아웃 변화 | 핀치 진행도에 따라 점진적 | 회전 애니메이션과 동기화 |
| 완료 시 | 끊김 없이 snap | 끊김 없이 완료 |

#### 회전 시작 시
1. 앵커 셀 결정 (화면 중앙 셀)
2. **즉시** 나머지 모든 셀에 목표 이미지 fadeIn 시작
3. 회전 애니메이션과 함께 레이아웃이 자연스럽게 변화

#### 회전 완료 시
- 끊김/새로고침 없이 최종 레이아웃 상태
- 앵커 셀이 화면에 유지됨

### 2.4 기준점 판단 로직 (손을 뗄 때)

#### Case 1: 단방향 핀치 (10% 규칙)

한 방향으로만 핀치한 경우:

```
3열에서 축소 10% 이상 진행 → 손을 뗌
→ 의도 명확 → 5열로 완료
```

**10% 이상 진행하면 무조건 그 방향으로 진행**

#### Case 2: 방향 전환 (50% 규칙)

핀치 도중 방향이 바뀐 경우:

```
3열에서 축소 70% → 다시 확대 40%
→ 최종 위치: 30% (축소 방향)
→ 50% 미만 → 3열 유지

3열에서 축소 80% → 다시 확대 20%
→ 최종 위치: 60% (축소 방향)
→ 50% 이상 → 5열로 완료
```

**방향 전환 시 최종 위치 기준 50%로 판단**

#### v6 수정: 방향 전환 감지 threshold

**미세한 scale 변화는 방향 전환으로 간주하지 않음**:
```
virtualColumns 변화량 < 0.05 → 무시
virtualColumns 변화량 >= 0.05 → 방향 전환 감지
```

#### 정리

| 상황 | 기준 | 결과 |
|-----|------|------|
| 단방향 10% 이상 | 10% | 그 방향으로 진행 |
| 방향 전환 | 50% | 최종 위치 기준 판단 |

## 3. 구현 방안

### 3.1 핵심 전략 (결론)
**연속 열 수(실수) 기반 커스텀 `UICollectionViewLayout`**으로 핀치 진행도에 따라
레이아웃을 실시간 재계산하고, 기준점 통과 시 fadeIn을 트리거한다.

핵심 이유:
- "점진적 레이아웃 변화" 요구를 만족하려면 열 수가 실수로 변해야 함
- 셀 개별 이동 애니메이션 없이 레이아웃 재계산만으로 연속 모션 제공 가능
- 다단계(5→3→1) 전환을 끊김 없이 처리 가능

---

### 3.2 아키텍처 구성

1) **ContinuousGridLayout (커스텀 레이아웃)**
- `virtualColumns: CGFloat`(예: 3.0 → 2.6 → 1.2)
- 값 변경 시 `invalidateLayout()`로 즉시 반영
- **v6 추가**: `layoutAttributesForElements`에서 rect 확장

2) **PinchController (제스처/상태 관리)**
- 핀치 진행도 → `virtualColumns` 변환
- 기준점 통과 감지 → fadeIn 트리거
- 앵커 셀 고정용 contentOffset 보정
- **v6 추가**: X축 보정 (contentInset.left)
- **v6 추가**: 방향 전환 threshold

3) **Cell FadeIn 레이어**
- 앵커 셀 제외 visible 셀에 목표 이미지 fadeIn
- 비동기 로딩 경쟁 조건 방지 (token/assetID 검증)
- **v6 수정**: isDegraded 조건 제거

---

### 3.3 레이아웃 계산 원리

**핵심 아이디어**: 열 수를 정수로 끊지 말고 실수로 유지.

```swift
columnWidth = (bounds.width - spacing * (virtualColumns - 1)) / virtualColumns
```

- `virtualColumns`가 3.0 → 2.4 → 1.8처럼 연속적으로 변하며
  셀 크기/간격이 매 프레임 부드럽게 변화한다.
- 셀 배치는 "정수 열 수"가 아니라 **실수 열 수로 계산된 폭**에 맞춰
  `x = colIndex * (columnWidth + spacing)` 방식으로 배치한다.

---

### 3.4 앵커 셀 고정 (contentOffset 보정) - v6 수정

**목표**: 핀치 중심점 셀이 화면에서 X/Y 모두 고정된 것처럼 보이도록 유지.

```swift
// Y축: contentOffset으로 보정 (기존)
contentOffset.y = anchorCenterInContent.y - anchorPointInView.y

// X축: contentInset.left로 보정 (v6 추가)
// - collectionView가 가로 스크롤 없으므로 contentOffset.x는 항상 0
// - 대신 contentInset.left를 조정하여 전체 그리드를 좌우로 이동
let xShift = anchorPointInView.x - anchorCenterInContent.x
collectionView.contentInset.left = max(0, xShift)
```

**주의**: X축 보정은 contentInset을 사용하므로 레이아웃 계산에 영향 없음

---

### 3.5 기준점 통과 감지 (다단계)

빠른 핀치로 5→1 이동 시 기준점 5,3,1 모두 통과 가능.
따라서 **통과한 기준점을 배열로 반환**하고 방향 순서대로 처리한다.

- 반환된 순서대로 fadeIn 트리거
- 동일 단계 중복 방지 (`currentTargetColumns` 체크)

---

### 3.6 fadeIn 트리거 규칙 - v6 수정

**대상 구분**:
- **fadeIn 대상**: 트리거 시점에 이미 visible이었던 셀 (앵커 제외)
- **즉시 표시**: 핀치 중 새로 visible 되는 셀 (fadeIn 금지)

**트리거 시점**:
- **방향 결정 시**: 첫 목표 열 수 이미지로 fadeIn 시작 (`.changed` 첫 프레임)
- **기준점 통과 시**: 다음 단계 목표 이미지로 fadeIn 재시작

**안전 규칙**:
- **앵커 셀 제외**
- **비동기 로딩 검증**: 요청 시점의 열 수/assetID/token 확인 후 적용
- **v6 수정**: isDegraded 조건 제거 - 저해상도라도 일단 표시

---

### 3.7 손을 뗄 때 스냅 로직 (10% / 50% 규칙) - v6 수정

**단방향**: 10% 이상 진행 → 해당 방향으로 완료
**방향 전환**: 최종 위치 50% 이상 → 해당 방향으로 완료

**v6 추가**: 방향 전환 감지 threshold (0.05)
- 미세한 scale 변화는 방향 전환으로 간주하지 않음
- 이를 통해 10% 규칙이 정상 작동

완료 애니메이션은 `virtualColumns`를 목표 기준점까지 보간하며,
동일한 레이아웃 계산을 유지해 끊김을 방지한다.

### 3.10 클리핑 처리 (v6 추가)

```swift
// BaseGridViewController에서
collectionView.clipsToBounds = false

// ContinuousGridLayout에서
override func layoutAttributesForElements(in rect: CGRect) -> [UICollectionViewLayoutAttributes]? {
    // rect를 50% 확장하여 화면 바깥 셀도 반환
    let expandedRect = rect.insetBy(dx: -rect.width * 0.5, dy: -rect.height * 0.5)
    return cachedAttributes.filter { $0.frame.intersects(expandedRect) }
}
```

### 3.11 좌표계 변환 함수 (v6 추가)

```swift
/// bounds 좌표 → content 좌표
func boundsToContent(_ point: CGPoint) -> CGPoint {
    CGPoint(
        x: point.x + collectionView.contentOffset.x,
        y: point.y + collectionView.contentOffset.y
    )
}

/// content 좌표 → bounds 좌표
func contentToBounds(_ point: CGPoint) -> CGPoint {
    CGPoint(
        x: point.x - collectionView.contentOffset.x,
        y: point.y - collectionView.contentOffset.y
    )
}
```
