**Findings**
1. `[High]` SDK 호출 시그니처가 현재 문서대로면 일부 컴파일/호출 불일치 위험이 큽니다.  
`docs/db/260212db-Archi.md:63`, `docs/db/260212db-Archi.md:65`, `docs/db/260212db-Archi.md:129`  
`TelemetryDeck.signal`의 `parameters`는 공식 Swift 가이드에서 `[String: CustomStringConvertible]`로 안내되고, Duration API 예시는 `additionalPayload` 라벨을 사용합니다. 문서의 `[String: String]` 고정/`parameters:` 라벨은 최신 가이드와 어긋납니다.

2. `[High]` “SDK 세션 5분 타임아웃” 전제가 근거 불충분하며 설계 판단에 직접 영향을 줍니다.  
`docs/db/260212db-Archi.md:89`, `docs/db/260212db-Archi.md:96`, `docs/db/260212db-Archi.md:152`  
공식 문서/README 계열에서는 “백그라운드에서 돌아오면 session identifier 갱신” 설명이 보이고, 5분 규칙은 확인 근거가 약합니다. 이 전제가 틀리면 “SDK 세션 ≠ 우리 세션” 결론과 요약 이벤트 설계가 달라질 수 있습니다.

3. `[Medium]` 자동 메타데이터 키/타입 정리가 최신 문서 체계와 다릅니다.  
`docs/db/260212db-Archi.md:103`, `docs/db/260212db-Archi.md:116`  
공식 `default parameters` 문서는 `TelemetryDeck.AppInfo.version` 같은 네임스페이스 키(프리픽스 configurable)를 사용합니다. 현재 문서의 비네임스페이스 키 표기는 대시보드 쿼리 시 혼선을 만들 수 있습니다.

4. `[Medium]` 래퍼 방향은 타당하지만, AppCore 주입 포인트가 `optional`이면 무음 데이터 유실 가능성이 큽니다.  
`docs/db/260212db-Archi.md:223`, `docs/db/260212db-Archi.md:225`, `docs/db/260212db-Archi.md:254`  
`Analytics.reporter?`가 누락되면 내부 오류 이벤트가 조용히 사라집니다. `NoOp 기본 구현` + 주입 검증 로그가 필요합니다. 또한 `result/screen/category/item`을 `String`으로 열어두면 cardinality 폭증/오타 리스크가 있습니다(열거형 권장).

5. `[Medium]` 섹션 4/5/6이 비어 있어 “구현 착수” 기준으로는 아직 불충분합니다.  
`docs/db/260212db-Archi.md:298`, `docs/db/260212db-Archi.md:304`, `docs/db/260212db-Archi.md:310`  
세션 종료 시점 플러시(`handleSessionEnd`)만 정의되어 있고, 세션 상태머신/이벤트 수집기 모델/파일 배치가 없어 실제 코드 단계에서 결정 공백이 큽니다.

**질문하신 4가지 포인트 결론**
1. SDK API 정확성: 현재 문서는 일부 핵심 시그니처/키 체계가 부정확하거나 근거가 약합니다. 먼저 API 표를 정정해야 합니다.  
2. 래퍼 구조 적절성: 방향성(의존성 격리, AppCore 경량 프로토콜)은 좋습니다. 다만 `optional reporter`, `String 파라미터 남용`은 수정 권장입니다.  
3. 다음 단계 진행 가능성: “다음 설계 단계(세션/수집기 설계)”로는 진행 가능, “구현”으로 바로 가기엔 아직 불가입니다.  
4. 누락 고려사항: 옵트아웃 저장/즉시 반영, 이벤트 스키마 버전(`schema_version`), 백그라운드 전송 보장(Background Task), 테스트 전략(단위/통합), cardinality 가드가 빠져 있습니다.

**권장 보완 순서**
1. SDK API 정정 섹션 먼저 확정(시그니처/키/세션 규칙 근거 링크 포함).  
2. 세션 상태머신 1페이지 정의(`foreground`, `background`, `flush`, `reset` 원자성).  
3. 이벤트 수집기 모델(카운터 저장소, enum 기반 파라미터, 직렬화 규칙) 확정.  
4. 파일 구조 확정 후 구현 착수.
