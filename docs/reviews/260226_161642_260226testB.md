**검토 결과**

핵심 결론: 문서의 Phase 설계와 카운터 합산 논리는 전반적으로 잘 맞지만, 현재 계획 그대로는 "안정적으로 E2E 검증 성공"까지 보장하기엔 부족합니다. 특히 검증 스크립트의 이벤트 식별 방식과 XCUITest 안정성에서 실패/오검출 가능성이 큽니다.

**주요 Findings (중요도 순)**
1. `--manual` 모드가 실제로는 자동 검증도 함께 실행하는 구조라 문서 설명과 동작이 충돌합니다.
- 문서 사용법은 `--manual`이 수동 cleanup 검증용인 것처럼 설명하지만 `verify-test-b.sh` 본문은 자동 항목 검증(`session.*`, `similar.*`)을 먼저 모두 수행한 뒤에만 manual 분기를 탑니다.
- 근거: `docs/db/260226testB.md:479`, `docs/db/260226testB.md:560`, `docs/db/260226testB.md:591`
- 영향: 수동 테스트만 확인하려고 실행해도 자동 기대값 실패로 전체 실패가 날 수 있습니다.
- 보완: `if [ "${1:-}" = "--manual" ]`를 초반에 분기해서 cleanup 이벤트만 검사하도록 분리.

2. Supabase 검증이 "이번 테스트 실행분"을 식별하지 못해 오검출 위험이 큽니다.
- 현재 쿼리는 `created_at >= 최근 2분` + `event_name` + 최신 1건(`limit=1`)만 봅니다.
- 근거: `docs/db/260226testB.md:485`, `docs/db/260226testB.md:496`
- 영향: 직전 실행/수동 테스트/다른 백그라운드 진입의 동일 이벤트를 잘못 집어올 수 있습니다. 특히 manual 검증과 auto 검증을 같은 스크립트가 다루는 구조에서 더 취약합니다.
- 보완: 테스트 시작 시각을 `run-test-b.sh`에서 캡처해 verify에 전달하고, 가능하면 `test_run_id`(launch arg/env로 앱이 이벤트 params에 포함)까지 넣어 필터링.

3. XCUITest `setUp()`의 `app.tap()`이 권한 팝업이 없을 때 실제 UI를 오작동시킬 가능성이 큽니다.
- 문서 사전조건은 권한/코치마크가 이미 완료된 상태를 전제합니다.
- 근거: `docs/db/260226testB.md:66`, `docs/db/260226testB.md:263`
- 영향: 권한 팝업이 없으면 중앙 탭이 사진 셀 탭으로 해석되어 테스트 시작 상태가 깨질 수 있습니다(카운터 기대값도 틀어짐).
- 보완: `app.tap()` 제거 또는 alert 존재 시에만 트리거. 시스템 alert는 `springboard.alerts` 기반으로 명시적으로 처리.

4. UI 요소 선택 전략이 아직 불안정합니다 (`firstMatch`, 전역 swipe).
- 문서는 버튼 identifier만 추가하고, 나머지는 `collectionViews.firstMatch`/`app.swipeLeft()`에 의존합니다.
- 근거: `docs/db/260226testB.md:213`, `docs/db/260226testB.md:302`, `docs/db/260226testB.md:334`, `docs/db/260226testB.md:363`
- 영향: 비교 화면/뷰어/그리드가 겹치는 시점에 잘못된 컬렉션/뷰에 제스처가 들어가 flaky 해집니다.
- 보완: 비교 화면 컬렉션, 보관함 그리드, 삭제대기함 그리드, 뷰어 페이지 영역에도 `accessibilityIdentifier` 추가.

5. 고정 `sleep` 중심의 동기화는 실기기에서 flaky 가능성이 높습니다.
- 여러 phase에서 `Thread.sleep`과 shell `sleep 8`에 의존합니다.
- 근거: `docs/db/260226testB.md:275`, `docs/db/260226testB.md:304`, `docs/db/260226testB.md:358`, `docs/db/260226testB.md:413`, `docs/db/260226testB.md:462`
- 영향: 애니메이션/시스템 삭제 확인 팝업/비동기 삭제 완료/네트워크 전송 속도 변화에 따라 실패 가능.
- 보완: `waitForExistence`, predicate expectation, 셀 개수 변화/버튼 상태 변화 기반 대기, verify 스크립트는 polling 재시도(예: 30초 timeout)로 변경.

6. cleanup 이벤트 설명에 개념 오류가 있습니다 (flush 필요 여부).
- 문서는 수동 cleanup 후 Home 버튼으로 background 보내야 flush된다고 설명하지만, cleanup 이벤트는 현재 구현에서 즉시 전송(`sendEvent`) 경로입니다.
- 근거 문서: `docs/db/260226testB.md:187`, `docs/db/260226testB.md:665`
- 근거 코드: `PickPhoto/PickPhoto/Shared/Analytics/AnalyticsService+Cleanup.swift:39`, `PickPhoto/PickPhoto/Shared/Analytics/AnalyticsService+Cleanup.swift:49`, `PickPhoto/PickPhoto/Shared/Analytics/AnalyticsService.swift:231`
- 영향: 절차 설명이 구현과 어긋나 혼동 유발. (즉시 전송이라도 백그라운드로 보내면 전송 완료 시간을 벌 수는 있음)
- 보완: "cleanup 이벤트는 즉시 전송, Home은 전송 완료 안정성 확보용(선택적/권장)"으로 표현 수정.
