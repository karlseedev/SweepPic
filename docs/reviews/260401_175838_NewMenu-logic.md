1. 현재 설계대로면 분석 완료 후 아직 처리하지 않은 그룹이 그대로 유실됩니다. 문서는 “분석 완료 후 위치만 저장, 그룹 데이터는 저장 안 함”으로 정의했는데, 이 경우 사용자가 목록에서 일부만 보고 나가도 다음 “이어서 정리”는 저장된 `lastAssetDate` 이후부터 다시 시작해서, 방금 찾아둔 나머지 그룹들을 다시 보여줄 방법이 없습니다. 이 기능의 목표가 “찾아둔 인물사진 그룹을 차례로 비교·정리”라면, 그룹 스냅샷과 처리 상태를 세션에 같이 저장하거나, 최소한 현재 목록을 모두 소진하기 전에는 커서를 전진시키면 안 됩니다.

2. `deletedAssetsByGroup`만 메모리에 들고 있는 구조로는 “dim 그룹 재진입 후 복원”을 안정적으로 구현할 수 없습니다. 현재 비교 화면은 삭제 시점에 실제 `SimilarityCache` 그룹 멤버를 제거하고, 그룹이 3장 미만이 되거나 유효 슬롯이 사라지면 그룹 자체를 무효화합니다. 그러면 문서가 기대하는 “dim 그룹 재진입” 시점에는 원본 그룹 구성이 이미 없어질 수 있습니다. 복원/재진입을 지원하려면 live cache와 별개로 FaceScan 전용 그룹 스냅샷(원본 멤버, 얼굴 정보, valid person list)을 고정해서 들고 있어야 합니다.

3. 삭제대기함 비어있음 체크를 없애면서도 “복원 diff”를 넣으려면, 어떤 사진이 이 세션에서 삭제된 것인지 provenance가 필요합니다. 지금 `TrashStore`는 단순히 `Set<String>`만 저장하므로, 이미 다른 흐름(스와이프 삭제, 뷰어 삭제, 자동정리)에서 들어 있던 사진과 FaceScan에서 방금 넣은 사진을 구분할 수 없습니다. 이 상태에서 문서대로 “체크 해제 = 삭제대기함에서 복원”을 하면, 사용자가 다른 이유로 trash에 넣어둔 사진도 FaceScan 재진입에서 의도치 않게 복원될 수 있습니다. 최소한 “이 세션이 넣은 assetID 집합”을 별도로 저장해야 합니다.

**결론**

현재 문서는 아이디어와 UX 스케치는 괜찮지만, 목표를 달성하기엔 핵심 상태 모델이 부족합니다. 특히 `세션 저장 범위`, `그룹 스냅샷 보존`, `삭제 provenance`, `continue cursor 정밀도`, `FaceComparisonVC 계약 변경`, `전용 스캔 서비스 정의`가 빠져 있어서 그대로 구현하면 재진입/복원/이어서 정리에서 틀어질 가능성이 높습니다.

원하면 다음 단계로, 이 문서를 바로 수정할 수 있게 “필수 추가 섹션 목록” 형태로 재작성안까지 정리해드리겠습니다.
