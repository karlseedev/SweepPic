웹검색은 필요 없었습니다. 로컬 문서들만으로 매핑 정확도 검증이 가능합니다.

**리뷰 결과 (핵심 Findings 우선)**

1. `hybrid.md` 섹션 매핑이 “누락 없이”는 아닙니다.  
`docs/db/260217db-docu-write.md:235`의 `hybrid.md` 매핑표는 `Phase 1~5`와 `검토 기록`만 다루는데, 실제 `hybrid.md`에는 `Context`, `구현 전 체크리스트`, `파일 변경 요약`도 top-level 섹션으로 존재합니다 (`docs/db/260217db-hybrid.md:3`, `docs/db/260217db-hybrid.md:11`, `docs/db/260217db-hybrid.md:633`).  
또한 Phase 3 내부의 `Credentials`, `photo_bucket 처리`, `볼륨 추정`은 목차/상단 설명에는 반영되지만 매핑표에는 명시적으로 분해되어 있지 않습니다.

2. `Archi.md §7`의 목적지 매핑이 표와 목차에서 서로 다릅니다.  
매핑표에서는 `Archi §7` 전체를 `API §1`로 보냅니다 (`docs/db/260217db-docu-write.md:218`). 하지만 같은 문서의 API 목차는 `Claude 분석 워크플로우`를 `API §4`로 분리하고 (`docs/db/260217db-docu-write.md:176`), 주의사항도 `API §5`로 분리합니다 (`docs/db/260217db-docu-write.md:180`). 실제 `Archi §7`에는 워크플로우/비용·접근 제한이 포함됩니다 (`docs/db/260212db-Archi.md:1416`, `docs/db/260212db-Archi.md:1466`).  
즉, 현재 표의 `API §1` 단일 매핑은 부정확합니다.

3. API 목차 주석과 섹션 매핑표가 충돌합니다 (`API-impl.md`).  
API 목차의 TD 스크립트 섹션 주석은 `API-impl 전체` 흡수를 말합니다 (`docs/db/260217db-docu-write.md:166`). 그런데 아래 매핑표에서는 `API-impl`의 `§1, §6`를 archive로 분류합니다 (`docs/db/260217db-docu-write.md:230`).  
이건 실행자 입장에서 혼동을 일으키는 명백한 문서 내부 불일치입니다.

4. `DBplan1` 미흡수 섹션 설명이 불완전합니다.  
`DBplan1 처리` 설명에서는 미흡수로 `§1, §8, §9`만 언급합니다 (`docs/db/260217db-docu-write.md:23`). 하지만 실제 `DBplan1`에는 `§2 경쟁사 분석`과 `참고 자료`도 존재하며 (`docs/db/260211DBplan1.md:82`, `docs/db/260211DBplan1.md:737`), 신규 기술 문서 범위 밖이라면 함께 명시하는 편이 맞습니다.

5. 문서 개수 표현이 일관되지 않아 작업 단계에서 혼선 가능성이 있습니다.  
목표에서는 `기존 7개 문서`라고 하고 (`docs/db/260217db-docu-write.md:35`), 파일명 섹션에서는 archive 대상을 `8개`로 적으며 이 문서를 포함합니다 (`docs/db/260217db-docu-write.md:65`). 그런데 작업 순서 Step 4에서 다시 `기존 8개` 이동 후 “이 문서도 archive 이동”을 별도로 적어 중복 해석 여지가 생깁니다 (`docs/db/260217db-docu-write.md:274`, `docs/db/260217db-docu-write.md:275`).

6. 매핑표의 `줄` 수치는 정확값처럼 보이지만 실제 섹션 길이와 다수 불일치합니다.  
예: `Spec`의 `1. 솔루션 선정`은 `54줄`로 표기되어 있으나 (`docs/db/260217db-docu-write.md:198`), 실제 top-level 시작/다음 섹션 시작 기준은 `docs/db/260212db-Spec.md:24`와 `docs/db/260212db-Spec.md:82`입니다.  
치명적 문제는 아니지만 “정확한 매핑 검증” 목적이라면 `대략` 표기로 바꾸거나 계산 기준을 명시해야 신뢰도가 올라갑니다.

**총평**

계획의 방향성과 3문서 구조(Spec/Archi/API) 자체는 좋고, Supabase 하이브리드 완료 상태에서 문서 정리 목표를 달성하는 데 전반적으로 충분합니다.  
다만 현재 버전은 “매핑표의 완전성/정확성”에서 몇 군데 빠진 항목과 내부 불일치가 있어, 이 상태로 작업 시작하면 작성 중 재판단이 생길 가능성이 큽니다.
