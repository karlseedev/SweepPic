**총평**
계획 방향은 맞지만, 현재 문서만으로는 목표(앱 백그라운드 배치 전송 + 안정 저장 + 저비용 운영)를 **충분히 보장하기 어렵습니다**. 아래 항목은 우선 수정이 필요합니다.

**핵심 이슈 (심각도 순)**
1. **[Critical] 배치 insert 실패 가능성(키 불일치)**
`docs/db/260217db-hybrid.md:112`의 배열 bulk insert는 맞지만, PostgREST는 bulk JSON에서 객체 키가 동일해야 안정적입니다. Swift `Codable`의 optional은 `nil`일 때 키를 생략할 수 있어 실패할 수 있습니다.
수정: `sendBatch()`에서 모든 row에 동일 키 강제(없으면 `null`) 또는 `?columns=...` + `Prefer: missing=default` 전략 명시.

2. **[Critical] 백그라운드 전송 보장 설계가 불충분**
`docs/db/260217db-hybrid.md:119`~`126`은 “확인 후 필요시 추가” 수준이라 실제로는 suspend 전에 요청이 끊길 수 있습니다.
수정: `beginBackgroundTask` 시작/종료/만료 핸들러, `flush` completion 대기, timeout 기준을 계획에 명시해야 합니다.

3. **[High] RLS 정책이 너무 개방적(비용/오염 리스크)**
`docs/db/260217db-hybrid.md:42`의 `WITH CHECK (true)`는 anon이 임의 payload를 무제한 삽입할 수 있어 무료 티어 소진/오염 위험이 큽니다.
수정: 이벤트명 화이트리스트, payload 크기 제한, pre-request 기반 rate limit(또는 앱단 서명/검증) 추가 권장.

4. **[High] 키 전략이 레거시 중심**
`docs/db/260217db-hybrid.md:102`, `165`는 `anon/service_role` 중심인데, Supabase는 `publishable/secret` 키 전환을 권장합니다.
수정: 앱은 publishable(또는 기존 anon 유지 시 전환 계획 명시), 스크립트는 secret key 우선으로 문서 업데이트.

5. **[Medium] 무료 티어 pause 설명의 근거 보강 필요**
`docs/db/260217db-hybrid.md:17`의 “7일 무호출 시 자동 pause”는 제가 확인한 공식 docs에서 문장 그대로는 찾지 못했습니다(다만 Free는 pause되고 Paid는 pause되지 않음은 확인).
수정: 문구를 “Free 프로젝트는 자동 pause될 수 있음”으로 완화하고, 실제 값은 가격/대시보드 기준으로 주기적 확인 명시.

6. **[Medium] pg_cron 안내 경로가 최신 표현과 다름**
`docs/db/260217db-hybrid.md:92`는 Extensions 활성화 경로인데, 최신 문서는 Supabase Cron(Integrations)을 전면에 둡니다(내부는 pg_cron).
수정: “Integrations > Cron(내부 pg_cron)”로 갱신.  
추론: 문서상 유료 전용 제한은 명시되지 않았지만, **프로젝트가 pause 상태면 cron job 실행도 멈춘다**는 운영 가정이 필요합니다.

7. **[Low] `.env.example` 상태 기술 불일치**
`docs/db/260217db-hybrid.md:200`은 “현재 미존재”라고 했지만 실제 `scripts/analytics/.env.example:1` 파일이 이미 있습니다.
수정: 신규 생성이 아니라 확장/이동 전략으로 정정.

**요청하신 항목 검증 요약**
- 무료 티어 제한: 공식 문서에서 `DB 500MB/project`, `Egress 5GB`, `2 free projects`, Free 프로젝트 pause 존재 확인.
- PostgREST API: `POST /rest/v1/<table>` 배열 bulk insert 가능 확인.
- RLS: `anon INSERT` 정책 문법/방향은 맞음. 다만 현재 정책 강도가 낮음.
- pg_cron: Supabase Cron 문서에서 pg_cron 기반임을 확인. 사용 가능성은 높지만 문서에 플랜별 명시 제한은 찾지 못함(추론).
