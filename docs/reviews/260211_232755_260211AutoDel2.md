결론부터 말하면, **방향성은 좋지만 현재 문서만으로는 목표 달성에 부족**합니다.  
특히 `분석`과 `휴지통 이동`을 분리하는 서비스/데이터 모델 설계가 빠져 있어 구현 단계에서 막힐 가능성이 큽니다.

**웹 검증으로 확인한 사실 보정 (2026-02-11 기준)**
- `Apple Photos (iOS 18.1)` 설명은 수정 필요합니다. iOS 18.1의 Photos는 `Photos search`, `Clean Up(사진 내 방해 요소 제거)`, `Memory movies` 중심이며, 문서에 적힌 “정리 민감도 조절”은 공식 근거를 확인하지 못했습니다.
- `Gemini Photos`는 현재 독립 제품보다는 **CleanMyPhone으로 전환된 맥락**이 맞습니다.
- `Google Photos`는 카테고리/스택 기반 정리가 맞습니다(사진 스택, 스크린샷/문서 자동 분류). 다만 “스와이프 리뷰”가 핵심 UX라는 공식 근거는 약합니다.
- `Slidebox`의 “한 장씩 스와이프 정리”는 맞습니다.
- `CleanMyPhone`도 카테고리/그룹 기반 + 사용자 확인 흐름이 맞습니다.

**계획의 핵심 부족점**
- `서비스 경계 미정의`: 현재 코드는 정리 완료 시 즉시 이동합니다. `GridViewController+Cleanup.swift:174`, `CleanupService.swift:389`, `CleanupService.swift:418`.  
  문서에 `analyzeOnly`와 `applyCleanup`를 분리한 API/상태 정의가 필요합니다.
- `데이터 모델 충돌`: 지금은 `found == trashed` 전제가 강합니다. `CleanupSession.swift:302`.  
  새 UX는 `found`, `selected`, `excluded`, `trashed`를 분리해야 합니다.
- `수량 제한 충돌`: 현재 상수는 `maxFoundCount = 50`입니다. `CleanupConstants.swift:26`.  
  문서 예시(47→83→124)와 충돌하므로, 제한 정책(미리보기에서는 124 허용? 스캔 제한은?)을 명시해야 합니다.
- `구현 난이도 과소평가`: 문서에서 방향 3을 “낮은 복잡도”로 봤지만, 기존 그리드는 `asset-only` 구조라 배너 인라인 삽입이 단순하지 않습니다. `BaseGridViewController.swift:623`.
- `모드 체계 정합성`: 실제 프로덕션 모드는 아직 `precision/recall` 2개 중심이고 호출도 고정 precision입니다. `JudgmentMode.swift:22`, `GridViewController+Cleanup.swift:176`.  
  3단계 UX를 적용하려면 모드 enum/세션/테스트 전부 확장 계획이 필요합니다.
- `신뢰 장치 부족`: “왜 잡혔는지” 이유(블러/노출/미적점수/유틸리티 제외 여부) 표시가 없으면 단계 확장 신뢰가 떨어집니다.
- `검증 계획 부족`: 성공 기준(KPI), 실패 기준, A/B 또는 사용자 테스트 기준이 없습니다.

**문서에 바로 추가하면 좋은 최소 항목**
1. `상태 머신`: `Scanning -> Preview(stage1) -> Expand(stage2/3) -> Confirm -> Apply`.
2. `데이터 모델`: `Candidate(assetID, stage, reasons, score)` + `PreviewSession(selectedIDs, excludedIDs)`.
3. `API 분리`: `startAnalysis()`와 `applyCleanup(selection)` 분리.
4. `제한 정책`: 50/2000 제한이 미리보기와 적용 단계에서 각각 어떻게 동작하는지.
5. `버전 정책`: iOS 16~17에서 단계 버튼을 숨길지/동일 결과임을 안내할지.
6. `접근성`: 자동 스크롤 시 VoiceOver/Reduce Motion 대응.
7. `테스트`: 단조성(light ⊂ standard ⊂ deep), 인덱스-에셋 매핑, 취소/재시작, 선택 제외 반영.
