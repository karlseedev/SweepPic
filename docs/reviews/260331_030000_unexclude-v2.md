**Findings**
- 중간: 계획서의 핵심 전제는 조금 과장돼 있습니다. `[zippy-doodling-otter.md](/Users/karl/.claude/plans/zippy-doodling-otter.md#L7)` 에서는 `isTrashed=true`면 `PhotoCell` 메서드들이 “올바르게 동작”한다고 적었지만, 실제로는 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L999)` 와 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L1037)` 이 `isTrashed`만 보고 마룬+아이콘 상태로 복귀합니다. 즉 이 전략은 맞는 방향이지만, `cancelDimmedAnimation`/`clearDimmed`가 호출되는 모든 경로에서 녹색 재적용이 빠지면 바로 회귀합니다. 계획서에는 이 점을 “기본적으로 해결됐다”가 아니라 “보완 코드가 반드시 필요한 제약”으로 적는 게 맞습니다.
- 중간: 다중 해제의 검증 기준 하나가 사실과 다릅니다. `[zippy-doodling-otter.md](/Users/karl/.claude/plans/zippy-doodling-otter.md#L196)` 의 “다중 해제 → 모두 fade out”은 현재 제안된 흐름과 맞지 않습니다. 계획대로면 비-커튼 셀은 드래그 중에 이미 `setRestoredPreview()`로 `alpha = 0` 상태가 되고, confirm 시 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L930)` 에서 다시 `alpha = 0`을 주기 때문에 실제로 페이드가 보일 셀은 많아야 현재 커튼 셀뿐입니다. 이건 구현 오류는 아니지만 acceptance criteria가 잘못 적혀 있습니다.
- 낮음: 앵커 셀을 `enterMultiSwipeMode`에서 즉시 `setRestoredPreview()`로 보내는 건 기능상 가능하지만 UX 트레이드오프가 있습니다. 기존 베이스 구현은 `[BaseMultiSwipeDelete.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/BaseMultiSwipeDelete.swift#L52)` 처럼 앵커도 커튼을 이어서 애니메이션하고, 제안안은 `[zippy-doodling-otter.md](/Users/karl/.claude/plans/zippy-doodling-otter.md#L96)` 처럼 즉시 복원 프리뷰로 스냅시킵니다. 첫 행 CA 충돌 회피용으로는 이해되지만, “문제 없음”으로 단정할 건 아니고 시각적 점프를 수용하는지 별도 확인이 필요합니다.

**질문별 확인**
- 1) `cancelDimmedAnimation` completion에서 마룬 뒤 바로 green 재적용하면, 코드상으로는 같은 completion 스택 안에서 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L999)` 의 마룬 리셋 직후 다시 덮어쓰게 되므로 보통은 마룬이 렌더되지 않을 가능성이 높습니다. 다만 이건 UIKit 렌더 타이밍에 대한 추론이지 API 보장은 아닙니다. 계획서에는 “렌더되지 않는다”보다 “동일 run loop에서 덮어써 플래시는 낮을 것으로 예상, 실기기 검증 필요”가 더 정확합니다.
- 2) `confirmDimmedAnimation(toTrashed:false)` 중 마스크가 남아 있어도, `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L930)` 에서 오버레이 전체 `alpha`를 0으로 내리므로 시각적으로는 안 보입니다. 이 가정은 맞습니다. 다만 `confirm/cancel`이 `dimmedOverlayView.layer.mask = nil`을 직접 안 하는 건 코드 청결상 찜찜한 부분입니다.
- 3) 다중 스와이프 해제에서 앵커를 즉시 `setRestoredPreview()` 처리하는 건 동작 자체는 성립합니다. 대신 단일 스와이프에서 보던 “부분 커튼이 이어지다가 다중으로 확장”되는 연속감은 사라집니다. 그래서 기능 버그라기보다 UX 리스크입니다.
- 4) `exclude → un-exclude confirm → re-exclude → cancel` 사이클의 `isTrashed` 관리는 제안안 기준으로 맞습니다. 첫 exclude confirm에서 `true`, un-exclude confirm에서 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L946)` 로 `false`, 그 뒤 re-exclude를 취소하면 `[PhotoCell.swift](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/Grid/PhotoCell.swift#L980)` 경로가 현재 `isTrashed == false`를 기준으로 복귀하므로 최종 상태는 “딤드 없음”이 됩니다. 이 사이클 자체의 상태 머신 모순은 보이지 않았습니다.

결론적으로, 핵심 전략 자체는 목표 달성 가능성이 높습니다. 다만 계획서는 `isTrashed=true`가 만능 해결처럼 보이게 쓰여 있고, 다중 해제의 시각적 결과를 일부 잘못 기술하고 있습니다. 그 두 점과 앵커 셀 UX 트레이드오프를 명시하면 더 안전한 계획이 됩니다.
