**Findings**

- 높음: 계획은 재분석 시작 시 `progressBar.reset()`만 추가하지만, 이전 분석 완료 시 예약된 fade-out 작업을 취소하지 않습니다. 현재 완료 후 5초 뒤 진행바를 숨기는 비동기 블록이 남아 있고 `[FaceScanListViewController.swift#L560](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/FaceScan/UI/FaceScanListViewController.swift#L560)`, `"다음 분석"`은 그 전에 바로 눌릴 수 있습니다 `[FaceScanListViewController.swift#L395](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/FaceScan/UI/FaceScanListViewController.swift#L395)`. 그래서 새 분석이 시작된 뒤에도 이전 run의 예약 작업이 진행바를 다시 `alpha=0`, `isHidden=true`로 만들어 버릴 수 있습니다. 계획의 `reset()` 제안 `[mellow-marinating-liskov.md#L216](/Users/karl/.claude/plans/mellow-marinating-liskov.md#L216)`만으로는 이 문제가 해결되지 않습니다. `DispatchWorkItem`이나 run token을 저장해서 재시작 시 취소하는 항목이 추가되어야 합니다.

- 중간: 계획의 `scannedSoFar = processedRawGroupCount / rawGroups.count * photos.count`는 텍스트 `"N / N장 검색"`의 의미와 맞지 않습니다 `[mellow-marinating-liskov.md#L157](/Users/karl/.claude/plans/mellow-marinating-liskov.md#L157)`. `rawGroups`는 전체 사진이 아니라 `formGroups()`가 뽑아낸 “3장 이상 후보 그룹”만 반환하고 `[SimilarityAnalyzer.swift#L305](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/SimilarPhoto/Analysis/SimilarityAnalyzer.swift#L305)`, 각 그룹 크기도 제각각입니다. 따라서 그룹 1개 처리마다 동일 비율로 `photos.count`를 배분하면 실제 처리량과 무관한 큰 점프가 생길 수 있습니다. 이 값은 “체감용 heuristic”로는 쓸 수 있지만, 그대로 `"검색한 사진 수"`로 노출하면 부정확합니다. 문구를 heuristic에 맞게 바꾸거나, 적어도 문서에 “정확한 사진 수가 아니라 Phase C 체감 진행률용 환산값”이라고 명시해야 합니다.

- 중간: 계획은 `FaceScanProgress.maxScanCount`의 의미를 상수 1000에서 런타임 `actualPhotosCount`로 바꾸겠다고 했는데 `[mellow-marinating-liskov.md#L45](/Users/karl/.claude/plans/mellow-marinating-liskov.md#L45)`, 현재 모델 이름과 주석은 여전히 “최대 검색 수”, 상수 의미를 전제로 합니다 `[FaceScanProgress.swift#L32](/Users/karl/Project/Photos/iOS/SweepPic/SweepPic/Features/FaceScan/Models/FaceScanProgress.swift#L32)`. 구현은 가능하지만 의미 충돌이 큽니다. 특히 VC에서 `progress.maxScanCount`를 “실제 분석 대상 수”로 읽는 순간 `[mellow-marinating-liskov.md#L243](/Users/karl/.claude/plans/mellow-marinating-liskov.md#L243)` 모델 명세와 사용처가 어긋납니다. `totalPhotoCount` 같은 별도 필드로 분리하는 편이 안전합니다.
