**검토 결과 (심각도순)**
1. `Phase 2`의 설정 탭 추가 범위가 과소추정되어 있습니다.  
`docs/bm/260212bm-plan.md:134`는 `TabBarController`만 수정으로 되어 있지만, 현재 iOS 16~25는 커스텀 플로팅 탭이 3개 고정입니다 (`PickPhoto/PickPhoto/Shared/Navigation/TabBarController.swift:135`, `PickPhoto/PickPhoto/Shared/Components/LiquidGlassTabBar.swift:107`, `PickPhoto/PickPhoto/Shared/Components/FloatingOverlayContainer.swift:202`).  
이 상태에서 4번째 탭 추가는 UI/탭 동기화 불일치가 납니다.

2. 리워드 일일 3회 제한(핵심 BM)이 계획서에 실행 로직으로 명확히 반영되지 않았습니다.  
게이트 플로우에 cap 체크가 없습니다 (`docs/bm/260212bm-plan.md:82`). “3회 소진 전환 유도”는 `Phase 3` UI 변경으로만 적혀 있어(`docs/bm/260212bm-plan.md:161`) BM 4.1 요구와 시점이 어긋납니다.

3. “오래된 50장(FIFO) 삭제”는 현재 AppCore API로 바로 구현이 어렵습니다.  
계획은 `trashDates` 기준 선택을 전제하지만(`docs/bm/260212bm-plan.md:201`), `TrashStore` 내부 상태가 private이고 외부 조회 API가 없습니다 (`Sources/AppCore/Stores/TrashStore.swift:111`).  
`oldestTrashedAssetIDs(limit:)` 같은 읽기 API + 원자적 삭제 경로를 먼저 설계해야 안전합니다.

4. `Phase 1`에서 `Plus 업그레이드 → PaywallVC`를 명시했는데(`docs/bm/260212bm-plan.md:92`), Paywall은 `Phase 2`에 있습니다 (`docs/bm/260212bm-plan.md:126`).  
출시 시점에 dead route/crash risk가 있으니 feature flag로 숨기거나 임시 CTA 처리 필요합니다.

5. 개인정보/광고 컴플라이언스(UMP) 계획이 빠져 있습니다.  
현재 문서는 ATT 프리프롬프트 중심(`docs/bm/260212bm-plan.md:30`, `docs/bm/260212bm-plan.md:100`)인데, AdMob 측은 UMP 기반 동의 플로우(EEA/US state 포함)와 privacy options entry point를 권장합니다. 출시 체크리스트에 넣어야 합니다.

6. AdMob SDK 버전 표기가 구버전 기준입니다.  
문서는 `11.x` 고정(`docs/bm/260212bm-plan.md:99`)인데, 최신 문서 기준 iOS SDK 12.x 라인입니다(마이그레이션 breaking change 존재). 버전 전략(11 유지 이유 or 12 채택)을 명시하세요.

7. 카운터 소모 타이밍이 삭제 경로에서 불명확합니다.  
`recordCleanup()`은 취소 제외로 명시돼 있으나(`docs/bm/260212bm-plan.md:205`), `recordDelete()`는 iOS 시스템 삭제 팝업 취소 시 처리 규칙이 없습니다 (`docs/bm/260212bm-plan.md:87`). “실제 삭제 성공 후만 소모”를 명문화해야 합니다.

8. 앱 오픈 광고 트리거가 과단순합니다.  
`sceneDidBecomeActive` 훅만 명시(`docs/bm/260212bm-plan.md:46`)되어 있어, modal 복귀/광고 재진입/냉시작 타이밍 문제를 막기 위한 `isShowing`, `loadTime`, 최소간격, cold-start 가드 조건이 계획서에 필요합니다.

**질문하신 4가지 기준 요약**
1. 게이트 삽입 지점 커버리지: 현재 코드 기준 `permanentlyDelete/emptyTrash` 직접 호출 지점은 계획서가 거의 정확히 커버합니다.  
2. Phase 순서/의존성: `Phase 1`이 `Phase 2` Paywall에 의존하는 부분은 순서 오류입니다.  
3. 기술 실현 가능성: 전체 방향은 가능하지만, FIFO 50장/탭4개/광고동의(UMP) 설계를 보강해야 실제 구현 가능합니다.  
4. 엣지 케이스: 광고 no-fill, 삭제 팝업 취소, foreground 자정 리셋, 다중 scene 동시성, ad-presenting VC 충돌 시나리오가 테스트 항목에 추가되어야 합니다.

**권장 수정(문서에 바로 반영)**
1. `Phase 1`의 Plus 버튼은 feature flag로 숨기고, `Phase 2`에서 활성화.  
2. `UsageLimit`에 `rewardViewsToday` 포함 + cap(3/day) 체크를 `Phase 1`에 명시.  
3. AppCore에 `oldestTrashedAssetIDs(limit:)`/원자 삭제 API 추가.  
4. `Phase 2`에 `LiquidGlassTabBar`/`FloatingOverlayContainer` 4탭 확장 작업을 명시.  
5. UMP 동의 플로우(초기 요청 + privacy options entry point) 추가.  
6. 삭제/정리 카운트는 “성공 커밋 후 증가” 규칙으로 통일.
