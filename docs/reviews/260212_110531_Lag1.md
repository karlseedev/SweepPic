**핵심 결론**
현재 `docs/260212Lag1-1.md`의 권장안(C)은 **그대로는 목표(메인 스레드 랙 해소)를 달성하기에 불충분**합니다. 가장 큰 이유는 이 프로젝트의 동시성 기본 설정과 충돌하기 때문입니다.

**주요 리뷰 결과 (심각도 순)**
1. **[치명적] `Task.detached`만으로는 메인 액터 이탈이 보장되지 않습니다.**  
`docs/260212Lag1-1.md:117`, `docs/260212Lag1-1.md:160`, `docs/260212Lag1-1.md:168`의 “즉시/완전 해결” 판단은 현재 빌드 설정과 맞지 않습니다.  
`SweepPic/SweepPic.xcodeproj/project.pbxproj:328`, `SweepPic/SweepPic.xcodeproj/project.pbxproj:361`에 `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`가 켜져 있어, 앱 타깃의 비표기 선언이 MainActor 격리로 동작합니다.  
즉 `Task.detached`에서 호출해도 `CleanupPreviewService.analyze`(`CleanupPreviewService.swift:125`), `CleanupService.startCleanup`(`CleanupService.swift:106`), `QualityAnalyzer.analyze`(`QualityAnalyzer.swift:103`) 등으로 들어갈 때 다시 MainActor hop이 발생할 수 있습니다.

2. **[높음] 계획대로 오프메인 전환 시, 공유 상태 데이터 레이스 위험이 큽니다.**  
`CleanupService`의 `currentSession`(`CleanupService.swift:47`), `foundAssetIDs`(`CleanupService.swift:69`)는 스캔 루프(`CleanupService.swift:317`)와 취소 처리(`CleanupService.swift:147`)에서 동시 접근 가능성이 있습니다.  
`QualityAnalyzer.mode`(`QualityAnalyzer.swift:58`, `QualityAnalyzer.swift:90`)도 싱글톤 가변 상태라 병렬 경로에서 충돌 여지가 있습니다.

3. **[높음] UI 스레드 전환 보강이 수정 목록에 빠져 있습니다.**  
문서 주의사항에는 적혀 있지만, 실제 수정 대상 표(`docs/260212Lag1-1.md:176`)에는 `startPreviewCleanup`의 `progressView.update`(`GridViewController+Cleanup.swift:358`) 보강이 포함되지 않았습니다.  
서비스가 오프메인으로 바뀌면 이 지점은 즉시 문제(오프메인 UI 접근)가 됩니다.

4. **[중간] 랙은 줄어도 체감 속도 저하는 남을 수 있습니다.**  
중복 작업(이미지 로드 + 텍스트 감지)이 그대로입니다.  
`CleanupPreviewService.swift:203`(기존 분석) + `CleanupPreviewService.swift:212`(재로딩) + `CleanupPreviewService.swift:220`(텍스트 감지)와 `QualityAnalyzer.swift:186`(텍스트 감지)로 중복이 큽니다.  
UI 프리즈 해소와 별개로 “정리 시작 후 오래 걸림”은 지속될 수 있습니다.

5. **[중간] 방향 B의 `DispatchQueue.global` 래핑은 구조화 동시성 이점을 약화시킵니다.**  
`docs/260212Lag1-1.md:124` 방식은 취소/우선순위/추적이 분산됩니다. 가능하면 서비스 격리 자체를 정리하고, queue 래핑은 최소화하는 쪽이 안전합니다.

**보완 권장안**
1. 먼저 동시성 모델을 정리하세요: `Task.detached` 이전에 분석 경로를 명시적으로 MainActor 밖으로 빼야 합니다. (`@concurrent` 적용 또는 별도 worker actor/타깃)
2. `CleanupService`/`QualityAnalyzer`의 가변 싱글톤 상태를 actor화하거나 락 범위를 확장해 레이스를 막으세요.
3. `progressHandler`의 메인 스레드 계약을 파일 수준으로 강제하세요. (`await MainActor.run` 또는 서비스가 메인에서만 콜백)
4. 중복 이미지 로드/텍스트 감지는 “별도 이슈”로 미루지 말고 같은 작업 묶음에 넣는 게 목표 달성에 유리합니다.
