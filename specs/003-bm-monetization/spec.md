## User Scenarios & Testing

### User Story 1 — 일일 삭제 한도 & 게이트 팝업 (Priority: P1)

무료 사용자가 삭제대기함을 비울 때, 일일 한도를 초과하면 게이트 팝업가 표시되어 추가 삭제 방법(광고 시청/구독)을 안내한다. 한도 이내이면 게이트 없이 바로 삭제가 진행된다.

**Why this priority**: 수익화의 핵심 메커니즘. 이것 없이는 모든 사용자가 무제한 무료 사용 → 수익 제로.

**Independent Test**: 삭제대기함에 11장 이상 담긴 상태에서 "비우기" 탭 → 게이트 팝업 표시·닫기를 광고/구독 없이도 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 무료 사용자, 삭제대기함 8장, 오늘 남은 한도 10장, **When** "비우기" 탭, **Then** 게이트 없이 바로 시스템 삭제 확인 팝업 표시
2. **Given** 무료 사용자, 삭제대기함 15장, 남은 한도 5장, 광고 2회 가능, **When** "비우기" 탭, **Then** 게이트 팝업 — "광고 2회 보고 15장 삭제" 옵션 활성
3. **Given** 무료 사용자, 삭제대기함 50장, 남은 한도 0장, 광고 소진, **When** "비우기" 탭, **Then** 게이트 팝업 — "Plus만 가능" 안내, 광고 옵션 비활성
4. **Given** 게이트 팝업에서 "닫기" 탭, **Then** 팝업 닫힘, 삭제 미진행
5. **Given** Plus 구독자, **When** "비우기" 탭, **Then** 게이트 없이 바로 삭제 진행
6. **Given** 삭제대기함 화면, **Then** 상단에 남은 한도 프로그레스 바(게이지) 표시 ("N / M장 남음")
7. **Given** 게이지 탭, **Then** 상세 팝업 (한도 상태 + 광고 잔여 횟수 + "광고 보기" 버튼)
8. **Given** Plus 구독자 또는 Grace Period 중, **Then** 게이지 미표시

### User Story 2 — 리워드 광고로 한도 확장 (Priority: P1)

무료 사용자가 게이트 팝업에서 광고 시청을 선택하면, 리워드 광고 후 일일 한도가 +10장 추가되고 삭제가 자동 진행된다.

**Why this priority**: 게이트(US-1)의 핵심 탈출구이자 전체 광고 수익의 69%를 차지하는 핵심 수익원.

**Independent Test**: 게이트 팝업 → "광고 보고 삭제" 탭 → 광고 시청 → 삭제 실행까지의 전체 흐름 검증.

**Acceptance Scenarios**:

1. **Given** 광고 버튼 "Ready" 상태, **When** "광고 N회 보고 X장 삭제" 탭, **Then** 광고 재생 시작
2. **Given** 광고 시청 완료 + 시스템 삭제 팝업 확인, **Then** 한도 +10장 추가, 리워드 1회 차감, 삭제 실행
3. **Given** 광고 시청 완료 후 시스템 삭제 팝업에서 취소, **Then** 리워드 횟수 미차감, 한도 미증가
4. **Given** 오늘 리워드 2회 소진, **When** 게이트 팝업 표시, **Then** 광고 옵션 비활성 + 골든 모먼트(구독 전환 유도)
5. **Given** 광고 로딩 중, **Then** 버튼 "광고 준비 중..." 비활성 + 스피너
6. **Given** 광고 로드 실패(no-fill), **When** 버튼 탭, **Then** 스피너 10초 대기 → 실패 시 재시도/취소 팝업
7. **Given** 생애 최초 no-fill, **Then** 1회 무료 +10장 제공 (이후 no-fill은 재시도/취소만)

### User Story 3 — Grace Period (Priority: P1)

신규 사용자가 설치 후 3일간 모든 제한(게이트, 광고, 카운터) 없이 앱을 체험하며, 단계별 사전 안내 배너로 무료 체험 종료를 예고받는다.

**Why this priority**: 사용자가 핵심 기능을 경험하고 가치를 느끼게 하는 전환 기반. 체험 없이 바로 게이트 → 설치 직후 이탈 위험.

**Independent Test**: 앱 설치 직후 삭제대기함 비우기 → 게이트/광고 없이 무제한 사용 확인. Day 4 전환 확인.

**Acceptance Scenarios**:

1. **Given** 설치 후 0~2일, **When** 삭제대기함 비우기, **Then** 게이트 없이 무제한 삭제
2. **Given** 설치 후 0~2일, **Then** 모든 광고 미표시 (전면, 배너, 리워드 포함)
3. **Given** 설치 후 0~2일, **Then** 삭제대기함 화면에 게이지 대신 "무료 체험 중 — N일 남음" 배너 표시
4. **Given** 모든 Day, **Then** 배너에 "무료 체험 중 — N일 남음" 텍스트만 표시 (CTA 없음, 배너 탭 시 페이월 이동)
7. **Given** Day 4 이후, **Then** 배너가 카운터 게이지로 전환 + 게이지 첫 표시 시 1회 툴팁
8. **Given** 배너 탭 (전 일차), **Then** 페이월 화면으로 이동

### User Story 4 — Plus 구독 & 페이월 (Priority: P2)

사용자가 Plus 구독을 구매하여 일일 한도/광고/게이트를 모두 제거하고, 페이월 화면에서 가격 비교 후 결제할 수 있다.

**Why this priority**: 게이트의 "구독하기" 버튼 대상. P1(게이트+광고)만으로도 무료 사용자 수익화 가능하므로 P2이나, 게이트 팝업의 핵심 전환 경로.

**Independent Test**: 페이월 → 구독 구매 → Plus 활성화 → 게이트 스킵 + 광고 미표시 확인.

**Acceptance Scenarios**:

1. **Given** 페이월 화면, **Then** "쌓인 사진, 한 번에 비우세요" 헤드라인 + 무료/Plus 비교표 표시
2. **Given** 페이월 화면, **Then** 연간($19.99, 메인) / 월간($2.99, 보조) 가격 표시, 취소선 정가 포함
3. **Given** "연간 구독하기" 탭 + 결제 성공, **Then** Plus 즉시 활성화, 페이월 dismiss
4. **Given** 결제 실패(네트워크), **Then** "네트워크 연결 확인" 알림, 페이월 유지
5. **Given** Ask to Buy(미성년), **Then** "구매 요청 전송됨, 승인 시 자동 활성화" 알림, 페이월 dismiss
6. **Given** "이전 구매 복원" 탭 + 성공, **Then** Plus 즉시 활성화
7. **Given** "이전 구매 복원" 탭 + 실패, **Then** "복원에 실패했습니다" 토스트
8. **Given** "리딤 코드" 탭 + 코드 입력 성공, **Then** 구독 활성화
9. **Given** "리딤 코드" 시스템 시트 실패/타임아웃(10초), **Then** "App Store에서 입력" 폴백 버튼
10. **Given** 앱 실행, **Then** 구독 상태 자동 확인 + 미완료 트랜잭션 처리
11. **Given** 구독 환불/만료, **When** 앱 실행, **Then** Plus 해제 → Free tier 복귀 (게이트 재활성화, 광고 복원, 조용히 처리)
12. **Given** 자동 갱신 실패, **Then** 전체메뉴 아이콘에 ⚠️ 뱃지 + 구독 메뉴에 ⚠️ 뱃지 → 탭 시 결제 수단 확인 안내
13. **Given** 페이월 하단, **Then** 자동 갱신 고지 + 해지 방법 + 이용약관/개인정보처리방침 링크 (스크롤 없이 보이는 영역)

### User Story 5 — 전면 광고 & 배너 광고 (Priority: P2)

정리 완료 시점에 전면 광고가, 사진 분석 대기 중에 배너 광고가 표시되어 추가 광고 수익을 창출한다.

**Why this priority**: 리워드(P1) 외 추가 광고 수익원. 전면 + 배너가 전체 광고 수익의 ~13% 차지.

**Independent Test**: 유사사진 삭제 완료 2회차 → 전면 광고 확인. 분석 시작 → 배너 확인.

**Acceptance Scenarios**:

1. **Given** 무료 사용자, 유사사진 삭제 완료 짝수 회차, **Then** 전면 광고 표시 (스킵 가능)
2. **Given** 무료 사용자, 자동정리 완료 짝수 회차, **Then** 전면 광고 표시 (스킵 가능)
3. **Given** 홀수 회차, **Then** 전면 광고 미표시
4. **Given** 사진 분석 대기 화면, **Then** 하단에 배너 광고 표시
5. **Given** Plus 구독자, **Then** 전면/배너 모두 미표시
6. **Given** Grace Period 중, **Then** 전면/배너 모두 미표시

### User Story 6 — ATT 동의 흐름 (Priority: P2)

Grace Period 종료 후 첫 앱 실행 시, 광고 추적 동의 프리프롬프트와 시스템 팝업을 표시하여 맞춤형 광고 단가를 확보한다. 게이트와 분리하여 부정적 경험이 중첩되지 않도록 한다.

**Why this priority**: 옵트인율이 광고 eCPM에 ~35% 영향. 프리프롬프트로 옵트인율 ~50% 목표.

**Independent Test**: Grace Period 만료 후 첫 앱 실행 → 프리프롬프트 → 시스템 팝업 순서 확인 (게이트와 별개 시점).

**Acceptance Scenarios**:

1. **Given** Grace Period 종료 후 첫 앱 실행, **Then** 프리프롬프트 화면 표시 (생애 1회)
2. **Given** 프리프롬프트에서 "계속" 탭, **Then** 시스템 ATT 팝업 즉시 표시
3. **Given** 프리프롬프트에서 "건너뛰기" 탭, **Then** 시스템 팝업 미표시. 다음 앱 실행 시 1회 재표시, 또 건너뛰면 영구 미표시 (총 2회 기회)
4. **Given** ATT 거부, **Then** 비개인화 광고 정상 동작, 기능 제한 없음

### User Story 7 — 축하 화면 (Priority: P2)

삭제대기함 비우기 성공 후, 삭제 통계와 확보 용량을 보여주는 축하 화면으로 성취감을 제공한다.

**Why this priority**: 사용자 만족도와 재방문 동기. 구독 유도/광고 없는 순수 긍정 경험.

**Independent Test**: 삭제대기함 비우기 완료 → 축하 화면 표시 → 통계 정확성 확인.

**Acceptance Scenarios**:

1. **Given** 삭제대기함 비우기 성공 (무료/Plus 동일), **Then** 축하 화면 표시
2. **Given** 축하 화면, **Then** "N장 삭제 완료!" (이번 세션) + "총 M장 삭제" (누적) + "X.XGB 확보" (누적 기준)
3. **Given** "확인" 탭, **Then** dismiss, 이전 화면 복귀

### User Story 8 — 전체 메뉴 & 고객센터 (Priority: P2)

전체 메뉴에 프리미엄(구독 관리/복원/리딤 코드)과 고객센터(피드백/FAQ/이용약관/개인정보처리방침/사업자 정보)를 구현한다.

**Why this priority**: 구독 관리 접근성 + 법적 필수 요구사항 (전자상거래법 사업자 정보, App Store 가이드라인).

**Independent Test**: 전체 메뉴 → 각 하위 항목 탭 → 해당 화면 정상 표시 확인.

**Acceptance Scenarios**:

1. **Given** 전체 메뉴 오픈, **Then** "프리미엄" 서브메뉴 (구독 관리, 구독 복원, 리딤 코드) 표시
2. **Given** 무료 사용자 + "구독 관리" 탭, **Then** 페이월 표시
3. **Given** Plus 사용자 + "구독 관리" 탭, **Then** 시스템 구독 관리 화면
4. **Given** "구독 복원" 탭 + Plus 이미 활성, **Then** "이미 구독 중입니다" 토스트
5. **Given** 고객센터 > "피드백 보내기" 탭, **Then** 이메일 작성 화면 (받는 사람/제목/기기 정보 프리셋)
6. **Given** 고객센터 > "자주 묻는 질문" 탭, **Then** FAQ 아코디언 리스트 (오프라인 지원)
7. **Given** 고객센터 > "이용약관" 또는 "개인정보처리방침" 탭, **Then** 인앱 브라우저로 웹 링크
8. **Given** 고객센터 > "사업자 정보" 탭, **Then** 상호/대표자/등록번호/연락처 표시

### User Story 9 — 리뷰 요청 (Priority: P3)

사용자가 앱에 만족할 가능성이 높은 시점에 App Store 리뷰를 요청하여 평점을 확보한다.

**Why this priority**: ASO에 직접 영향. 하지만 수익화 기능 구축 후 도입해도 무방.

**Independent Test**: 조건 충족 상태에서 트리거 이벤트 발생 → 리뷰 팝업 표시 확인.

**Acceptance Scenarios**:

1. **Given** 설치 후 3일 경과 + 3세션 이상 + 누적 30장 이상 삭제대기함 이동 + 버전당 미요청 + 90일 쿨다운, **When** 트리거 이벤트 발생, **Then** 리뷰 팝업 표시
2. **Given** 광고 직후 / 결제 직후 / 에러 세션 / 게이트 직후, **Then** 리뷰 요청 안 함
3. **Given** Grace Period 이후라도 조건 충족 + 금지 타이밍 아님, **Then** 리뷰 요청 가능

### User Story 10 — 분석 이벤트 로깅 (Priority: P3)

핵심 KPI(전환율, 광고 소비, 리텐션) 측정을 위한 비즈니스 이벤트를 기록한다.

**Why this priority**: 데이터 없이는 A/B 테스트·한도 최적화 불가. 하지만 수익화 기능이 먼저.

**Independent Test**: 각 이벤트(게이트 노출, 광고 시청, 페이월 표시, 구독 완료 등) 발생 시 로그 기록 확인.

**Acceptance Scenarios**:

1. **Given** 게이트 팝업 노출, **Then** 이벤트 기록 (삭제대기함 장수 + 남은 한도)
2. **Given** 게이트 팝업에서 선택 (광고/구독/닫기), **Then** 이벤트 기록
3. **Given** 리워드/전면/배너 광고 노출, **Then** 이벤트 기록
4. **Given** 페이월 노출 / 구독 완료, **Then** 이벤트 기록
5. **Given** ATT 동의/거부, **Then** 이벤트 기록
6. **Given** Grace Period 종료 후 첫 세션, **Then** 이벤트 기록

### User Story 11 — 해지 방어 & Exit Survey (Priority: P4 — Phase 2)

구독 해지를 시도하는 사용자에게 누적 사용 통계를 보여주고, 해지 완료 시 사유를 수집한다.

**Why this priority**: Phase 2 항목. 구독자 기반이 충분히 쌓인 후 의미 있음.

**Independent Test**: 전체메뉴 > 구독 관리 → Save Offer → 시스템 구독 관리 → 해지 감지 → Exit Survey.

**Acceptance Scenarios**:

1. **Given** Plus 사용자 + "구독 관리" 탭, **Then** Save Offer 화면 (누적 통계 + 해지 시 변경 안내)
2. **Given** Save Offer에서 "계속 사용하기" 탭, **Then** dismiss
3. **Given** Save Offer에서 "구독 관리로 이동" 탭, **Then** 시스템 구독 관리 시트
4. **Given** 시스템 시트에서 해지 완료, **Then** Exit Survey 표시 (5개 선택지 + 기타)
5. **Given** Exit Survey 제출, **Then** 사유 기록 (로컬 + 분석 서비스)

### User Story 12 — 첫 구매 후 축하 (Priority: P4 — Phase 2)

구독 완료 직후 환영 화면으로 프리미엄 혜택을 확인시키고, 첫 24시간 사용 패턴을 추적한다.

**Why this priority**: Phase 2 항목. 구독 인프라 안정화 후 추가.

**Independent Test**: 구독 구매 완료 → 환영 화면 표시 → 24시간 이벤트 기록 확인.

**Acceptance Scenarios**:

1. **Given** 구독 구매 완료, **Then** "무제한 정리 활성화 완료!" 환영 화면
2. **Given** 환영 화면 "확인" 탭, **Then** dismiss, 이전 화면 복귀 (기존 동작 자동 이어붙임 없음)
3. **Given** 구독 후 24시간 이내, **Then** 첫 삭제 소요 시간, 삭제 장수, 비우기 횟수, 자동정리 사용 여부 추적

### Edge Cases

- **오프라인 + 한도 초과**: 게이트 팝업에서 광고/구독 버튼 비활성 + "인터넷 연결이 필요합니다" 안내, "나중에"만 활성
- **오프라인 + 구독자**: 로컬 캐시 기반 구독 확인 → 모든 기능 정상 동작
- **앱 삭제 후 재설치**: 일일 한도가 보안 저장소에 유지되어 리셋 불가
- **기기 시계 조작**: 온라인 시에만 서버 시간 기준 한도 리셋 → 오프라인 시계 변경으로 리셋 불가
- **광고 no-fill 연속**: 생애 최초 1회 무료 → 이후 재시도/취소 팝업 (횟수 무제한)
- **광고 사전 로드 실패**: 지수 백오프 재시도 (2→4→8초), 즉시 재로드 금지
- **Ask to Buy (미성년)**: 결제 pending → 안내 후 페이월 dismiss, 승인 시 자동 활성화
- **결제 검증 실패**: Plus 미활성화 + "다시 시도해주세요" 알림 + 로그 기록
- **구독 환불**: Plus 즉시 해제, 조용히 처리 (사용자 알림 없음)
- **자동 갱신 실패**: 결제 유예 기간(16일) 동안 Plus 유지 + ⚠️ 뱃지로 결제 수단 확인 유도
- **Grace Period 중 구독**: Grace Period 즉시 종료, Plus로 전환
- **카운터 게이지 첫 표시**: Day 4 전환 시 1회 툴팁 "오늘의 무료 삭제 한도예요. 탭해서 자세히 볼 수 있어요"
- **피드백 이메일 미지원 기기**: mailto: URL 스킴으로 폴백
- **비우기는 항상 전체 비우기**: 부분 삭제 없음. 한도 초과 시 사용자가 직접 삭제대기함 장수를 줄이거나, 광고 시청 또는 구독으로 해결
- **App Store 심사 대응**: Review Notes에 테스트 시나리오 상세 기술 (Grace Period 3일 자동 부여로 핵심 기능 무제한 테스트 가능, 메뉴에서 페이월 진입 → Sandbox 구독 구매 안내). 코드 레벨 바이패스 불필요 — 리뷰어 클린 설치 시 Grace Period 내에서 삭제 기능 자유롭게 검증, 구독은 Sandbox IAP로 별도 검증

## Requirements

### Functional Requirements

**한도 & 게이트**

- **FR-001**: 무료 사용자의 일일 영구 삭제 한도는 10장이어야 한다 (시작값, A/B 테스트로 조정 가능)
- **FR-002**: 리워드 광고 1회 시청 시 일일 한도가 +10장 추가되어야 한다
- **FR-003**: 리워드 광고는 일 최대 2회까지 시청 가능해야 한다 (일일 최대 30장)
- **FR-004**: 일일 한도는 사용자 기기 시간 기준 자정에 리셋되어야 한다
- **FR-005**: 한도 리셋은 이중 체크로 수행해야 한다 — 앱 포그라운드 진입 시 + 자정 시스템 알림 수신 시
- **FR-006**: 삭제대기함 비우기는 항상 전체 비우기여야 한다 (부분 삭제 없음)
- **FR-007**: 삭제대기함 장수 ≤ 남은 한도이면 게이트 없이 바로 삭제가 진행되어야 한다
- **FR-008**: 삭제대기함 장수 > 남은 한도이면 게이트 팝업를 표시해야 한다
- **FR-009**: 게이트 판단 시 광고 시청 가능 여부에 따라 팝업 내용이 달라야 한다 — 광고로 해결 가능 vs Plus만 가능. 필요 광고 횟수 N = ceil((삭제대기함 장수 - 남은 무료 한도) / 10), 단 N ≤ 남은 리워드 횟수(최대 2회). N이 남은 리워드를 초과하면 "Plus만 가능" 분기
- **FR-010**: 삭제대기함 화면 상단에 남은 한도 프로그레스 바(게이지)를 표시해야 한다
- **FR-011**: 게이지 탭 시 상세 팝업(한도 상태, 광고 잔여 횟수, 광고 보기 버튼)을 표시해야 한다. "광고 보기" 버튼 탭 시 리워드 광고를 바로 재생하고, 시청 완료 시 한도 +10장 추가한다 (게이트 팝업 거치지 않음)
- **FR-012**: "비우기" 버튼 텍스트는 상태와 무관하게 항상 "비우기"여야 한다

**광고**

- **FR-013**: 리워드 횟수는 삭제 성공 후에만 차감되어야 한다 (시스템 팝업 취소 시 미차감)
- **FR-014**: 리워드 2회 소진 시 골든 모먼트(구독 전환 유도)를 표시해야 한다
- **FR-015**: 전면 광고는 유사사진 삭제 완료 / 자동정리 완료의 짝수 회차에만 표시해야 한다 (각 트리거별 독립 카운터 — 유사사진 2,4,6... / 자동정리 2,4,6... 별도 집계)
- **FR-016**: 전면 광고는 스킵 가능해야 한다
- **FR-017**: 배너 광고는 사진 분석 대기 화면 하단에 표시해야 한다
- **FR-018**: 광고 버튼은 3단계 상태(Ready/Loading/Failed)를 시각적으로 구분해야 한다
- **FR-019**: 광고는 앱 시작 시 사전 로드 + 시청 완료 후 즉시 다음 광고 로드해야 한다
- **FR-020**: 광고 로드 실패 시 지수 백오프(2→4→8초) 재시도, 즉시 재로드 금지
- **FR-021**: 생애 최초 no-fill 시 1회 무료 +10장 제공해야 한다
- **FR-022**: 정리 행위 도중(그리드, 뷰어, 비교 화면)에는 광고가 없어야 한다

**Grace Period**

- **FR-023**: 설치 후 3일간 게이트, 광고, 카운터 없이 무제한 사용 가능해야 한다
- **FR-024**: Grace Period 중 삭제대기함 화면에 안내 배너를 표시해야 한다 (모든 Day 동일: "무료 체험 중 — N일 남음" + "체험 종료 후 일일 무료 삭제 한도가 적용됩니다", CTA 없음)
- **FR-025**: Grace Period 배너 탭 시 페이월로 이동해야 한다

**구독**

- **FR-026**: 월간/연간 자동 갱신 구독을 제공해야 한다 (런칭 할인가: 월 $2.99, 연 $19.99)
- **FR-027**: Plus 구독자는 모든 광고, 게이트, 한도가 제거되어야 한다
- **FR-028**: 앱 시작 시 구독 상태를 확인하고 미완료 트랜잭션을 처리해야 한다
- **FR-029**: 앱 실행 중 구독 상태 변경을 실시간으로 감지해야 한다
- **FR-030**: 구독 복원 기능을 페이월과 전체 메뉴에서 제공해야 한다
- **FR-031**: 리딤 코드 입력 기능을 페이월과 전체 메뉴에서 제공해야 한다
- **FR-032**: 리딤 코드 시스템 시트 실패 시 App Store URL 폴백을 제공해야 한다
- **FR-033**: 환불 감지 시 Plus를 즉시 해제하고 Free tier로 복귀해야 한다
- **FR-034**: 자동 갱신 실패 시 전체메뉴에 ⚠️ 뱃지를 표시해야 한다

**페이월**

- **FR-035**: 페이월에 가치 헤드라인 + 무료/Plus 비교표를 표시해야 한다
- **FR-036**: 연간 구독을 메인 CTA로, 월간을 보조로 표시해야 한다 (연간 크게, 취소선 정가)
- **FR-037**: 페이월 하단에 자동 갱신 고지, 해지 방법, 이용약관/개인정보처리방침 링크를 스크롤 없이 보이는 영역에 표시해야 한다
- **FR-038**: 결제 실패 시나리오별 사용자 안내를 제공해야 한다 (네트워크 오류 / Ask to Buy / 검증 실패)

**축하**

- **FR-039**: 삭제대기함 비우기 성공 후 축하 화면을 표시해야 한다 (이번 삭제 장수 + 누적 장수 + 확보 용량)
- **FR-040**: 확보 용량은 삭제 시점에 각 사진/동영상의 실제 파일 크기를 합산하여 누적 기록해야 한다
- **FR-040a**: 삭제 통계(누적 삭제 수, 누적 확보 용량, 마지막 갱신 시각)는 전용 파일에 영구 저장해야 한다 (TrashStore 패턴과 동일한 JSON 파일 방식). 파일 손상/삭제 시 0으로 초기화하고 새로 시작한다 (로컬 전용, 서버 백업 없음 — 부가 정보이므로 데이터 손실 허용)
- **FR-040b**: 이번 삭제 건의 삭제 수와 확보 용량은 축하 화면에 전달하되, 별도 영구 저장하지 않는다 (인메모리)

**ATT**

- **FR-041**: Grace Period 종료 후 첫 앱 실행 시 프리프롬프트를 1회 표시해야 한다 (게이트 진입과 분리 — 부정적 경험 중첩 방지)
- **FR-042**: 프리프롬프트 문구: "허용하시면 관련 없는 광고가 줄어듭니다. 데이터는 외부에 판매하지 않습니다."

**메뉴 & 고객센터**

- **FR-043**: 전체 메뉴에 프리미엄(구독 관리/복원/리딤 코드)과 고객센터(피드백/FAQ/약관/처리방침/사업자정보) 하위 항목을 추가해야 한다
- **FR-044**: 프리미엄 메뉴는 구독 상태와 무관하게 항상 동일 구조로 표시되어야 한다 (내부에서 분기)
- **FR-045**: 피드백 보내기는 이메일 작성 화면(받는 사람/제목/기기 정보 자동 첨부)으로 구현해야 한다
- **FR-046**: FAQ는 인앱 정적 리스트(아코디언/펼침)로 오프라인 지원해야 한다. 콘텐츠는 `docs/bm/260227Marketing.md §7` FAQ 테이블(3카테고리 12항목)을 인앱 하드코딩
- **FR-047**: 이용약관/개인정보처리방침은 인앱 브라우저로 웹 링크를 열어야 한다
- **FR-048**: 사업자 정보 화면에 상호/대표자/등록번호/연락처를 표시해야 한다 (전자상거래법 제10조). 초기 구현은 더미 데이터, 출시 전 실제 값으로 교체

**리뷰 요청**

- **FR-049**: 5개 조건(설치 후 3일 경과, 3세션+, 30장+ 삭제대기함 이동, 버전당 1회, 90일 쿨다운) 충족 시 트리거 이벤트에서 리뷰 팝업을 표시해야 한다
- **FR-050**: 금지 타이밍(광고 직후, 결제 직후, 에러 세션, 게이트 직후)에는 리뷰 요청을 하지 않아야 한다

**악용 방지**

- **FR-051**: 일일 한도는 앱 삭제/재설치에도 유지되는 보안 저장소에 저장해야 한다
- **FR-051a**: Grace Period 사용 여부(hasUsedGracePeriod)를 Keychain에 저장하여 같은 기기에서 앱 삭제/재설치로 Grace Period를 반복 획득할 수 없어야 한다. 단, 새 기기 설치 시에는 Grace Period를 정상 부여한다 (기기별 독립, iCloud Keychain 동기화 안 함)
- **FR-051b**: 기기 이전(iCloud 복원, Quick Start 등) 시 Keychain 데이터(UsageLimit, hasUsedGracePeriod)는 iOS 기본 정책을 따르며 별도 마이그레이션/동기화 처리하지 않는다
- **FR-051c**: Keychain 접근 실패(읽기) 시 한도 내로 간주하여 게이트 없이 삭제를 허용한다. 쓰기 실패 시 삭제는 진행하되 다음 접근 시 재시도한다 — 정상 사용자 경험 우선
- **FR-052**: 한도 리셋은 기기 시계 조작으로 우회할 수 없어야 한다 (온라인 시 서버 시간 기준). 서버 시간 확인 실패(네트워크 오류, 서버 다운) 시 로컬 시간으로 폴백하여 리셋 허용 — 정상 사용자 불편 방지 우선
- **FR-052a**: 오프라인 시계 조작 후 온라인 복귀 시 서버 시간 기준으로 현재 한도를 재계산한다. 단, 이미 실행된 삭제는 소급하지 않는다 (100% 방지 불가, 합리적 수준 방어 원칙)

**오프라인**

- **FR-053**: 오프라인 구독자는 StoreKit 2 트랜잭션의 expirationDate 기반으로 모든 기능을 정상 사용할 수 있어야 한다 (별도 캐시 만료 정책 없음 — 구독 유효기간 내 Plus 유지, 만료일 경과 시 자동 무료 전환)
- **FR-054**: 오프라인 무료 사용자 한도 내 삭제는 정상 동작해야 한다
- **FR-055**: 오프라인 한도 초과 시 게이트 팝업에서 광고/구독 버튼 비활성 + "인터넷 연결 필요" 안내

**접근성**

- **FR-057**: 모든 신규 BM UI(게이트 팝업, 게이지, 배너, 페이월 등)는 기존 앱 수준의 기본 접근성을 준수해야 한다 — accessibilityLabel 설정, VoiceOver 활성화 시 적절한 동작. Dynamic Type 등 전체 앱 접근성 개선은 별도 과제로 분리

**분석**

- **FR-056**: BM 명세 §11에 정의된 모든 이벤트(게이트 노출, 게이트 선택, 광고 시청, 페이월 노출, 구독 완료, 삭제 장수, Grace Period 종료 후 첫 세션, ATT 결과)를 기록해야 한다

### Key Entities

- **UsageLimit**: 일일 삭제 카운터 — 오늘 삭제 수, 오늘 광고 시청 수, 마지막 리셋 날짜, 생애 무료 제공 사용 여부
- **GracePeriod**: 설치일, 만료 여부, 남은 일수
- **SubscriptionState**: 구독 티어 (Free/Plus), 활성 여부, 자동 갱신 여부, 결제 문제 여부
- **DeletionStats** (영구 저장, 전용 JSON 파일): 누적 삭제 수 (Int), 누적 확보 용량 (Int64, bytes), 마지막 갱신 시각 (Date)
- **CelebrationResult** (인메모리, 축하 화면 전달용): 이번 삭제 수, 이번 확보 용량, DeletionStats 참조 (누적 표시용)
- **ReviewTracker**: 세션 수, 누적 삭제대기함 이동 수, 마지막 요청 날짜, 요청한 버전

## Clarifications

### Session 2026-03-02

- Q: 전면 광고 "짝수 회차" 카운터가 트리거별 독립인지 통합인지? → A: 각 트리거별 독립 카운터 (유사사진 완료 카운터 / 자동정리 완료 카운터 별도)
- Q: 축하 화면 "확보 용량" 계산 방식? → A: 실제 파일 크기 합산 (삭제 시점에 각 사진/동영상의 실제 용량을 누적 기록)
- Q: ATT 프리프롬프트 "건너뛰기" 후 재시도 시기? → A: 다음 앱 실행 시 1회 재표시, 또 건너뛰면 영구 미표시 (총 2회 기회)
- Q: ATT 프리프롬프트 표시 시점? → A: Grace Period 종료 후 첫 앱 실행 시 (게이트 진입 시점이 아님). 게이트는 부정적 경험이므로 ATT와 분리하여 스트레스 중첩 방지
- Q: 삭제 통계(확보 용량 등) 저장 방식? → A: 전용 JSON 파일 (DeletionStats.json, TrashStore 패턴). 누적 삭제 수 + 누적 확보 용량 + 마지막 갱신 시각. 이번 건 통계는 인메모리로 축하 화면에 전달
- Q: Grace Period 재설치 악용 방지? → A: Keychain에 hasUsedGracePeriod(Bool) 저장. 같은 기기 삭제/재설치 시 Grace Period 차단. 새 기기는 정상 부여 (기기별 독립, iCloud Keychain 미동기화)
